🔹 Q1. Explain Angular architecture in detail.
=============================================================================================================
🧱 Angular is a component-based, TypeScript-driven SPA framework that follows a modular, layered architecture.

1️⃣ High-Level Angular Architecture Overview
 Angular consists of: Modules, Components, Templates, Directives, Services, Dependency Injection, Routing, RxJS (Reactive Programming)

2️⃣ Core Building Blocks
	2.1 🧩 NgModules (Modular Architecture)
		Angular applications are modular.
			Main module: AppModule (root module)
			Types of modules: Root module, Feature modules, Shared module, Core module, Lazy-loaded modules
			Example:
					@NgModule({
					  declarations: [AppComponent],
					  imports: [BrowserModule],
					  bootstrap: [AppComponent]
					})
					export class AppModule {}

			Why Modules? Encapsulation, Lazy loading, Separation of concerns, Micro-frontend ready structure

	2.2 🧱 Components (UI Building Blocks)
			A component consists of: TypeScript class (logic), HTML template (view), CSS (style), Metadata decorator
			Example:
					@Component({
					  selector: 'app-user',
					  templateUrl: './user.component.html',
					  styleUrls: ['./user.component.css']
					})
					export class UserComponent {}

	2.3 🖼 Templates & Data Binding
			| Binding Type     | Syntax               | Direction |
			| ---------------- | -------------------- | --------- |
			| Interpolation    | `{{value}}`          | TS → HTML |
			| Property Binding | `[value]="data"`     | TS → DOM  |
			| Event Binding    | `(click)="method()"` | DOM → TS  |
			| Two-way Binding  | `[(ngModel)]`        | Both      |
			
			Example: <input [(ngModel)]="username">
			Under the hood: Uses Zone.js, Change Detection cycle triggers updates
	
	2.4 🧠 Services & Dependency Injection
			Angular uses Hierarchical Dependency Injection.
			Example:
					@Injectable({ providedIn: 'root' })
					export class UserService {}
		
			Types of providers: Root level, Module level, Component level
			Why Important? Singleton services, Testability, Loose coupling

3️⃣ Angular Runtime Architecture
	3.1 Bootstrapping Process: platformBrowserDynamic().bootstrapModule(AppModule)
			main.ts loads, Angular platform initializes, Root module bootstraps, Root component renders, Component tree created
		
	3.2 Change Detection
			Angular uses: Zone.js, Unidirectional data flow, Tree-based change detection
			Strategies: Default, OnPush (performance optimized), changeDetection: ChangeDetectionStrategy.OnPush
			For enterprise apps: Prefer OnPush, Use immutable data

4️⃣ Routing Architecture: Angular Router manages SPA navigation.
		const routes: Routes = [
		  { path: 'users', component: UserComponent }
		];
	
	Supports: Lazy loading, Route guards, Resolvers, Nested routing, Preloading strategies
	Enterprise structure:
						app/
						 ├── core/
						 ├── shared/
						 ├── features/
						 │     ├── user/
						 │     ├── order/
						 │     └── payment/

5️⃣ Reactive Architecture (RxJS)
	Angular heavily uses RxJS
		Core concepts: Observable, Subject, BehaviorSubject, Operators (map, switchMap, mergeMap)
		Example:
				this.http.get('/api/users')
				  .pipe(map(res => res))
				  .subscribe(data => console.log(data));
		
		Why important? Async HTTP calls, Event streams, WebSockets, State management
		👉 Very similar to reactive programming in Spring WebFlux.

6️⃣ Angular Compilation Architecture
	Angular uses: AOT (Ahead-of-Time) compilation, JIT (Just-in-Time)
	Modern Angular uses: Ivy compiler (default), Tree shaking, Smaller bundle size
	Build pipeline: TypeScript → JS, HTML templates compiled, Optimized bundles generated

	Best practices: Smart vs Dumb components, Shared state management, Feature-based folder structure, Strict typing

8️⃣ State Management Architecture
	Options: Services with BehaviorSubject, NgRx (Redux pattern), Akita, Signals (newer Angular)
	Example flow: Component → Action → Reducer → Store → Selector → Component
	Used for: Large enterprise apps, Micro-frontends, Complex workflows

9️⃣ How Angular Compares Architecturally
	| Concept    | Angular        | Spring Boot Equivalent |
	| ---------- | -------------- | ---------------------- |
	| Module     | NgModule       | Configuration module   |
	| Component  | UI Controller  | RestController + View  |
	| Service    | Injectable     | @Service               |
	| DI         | Injector       | ApplicationContext     |
	| Router     | Angular Router | DispatcherServlet      |
	| Observable | RxJS           | WebFlux Mono/Flux      |

🔟 Real Enterprise Angular Architecture Example
	For a Banking App:
				app/
				 ├── core/ (interceptors, guards)
				 ├── shared/ (common components)
				 ├── auth/ (lazy loaded)
				 ├── dashboard/
				 ├── transactions/
				 ├── payments/

	Performance optimizations: Lazy loading, OnPush, TrackBy, Pure pipes, Memoized selectors

🏁 Final Architecture Summary
	Angular follows: 
				✔ Component-based architecture
				✔ Modular structure
				✔ Hierarchical Dependency Injection
				✔ Reactive data flow
				✔ Tree-based change detection
				✔ Router-driven SPA navigation
				✔ AOT compiled production builds

🔹 Q2. What happens internally when Angular app boots?
=============================================================================================================
1️⃣ Browser Loads index.html
	When you run: ng serve or deploy to production, the browser loads: index.html
			<!DOCTYPE html>
			<html>
			  <body>
				<app-root></app-root>
				<script src="main.js"></script>
			  </body>
			</html>
		
		Important: <app-root> is just a placeholder. main.js contains the compiled Angular app.
		Nothing Angular-specific has executed yet.

2️⃣ main.ts Executes (Entry Point)
	Angular starts from: platformBrowserDynamic().bootstrapModule(AppModule)
	What this does internally: 
		Creates Angular platform, Creates root injector, Compiles root module (JIT or precompiled AOT), Instantiates AppModule

3️⃣ Platform Initialization
	Angular creates a browser platform via: platformBrowserDynamic()
	This: Sets up DOM renderer, Registers core services, Initializes global dependency injection container, Installs Zone.js hooks

4️⃣ Zone.js Setup (Very Important): Angular uses Zone.js
	Zone.js: Patches async APIs, setTimeout, Promises, DOM events, XHR
	Why? So Angular knows when async operations finish and can trigger change detection.
	Without Zone.js, Angular wouldn’t know when to update the UI.

5️⃣ Root Module Compilation
	Now Angular bootstraps: bootstrapModule(AppModule)
	Internally: Reads @NgModule metadata
	Resolves: declarations, imports, providers, Creates module injector, Compiles component factories
	If using AOT: Compilation already done at build time
	If JIT: Compiler runs in browser

6️⃣ Root Injector Creation
	Angular creates a hierarchical injector tree. Hierarchy:
			Platform Injector
				 ↓
			Root Injector
				 ↓
			Module Injector
				 ↓
			Component Injectors

7️⃣ Root Component Creation
	Angular now looks at: bootstrap: [AppComponent]
	It: 
		Finds <app-root> in index.html, Creates AppComponent instance, Creates its injector
		Resolves constructor dependencies, Instantiates child components recursively, Now your component tree is built.

8️⃣ Initial Change Detection Cycle: Angular runs first change detection pass.
	This: Evaluates all bindings, Renders template, Interpolates values, Applies property bindings, Registers event listeners
	Now the UI becomes visible.

9️⃣ Lifecycle Hooks Triggered
	Order: 
		constructor()
		ngOnChanges()
		ngOnInit()
		ngAfterContentInit()
		ngAfterViewInit()
	This happens for each component in hierarchy.

🔟 Router Initialization (If Used)
	If your app uses Router: 
		Router parses current URL, Matches route config, Instantiates routed component, Runs guards, Runs resolvers, Inserts component into <router-outlet>

1️⃣1️⃣ Application Becomes Stable
	Angular waits for all macrotasks complete, Then marks app as "stable".
	This is important for: SSR hydration, Testing, Performance tracking

🧠 Complete Boot Flow Summary (Internals)
			Browser loads index.html
					↓
			main.ts executes
					↓
			Angular platform created
					↓
			Zone.js patches async APIs
					↓
			Root module compiled
					↓
			Injector tree created
					↓
			Root component instantiated
					↓
			Component tree built
					↓
			Initial change detection
					↓
			Router activated
					↓
			App stable

🔥 Deep Technical Internals (Senior Level)
	Angular Creates: Renderer (DOM abstraction), View tree, TView & LView (Ivy internal structures), Change detection tree
	Ivy stores: Static template info → TView, Runtime state → LView
	This is how Angular makes rendering fast.

🏢 Enterprise Considerations During Boot: In large Angular apps: 
	You may also see: 
		APP_INITIALIZER execution, HTTP config loading, Auth token refresh, Global interceptors registration
		Feature module lazy loading, State store initialization
		Example:
				{
				  provide: APP_INITIALIZER,
				  useFactory: loadConfig,
				  multi: true
				}
		Angular waits for these to complete before finishing boot.

🔹 Q3. What is the role of main.ts, AppModule, and platformBrowserDynamic()?
=============================================================================================================
1️⃣ main.ts – The Entry Point
	📌 Role: main.ts is the starting file of the Angular application.
		What it does: Imports the Angular platform, Loads the root module, Starts the bootstrap process
		Important: This file does NOT create components, It only tells Angular which module to start with
		Think of it as: “Start Angular and use AppModule as root configuration.”
	
2️⃣ AppModule – Root Module
	📌 Role: AppModule is the root configuration container of the application.
		Example:
				@NgModule({
				  declarations: [AppComponent],
				  imports: [BrowserModule],
				  providers: [],
				  bootstrap: [AppComponent]
				})
				export class AppModule {}

		What It Defines:
			1. Declarations: Which components belong to this module.
			2. Imports: Which modules are used. Example: BrowserModule, FormsModule, HttpClientModule
			3. Providers: Services available for dependency injection.
			4. Bootstrap: Which component should be loaded first. 
				bootstrap: [AppComponent] This tells Angular: Start rendering from AppComponent.

3️⃣ platformBrowserDynamic() – Creates Angular Platform
	📌 Role: Creates a browser-specific Angular platform capable of running in a web environment.
		It prepares: DOM renderer, Event manager, Change detection setup, Zone.js integration, Root injector
		Why "Browser"? Angular supports multiple platforms:
					| Platform   | Use Case                |
					| ---------- | ----------------------- |
					| Browser    | Standard web app        |
					| Server     | Angular Universal (SSR) |
					| Web Worker | Background thread       |
		
		For browser apps: platformBrowserDynamic()
		For SSR: platformServer()
		Why "Dynamic"? 
			Because this uses JIT (Just-In-Time) compilation. It compiles Angular templates in the browser.
			In production (AOT): You typically use: platformBrowser() (AOT precompiles templates at build time.)

🔹 Q4. Difference between JIT and AOT compilation?
=============================================================================================================
🔥 Angular supports two compilation modes: JIT – Just-In-Time, AOT – Ahead-Of-Time
	Both are ways Angular converts: TypeScript + HTML templates → JavaScript executable code

1️⃣ What is JIT (Just-In-Time) Compilation?
	📌  In JIT, Angular compiles templates in the browser at runtime.
		Browser loads Angular app, Angular compiler runs in browser, Templates compiled to JS, App starts
		
		JIT Boot Example: platformBrowserDynamic().bootstrapModule(AppModule);
		The keyword Dynamic indicates JIT.
		
		Pros: ✔ Faster development build, Easier debugging, Good for development mode
		Cons: ❌ Slower app startup, Larger bundle size, Compiler shipped to browser, Slightly less secure (templates compiled at runtime)
		
		When Used? ng serve, Development builds

2️⃣ What is AOT (Ahead-Of-Time) Compilation?
	📌 In AOT, Angular compiles templates during build time, before deployment.
		Run ng build --prod, Angular compiler runs at build time, Templates converted into JS
		Final optimized JS shipped to browser, No runtime compiler needed
		
		AOT Boot Example: platformBrowser().bootstrapModule(AppModule);
		(No “Dynamic” — because compilation already done.)
		
		Pros: 
			✔ Faster startup time, Smaller bundle size, Better tree shaking, No compiler in browser
			✔ Better security, Early template error detection
		
		Cons: ❌ Slower build time, Harder to debug template issues

🧠 Internal Difference (Important)
🔹 In JIT: Angular ships the compiler to browser. Browser does: Template HTML → JavaScript functions

🔹 In AOT: Angular CLI converts:
				@Component({
				  template: `<h1>{{name}}</h1>`
				})
		
		Into optimized JS like:
				ɵɵelementStart(0, "h1");
				ɵɵtext(1);
				ɵɵelementEnd();
		This is Ivy’s internal rendering instruction.

📦 Bundle Size Comparison
	| Mode | Bundle Size | Startup |
	| ---- | ----------- | ------- |
	| JIT  | Larger      | Slower  |
	| AOT  | Smaller     | Faster  |

🔹 Q5. What is Ivy? How is it different from View Engine?
=============================================================================================================
🔥 Ivy is Angular’s next-generation compiler and rendering engine, introduced in Angular 9 as the default.
	It replaced the older rendering engine called View Engine.
	Ivy is responsible for: Compiling components, Rendering templates, Change detection execution, Optimizing bundle size

🏗 Angular Rendering Engines: Angular has had two rendering engines:
| Engine      | Introduced | Default Until | Status     |
| ----------- | ---------- | ------------- | ---------- |
| View Engine | Angular 2  | Angular 8     | Deprecated |
| Ivy         | Angular 9  | Present       | Default    |

🧠 High-Level Difference
| View Engine                   | Ivy                               |
| ----------------------------- | --------------------------------- |
| Metadata-based compilation    | Instruction-based compilation     |
| Generates large factory files | Generates compact JS instructions |
| Whole-program compilation     | Local compilation                 |
| Harder tree shaking           | Better tree shaking               |
| Larger bundle                 | Smaller bundle                    |
| Slower builds                 | Faster builds                     |

🏗 How View Engine Worked
	View Engine compiled components into: ComponentFactory, NgFactory, Metadata JSON files
	Example (simplified output): AppComponent.ngfactory.js, AppModule.ngfactory.js
	Problems: Large generated code, Entire app recompiled for small changes, Poor tree shaking, Hard to debug, Complex library packaging
	
	
🚀 How Ivy Works
	Ivy compiles templates into JavaScript instructions.
	Instead of generating factories, it generates instructions like:
			ɵɵelementStart(0, "h1");
			ɵɵtext(1);
			ɵɵelementEnd();
	
	These instructions directly manipulate DOM.

🧩 Ivy Internal Architecture
	Ivy introduced: 
		TView → Template blueprint (static)
		LView → Live view instance (runtime state)
	
	Think like:
		TView = Class definition
		LView = Object instance

	This makes rendering extremely efficient.

📦 Example Comparison
			@Component({
			  selector: 'app-root',
			  template: `<h1>{{title}}</h1>`
			})
			export class AppComponent {
			  title = "Hello";
			}

	View Engine: Creates ComponentFactory, Creates view classes, Uses reflective metadata
	Ivy: Generates something like:
				function AppComponent_Template(rf, ctx) {
				  if (rf & 1) {
					ɵɵelementStart(0, "h1");
					ɵɵtext(1);
					ɵɵelementEnd();
				  }
				  if (rf & 2) {
					ɵɵtextBinding(1, ctx.title);
				  }
				}
		Two phases: Creation mode (rf & 1), Update mode (rf & 2) This makes change detection very fast.


🔥 Key Architectural Differences
1️⃣ Compilation Scope
	View Engine: Whole application compiled together
	Ivy: Local compilation, Each component compiled independently
	👉 Enables faster incremental builds

2️⃣ Tree Shaking
	View Engine: Difficult to remove unused code
	Ivy: Better tree shaking, Smaller bundles, Dead code elimination improved

3️⃣ Bundle Size
	Ivy reduces bundle size significantly because: No factory files, No metadata JSON, No runtime compiler needed (AOT)

4️⃣ Debugging
	Ivy: Stack traces are cleaner, Errors point directly to component
	View Engine: Errors inside generated factory files

5️⃣ Library Distribution (Big One)
	With View Engine: Libraries needed special compilation (ngcc)
	With Ivy: Libraries compiled once, No compatibility compiler needed (modern Angular), Huge improvement for enterprise dependency management.

🏢 Enterprise-Level Advantages of Ivy
	✔ Faster builds, Smaller bundles, Better lazy loading, Improved debugging, Better type checking
	✔ Enables standalone components, Foundation for future features (Signals, hydration, SSR improvements)

🔹 Q6. How does Angular dependency injection work internally?
=============================================================================================================
🧠 Angular DI is a hierarchical, tree-based injector system that:
		Creates services, Manages service lifecycle, Resolves dependencies, Provides scope control

🏗 High-Level Architecture
	Angular DI is based on: Injectors, Providers, Tokens, Resolution hierarchy

📦 1️⃣ What Happens When You Write This?
	@Injectable({ providedIn: 'root' })
	export class UserService {}
	
	Internally Angular: Registers UserService with the root injector, Associates it with a token, Prepares a factory function to create it lazily

🧩 Injector Hierarchy
	Angular DI is hierarchical, not flat. Hierarchy looks like:
			Platform Injector
				  ↓
			Root Injector
				  ↓
			Module Injector
				  ↓
			Component Injector
	This means: A child injector can access parent services, Parent cannot access child services

🏗 2️⃣ How Angular Creates Injectors Internally
	When Angular boots: Platform injector created, Root injector created, Module injectors created, Component injectors created per component instance
	Each injector stores: Provider records, Factory functions, Instance cache

🔎 3️⃣ How Dependency Resolution Works Internally
	Let’s say: constructor(private userService: UserService) {}
	
	When Angular creates the component:
		Step 1: Check Component Injector
			Does this component injector have UserService?
				If yes → return instance, If no → go to parent injector
		
		Step 2: Traverse Upwards
			Angular walks upward: 
								Component Injector
								   → Module Injector
									   → Root Injector
										   → Platform Injector

			This is called Hierarchical Resolution Strategy.
			If found: Instance created (if not cached), Cached, Returned
			If not found: Throws NullInjectorError

🧠 Internally in Ivy
	Ivy uses: NodeInjector, EnvironmentInjector, Bloom filters for fast lookup

🌸 4️⃣ Bloom Filter Optimization (Important)
	To make dependency lookup fast, Angular uses Bloom Filters.
	Instead of scanning all providers: Angular hashes token into bit position, Checks bitmask
	If bit not set → immediately skip injector, This makes DI lookup O(1) instead of O(n). Huge performance improvement.

📦 5️⃣ Provider Types (How Angular Stores Them)
	Angular supports different provider formats:
		providers: [
		  UserService,
		  { provide: API_URL, useValue: 'http://localhost' },
		  { provide: Logger, useClass: FileLogger },
		  { provide: AuthService, useExisting: UserService },
		  { provide: Config, useFactory: configFactory }
		]

	Internally, Angular converts them into:
			ProviderRecord {
			   token
			   factory
			   value (cached instance)
			}

🧩 6️⃣ Injection Tokens
	If injecting primitive or interface:
		export const API_URL = new InjectionToken<string>('apiUrl');
	Internally: Token acts as key, Injector maps token → factory/value

🔁 7️⃣ Service Lifecycle
	When is service created? ✔ Lazy creation, Created only when first injected, Singleton at injector level
	Example:
		providedIn: 'root' → singleton app-wide
		providers: [UserService] in component → new instance per component

🏗 8️⃣ Component-Level Providers
		@Component({
		  providers: [UserService]
		})
		
		Now each component instance gets its own UserService.
		Angular creates: New injector for that component

🧠 9️⃣ Constructor Injection Process Internally
	When Angular creates a component: 
		Reads constructor parameter metadata, Resolves tokens, Calls injector.get(token)
		Passes resolved instances to constructor, Reflection is NOT used in Ivy — metadata is compiled.

⚡ 1️⃣0️⃣ Advanced: Resolution Modifiers
	Angular supports:
		constructor(
		  @Optional() service: UserService,
		  @Self() service: UserService,
		  @SkipSelf() service: UserService,
		  @Host() service: UserService
		)
	These modify how Angular traverses injector tree.

🏢 Enterprise Architecture Strategy
	Best practice:
			CoreModule → Singleton services
			SharedModule → Stateless utilities
			FeatureModule → Feature-specific services
			Component providers → Scoped services

🧠 Deep Comparison With Spring
| Angular             | Spring             |
| ------------------- | ------------------ |
| Injector            | ApplicationContext |
| providedIn: 'root'  | @Singleton         |
| Component providers | Prototype scope    |
| InjectionToken      | @Qualifier         |
| Factory providers   | @Bean method       |

🔹 Q7. What is hierarchical injector?
=============================================================================================================
🧠 What is a Hierarchical Injector?
	Angular maintains a tree of injectors where each injector can access providers from its parent, but not from its children.
	Instead of one global container, Angular creates multiple injectors arranged in a hierarchy.

🏗 Why Angular Uses Hierarchical Injectors
	Scope control (singleton vs per-component instance), Encapsulation, Lazy loading support, Performance optimization, Micro-frontend friendliness

🌳 Injector Tree Structure
							Platform Injector
									↓
							Root Injector (Application level)
									↓
							Feature Module Injector (Lazy modules)
									↓
							Component Injector
									↓
							Child Component Injector

🔎 How Dependency Resolution Works
	When Angular tries to resolve a dependency: constructor(private userService: UserService) {}
		It searches like this: 
			Check current component injector
				If not found → go to parent injector, Continue upward
				If found → return instance
				If not found anywhere → throw error
			This is called Hierarchical Resolution Strategy.

	🧩 Example 1 – Root Level Service
		@Injectable({ providedIn: 'root' })
		export class UserService {}
		
		✔ Registered in root injector, Single instance for entire app, All components share the same instance.
	
	🧩 Example 2 – Component-Level Service
		@Component({
		  providers: [UserService]
		})

		✔ A new injector is created for this component, New instance per component instance
			If you render component 3 times → 3 separate UserService instances.

	🧠 Important Rule
		👉 Parent injector cannot see child providers
		👉 Child injector can see parent providers
		So: Child → Parent  ✅ Allowed, Parent → Child  ❌ Not Allowed
	
	🏗 Lazy Loaded Module Case. For lazy modules:

			{
			  path: 'admin',
			  loadChildren: () => import('./admin/admin.module')
			}
		
		Angular creates a separate injector for that module.
		That means:
			✔ Services provided in lazy module are scoped to that module
			✔ Not shared globally unless providedIn: 'root'
		This is very important for large enterprise apps.

🧠 Internal Implementation (Ivy Level)
	Internally Angular uses: NodeInjector (component-level), EnvironmentInjector (module-level), Bloom filters for fast lookup
	When resolving: Angular checks bloom filter bit
		If bit not set → skips injector immediately
		If set → resolves provider
		This makes lookup very fast.

🏢 Real Enterprise Example
	Imagine: 
			AppComponent
			   ├── DashboardComponent
			   │       └── WidgetComponent
			   └── AdminComponent

	If AdminComponent provides: providers: [AdminService]
	Only Admin and its children can access it, Dashboard cannot. This ensures encapsulation.

🧩 Difference from Single Injector System
	If Angular had only one injector: Every service would be global, No per-component isolation, No lazy module scoping, Harder state isolation
	Hierarchical injector solves that.

🔹 Q8. What is zone.js and why Angular needs it?
=============================================================================================================
🧠 Zone.js is a JavaScript library that: Intercepts and tracks all asynchronous operations and notifies Angular when they complete.
	It monkey-patches browser async APIs like: setTimeout, setInterval, Promise, XMLHttpRequest, DOM events (click, input), WebSocket, fetch
	
	🚨 The Core Problem Angular Solves: JavaScript is asynchronous.
		Example:
				setTimeout(() => {
				  this.name = "Munna";
				}, 1000);
		Without help, Angular would NOT know: When async code finished, When to update the UI, When to run change detection
		Angular needs to know: “Hey, something changed — re-render the UI.” That’s where Zone.js comes in.

🏗 What is a “Zone”?
	A zone is like: An execution context that survives across async tasks.
	Think of it like a thread-local context in backend systems.
	Zone.js creates a special zone called: NgZone
	Angular runs your entire application inside this zone.

🔄 How Zone.js Works Internally
	Step 1 – Patch Async APIs
		Zone.js overrides: window.setTimeout, Promise.then, addEventListener
				It wraps them like:
							originalSetTimeout(() => {
							   // notify Angular before
							   callback();
							   // notify Angular after
							});
		
		Step 2 – Track Microtasks & Macrotasks
				JavaScript event loop has:
							Microtasks → Promises
							Macrotasks → setTimeout, events
						Zone.js hooks into both queues.

		Step 3 – Notify Angular
				When async task finishes: 
					Zone.js triggers: ngZone.onMicrotaskEmpty
					Angular then runs: Change Detection Cycle
					
		🔁 Flow Example:
						this.http.get('/api/users')
						  .subscribe(data => {
							this.users = data;
						  });

			Flow: 
				HTTP request sent, Zone.js tracks XHR, Response arrives, Zone.js detects task completion, 
				Angular runs change detection, UI updates
			
			Without Zone.js: ❌ Angular wouldn’t know to refresh UI.

🧠 Why Angular Needs Zone.js
	Angular uses automatic change detection.
	Unlike React (manual state triggers), Angular:
		✔ Automatically runs change detection after async tasks
		✔ Doesn’t require developer to call re-render
		✔ Keeps UI in sync transparently
	Zone.js makes this possible.

🔥 What Happens Without Zone.js?
	If Zone.js is removed: Angular won’t auto detect async changes
	You must manually call: ChangeDetectorRef.detectChanges()
	This is called zone-less Angular (modern optimization approach).

🏢 Enterprise Perspective
	Zone.js makes development easy: ✔ No need to manually track async state, Less boilerplate, Simplifies large apps
	But: ❌ It adds performance overhead, Every async task triggers change detection, Can cause unnecessary checks

	That’s why Angular now supports: OnPush strategy, Signals, Zone-less apps (Angular 16+)

⚡ Deep Internal Detail (Ivy + Zone)
	Angular bootstraps inside: ngZone.run(() => bootstrapApplication())
	Zone monitors: Microtask queue empty, Macrotask completion
	When both are stable: Angular marks app as "stable"

🧩 Comparison With React
| Angular              | React                         |
| -------------------- | ----------------------------- |
| Zone.js tracks async | React relies on state updates |
| Automatic detection  | Explicit setState             |
| Global patching      | No global patching            |

🔹 Q9. How Angular change detection works internally?
=============================================================================================================
🧠 Change Detection is the mechanism Angular uses to: Detect data changes in components and update the DOM accordingly.
	Unlike React (explicit state updates), Angular:
		✔ Automatically checks bindings, Traverses component tree, Updates only changed values

🚀 When Does Change Detection Run?
		Angular runs change detection when: 
			User events (click, input), HTTP responses, setTimeout / setInterval, Promise resolution
			Observable emission, Manual trigger (detectChanges())
		
		These are detected via Zone.js (in zone-based apps).

🏗 High-Level Flow
		Async event occurs
				↓
		Zone.js detects task completion
				↓
		Angular calls ApplicationRef.tick()
				↓
		Angular walks component tree
				↓
		Checks bindings
				↓
		Updates DOM if value changed

🌳 Component Tree Traversal
	Angular maintains a change detection tree that mirrors component hierarchy.
	Example:
		AppComponent
		   ├── HeaderComponent
		   ├── DashboardComponent
		   │       ├── ChartComponent
		   │       └── TableComponent
		   └── FooterComponent

		During change detection: Angular starts from root, Traverses depth-first, Checks each component


🔍 How Binding Checking Works
	Example: <h1>{{title}}</h1>
	Angular internally generates something like: ɵɵtextBinding(1, ctx.title);
	During change detection: Angular compares previous value with current value
		If changed → update DOM, If same → skip
		Comparison is: === (reference check)
		No deep comparison.

	⚡ Important: It’s NOT Dirty Checking
		Angular does NOT scan entire object deeply.
		It: Re-evaluates expressions, Compares old vs new reference, That’s why immutability improves performance.

🔹 Q10. What is Angular lifecycle hook execution order?
=============================================================================================================
🧠 Angular lifecycle hooks are methods that allow you to: Tap into different stages of a component’s lifecycle — from creation to destruction.
	They are defined as interfaces like: OnInit, OnChanges, AfterViewInit, OnDestroy

🚀 Complete Lifecycle Hook Execution Order
	Here is the correct execution order for a component:
	🔹 1️⃣ Creation Phase: When Angular creates a component instance: constructor()
	🔹 2️⃣ Change Detection Phase (Initialization)
			ngOnChanges(): (Only if component has @Input properties)
			ngOnInit()
			ngDoCheck()
	
	🔹 3️⃣ Content Projection Phase: (When using <ng-content>)
			ngAfterContentInit()
			ngAfterContentChecked()
	
	🔹 4️⃣ View Initialization Phase: (After component’s template is initialized)
			ngAfterViewInit()
			ngAfterViewChecked()
	
	🔹 5️⃣ Destruction Phase: ngOnDestroy()

📌 Final Execution Order (Memorize This)
	constructor, ngOnChanges, ngOnInit, ngDoCheck, ngAfterContentInit, ngAfterContentChecked, ngAfterViewInit, ngAfterViewChecked, ngOnDestroy

🧠 Important Details (Senior-Level Understanding)
	🔹 constructor(): Pure TypeScript, No Angular bindings ready, Avoid business logic, Use for dependency injection only
	
	🔹 ngOnChanges(): Runs before ngOnInit, Runs every time an @Input changes, Receives SimpleChanges 
		ngOnChanges(changes: SimpleChanges) {}
	
	🔹 ngOnInit(): Runs once, After first ngOnChanges, Safe place for initialization logic, API calls usually placed here
	
	🔹 ngDoCheck(): Runs on every change detection cycle, Used for custom change detection, Rarely needed in modern apps
	
	🔹 ngAfterContentInit(): After projected content (ng-content) initialized, Runs once
	
	🔹 ngAfterContentChecked(): Runs after every content check

	🔹 ngAfterViewInit(): After component’s view and child views initialized, Safe to access: @ViewChild()
	
	🔹 ngAfterViewChecked(): Runs after every view check
	
	🔹 ngOnDestroy(): Runs before component removed, Used for: Unsubscribing, Clearing timers, Cleanup

🏗 Real Example
	Parent: <app-child [data]="value"></app-child>
		Child lifecycle: constructor, ngOnChanges, ngOnInit, ngDoCheck, ngAfterContentInit, ngAfterContentChecked, ngAfterViewInit, ngAfterViewChecked
		If input changes again: ngOnChanges, ngDoCheck, ngAfterContentChecked, ngAfterViewChecked

🧠 Dev Mode Important Detail
	In development mode: Angular runs change detection twice, To detect unstable bindings
	So hooks like ngAfterViewChecked may run multiple times

🏢 Enterprise Best Practices
	✔ API calls → ngOnInit
	✔ @ViewChild access → ngAfterViewInit
	✔ Cleanup → ngOnDestroy
	✔ Avoid heavy logic in ngDoCheck
	✔ Avoid modifying data inside AfterView hooks (can cause ExpressionChangedAfterItHasBeenCheckedError)

🔹 Q1. What is Change Detection Strategy?
=============================================================================================================
🧠 Change Detection Strategy defines: When and how Angular checks a component for data changes and updates the DOM.
	Angular provides two strategies: Default, OnPush

1️⃣  ChangeDetectionStrategy.Default : This is the default behavior.
			@Component({
			  selector: 'app-user',
			  templateUrl: './user.component.html',
			  changeDetection: ChangeDetectionStrategy.Default
			})

	🔎 Every time change detection runs, Angular checks this component, And all of its children, Regardless of whether data changed
		Triggered by: User events (click, input), HTTP response, setTimeout / Promise, Observable emission, Parent change detection cycle
		Because Angular (via Zone.js) detects async completion and triggers a full tree check.

	🟢 Pros: ✔ Simple, Less thinking required, Safe for beginners
	🔴 Cons: ❌ Can be inefficient in large component trees, Unnecessary checks

2️⃣ ChangeDetectionStrategy.OnPush
			@Component({
			  changeDetection: ChangeDetectionStrategy.OnPush
			})

	🔎 How it works: Angular checks this component ONLY if one of these happens:
		An @Input() reference changes, An event occurs inside this component, An observable emits (async pipe)
		You manually call:
			markForCheck()
			detectChanges()
		
		⚡ Important Rule; Angular compares @Input() values using: === (reference equality)
			So this works: this.user = { ...this.user, name: 'Munna' }; // new reference
			But this does NOT trigger update: this.user.name = 'Munna'; // same reference
			Because reference didn't change.

🧩 Internal Behavior
	With Default Strategy: Root → Check entire tree every cycle
	With OnPush: Root → Check only dirty components
	Angular marks component as “dirty” when: Input reference changes, Event happens, markForCheck() called

🧠 What Happens Internally (Ivy Level)
	Each component has a flag in its internal LView:
		CheckAlways (Default)
		CheckOnce (OnPush)
	During change detection:
		Default → always evaluated
		OnPush → skipped unless marked dirty
	Angular skips entire subtree if clean. This is major performance gain.

🏢 Enterprise Recommendation
	For large enterprise apps (like dashboards, banking UIs):
	✔ Use OnPush, Use immutable state, Use async pipe, Avoid direct object mutation, Use trackBy in ngFor
	Default strategy is fine for small apps.

🔹 Q2. What is ChangeDetectorRef? How to detach and reattach change detector?
=============================================================================================================
🧠 ChangeDetectorRef is an Angular service that gives you: Manual control over change detection for a component.
	It allows you to: Trigger change detection manually, Mark component for check, Detach component from change detection tree, Reattach it later
	You inject it like this: 
		import { ChangeDetectorRef } from '@angular/core';
		constructor(private cd: ChangeDetectorRef) {}

🏗 Why Do We Need It?
	Normally, Angular automatically runs change detection using Zone.js.
	But in large enterprise apps: You may have thousands of components, High-frequency updates (charts, timers, websockets), Performance bottlenecks
	In such cases, you may want manual control.

🚀 Methods of ChangeDetectorRef
	Angular provides these important methods:
	| Method             | Purpose                                      |
	| ------------------ | -------------------------------------------- |
	| `markForCheck()`   | Marks component as dirty (OnPush only)       |
	| `detectChanges()`  | Runs change detection immediately            |
	| `detach()`         | Removes component from change detection tree |
	| `reattach()`       | Adds component back to tree                  |
	| `checkNoChanges()` | Dev mode validation                          |


	🔥 1️⃣ detectChanges(): Runs change detection only for: This component, Its children 
			this.cd.detectChanges();
			Use case:
					setTimeout(() => {
					  this.data = "Updated";
					  this.cd.detectChanges();
					}, 1000);

	🔥 2️⃣ markForCheck(): Used with OnPush strategy. 
			this.cd.markForCheck();
			It tells Angular: “Check this component in next detection cycle.”
			It does NOT run detection immediately.

	🔥 3️⃣ detach(): This removes the component from Angular’s change detection tree.
			this.cd.detach();
			After this: Angular will NOT check this component, Even if parent runs detection, Even if async event happens
	
	🔥 4️⃣ reattach(); Adds component back into change detection tree.
			this.cd.reattach();
			Now Angular resumes normal detection.

🧩 Example – detectChanges, Detach and Reattach. Imagine high-frequency data updates:

			@Component({
			  selector: 'app-clock',
			  template: `{{time}}`
			})
			export class ClockComponent {

			  time = new Date();

			  constructor(private cd: ChangeDetectorRef) {
				this.cd.detach();  // stop auto detection

				setInterval(() => {
				  this.time = new Date();
				  this.cd.detectChanges(); // manually update, More Controlled 
				}, 1000);
			  }
			  
			  resume() {
				// Reattach to change detection tree
				this.cd.reattach();
			  }
			}
		
		Here: ✔ Component does NOT run full change detection tree, Only updates itself manually, Improves performance
		
		🧠 Internally What Happens? 
			Each component has a flag in its internal LView: Attached, Detached, Dirty, Clean
			When you call: detach(), Angular marks that view as Detached.
			During traversal: if (view.detached) → skip subtree
			That’s how Angular skips it.

🏢 Enterprise Use Cases
	Detach useful when: ✔ Real-time dashboards, Stock market UI, Live charting, Heavy tables, Infinite scrolling, Large form with partial updates

⚠ Important Warnings: 	❌ If you detach and forget to reattach → UI won’t update, Be careful with nested child components, Harder to debug

🧠 Comparison Summary
| Method        | Runs Immediately? | Used With        |
| ------------- | ----------------- | ---------------- |
| detectChanges | Yes               | Default & OnPush |
| markForCheck  | No                | OnPush           |
| detach        | Stops detection   | Any              |
| reattach      | Resume detection  | Any              |

🔹 Q3. What are performance issues due to change detection?
=============================================================================================================
🚨 Angular change detection: Traverses component tree, Re-evaluates bindings, Runs frequently (after every async event)
	In large apps, this can cause serious performance problems.

	🧠 1️⃣ Full Tree Re-Rendering (Default Strategy Problem)
			With Default strategy, Angular checks: Entire component tree, on every async event triggered by Zone.js.
			Problem:
				If your app has: 2000+ components, Deep nested tree, Frequent async updates
				Then: ❌ Even small changes trigger full traversal, Wasted CPU cycles

	🧠 2️⃣ High-Frequency Async Events
			Examples: setInterval running every 100ms, WebSocket updates, Rapid user input, Mouse move events, Scroll events
			Each event triggers: Zone.js → ApplicationRef.tick() → Full change detection
			This can: ❌ Freeze UI, Spike CPU usage, Drain battery (mobile apps)

	🧠 3️⃣ Heavy Template Expressions
			Bad practice: <div>{{ calculateTotal() }}</div>
			That function runs: On every change detection cycle, Even if nothing changed
			If function is heavy: ❌ Performance degrades
			Better: Precompute value, Use pure pipes

	🧠 4️⃣ Large Lists Without trackBy
			Without trackBy: <div *ngFor="let item of items">, Angular recreates entire DOM list, Even if only one item changes
			With trackBy: <div *ngFor="let item of items; trackBy: trackById">, Angular reuses DOM elements.

	🧠 5️⃣ Object Mutation with OnPush
			this.user.name = "New Name";
			With OnPush: Reference unchanged, Angular skips component, UI appears stale
			Developers then call: detectChanges(), Excessively → performance issues.

	🧠 6️⃣ Nested Component Trees
			Imagine:
					Dashboard
					  ├── Chart
					  ├── Table
					  ├── Filters
					  ├── Summary
					  ├── Notifications

				If Dashboard changes: ✔ All children checked, Even if only Filters changed
				This becomes expensive in large enterprise dashboards.

	🧠 7️⃣ Dev Mode Double Change Detection
			In development mode: Angular runs change detection twice to detect unstable bindings, 
			This doubles work. Production mode removes this.

🏢 Real Enterprise Example
	Stock trading app: Live price updates every 200ms, 100+ rows, Charts, Notifications
	Without optimization: Change detection runs 5 times per second, Entire app checked, UI lags

🚀 How to Fix These Issues
	✅ 1️⃣ Use OnPush Strategy:
			changeDetection: ChangeDetectionStrategy.OnPush
			Reduces unnecessary subtree checks.
	
	✅ 2️⃣ Use trackBy in ngFor: Prevents DOM recreation.
	
	✅ 3️⃣ Avoid Functions in Templates
			Bad: {{ heavyCalculation() }}
			Good: total = computeOnce();
	
	✅ 4️⃣ Use Pure Pipes: Pure pipes only run when input reference changes.
	
	✅ 5️⃣ Use ChangeDetectorRef.detach() for High-Frequency Components, Manual control improves performance.
	
	✅ 6️⃣ Use runOutsideAngular(): Prevents unnecessary cycles.
					this.ngZone.runOutsideAngular(() => {
					   setInterval(() => {
						  // no change detection triggered
					   }, 100);
					});

	✅ 7️⃣ Use Signals (Angular 16+): Signals reduce need for full tree traversal. More granular reactivity.

🔹 Q1. Difference between Observable and Promise?
=============================================================================================================
🧠 Basic Definition
	🔹 Promise: Represents a single future value.
	🔹 Observable: Represents a stream of multiple values over time.

🚀 Core Differences
| Feature         | Promise               | Observable                              |
| --------------- | --------------------- | --------------------------------------- |
| Values          | Single value          | Multiple values                         |
| Execution       | Eager                 | Lazy                                    |
| Cancelable      | ❌ No    	          | ✅ Yes             	                    |
| Operators       | Limited (then, catch) | Rich operators (map, filter, switchMap) |
| Retry           | Hard                  | Easy                                    |
| Used in Angular | Rare                  | Everywhere (HTTP, Router, Forms)        |

🔍 1️⃣ Single vs Multiple Values
	Promise Example: ✔ emit one value
					const promise = new Promise(resolve => {
					  resolve("Data");
					});
					promise.then(data => console.log(data));
			
	Observable Example: ✔ Can emit multiple values, Can emit over time
					const observable = new Observable(observer => {
					  observer.next("First");
					  observer.next("Second");
					  observer.complete();
					});
			
🔥 2️⃣ Eager vs Lazy Execution
	Promise (Eager): Runs immediately when created.
	Observable (Lazy): Does NOT execute until: obs.subscribe(); 


🔁 3️⃣ Cancellation
	Promise: ❌ Cannot cancel once started.
	
	Observable ✔ Can cancel using: Very important for: Avoiding memory leaks, Canceling HTTP calls, Component destruction
				const subscription = obs.subscribe();
				subscription.unsubscribe();

🔄 4️⃣ Operators & Power
	Promise: fetch().then().catch()
	
	Observable: Observables support powerful reactive operators.
				http.get()
				  .pipe(
					map(),
					filter(),
					debounceTime(),
					switchMap()
				  )
				  .subscribe();
		
🧠 5️⃣ Error Handling
	Promise: promise.catch(err => {});
		
	Observable: More flexible error handling.
				observable.subscribe({
				  next: val => {},
				  error: err => {},
				  complete: () => {}
				});
				
				Or: .pipe(catchError())

🏗 6️⃣ Angular Usage
	Angular uses Observables in: HttpClient, Router events, Reactive forms, WebSocket, Async pipe
	Example: this.http.get('/api/users'), Returns Observable — not Promise.

🏢 Real Enterprise Example
	Promise Case: Login API → Single response → Promise is enough.
	Observable Case: Live stock price feed → Continuous stream → Observable required.

🔥 Advanced Differences
	🔹 1️⃣ Cold vs Hot
			Observables can be: Cold (each subscriber gets new execution), Hot (shared execution)
			Promise: Always single execution
	
	🔹 2️⃣ Completion
			Promise: Automatically completes after resolve/reject
			Observable: May never complete (e.g., WebSocket)
	
	🔹 3️⃣ Retry
			Promise: Must recreate promise manually
			Observable: .pipe(retry(3))
			Simple and powerful.

🔹 Q2. What are cold vs hot observables?
=============================================================================================================
🧠 Simple Definition
	🔹 Cold Observable: Each subscriber gets its own independent execution.
	🔹 Hot Observable: All subscribers share the same execution source.

❄️ Cold Observable
	Think of it like: Netflix — every user streams their own copy.
	Each subscription: Starts execution from beginning, Gets independent data stream, No sharing between subscribers
	Example – HTTP in Angular, const users$ = this.http.get('/api/users');
		users$.subscribe();  // API call #1, users$.subscribe();  // API call #2
		
		Each subscription triggers: ✔ New HTTP request, Separate execution
		Angular HttpClient returns Cold Observables by default.
		
		Timeline Visualization
			Subscriber A: 1 --- 2 --- 3
			Subscriber B: 1 --- 2 --- 3  (separate execution)

🔥 Hot Observable
	Think of it like: Live cricket match — everyone watches same broadcast.
	Source runs independently of subscribers, Subscribers join ongoing stream, Values are shared
	Example – DOM Event, const clicks$ = fromEvent(button, 'click');
		clicks$.subscribe();  // listens to same click stream, clicks$.subscribe();
		Both subscribers: ✔ Share same click source, ✔ No new event producer created
		
		Timeline Visualization
			Source: ----1----2----3----4----
			Sub A:       2----3----4
			Sub B:             3----4

		Subscribers only receive future values.

🧩 How to Convert Cold → Hot
	Use operators like: share(), shareReplay(), publish(), multicast()
	Example:
		const shared$ = this.http.get('/api/users').pipe(
		  shareReplay(1)
		);
	Now:
		shared$.subscribe();  // API call #1
		shared$.subscribe();  // NO new API call
	Execution shared.

🚀 Core Differences
| Feature              | Cold           | Hot                    |
| -------------------- | -------------- | ---------------------- |
| Execution            | Per subscriber | Shared                 |
| Starts on subscribe? | Yes            | May already be running |
| HTTP example         | Yes            | No                     |
| DOM events           | No             | Yes                    |
| Memory               | Separate       | Shared                 |
| Use case             | API calls      | Real-time streams      |


🏗 Real Enterprise Examples
	Cold: ✔ HTTP API, Database query, File read, Each consumer needs separate data.
	Hot: ✔ WebSocket live feed, User clicks, Stock ticker, Notification stream, Everyone listens to same source.

🧠 Advanced Detail
	Cold Observables: Unicast, Lazy, Independent execution
	
	Hot Observables: Multicast, Shared producer, Independent of subscribers

🔥 Subject = Hot Observable
	In RxJS: const subject = new Subject();
		Subject: ✔ Is hot, Shares execution, Multicasts values

🔹 Q3. What is Subject, BehaviorSubject, ReplaySubject?
=============================================================================================================
🧠 What is a Subject? A Subject is: Both an Observable AND an Observer.
	Meaning: ✔ You can subscribe to it, You can emit values using .next()
	
	🔹 1️⃣ Subject (Basic Subject)
			Behavior: Does NOT store previous value, New subscribers only get future emissions, It is hot (shared source)
			
			Example:
					import { Subject } from 'rxjs';
					
					const subject = new Subject<number>();
					subject.subscribe(val => console.log('A:', val));
					subject.next(1);
					subject.next(2);

					subject.subscribe(val => console.log('B:', val));
					subject.next(3);

					Output:
							A: 1
							A: 2
							A: 3
							B: 3

				Notice: Subscriber B did NOT get 1 or 2, It only receives values after subscription

	🔹 2️⃣ BehaviorSubject
			A BehaviorSubject: Stores the latest value and emits it immediately to new subscribers.
			✔ Requires an initial value, Always holds current state, Good for state management
		
			Example:
					import { BehaviorSubject } from 'rxjs';

					const subject = new BehaviorSubject<number>(0);
					subject.subscribe(val => console.log('A:', val));
					subject.next(1);
					subject.next(2);

					subject.subscribe(val => console.log('B:', val));

					Output:
					A: 0
					A: 1
					A: 2
					B: 2
				
				Notice: B immediately receives the latest value (2)

	🔹 3️⃣ ReplaySubject
			A ReplaySubject: Stores multiple previous values and replays them to new subscribers.
			You define buffer size: new ReplaySubject(3)
		
			Example:
				import { ReplaySubject } from 'rxjs';

				const subject = new ReplaySubject<number>(2);
				subject.next(1);
				subject.next(2);
				subject.next(3);

				subject.subscribe(val => console.log('A:', val));

				Output:
				A: 2
				A: 3

			It replayed last 2 values.

🚀 Core Differences
| Feature                   | Subject | BehaviorSubject | ReplaySubject    |
| ------------------------- | ------- | --------------- | ---------------- |
| Initial Value             | ❌ No	  | ✅ Required 	| ❌ No 	       |
| Stores Latest Value       | ❌ No	  | ✅ Yes  	    | ✅ Yes (multiple)|
| Replays to New Subscriber | ❌ No	  | Latest only     | Multiple values  |
| Common Use Case           | Events  | State           | Cached history   |

🏗 Real Angular Use Cases
	🔹 Subject: ✔ Event bus, Trigger refresh, Notify components
	
	🔹 BehaviorSubject: ✔ Application state, User authentication status, Shared data between components
			Example:
				private userSubject = new BehaviorSubject<User | null>(null);
				user$ = this.userSubject.asObservable();

	🔹 ReplaySubject: ✔ Chat message history, Caching previous API results, Replay previous notifications

🧠 Advanced Concepts
	Subject = Multicast (Hot), All subscribers share same source.
	BehaviorSubject = Mini Store, Used often for simple state management without NgRx.
	ReplaySubject = Memory Heavy, Be careful with large buffers.

⚠ Common Mistake
	Don’t expose Subject directly:
		❌ Bad: public subject = new Subject();
		✔ Good:
			private subject = new Subject();
			public observable$ = this.subject.asObservable();

	Encapsulation is important.

🔹 Q4. Difference between mergeMap, switchMap, concatMap, exhaustMap?
=============================================================================================================
🧠 What Are These Operators?
	All four are higher-order mapping operators.
	They: Map a source value to an inner Observable and flatten the result.
	
	Example pattern: source$.pipe(operator(value => httpCall$(value))).subscribe();
	The difference is: 👉 How they handle multiple inner subscriptions.

🚀 1️⃣ mergeMap (Parallel Execution)
		Behavior: Subscribes to all inner Observables immediately, Runs them in parallel, Does NOT cancel previous ones, Order NOT guaranteed
		Example:
				click$
				  .pipe(
					mergeMap(() => this.http.get('/api/data'))
				  )
				  .subscribe();
		
		If user clicks 5 times: ✔ 5 HTTP calls run in parallel, All responses processed
		Use Case: ✔ Fire-and-forget operations, Parallel API calls, Save multiple items

	Risk: ❌ Can cause memory leaks if too many subscriptions, Race conditions possible

🔄 2️⃣ switchMap (Cancel Previous)
	Behavior: Cancels previous inner Observable, Only keeps latest, Ideal for search/autocomplete
	Example:
		searchInput$
		  .pipe(
			switchMap(term => this.http.get(`/api/search?q=${term}`))
		  )
		  .subscribe();

		If user types fast: ✔ Previous HTTP calls canceled, Only latest result processed
		Use Case: ✔ Autocomplete, Search box, Typeahead, Real-time filtering
		Important: SwitchMap = “Switch to latest”

📦 3️⃣ concatMap (Sequential Execution)
	Behavior: Queues inner Observables, Runs one at a time, Preserves order, Waits for previous to complete
	Example:
			buttonClicks$
			  .pipe(
				concatMap(() => this.http.post('/api/save'))
			  )
			  .subscribe();

		If user clicks 3 times: ✔ 1st call runs, 2nd waits, 3rd waits, Order guaranteed.
		Use Case: ✔ Payment processing, File uploads, Order submission, Operations that must preserve sequence

🚫 4️⃣ exhaustMap (Ignore New While Running)
	Behavior: Ignores new emissions while inner Observable active, Does NOT cancel, Ignores repeated triggers
	Example:
			loginClick$
			  .pipe(
				exhaustMap(() => this.authService.login())
			  )
			  .subscribe();
		If user clicks login multiple times: ✔ First request runs, Other clicks ignored, Prevents double submission
		Use Case: ✔ Login button, Prevent double API calls, Form submit protection
		
🧠 Quick Comparison Table
	| Operator   | Parallel? | Cancels Previous? | Preserves Order? | Ignores New? |
	| ---------- | --------- | ----------------- | ---------------- | ------------ |
	| mergeMap   | ✅ Yes	 | ❌ No    	     | ❌ No    	    | ❌ No   	   |
	| switchMap  | ❌ No 	 | ✅ Yes 	         | ❌ No 	        | ❌ No  	   |
	| concatMap  | ❌ No 	 | ❌ No 	         | ✅ Yes 	        | ❌ No   	   |
	| exhaustMap | ❌ No 	 | ❌ No  	         | ✅ First only 	| ✅ Yes  	   |

🏢 Real Enterprise Examples
	| Scenario                | Best Operator |
	| ----------------------- | ------------- |
	| Autocomplete            | switchMap     |
	| Parallel API batch      | mergeMap      |
	| Save items sequentially | concatMap     |
	| Prevent double login    | exhaustMap    |

🔹 Q5. How to handle API cancellation?
=============================================================================================================
🧠 Important Concept: Angular HttpClient returns a cold Observable.
	That means: If you unsubscribe → HTTP request is cancelled.
	Under the hood, Angular uses: XMLHttpRequest (XHR) or Fetch with AbortController, Unsubscription aborts the request.

🚀 1️⃣ Best Way – Use switchMap (Automatic Cancellation)
	Most common and recommended approach.
	🔹 Example: Search API
				searchInput$
				  .pipe(
					debounceTime(300),
					switchMap(term =>
					  this.http.get(`/api/search?q=${term}`)
					)
				  )
				  .subscribe(result => {
					this.results = result;
				  });

			What Happens: User types quickly, Previous API call is automatically cancelled, Only latest request survives
		
		✔ No race condition, No manual unsubscribe needed, Clean code

		🧠 Why switchMap Cancels? When a new value arrives:
			switchMap: unsubscribe previous inner observable, subscribe to new one, Unsubscribe → HTTP aborted.

🚀 2️⃣ Manual Cancellation Using Subscription
	If not using switchMap:
		private subscription!: Subscription;

		callApi() {
		  if (this.subscription) {
			this.subscription.unsubscribe(); // cancel previous
		  }

		  this.subscription = this.http.get('/api/data').subscribe();
		}
	
		✔ Works, ❌ More manual work

🚀 3️⃣ Cancel on Component Destroy (Memory Leak Prevention)
	Use takeUntil pattern.
		private destroy$ = new Subject<void>();

		ngOnInit() {
		  this.http.get('/api/data')
			.pipe(takeUntil(this.destroy$))
			.subscribe();
		}

		ngOnDestroy() {
		  this.destroy$.next();
		  this.destroy$.complete();
		}

	✔ Automatically cancels when component destroyed, Prevents memory leaks

🚀 4️⃣ Using AbortController (Advanced / Angular 16+)
	Angular supports fetch-based cancellation:

		const controller = new AbortController();

		this.http.get('/api/data', {
		  signal: controller.signal
		}).subscribe();

		controller.abort();
	Used in more advanced cases.

🏢 Real Enterprise Scenarios
| Scenario            | Best Solution |
| ------------------- | ------------- |
| Search box          | switchMap     |
| Route change        | takeUntil     |
| Form submit         | exhaustMap    |
| Cancel previous API | switchMap     |
| Component destroy   | takeUntil     |

⚠ Common Mistake: ❌ Using mergeMap for search: mergeMap(term => http.get(...))
	Problem: Multiple parallel calls, Responses may return out of order, UI shows stale data
	switchMap solves this.

🔹 Q6. How to avoid memory leaks in Angular? What is takeUntil pattern?
=============================================================================================================
🧠 Why Memory Leaks Happen in Angular?
	Most common cause: Subscriptions that are not unsubscribed.
	Angular heavily uses RxJS, and if you subscribe manually without cleanup, the subscription stays alive even after the component is destroyed.

	That means: Observables keep emitting, Callbacks still execute, Component stays in memory, Memory usage grows

🚨 Common Memory Leak Sources
	Manual .subscribe() without unsubscribe, setInterval not cleared, Event listeners not removed, Subjects never completed, Nested subscriptions
	Long-lived services holding component references

🔥 Bad Example (Memory Leak)
		ngOnInit() {
		  this.http.get('/api/data')
			.subscribe(data => {
			  this.data = data;
			});
		}
	
	If component is destroyed before request completes: ❌ Subscription remains, Memory leak possible
	
✅ Best Practice – The takeUntil Pattern
	The takeUntil pattern is the most recommended solution in Angular.

🧩 What is takeUntil?
	takeUntil() is an RxJS operator that: Automatically unsubscribes when another observable emits.
	🚀 Implementation Pattern
		private destroy$ = new Subject<void>();

		ngOnInit() {
		  this.http.get('/api/data')
			.pipe(
			  takeUntil(this.destroy$)
			)
			.subscribe(data => {
			  this.data = data;
			});
		}

		ngOnDestroy() {
		  this.destroy$.next();
		  this.destroy$.complete();
		}

	🧠 How It Works: destroy$ is a notifier
		When component destroys: destroy$.next() =>  takeUntil receives signal
		Automatically unsubscribes; ✔ Clean, Safe, Scalable, Enterprise standard

🚀 Alternative Methods
	1️⃣ Async Pipe (Best for Templates)
		<div *ngIf="data$ | async as data">
		  {{ data }}
		</div>
		
		✔ Auto subscribes, Auto unsubscribes, No manual cleanup, Best for UI bindings.

	2️⃣ Store Subscription & Unsubscribe Manually: Works but not scalable for multiple streams.
		private subscription!: Subscription;

		ngOnInit() {
		  this.subscription = this.service.getData()
			.subscribe();
		}

		ngOnDestroy() {
		  this.subscription.unsubscribe();
		}
	
	3️⃣ take(1) for One-Time Calls: Auto completes after first emission. Good for HTTP.
			this.http.get('/api/data')
			  .pipe(take(1))
			  .subscribe();

	4️⃣ Using exhaustMap / switchMap Properly
		Avoid nested subscriptions:
			❌ Bad:
				this.route.params.subscribe(params => {
				  this.http.get(...).subscribe();
				});
			
			✔ Good:
				this.route.params
				  .pipe(
					switchMap(params => this.http.get(...))
				  )
				  .subscribe();

🏢 Enterprise Best Practices
	✔ Use async pipe wherever possible
	✔ Use takeUntil for manual subscriptions
	✔ Avoid nested subscriptions
	✔ Avoid global Subjects without completion
	✔ Use OnPush strategy
	✔ Clean up intervals & listeners

🧠 Modern Angular (Angular 16+)
	Angular now provides:
		import { takeUntilDestroyed } from '@angular/core/rxjs-interop';
		
	Cleaner version: No manual Subject needed.
		this.http.get('/api/data')
		  .pipe(takeUntilDestroyed())
		  .subscribe();

🔹 Q7. What is async pipe and how it works internally?
=============================================================================================================
🧠 The Async Pipe is an Angular template pipe that: Automatically subscribes to an Observable or Promise and unsubscribes when the component is destroyed. It is commonly used like this:

	<div *ngIf="users$ | async as users">
	  {{ users.length }}
	</div>

	Instead of:
				this.users$.subscribe(users => {
				  this.users = users;
				});

🚀 What Problems Does It Solve?
	Without async pipe: You must subscribe manually, You must unsubscribe manually, Risk of memory leaks
	With async pipe: ✔ Automatic subscription, Automatic unsubscription, Triggers change detection, Cleaner template

🧩 How Async Pipe Works (High-Level Flow)
	Template renders
		 ↓
	Async pipe subscribes to observable
		 ↓
	Value emitted
		 ↓
	Async pipe stores latest value
		 ↓
	Marks component for change detection
		 ↓
	UI updates
		 ↓
	Component destroyed
		 ↓
	Async pipe unsubscribes automatically

🔥 Internal Implementation Details
	Async pipe internally:
		Checks if value is: Observable, Promise, Subscribes to it, Stores latest emitted value
		Calls: ChangeDetectorRef.markForCheck(), Returns latest value to template
		On destroy → unsubscribes

🧠 Why markForCheck?
	Because Angular change detection may be using OnPush strategy.
	Async pipe ensures: Even OnPush components re-render when new value arrives.
	It internally interacts with change detection system (which may be triggered via Zone.js in zone-based apps).

🏗 Simplified Internal Code (Conceptual)
	Angular async pipe internally looks roughly like:

		transform(obj: Observable<any>) {
		  if (!this.subscription) {
			this.subscription = obj.subscribe(value => {
			  this.latestValue = value;
			  this.cd.markForCheck();
			});
		  }

		  return this.latestValue;
		}

	And on destroy:
		ngOnDestroy() {
			this.subscription.unsubscribe();
		}

🚀 Important Behavior Details
	🔹 1️⃣ Auto-Unsubscribe
			When component is destroyed: ✔ Async pipe unsubscribes, Prevents memory leaks
	
	🔹 2️⃣ Handles Observable Changes
			If input Observable reference changes: {{ newStream$ | async }}
			Async pipe: ✔ Unsubscribes old, Subscribes new
	
	🔹 3️⃣ Works With Promise Too: {{ promise$ | async }}
			Promise resolves once → value displayed → completed automatically.

🏢 Enterprise Best Practices
	✔ Prefer async pipe over manual subscribe
	✔ Combine with OnPush strategy
	✔ Avoid nested async pipes
	✔ Avoid heavy transformation in template

⚠ Common Mistakes
	❌ Doing this: {{ http.get('/api') | async }}, This creates new Observable every change detection cycle.
	Instead: data$ = this.http.get('/api');
		Then use: {{ data$ | async }}

🧠 Async Pipe vs Manual Subscribe
	| Feature           | Async Pipe | Manual Subscribe |
	| ----------------- | ---------- | ---------------- |
	| Auto unsubscribe  | ✅ 	     | ❌      	        |
	| Memory leak safe  | ✅ 	     | Depends          |
	| Cleaner code      | ✅  	     | ❌        	    |
	| Template friendly | ✅ 	     | ❌       	    |

🔹 Q8. How do you implement global error handling using RxJS?
=============================================================================================================
🧠 Without centralized handling: Every .subscribe() needs error callback, Code duplication, Inconsistent user messaging, Hard to log errors
	Hard to handle auth failures (401)

🚀 1️⃣ Basic RxJS Error Handling – catchError
		RxJS provides: catchError()
			Example:
					this.http.get('/api/users')
					  .pipe(
						catchError(error => {
						  console.error(error);
						  return of([]); // fallback value
						})
					  ).subscribe();

		But this is still local handling.

🔥 2️⃣ Global Error Handling Using HTTP Interceptor (Recommended)
		This is the enterprise standard approach. Angular provides HttpInterceptor.
		Example:
			@Injectable()
			export class GlobalErrorInterceptor implements HttpInterceptor {

			  intercept(req: HttpRequest<any>, next: HttpHandler) {
				return next.handle(req).pipe(
				  catchError((error: HttpErrorResponse) => {

					if (error.status === 401) {
					  // redirect to login
					}

					if (error.status === 500) {
					  // show global error message
					}

					return throwError(() => error);
				  })
				);
			  }
			}
		
		Register it:
			providers: [
			  { provide: HTTP_INTERCEPTORS, useClass: GlobalErrorInterceptor, multi: true }
			]
		
		Now: ✔ All HTTP errors handled centrally, No duplication, Cleaner components

🧠 How It Works Internally
	Flow:
		Component
		   ↓
		HttpClient
		   ↓
		Interceptor
		   ↓
		Backend
		   ↑
		Interceptor catchError
		   ↑
		Component
	
	Interceptor wraps every HTTP request as an RxJS stream.

🚀 3️⃣ Global RxJS Error Operator (Custom Operator Pattern): 
	You can create reusable operator:
			export function handleError<T>() {
			  return (source: Observable<T>) =>
				source.pipe(
				  catchError(error => {
					console.error('Global error:', error);
					return throwError(() => error);
				  })
				);
			}
		
		Use it:
			this.http.get('/api/users')
			  .pipe(handleError())
			  .subscribe();

		Reusable across app.

🚀 4️⃣ Using ErrorHandler (Application-Level)
	Angular also provides global error handling via:
		import { ErrorHandler } from '@angular/core';
	
	Example:

		@Injectable()
		export class GlobalErrorHandler implements ErrorHandler {

		  handleError(error: any) {
			console.error('App Error:', error);
		  }
		}

	Register:
		providers: [
		  { provide: ErrorHandler, useClass: GlobalErrorHandler }
		]
	
	This catches: ✔ Uncaught exceptions, Runtime errors, Change detection errors
	But NOT HTTP errors unless rethrown.

🏢 Enterprise-Level Strategy
	In production apps:
		✔ Use HTTP Interceptor for API errors
		✔ Use catchError for stream-level recovery
		✔ Use Global ErrorHandler for runtime crashes
		✔ Log errors to monitoring service (Sentry, ELK)
		✔ Show user-friendly messages

🧠 Best Practice Pattern
	intercept(req, next) {
	  return next.handle(req).pipe(
		retry(1),
		catchError(error => {
		  this.notificationService.showError(error.message);
		  return throwError(() => error);
		})
	  );
	}

🔹 Q9. How to retry failed API calls?
=============================================================================================================
🧠 1️⃣ Simple Retry Using retry()
	The simplest way:

			this.http.get('/api/data')
			  .pipe(
				retry(3)
			  )
			  .subscribe();

		What happens: If request fails → retry, Retry up to 3 times, If still fails → error propagated
		✔ Easy ❌ No delay ❌ Retries for all errors (even 400)

🚀 2️⃣ Retry With Delay Using retryWhen()
	For real-world apps, you need delay.

			this.http.get('/api/data')
			  .pipe(
				retryWhen(errors =>
				  errors.pipe(
					delay(2000),
					take(3)
				  )
				)
			  )
			  .subscribe();

		What happens: Wait 2 seconds, Retry 3 times, Then fail

🔥 3️⃣ Retry With Exponential Backoff (Best Practice): Production-grade retry strategy.

			this.http.get('/api/data')
			  .pipe(
				retryWhen(errors =>
				  errors.pipe(
					scan((retryCount, error) => {
					  if (retryCount >= 3) {
						throw error;
					  }
					  return retryCount + 1;
					}, 0),
					delayWhen(retryCount =>
					  timer(Math.pow(2, retryCount) * 1000)
					)
				  )
				)
			  )
			  .subscribe();

			Retry pattern:
			1st retry → 2s
			2nd retry → 4s
			3rd retry → 8s
		
		✔ Prevents server overload, Used in distributed systems

🧠 4️⃣ Retry Only for Specific Status Codes
		Very important in enterprise apps.
		Never retry: 400 (Bad request), 401 (Unauthorized), 403 (Forbidden)
		Retry only: 500, 503, Network errors
		
		Example:
				this.http.get('/api/data')
				  .pipe(
					retryWhen(errors =>
					  errors.pipe(
						mergeMap((error, retryCount) => {
						  if (retryCount >= 3 || error.status < 500) {
							return throwError(() => error);
						  }
						  return timer(2000);
						})
					  )
					)
				  )
				  .subscribe();

🏗 Best Practice – Use Retry in HTTP Interceptor
	Instead of writing retry everywhere:
			intercept(req, next) {
			  return next.handle(req).pipe(
				retry({
				  count: 3,
				  delay: (error, retryCount) => {
					if (error.status >= 500) {
					  return timer(retryCount * 1000);
					}
					return throwError(() => error);
				  }
				})
			  );
			}

	✔ Centralized, Clean services, Enterprise-level solution

🏢 Real Enterprise Strategy
	In production systems:
		✔ Retry only idempotent calls (GET, PUT)
		❌ Avoid retrying POST payments
		✔ Use exponential backoff
		✔ Add circuit breaker logic
		✔ Combine with timeout

	Example:
		this.http.get('/api/data')
		  .pipe(
			timeout(5000),
			retry(2)
		  )

🔹 Q10. What is shareReplay and when to use?
=============================================================================================================
🧠 shareReplay is an RxJS operator that: Shares a single subscription among multiple subscribers and replays the last emitted value(s) to new subscribers.
	It combines: 
		share() → Multicasts (makes observable hot)
		replay() → Replays previous values

🚀 Why Do We Need shareReplay?
	By default, Angular HttpClient returns a cold observable.
		Example:
			const users$ = this.http.get('/api/users');
			users$.subscribe(); // API call 1
			users$.subscribe(); // API call 2
		
		Each subscription triggers a new HTTP request ❌

🔥 Solution Using shareReplay
			const users$ = this.http.get('/api/users')
			  .pipe(
				shareReplay(1)
			  );

			users$.subscribe(); // API call 1
			users$.subscribe(); // NO new API call
		
		✔ Only one HTTP call, All subscribers get same data, Cached value replayed

🧩 How shareReplay Works Internally
	Without shareReplay: Subscriber A → New execution, Subscriber B → New execution
	With shareReplay: Source executes once, Value stored in buffer, All subscribers receive same value
	
	Internally:
		Uses a ReplaySubject, Caches last N values, Shares subscription

🔢 What Does shareReplay(1) Mean?
	1 = buffer size
	It stores: Last 1 emitted value
	If you write: shareReplay(3)
	It stores last 3 values.

🏗 Real Angular Use Cases
	🔹 1️⃣ HTTP Caching (Most Common)
			getUsers() {
			  return this.http.get('/api/users')
				.pipe(shareReplay(1));
			}
		
		Prevents duplicate calls across components.

	🔹 2️⃣ Configuration Data=> Load app config once:
			config$ = this.http.get('/api/config').pipe(shareReplay(1));
			All components share config.

	🔹 3️⃣ Expensive Calculations: Share computed streams.

⚠ Important – Memory Leak Risk
	Default behavior of shareReplay may: Keep subscription alive, Never unsubscribe from source
	Safer version:
			shareReplay({
			  bufferSize: 1,
			  refCount: true
			})
	
	What refCount: true does: ✔ Automatically unsubscribe when no subscribers left, Prevents memory leaks

🔥 share vs shareReplay
	| Feature                 | share | shareReplay |
	| ----------------------- | ----- | ----------- |
	| Shares execution        | ✅	  | ✅  	    |
	| Replays previous values | ❌ 	  | ✅  	    |
	| Caches data             | ❌ 	  | ✅ 	        |
	| Good for HTTP caching   | ❌ 	  | ✅ 	        |

🏢 Enterprise Best Practice
	Always prefer: shareReplay({ bufferSize: 1, refCount: true })
	Especially for long-lived services.

🔥 State Management (Enterprise Level)

🔹 Q1. What is state management? How do you manage state in large Angular apps?
=============================================================================================================
🧠 State management is: The strategy of storing, organizing, updating, and sharing application data across components in a predictable way.
	State can include: Logged-in user, Selected filters, Shopping cart items, API responses, UI state (loading, error), Form data

🚨 Why State Becomes Complex in Large Apps
	In small apps:
			Parent → Child via @Input
			Child → Parent via @Output
			Works fine.

	In large apps: Component A → Component F → Component Z
	You start seeing: ❌ Prop drilling, Duplicate API calls, Inconsistent data, Race conditions, Hard debugging, Tight coupling
	That’s when you need structured state management.

🏗 Types of State: Understanding this is important.
	| Type                 | Example                 |
	| -------------------- | ----------------------- |
	| Local UI State       | Modal open/close        |
	| Component State      | Form values             |
	| Shared Feature State | Filters, search results |
	| Global App State     | Auth user, theme        |
	
	Not all state needs a global store.

🚀 State Management Approaches in Angular
	1️⃣ Simple Service + BehaviorSubject (Most Common): Using RxJS BehaviorSubject.
		Example:
				@Injectable({ providedIn: 'root' })
				export class UserStore {

				  private userSubject = new BehaviorSubject<User | null>(null);
				  user$ = this.userSubject.asObservable();

				  setUser(user: User) {
					this.userSubject.next(user);
				  }
				}
		
		Component: this.userStore.user$.subscribe();
			✔ Simple, Reactive, Good for medium apps

	2️⃣ Component Store (Feature-Level Store): Angular provides ComponentStore (NgRx package).
		Good for: Feature-level state, Localized state management, Avoiding global store complexity
	
	3️⃣ NgRx (Redux Pattern): Full enterprise state management.
		Architecture: Component → Action → Reducer → Store → Selector → Component
		Features:
			✔ Single source of truth, Immutable state, Predictable state transitions, DevTools support
			✔ Time-travel debugging, Used in very large applications.

	4️⃣ Signals (Modern Angular)
		Angular 16+ introduced Signals. More fine-grained reactivity:
			count = signal(0);
			increment() {
			  this.count.update(c => c + 1);
			}

		Signals: ✔ No need for manual subscription, No need for BehaviorSubject, Less boilerplate, Better performance
		Signals are becoming preferred modern solution.

🏢 How I Manage State in Large Angular Apps (Enterprise Strategy)
	🔹 1️⃣ Separate State Layers
			Core State (auth, config)
			Feature State (dashboard, orders)
			UI State (modal, loading)

	🔹 2️⃣ Keep State Immutable
			this.state = { ...this.state, user: newUser };
			Avoid direct mutation.

	🔹 3️⃣ Use OnPush Strategy: Improves performance.
	
	🔹 4️⃣ Use shareReplay for Cached Data: Avoid duplicate API calls.
	
	🔹 5️⃣ Avoid Global Store Unless Needed
			NgRx only if: Very large app, Complex workflows, Multiple teams
			Strict state traceability required

🧠 When to Use What?
| App Size         | Recommended               |
| ---------------- | ------------------------- |
| Small            | Service + BehaviorSubject |
| Medium           | ComponentStore            |
| Large Enterprise | NgRx                      |
| Modern Angular   | Signals                   |

	Banking app: 
		Auth state → Global store
		Dashboard data → Feature store
		Modal open state → Local component state
	Not everything goes into global store.

🔹 Q2. What is NgRx architecture? Difference between BehaviorSubject vs NgRx? How to avoid overusing NgRx?
=============================================================================================================
1️⃣ What is NgRx Architecture?
	NgRx is a Redux-inspired state management library for Angular. It follows: Unidirectional data flow + immutable state + centralized store.
	Built on top of RxJS.
	
	🔁 Unidirectional Data Flow:
								Component
								   ↓ dispatch
								Action
								   ↓
								Reducer
								   ↓
								Store (new state)
								   ↓ select
								Component

	🧩 Core Building Blocks of NgRx
		1️⃣ Store: Single source of truth.
				store.select(selectUsers)
				
		2️⃣ Actions: Plain objects describing “what happened”.
				export const loadUsers = createAction('[User] Load Users');

		3️⃣ Reducers: Pure functions that update state.
				const userReducer = createReducer(
				  initialState,
				  on(loadUsersSuccess, (state, { users }) => ({
					...state,
					users
				  }))
				);
			
			✔ Must be pure, No side effects, Immutable updates

		4️⃣ Effects: Handle side effects (API calls).
				loadUsers$ = createEffect(() =>
				  this.actions$.pipe(
					ofType(loadUsers),
					switchMap(() =>
					  this.userService.getUsers().pipe(
						map(users => loadUsersSuccess({ users }))
					  )
					)
				  )
				);
		
		5️⃣ Selectors: Read state efficiently.
				export const selectUsers = createSelector(
				  selectUserState,
				  state => state.users
				);
			
			Selectors are memoized → performance optimized.


2️⃣ BehaviorSubject vs NgRx
	
	🔹 BehaviorSubject (Service-Based Store)
				private userSubject = new BehaviorSubject<User[]>([]);
				users$ = this.userSubject.asObservable();
			
			Pros: ✔ Simple, Less boilerplate, Good for medium apps, Quick to implement
			Cons: ❌ No strict structure, No immutability enforcement, Hard to scale, No devtools, Easy to create spaghetti state
	
	🔹 NgRx
		Pros: 
		✔ Predictable state, Strict separation of concerns, DevTools support, Time-travel debugging, Scales to large teams, Immutable state enforced
		
		Cons: ❌ More boilerplate, Steeper learning curve, Overkill for small apps, 

🔥 Comparison Table
| Feature            | BehaviorSubject | NgRx     |
| ------------------ | --------------- | -------- |
| Boilerplate        | Low             | High     |
| Structure          | Flexible        | Strict   |
| DevTools           | ❌               | ✅        |
| Scalability        | Medium          | High     |
| Team Collaboration | Harder          | Easier   |
| Debugging          | Manual          | Powerful |


🏢 When to Use What?
	Small App: ✔ BehaviorSubject
	Medium App: ✔ Service + ComponentStore
	Large Enterprise App: ✔ NgRx

🚨 3️⃣ How to Avoid Overusing NgRx
	Many developers put EVERYTHING into store. That’s wrong.
	
	🔹 Rule 1: Not All State Should Be Global
			Don’t put: ❌ Modal open state, Form control value, UI-only flags, Temporary filter Into NgRx.
			Keep local state local.

	🔹 Rule 2: Separate State Layers
			Global State → Auth, User, Config
			Feature State → Orders, Dashboard
			Local State → Component-only
	
	🔹 Rule 3: Use NgRx Only When Needed
			Use NgRx when: 
				✔ Multiple unrelated components need same data, Complex workflows, Multi-step operations
				✔ Large team collaboration, Need audit/debug history
			
			Avoid NgRx when: ❌ App is small, No shared global state, Overhead outweighs benefit

	🔹 Rule 4: Feature Store, Not Giant Store
		Use: UserFeatureStore, OrderFeatureStore, PaymentFeatureStore
		Avoid one huge monolithic state.

🧠 Modern Alternative – Signals
	Angular Signals reduce need for NgRx in many apps: Simpler, less boilerplate.
	NgRx still useful for: Very large apps, Complex workflows, Enterprise governance

🔹 Q3. What are actions, reducers, effects?
=============================================================================================================
🧩 1️⃣ What is an Action?
	An Action is: A plain object that describes “what happened”. It does NOT contain business logic.
	Example:
		export const loadUsers = createAction('[User] Load Users');
	
	With payload:
				export const loadUsersSuccess = createAction(
				  '[User] Load Users Success',
				  props<{ users: User[] }>()
				);
				
	Action Structure:
				{
				  type: '[User] Load Users',
				  payload: ...
				}

	Role of Action: ✔ Trigger state change, Describe intent, Sent from component
	Component: this.store.dispatch(loadUsers());
	
🧩 2️⃣ What is a Reducer?
	A Reducer is: A pure function that takes current state and action, and returns new state.
	Rules: ✔ Must be pure, No API calls, No side effects, Must return new immutable state

	Example: 
			const initialState: UserState = {
			  users: [],
			  loading: false
			};

			export const userReducer = createReducer( initialState,
			  on(loadUsers, state => ({ ...state, loading: true })),
			  on(loadUsersSuccess, (state, { users }) => ({ ...state, users, loading: false }))
			);


🧩 3️⃣ What is an Effect?
	An Effect handles: Side effects like API calls, logging, navigation.
	Reducers cannot do this. Effects listen to actions and return new actions.

	Example
			@Injectable()
			export class UserEffects {

			  loadUsers$ = createEffect(() =>
				this.actions$.pipe(
				  ofType(loadUsers),
				  switchMap(() =>
					this.userService.getUsers().pipe(
					  map(users => loadUsersSuccess({ users }))
					)
				  )
				)
			  );
		
			constructor(private actions$: Actions, private userService: UserService) {}
			}
			
🧠 Full Execution Flow Example
	Step 1: Component: this.store.dispatch(loadUsers());
	Step 2: Effect detects loadUsers → calls API.
	Step 3: API returns → effect dispatches: loadUsersSuccess
	Step 4: Reducer updates store state.
	Step 5: Component uses selector to get updated state.

🔥 Quick Comparison Table
| Concept | Responsibility     | Contains Business Logic? | Side Effects? |
| ------- | ------------------ | ------------------------ | ------------- |
| Action  | Describe event     | ❌         	          | ❌   	      |
| Reducer | Update state       | ❌         	          | ❌  	      |
| Effect  | Handle async logic | ✅      	              | ✅  	      |

🏢 Real Enterprise Example
	E-commerce app:
			1️⃣ User clicks “Load Orders”
			2️⃣ Action dispatched
			3️⃣ Effect calls /orders API
			4️⃣ Success action dispatched
			5️⃣ Reducer stores orders
			6️⃣ UI updates
		Clean separation.

🧠 Why This Pattern Is Powerful
	✔ Predictable state changes, Easy debugging, Time-travel debugging, Scales for large teams, Clear separation of concerns

🔹 Q4. What is selector? How to manage caching of API responses?
=============================================================================================================
🧠 1️⃣ What is a Selector?
	In NgRx, a Selector is: A pure function used to read and derive data from the store.
	Selectors help: ✔ Extract state slices, Compute derived data, Optimize performance (memoization), Prevent unnecessary re-renders
	
	🏗 Basic Selector Example
		State:
			interface AppState {
			  users: User[];
			}

		Selector: export const selectUsers = (state: AppState) => state.users;
		
		Better version using createSelector:

						export const selectUserState = 
						  (state: AppState) => state.users;

						export const selectUsers = createSelector(
						  selectUserState,
						  (users) => users
						);

		Component:
				this.store.select(selectUsers)
				  .subscribe(users => {
					this.users = users;
				  });
				  
	🔥 Why Use createSelector? Because selectors are memoized.
		Meaning: If input state doesn’t change → selector does NOT recompute. This improves performance significantly.

	🧠 Derived Data Example
				export const selectActiveUsers = createSelector(
				  selectUsers,
				  users => users.filter(u => u.active)
				);
		
		✔ Runs only when users array reference changes, Not on every change detection cycle
	
🧠 2️⃣ How to Manage Caching of API Responses?
	Caching is critical for: Performance, Reducing duplicate API calls, Avoiding network overhead
	There are multiple strategies depending on complexity.
	
	🚀 Strategy 1️⃣ shareReplay (Simple Caching): Using RxJS
						getUsers() {
						  return this.http.get('/api/users')
							.pipe(
							  shareReplay({ bufferSize: 1, refCount: true })
							);
						}
			✔ First subscriber → API call, Next subscribers → cached value, Prevents duplicate calls
			Best for: Config data, Lookup tables, Static data
	
	🚀 Strategy 2️⃣ Store-Based Caching (NgRx Way): 
			Instead of calling API multiple times:
				1️⃣ Dispatch action
				2️⃣ Effect checks if data exists
				3️⃣ If exists → return cached
				4️⃣ Else → call API
			
			Example:
					loadUsers$ = createEffect(() =>
					  this.actions$.pipe(
						ofType(loadUsers),
						withLatestFrom(this.store.select(selectUsers)),
						switchMap(([_, users]) => {
						  if (users && users.length > 0) {
							return EMPTY; // Already cached
						  }
						  return this.userService.getUsers().pipe(
							map(users => loadUsersSuccess({ users }))
						  );
						})
					  )
					);
			✔ Prevents duplicate API calls, Store acts as cache

	🚀 Strategy 3️⃣ TTL (Time-Based Cache): Sometimes cache must expire.
					private cacheTime = 0;
					private cacheData: any;

					getUsers() {
					  const now = Date.now();

					  if (this.cacheData && (now - this.cacheTime < 60000)) {
						return of(this.cacheData);
					  }

					  return this.http.get('/api/users').pipe(
						tap(data => {
						  this.cacheData = data;
						  this.cacheTime = Date.now();
						})
					  );
					}

			✔ Cache for 60 seconds, Then refresh

	🚀 Strategy 4️⃣ HTTP Interceptor Caching
			Advanced pattern: Interceptor stores response, Next request returns cached version, Used in large enterprise apps.

🏢 Enterprise Strategy: In real projects:
| Data Type      | Strategy    |
| -------------- | ----------- |
| Auth user      | Store       |
| Config         | shareReplay |
| Lookup lists   | shareReplay |
| Dashboard      | NgRx store  |
| Search results | No cache    |

🧠 Important: Avoid Over-Caching
	Do NOT cache: ❌ Frequently changing data, Sensitive data without invalidation, Large datasets without cleanup

🔹 Q5. When should we use Component Store? How to handle optimistic updates?
=============================================================================================================
🧠 1️⃣ When Should We Use Component Store?
	Component Store (from NgRx) is: A lightweight, local state management solution for feature-level state.
	It sits between: Simple BehaviorSubject, Full NgRx Store, Built on top of RxJS

	🏗 When to Use Component Store
		Use it when:
			✔ State is limited to a feature/module, Multiple components share feature state
			✔ You want structured state but not global store, Avoid heavy NgRx boilerplate, Medium complexity
	
	🚫 When NOT to Use It
			❌ For simple local component state, For very small apps, For global auth/config (use global store)

🏢 Example Scenario; Feature: Product Management
	You have: Product list, Filters, Loading state, Pagination
	Instead of putting this in global NgRx store:
		✔ Create ProductStore using ComponentStore.

	🧩 Basic Component Store Example
					@Injectable()
					export class ProductStore extends ComponentStore<ProductState> {

					  constructor(private productService: ProductService) {
						super({
						  products: [],
						  loading: false
						});
					  }

					  readonly products$ = this.select(state => state.products);

					  readonly loadProducts = this.effect(() =>
						this.productService.getProducts().pipe(
						  tap(products => this.patchState({ products }))
						)
					  );
					}
			✔ Structured, Reactive, Encapsulated per feature

	📊 Component Store Position in Architecture
		Small App → Service + BehaviorSubject
		Medium Feature → Component Store
		Large Enterprise → NgRx Store

🚀 2️⃣ How to Handle Optimistic Updates?
	Optimistic update means: Update UI immediately before server confirms success. Improves UX dramatically.
	
	🧠 Example Scenario
		User clicks “Like” button. Instead of: Wait for API response Then update UI
		We: Update UI instantly, Call API, If API fails → rollback

	🧩 Optimistic Update with Component Store
	Step 1 – Update state immediately
				readonly likePost = this.effect((postId$: Observable<number>) =>
					  postId$.pipe(
						tap(postId => {
						  this.patchState(state => ({
							posts: state.posts.map(p =>
							  p.id === postId ? { ...p, likes: p.likes + 1 } : p
							)
						  }));
						}),
						switchMap(postId =>
						  this.api.likePost(postId).pipe(
							catchError(error => {
							  // Rollback on error
							  this.patchState(state => ({
								posts: state.posts.map(p =>
								  p.id === postId ? { ...p, likes: p.likes - 1 } : p
								)
							  }));
							  return EMPTY;
							})
						  )
						)
					  )
					);
	
	🏢 Optimistic Update in NgRx Store
		In full NgRx:
			1️⃣ Dispatch likePostOptimistic
			2️⃣ Reducer updates state immediately
			3️⃣ Effect calls API
			4️⃣ On failure → dispatch rollback action

	🧠 Important Rules for Optimistic Updates
		✔ Always store previous state, Always handle rollback, Only use for idempotent operations, Avoid for financial transactions

	🧠 When NOT to Use Optimistic Updates
		❌ Payment processing, Critical financial data, Irreversible operations

⚖ Component Store vs NgRx for Optimistic Updates
| Feature          | Component Store | NgRx             |
| ---------------- | --------------- | ---------------- |
| Boilerplate      | Low             | High             |
| Suitable For     | Feature-level   | Global workflows |
| Rollback Support | Manual          | Action-based     |

🔹 Q1. Difference between Template-driven vs Reactive forms? 
=============================================================================================================
1️⃣ Template-Driven Forms
	Template-driven forms are: Form logic defined mostly inside the HTML template.
	They rely on: ngModel, Template directives, Two-way binding
	
	Example:
			<form #form="ngForm" (ngSubmit)="onSubmit(form)">
			  <input name="username" ngModel required>
			  <button type="submit">Submit</button>
			</form>
			
		How It Works Internally: 
			Angular automatically creates FormControl
			ngModel binds input to model
			Form state managed by directives
			Relies heavily on change detection (via Zone.js)

		Pros: ✔ Simple, Less boilerplate, Easy for small forms
		Cons: ❌ Hard to scale, Hard to unit test, Less explicit control, Complex validation becomes messy

🏗 2️⃣ Reactive Forms
	Reactive forms are: Form logic defined explicitly in TypeScript.
	They are model-driven and use: FormControl, FormGroup, FormArray
		Example:
				form = new FormGroup({
				  username: new FormControl('', Validators.required)
				});
				<form [formGroup]="form" (ngSubmit)="onSubmit()">
				  <input formControlName="username">
				  <button type="submit">Submit</button>
				</form>

	How It Works Internally: Form model created in component class, Template binds to model, Form state is immutable, Built on top of RxJS
	Uses observable streams for valueChanges
	
⚖ Key Differences
		| Feature          | Template-Driven | Reactive Forms     |
		| ---------------- | --------------- | ------------------ |
		| Setup            | Template        | TypeScript         |
		| Control Creation | Automatic       | Manual             |
		| Validation       | Template-based  | Class-based        |
		| Scalability      | Low             | High               |
		| Testability      | Hard            | Easy               |
		| Dynamic Forms    | Difficult       | Easy               |
		| Async Validation | Limited         | Strong support     |
		| Observables      | No              | Yes (valueChanges) |

🧠 Validation Comparison
	Template-Driven Validation: <input name="email" ngModel required email>
		Validation defined in template.

	Reactive Validation:
			email: new FormControl('', [
			  Validators.required,
			  Validators.email
			])
		
		Validation defined in TS. ✔ More maintainable, Better for complex rules

🏢 Enterprise Recommendation
	Small Simple Form: ✔ Template-driven acceptable
	Medium/Large App: ✔ Use Reactive Forms
	
	Especially when you have: Dynamic forms, Conditional validation, Complex business rules, Multi-step forms, Enterprise dashboards
	
🚀 Advanced Capabilities in Reactive Forms
	✔ valueChanges observable, statusChanges observable, Custom validators, Async validators, FormArray for dynamic fields, Nested FormGroup
	Example:
		this.form.valueChanges.subscribe(value => {
			console.log(value);
		});

🔹 Q3. How to create custom validators? Async validator use case? How to implement cross-field validation?
=============================================================================================================
🧠 1️⃣ How to Create Custom Validators (Synchronous)
	A custom validator is: A function that takes AbstractControl and returns ValidationErrors | null.
	
	🔹 Example: Custom Password Strength Validator

			import { AbstractControl, ValidationErrors } from '@angular/forms';

			export function strongPasswordValidator(
			  control: AbstractControl
			): ValidationErrors | null {

			  const value = control.value;

			  if (!value) return null;

			  const hasUpperCase = /[A-Z]/.test(value);
			  const hasNumber = /\d/.test(value);
			  const hasMinLength = value.length >= 8;

			  const valid = hasUpperCase && hasNumber && hasMinLength;

			  return valid ? null : { weakPassword: true };
			}
			
	🔹 Use in Form
			form = new FormGroup({
			  password: new FormControl('', [ Validators.required, strongPasswordValidator ])
			});

	🔹 Template
			<div *ngIf="form.get('password')?.errors?.['weakPassword']">
					  Password must contain uppercase, number and 8+ characters
			</div>
			
🧠 2️⃣ Async Validator (Real Enterprise Use Case)
	Async validators are used when:
		✔ You need to call backend, Check uniqueness (username/email), Validate coupon code, Verify referral code
		
	🔹 Example: Username Already Exists
	
			import { AbstractControl, ValidationErrors } from '@angular/forms';
			import { Observable, of } from 'rxjs';
			import { map, catchError } from 'rxjs/operators';

			export function usernameExistsValidator( userService: UserService ) {

			  return (control: AbstractControl): Observable<ValidationErrors | null> => {
						return userService.checkUsername(control.value).pipe(
						  map(exists => { return exists ? { usernameTaken: true } : null; }),
						  catchError(() => of(null))
						);
				  };
			}
	
	🔹 Use in Form:
			form = new FormGroup({
			  username: new FormControl('', {
				validators: [Validators.required],
				asyncValidators: [usernameExistsValidator(this.userService)],
				updateOn: 'blur'
			  })
			});
			
	🔹 Why updateOn: 'blur'? Prevents API call on every keystroke. Better for performance.

🧠 3️⃣ Cross-Field Validation (Form-Level Validator)
	Used when validation depends on multiple fields.
	Example: ✔ Password & Confirm Password, Start Date < End Date, Minimum age based on country
	
	🔹 Example: Password Match Validator
			import { FormGroup, ValidationErrors } from '@angular/forms';

			export function passwordMatchValidator( group: FormGroup ): ValidationErrors | null {
					const password = group.get('password')?.value;
					const confirmPassword = group.get('confirmPassword')?.value;

					  if (password !== confirmPassword) {
						return { passwordMismatch: true };
					  }

					return null;
				}
	
	🔹 Apply at FormGroup Level
			form = new FormGroup({
			  password: new FormControl(''),
			  confirmPassword: new FormControl('')
			}, {
			  validators: passwordMatchValidator
			});
			
	🔹 Template
			<div *ngIf="form.errors?.['passwordMismatch']">  Passwords do not match </div>

	🧠 Advanced Cross-Field Example (Dynamic Rule)
		Example: End date must be after start date

				export function dateRangeValidator(group: FormGroup) {
				  const start = group.get('startDate')?.value;
				  const end = group.get('endDate')?.value;

				  return start && end && end < start
					? { invalidDateRange: true }
					: null;
				}

🏢 Enterprise Best Practices
	✔ Use pure functions for validators
	✔ Extract validators into separate files
	✔ Avoid heavy logic inside validators
	✔ Use async validator only when necessary
	✔ Debounce API calls if needed
	✔ Combine with OnPush strategy

⚠ Common Mistakes: 
	❌ Calling API inside sync validator, Not using updateOn: 'blur' for async
	❌ Mutating form values inside validator, Forgetting to return null when valid

🔹 Q4. How to disable form control dynamically? 
=============================================================================================================
🧠 1️⃣ Basic Way – Using disable() / enable()
	In Reactive Forms, every FormControl has built-in methods: control.disable();, control.enable();

	🔹 Example:
				form = new FormGroup({
				  username: new FormControl(''),
				  email: new FormControl('')
				});
		
		Disable dynamically: this.form.get('email')?.disable();
		Enable again: this.form.get('email')?.enable();
		
		✔ Angular updates form state, Control excluded from validation, Control excluded from form.value

🧠 Important Behavior
	When disabled: Control is removed from form.value, Validation is skipped, Status becomes DISABLED
	If you need disabled values included: this.form.getRawValue();

🚀 2️⃣ Disable Based on Another Control (Dynamic Condition)
	
	Example: Disable "state" if country ≠ "India"
	
		this.form.get('country')?.valueChanges.subscribe(value => {
		  if (value !== 'India') {
			this.form.get('state')?.disable();
		  } else {
			this.form.get('state')?.enable();
		  }
		});

	✔ Fully reactive, Common enterprise pattern

🚀 3️⃣ Disable During API Call (Loading State)
	
		this.form.disable();

		this.api.save(this.form.value).subscribe({
		  next: () => this.form.enable(),
		  error: () => this.form.enable()
		});

	✔ Prevents duplicate submissions, Better UX

🚀 4️⃣ Disable at Initialization
		form = new FormGroup({
		  username: new FormControl({ value: '', disabled: true }),
		  email: new FormControl('')
		});

🚀 5️⃣ Disable FormArray Items Dynamically
		(this.form.get('addresses') as FormArray)
		  .at(0)
		  .disable();
		
		Works the same way.

🧠 6️⃣ Advanced: Conditional Disable Using RxJS
	Cleaner approach for complex logic: 
	
			combineLatest([
			  this.form.get('role')!.valueChanges,
			  this.form.get('status')!.valueChanges
			]).subscribe(([role, status]) => {

			  if (role === 'admin' && status === 'active') {
				this.form.get('approval')?.enable();
			  } else {
				this.form.get('approval')?.disable();
			  }
			});
		
		Useful in enterprise forms with many dependencies.

🧠 7️⃣ Template Binding (Avoid for Reactive Forms)
	Avoid:
		<input [disabled]="true">
		
		Because: ❌ It does not sync with FormControl state, Causes inconsistency
		Always use disable() method.

🏢 Enterprise Best Practices
	✔ Always control state via FormControl, Avoid direct DOM disabling, Use valueChanges for reactive behavior
	✔ Use form.disable() during submission, Use getRawValue() when needed

🔹 Q5. How to build reusable form components?
=============================================================================================================
🧠 Why Build Reusable Form Components?
	In large apps you often have: 20+ forms, Repeated input fields (email, phone, date, currency)
	Same validation logic, Same error display pattern, Same styling
	Without reuse: ❌ Code duplication, Inconsistent validation, Hard maintenance
	Reusable form components solve this.

	🏗 Core Concept: ControlValueAccessor (CVA)
		To make a reusable form component work with Angular Reactive Forms, you must implement: ControlValueAccessor
		It allows your custom component to behave like a native form control.

🚀 Step-by-Step: Build Reusable Input Component
1️⃣ Create Reusable Input Component

						@Component({
						  selector: 'app-input',
						  template: `
							<label>{{ label }}</label>
							<input
							  [value]="value"
							  (input)="onChange($event.target.value)"
							  (blur)="onTouched()"
							/>
							<div *ngIf="error">{{ error }}</div>
						  `,
						  providers: [
							{
							  provide: NG_VALUE_ACCESSOR,
							  useExisting: forwardRef(() => InputComponent),
							  multi: true
							}
						  ]
						})
						export class InputComponent implements ControlValueAccessor {

						  @Input() label!: string;

						  value: any;
						  onChange = (_: any) => {};
						  onTouched = () => {};

						  writeValue(value: any): void {
							this.value = value;
						  }

						  registerOnChange(fn: any): void {
							this.onChange = fn;
						  }

						  registerOnTouched(fn: any): void {
							this.onTouched = fn;
						  }
						}

2️⃣ Use It Inside Reactive Form
				<form [formGroup]="form">
				  <app-input formControlName="email" label="Email">
				  </app-input>
				</form>
		
		✔ Works like native input, Fully integrated with validation, Reusable everywhere

🧠 How ControlValueAccessor Works Internally
	When Angular sees: <formControlName="email">
		It: Finds registered NG_VALUE_ACCESSOR, Connects FormControl to component
		Calls: writeValue(), registerOnChange(), registerOnTouched()
		That’s how custom components integrate with Reactive Forms.

🚀 Add Validation Support (Advanced): To access validation state inside reusable component:
	
	Inject NgControl:
				constructor(@Optional() @Self() public ngControl: NgControl) {
				  if (this.ngControl) {
					this.ngControl.valueAccessor = this;
				  }
				}
		Now you can show:
			<div *ngIf="ngControl?.control?.errors?.['required']"> This field is required </div>
			
🏢 Enterprise Best Practices
	🔹 1️⃣ Build Form Field Wrapper Component
			Instead of repeating: Label, Error message, Required indicator, Styling
			Create:
					<app-form-field>
					   <input ...>
					</app-form-field>
			Centralizes UI.

	🔹 2️⃣ Use Generic Error Display Component
			Reusable error renderer: <app-error [control]="form.get('email')"></app-error>
	
	🔹 3️⃣ Use Config-Driven Dynamic Forms
			fields = [
				  { type: 'text', name: 'email', label: 'Email' },
				  { type: 'date', name: 'dob', label: 'DOB' }
				];

			Render dynamically using reusable components.

🚀 Advanced Enterprise Pattern
	Create a Form Control Library:

			/shared/form-controls/
			   input/
			   select/
			   date-picker/
			   currency/
			   checkbox/
		All implementing ControlValueAccessor. Used across entire organization.

🧠 Common Mistakes
	❌ Not implementing registerOnTouched, Forgetting multi: true
	❌ Mutating value directly, Mixing ngModel with Reactive Forms, Not forwarding disabled state

	To handle disabled state:
		setDisabledState(isDisabled: boolean): void {
		  this.isDisabled = isDisabled;
		}

🔥 Routing (Important)

🔹 Q1. How Angular routing works internally? 
=============================================================================================================
🧠 Angular Router is a client-side navigation system that:
		Interprets browser URL → Matches route config → Loads component → Renders inside <router-outlet>.
		It’s a state machine managing navigation transitions.

🏗 1️⃣ High-Level Architecture
	When you define routes:
		const routes: Routes = [
		  { path: 'users', component: UsersComponent },
		  { path: 'orders', loadChildren: () => import('./orders/orders.module') }
		];
	
		Router creates: Route configuration tree, URL matcher, Navigation pipeline, ActivatedRoute tree

	🚀 2️⃣ What Happens When You Navigate?
		Example: this.router.navigate(['/users']);
		Or user clicks: <a routerLink="/users">Users</a>
		
🔁 Internal Navigation Lifecycle: Here’s what happens internally:
	1️⃣ URL Change Detected
		Router listens to browser events via: History API, popstate event
	
	2️⃣ Create Navigation Object
		Router creates a Navigation object containing: target URL, source URL, route state
	
	3️⃣ Apply URL Serializer
		Angular parses URL into: UrlTree
			Example: /users/10?active=true
			Becomes structured tree.

	4️⃣ Route Matching (Important)
		Router matches URL segments against route configuration.
		Matching algorithm: Compare path segments, Handle parameters (:id), Check wildcard routes (**), Match child routes recursively
		Example:
			{ path: 'users/:id', component: UserDetailComponent }
			If URL matches → route activated.
	
	5️⃣ Guards Execution
		Before activating route: Angular runs guards: canActivate, canDeactivate, canLoad, canMatch
		If any guard returns: 
			❌ false → Navigation cancelled
			✔ true → Continue
	
	6️⃣ Resolvers (Optional)
		Resolvers fetch data before route activates:
			resolve: { user: UserResolver }
		Router waits until resolver completes.

	7️⃣ Lazy Loading (If Needed)
		If route uses: loadChildren
		Router: ✔ Dynamically loads module chunk, Registers its routes, Continues navigation
		
	8️⃣ Create ActivatedRoute Tree
		Router builds: 
			RouterState
			  ├── ActivatedRoute
			  ├── Child ActivatedRoute

		Each route contains: params, queryParams, data, snapshot
	
	9️⃣ Render in <router-outlet>
		Router finds <router-outlet>.
		It: Creates component instance, Injects ActivatedRoute, Attaches view to DOM
		Internally it uses: ComponentFactory, ViewContainerRef
	
	🔟 Trigger Change Detection
		Angular runs change detection, UI updates, Navigation complete.

🧠 Navigation Lifecycle Events
	Router emits events like:

			this.router.events.subscribe(event => {
			  console.log(event);
			});

	Important events: NavigationStart, RoutesRecognized,  GuardsCheckStart, ResolveStart, NavigationEnd, NavigationCancel, NavigationError
	Useful for: ✔ Global loading indicators, Analytics tracking, Logging

🏗 3️⃣ Internal Router Data Structures
	Router maintains: ✔ RouteConfig tree, UrlTree, RouterState tree, ActivatedRoute tree
	These are immutable snapshots during navigation.

🚀 4️⃣ How Lazy Loading Works Internally
	When route uses: loadChildren: () => import('./admin/admin.module')
	Angular: Downloads separate JS chunk, Registers module routes, Creates new injector for module, Continues route activation
	This improves: ✔ Initial load time, Bundle size, Performance
	
🧠 5️⃣ How <router-outlet> Works
		router-outlet is a directive. 
		It: Holds a ViewContainerRef, Destroys old component, Creates new component dynamically, Attaches component to DOM
		Every time route changes: ✔ Previous component destroyed, New component created

🏢 Enterprise Considerations
	In large apps: 
		✔ Use lazy loading for feature modules, Use route guards for auth, Use resolvers for preloading data
		✔ Use preloading strategies, Avoid heavy resolvers blocking navigation

🔹 Q2. What is preloading strategy? What is lazy loading? How to implement lazy loading?
=============================================================================================================
🧠 1️⃣ What is Lazy Loading?
	Lazy loading means: Load feature modules only when the user navigates to them.
	Instead of loading entire app at startup, Angular splits code into separate bundles.

🚀 Why Lazy Load?
	Without lazy loading: Initial load → All modules loaded, Large bundle → Slow startup
	With lazy loading: Initial load → Only core module, Feature modules → Loaded on demand
	
		✔ Faster initial load, Smaller bundle size, Better performance, Scales for large apps

2️⃣ How to Implement Lazy Loading?
	🔹 Step 1: Create Feature Module
				ng generate module admin --route admin --module app.module
				This creates:
							admin/
							  admin.module.ts
							  admin-routing.module.ts
	
	🔹 Step 2: Configure Route with loadChildren
				const routes: Routes = [
				  {
					path: 'admin',
					loadChildren: () => import('./admin/admin.module').then(m => m.AdminModule)
				  }
				];
	
	🔹 What Happens Internally?
			When user navigates to /admin:
				1️⃣ Router detects route
				2️⃣ Dynamically imports JS chunk
				3️⃣ Registers module routes
				4️⃣ Creates new injector
				5️⃣ Renders component
			
			This uses browser dynamic import().

🧠 3️⃣ What is Preloading Strategy?
		Preloading strategy defines:
		When lazy-loaded modules should be loaded after app initialization.
		Instead of waiting for user navigation, Angular can preload modules in background.

🚀 Built-in Preloading Strategies
	Angular provides: NoPreloading (default), PreloadAllModules
	
	🔹 NoPreloading (Default): Lazy modules load only when navigated.
	🔹 PreloadAllModules: Preloads all lazy modules after app boot.
	
	Enable in router config:
				RouterModule.forRoot(routes, {
				  preloadingStrategy: PreloadAllModules
				});
	
	✔ Improves subsequent navigation speed, Slightly increases network usage

4️⃣ Custom Preloading Strategy (Advanced)
	You can create your own strategy. Example: Preload only modules with flag.
	
	Step 1️⃣ Add data flag in route
				{
				  path: 'admin',
				  loadChildren: () => import('./admin/admin.module').then(m => m.AdminModule),
				  data: { preload: true }
				}

	Step 2️⃣ Create Custom Strategy
				@Injectable({ providedIn: 'root' })
				export class CustomPreloadingStrategy implements PreloadingStrategy {

				  preload(route: Route, load: () => Observable<any>) : Observable<any> {
						return route.data?.['preload']? load() : of(null);
				  }
				}
				
	Step 3️⃣ Register It
				RouterModule.forRoot(routes, {
				  preloadingStrategy: CustomPreloadingStrategy
				});

		✔ Only selected modules preload, Better control

🏢 Enterprise Strategy
	In large apps: 
		✔ Lazy load all feature modules, Preload frequently used modules, Avoid preloading admin-only modules
		✔ Use custom preloading for performance tuning, Combine with route guards

⚖ Lazy Loading vs Preloading
| Feature                        | Lazy Loading  | Preloading              |
| ------------------------------ | ------------- | ----------------------- |
| When Loaded                    | On navigation | After app init          |
| Improves Initial Load          | ✅             | ❌                       |
| Improves Subsequent Navigation | ❌             | ✅                       |
| Best For                       | Large apps    | Frequently used modules |

🔹 Q3. What are route guards? Difference between CanActivate, CanLoad, CanDeactivate?
=============================================================================================================
🧠 Route guards are: Services that decide whether a route can be activated, loaded, or exited.
	They act as control points in Angular’s routing pipeline.
	Commonly used for: Authentication, Authorization, Preventing unsaved changes, Role-based access, Lazy module protection

🏗 Where Guards Fit in Routing Flow
	When navigation happens:
							URL change
							   ↓
							Route match
							   ↓
							Guards execute
							   ↓
							If allowed → Activate route
							Else → Cancel navigation

🚀 Types of Guards: Angular provides several guards, but we’ll focus on the most important three:
	
	🧩 1️⃣ CanActivate: What It Does
		Determines whether a route can be activated. Runs after route matching but before component creation.
			
		Example:
						@Injectable({ providedIn: 'root' })
						export class AuthGuard implements CanActivate {
							  constructor(private authService: AuthService,  private router: Router) {}

							  canActivate(): boolean {
									if (this.authService.isLoggedIn()) {
									  return true;
									}

									this.router.navigate(['/login']);
									return false;
								  }
							}
					
		Route Configuration:
							{
							  path: 'dashboard',
							  component: DashboardComponent,
							  canActivate: [AuthGuard]
							}
		
		Use Case: ✔ Protect logged-in routes, Role-based access, Admin dashboard

	🧩 2️⃣ CanLoad: What It Does
		Prevents lazy-loaded module from even being loaded. This is very important.
		If CanLoad returns false: ✔ Module bundle is NOT downloaded, Code is not loaded, Improves security
		Example
						@Injectable({ providedIn: 'root' })
						export class AdminGuard implements CanLoad {

						  constructor(private authService: AuthService) {}

						  canLoad(): boolean {
							return this.authService.isAdmin();
						  }
						}
						
		Route Configuration:
						{
						  path: 'admin',
						  loadChildren: () =>
							import('./admin/admin.module')
							  .then(m => m.AdminModule),
						  canLoad: [AdminGuard]
						}
		
		Why Important? If you only use CanActivate: ❌ Module still downloads, Code accessible via DevTools
		CanLoad prevents that.

	🧩 3️⃣ CanDeactivate: What It Does
		Prevents user from leaving a route.
		Used to: ✔ Prevent losing unsaved form data, Confirm navigation
		Example
					@Injectable({ providedIn: 'root' })
					export class UnsavedChangesGuard
					  implements CanDeactivate<FormComponent> {

					  canDeactivate(component: FormComponent): boolean {
						return component.isFormSaved
						  || confirm('You have unsaved changes. Leave?');
					  }
					}
					
		Route Configuration:
		
					{
					  path: 'edit',
					  component: EditComponent,
					  canDeactivate: [UnsavedChangesGuard]
					}

⚖ Key Differences
| Guard         | When It Runs             | Prevents Module Load? | Use Case                |
| ------------- | ------------------------ | --------------------- | ----------------------- |
| CanActivate   | Before route activates   | ❌ No                  | Auth check              |
| CanLoad       | Before lazy module loads | ✅ Yes                 | Secure lazy modules     |
| CanDeactivate | Before leaving route     | N/A                   | Unsaved form protection |


🧠 Advanced Guard Behavior
	Guards can return: boolean, UrlTree, Observable<boolean | UrlTree>, Promise<boolean | UrlTree>
	
	Returning UrlTree allows redirect: return this.router.parseUrl('/login');
	Better than navigating manually.

🏢 Enterprise Best Practice
	✔ Use CanLoad for lazy modules
	✔ Use CanActivate for route-level access
	✔ Use CanDeactivate for forms
	✔ Combine with role-based logic
	✔ Keep guards lightweight

🧠 Guard Execution Order (Important for Interviews)
	Execution order:
		CanMatch, CanLoad, CanActivate, Resolve, Component Instantiation
	CanLoad only applies to lazy routes.

🔹 Q4. How to implement role-based routing?
=============================================================================================================
🧠 Role-based routing means: Restricting access to routes based on user roles (Admin, Manager, User, etc.)
	Example: 
		Admin → Can access /admin
		Manager → Can access /reports
		User → Can access /dashboard

	🏗 Enterprise Architecture Overview
			Login → Get JWT → Extract roles → Store roles
					 ↓
			Route navigation
					 ↓
			Guard checks role
					 ↓
			Allow / Redirect
	
	
🚀 Step-by-Step Implementation
	1️⃣ Store Roles in Auth Service
		After login:
				@Injectable({ providedIn: 'root' })
				export class AuthService {

				  private userRoles: string[] = [];

				  setRoles(roles: string[]) {
					this.userRoles = roles;
				  }

				  getRoles(): string[] {
					return this.userRoles;
				  }

				  hasRole(expectedRoles: string[]): boolean {
					return expectedRoles.some(role =>
					  this.userRoles.includes(role)
					);
				  }
				}
		
		In real apps: Roles come from JWT token, Decode token, Store in memory or state management

	2️⃣ Create Role Guard
				@Injectable({ providedIn: 'root' })
				export class RoleGuard implements CanActivate {

				  constructor(private authService: AuthService, private router: Router) {}

				  canActivate(route: ActivatedRouteSnapshot): boolean | UrlTree {
						const expectedRoles = route.data['roles'];
						if (this.authService.hasRole(expectedRoles)) {
						  return true;
						}
						return this.router.parseUrl('/unauthorized');
					  }
				}

	3️⃣ Configure Routes with Role Metadata
					const routes: Routes = [
					  {
						path: 'admin',
						component: AdminComponent,
						canActivate: [RoleGuard],
						data: { roles: ['ADMIN'] }
					  },
					  {
						path: 'reports',
						component: ReportsComponent,
						canActivate: [RoleGuard],
						data: { roles: ['ADMIN', 'MANAGER'] }
					  }
					];

		Now: ✔ Admin can access both, ✔ Manager can access reports, ✔ User blocked

🔥 For Lazy Loaded Modules (Important): Use CanLoad or CanMatch.
	Example:
			{
			  path: 'admin',
			  loadChildren: () =>  import('./admin/admin.module').then(m => m.AdminModule),
			  canLoad: [RoleGuard],
			  data: { roles: ['ADMIN'] }
			}

		This prevents module from even downloading.

🧠 Best Practice: Combine Auth + Role Guard
	Often you need:
		1️⃣ Is user logged in?
		2️⃣ Does user have role?
	
	You can combine:
					canActivate(route: ActivatedRouteSnapshot) {

					  if (!this.authService.isLoggedIn()) {
						return this.router.parseUrl('/login');
					  }

					  if (!this.authService.hasRole(route.data['roles'])) {
						return this.router.parseUrl('/unauthorized');
					  }

					  return true;
					}
				
🏢 Enterprise-Level Improvements
	🔹 1️⃣ Use JWT Role Claims
		Decode JWT:
					const decoded = jwtDecode(token);
					this.setRoles(decoded.roles);
			Never trust only frontend — backend must also enforce roles.

	🔹 2️⃣ Use CanMatch (Angular 15+ Recommended)
		CanMatch replaces CanLoad in modern Angular. Prevents route match completely.
	
	🔹 3️⃣ Role-Based Menu Rendering 
		Hide UI links dynamically:
			<a *ngIf="authService.hasRole(['ADMIN'])" routerLink="/admin"> Admin Panel </a>

	⚠ Important Security Note 
		Frontend role-based routing is: ❌ Not security, ✔ UX improvement
		Real security must be enforced on backend.

🧠 Guard Return Types
	Guards can return: boolean, UrlTree, Observable<boolean | UrlTree>, Promise<boolean | UrlTree>
	For async role check:
			canActivate(): Observable<boolean> {
			  return this.authService.user$.pipe(
				map(user => user.roles.includes('ADMIN'))
			  );
			}

🔹 Q5. What is route resolver? How to pass data between routes? How to handle 404 routes?
=============================================================================================================
🧠 1️⃣  A Route Resolver is: A service that fetches data before a route is activated.
	It ensures: ✔ Component loads with data ready, No empty UI flicker, Better UX, Centralized data fetching

🏗 Why Use Resolver?
	Without resolver: Navigate → Component loads → API call → UI updates
	With resolver: Navigate → Resolver calls API → Data ready → Component loads
	Component renders with data immediately.

🚀 How to Implement Route Resolver
	🔹 Step 1: Create Resolver
				@Injectable({ providedIn: 'root' })
				export class UserResolver implements Resolve<User> {

				  constructor(private userService: UserService) {}

				  resolve(route: ActivatedRouteSnapshot): Observable<User> {
					const id = route.paramMap.get('id');
					return this.userService.getUser(id!);
				  }
				}

	🔹 Step 2: Add to Route
				{
				  path: 'users/:id',
				  component: UserDetailComponent,
				  resolve: { user: UserResolver }
				}
	
	🔹 Step 3: Access Resolved Data
			constructor(private route: ActivatedRoute) {}

			ngOnInit() {
			  this.route.data.subscribe(data => {
				this.user = data['user'];
			  });
			}
			
🧠 Important Behavior
	Resolvers run: After guards, Before component creation, Navigation waits until resolver completes
	If resolver fails → navigation can fail.

🏢 When to Use Resolver?
	✔ Detail pages, Dashboard data, Lookup data, Initial page load data
	Avoid heavy blocking API calls unless necessary.

🧠 2️⃣ How to Pass Data Between Routes?
	There are multiple ways.
	
	🔹 Method 1: Route Parameters (Most Common)
					this.router.navigate(['/users', 10]);
			
			Route: { path: 'users/:id', component: UserComponent }
			Read:	this.route.paramMap.subscribe(params => { const id = params.get('id'); });
	
	🔹 Method 2: Query Parameters
			this.router.navigate(['/users'], {  queryParams: { active: true } });

		URL: /users?active=true
		Read: this.route.queryParamMap.subscribe(params => { const active = params.get('active'); });
	
	🔹 Method 3: Navigation State (Hidden Data)
			this.router.navigate(['/users'], { state: { userData: user } });
		Read: const data = this.router.getCurrentNavigation()?.extras.state;

		Good for: ✔ Temporary data, Not exposing in URL, But lost on page refresh.

	🔹 Method 4: Shared Service (Best for Complex Data)
		Use service with BehaviorSubject.
		Best for: ✔ Large objects, Cross-module sharing, State management

	🔹 Method 5: Route Data (Static)
				{
				  path: 'admin',
				  component: AdminComponent,
				  data: { role: 'ADMIN' }
				}

		Read: this.route.snapshot.data['role'];
		
🧠 3️⃣ How to Handle 404 Routes?
	In Angular, 404 route is defined using wildcard: { path: '**', component: NotFoundComponent }
	Important: ✔ Must be last route, Matches any unmatched URL
	
	🚀 Example
			const routes: Routes = [
			  { path: '', component: HomeComponent },
			  { path: 'users', component: UsersComponent },
			  { path: '**', component: NotFoundComponent }
			];
		Now: /invalid-url → NotFoundComponent

🔥 Performance Optimization (Critical for Senior)
=============================================================================================================
🔹 Q1. How to optimize Angular app performance? What is lazy loading and code splitting?
=============================================================================================================
🧠 1️⃣  Performance optimization happens at multiple layers:
		Build level, Routing level, Change detection level, Template level, Network level, State management level. Let’s break it down.
		
	🚀 A. Use Lazy Loading (Routing-Level Optimization)
			Instead of loading entire app at once: App starts → All modules loaded ❌
			Use lazy loading:
				App starts → Only core module
				Feature modules → Load when needed ✅
				We’ll go deeper into this below.

	🚀 B. Enable Production Mode
		Always build with: ng build --configuration production
		This enables: ✔ AOT compilation, Minification, Tree shaking, Dead code elimination

	🚀 C. Use OnPush Change Detection
		Default change detection checks entire component tree.
		Use: changeDetection: ChangeDetectionStrategy.OnPush
		Benefits: ✔ Reduces unnecessary re-renders, Works well with immutable data, Improves large dashboard performance

	🚀 D. Avoid Unnecessary Subscriptions
		Use: ✔ async pipe, takeUntil pattern, Avoid nested subscriptions
		Reduces memory leaks and improves responsiveness.

	🚀 E. Optimize *ngFor with trackBy
		Without trackBy: Angular recreates DOM elements.
		Use: 
			*ngFor="let item of items; trackBy: trackById"
			trackById(index: number, item: any) {
			  return item.id;
			}
		Huge improvement for large lists.

	🚀 F. Virtual Scrolling
		For 1000+ rows: Use Angular CDK Virtual Scroll. Only renders visible items.

	🚀 G. Use Pure Pipes
		Pure pipes run only when input reference changes. Avoid heavy calculations in templates.

	🚀 H. Debounce User Input
		For search fields: valueChanges.pipe(debounceTime(300)), Reduces API calls.

	🚀 I. Use Preloading Strategy
		Preload important lazy modules in background. Improves navigation speed.

	🚀 J. Optimize Bundle Size: ✔ Remove unused libraries, Use dynamic imports, Avoid heavy polyfills, Use ESBuild (Angular 17+)

🧠 2️⃣ What is Lazy Loading?  
		Lazy loading means load feature modules only when user navigates to them. Instead of bundling everything into main bundle.
		Example
				{
				  path: 'admin',
				  loadChildren: () => import('./admin/admin.module').then(m => m.AdminModule)
				}

		When user visits /admin: ✔ Angular downloads separate JS chunk, Loads module, Renders component
		This improves: ✔ Initial load time, Time to Interactive (TTI)

🧠 3️⃣ What is Code Splitting? 
		Code splitting means breaking application into smaller bundles. Lazy loading uses dynamic import() to create separate chunks.
		Without code splitting: main.js → 3MB
		With code splitting:
			main.js → 800KB
			admin.js → 500KB
			reports.js → 400KB
		Browser loads only required chunks.

🏢 Enterprise Performance Strategy
	For large apps:
		✔ Lazy load all feature modules, Use OnPush everywhere, Use smart/dumb component pattern
		✔ Cache API responses (shareReplay/store), Use CDK virtual scroll, Avoid heavy resolvers
		✔ Use route preloading strategically, Use browser caching headers

🧠 Performance Debugging Tools
	✔ Angular DevTools, Chrome Performance tab, Source-map-explorer (bundle size), Lighthouse

🔹 Q2. What is tree shaking? How to reduce bundle size?
=============================================================================================================
🌳 1️⃣  Tree shaking is the process of removing unused code from the final JavaScript bundle.
	It works at build time.

🧠 Why Called "Tree Shaking"? Because the module dependency graph looks like a tree:
		AppModule
		 ├── FeatureA
		 ├── FeatureB
		 ├── UtilityX

	If FeatureB is never used: 👉 It gets “shaken off” like a dead branch.

🏗 How Angular Performs Tree Shaking
	Angular build process (production mode): TypeScript → AOT → Bundler (Webpack/ESBuild) → Minifier → Output
	Tree shaking is done by: ✔ ES module static imports, Dead code elimination, Terser/ESBuild optimizer
	
	Modern Angular (v16+) uses ESBuild by default for faster builds.
	🚀 Example of Tree Shaking
				// utils.ts
				export function usedFunction() {}
				export function unusedFunction() {}
				import { usedFunction } from './utils';

	Final bundle will include: ✔ usedFunction ❌ unusedFunction removed
	
	⚠ When Tree Shaking Fails
		Tree shaking works only with: 
			✔ ES Modules (import/export)
			❌ CommonJS require()

		Bad example: const lib = require('some-lib');
		This prevents effective tree shaking.

🚀 2️⃣ How to Reduce Bundle Size (Practical Guide): Reducing bundle size is a multi-layer strategy.

	✅ A. Always Use Production Build: ng build --configuration production
		Enables: ✔ AOT, Minification, Tree shaking, Dead code removal

	✅ B. Use Lazy Loading (Code Splitting)=> Instead of loading all features:
		loadChildren: () => import('./admin/admin.module')
		Creates separate chunks.
	
	✅ C. Avoid Large Third-Party Libraries
		Bad: import * as _ from 'lodash';
		Better: import debounce from 'lodash/debounce';
		Only import what you need.
	
	✅ D. Use Modern Libraries (ESM Compatible)
		Avoid libraries using CommonJS.
		Angular CLI warns about this.

	✅ E. Use Standalone Components (Angular 15+)
		Standalone components reduce module overhead.
	
	✅ F. Remove Unused Polyfills
		Check: polyfills.ts
		Remove unnecessary ones if targeting modern browsers.

	✅ G. Analyze Bundle
		Use: ng build --stats-json
		Then: npx webpack-bundle-analyzer dist/stats.json
		Identify heavy modules.
		
	✅ H. Use Differential Loading (Handled Automatically): Modern Angular builds separate bundles for modern browsers.
	
	✅ I. Optimize Images & Assets: ✔ Compress images, Use WebP, Lazy load images
	
	✅ J. Use Environment-Based Imports: Avoid importing dev-only tools in production.
			Example:
					if (environment.production) {
					  enableProdMode();
					}

🏢 Enterprise Bundle Optimization Strategy
	For large enterprise apps:
		✔ Lazy load every feature module, Use OnPush change detection, Avoid heavy UI frameworks
		✔ Use CDK instead of large UI libs if possible, Keep shared module lightweight
		✔ Preload only necessary modules, Use HTTP caching

🔹 Q3. What is Ahead of Time compilation benefit? How to improve initial load time?
=============================================================================================================
🧠 1️⃣ What is Ahead-of-Time (AOT) Compilation?
	Angular supports two compilation modes: JIT (Just-In-Time), AOT (Ahead-of-Time)

👉 AOT means: Angular templates are compiled into JavaScript at build time instead of in the browser.

🏗 Without AOT (JIT): Browser loads Angular compiler, Templates compiled in browser, Slower startup, Larger bundle
🏗 With AOT: Templates compiled during build, Browser gets optimized JS, No runtime compilation, Faster startup
	Modern Angular (v9+) uses AOT by default in production builds

🚀 2️⃣ Benefits of AOT Compilation
	✅ 1. Faster Initial Load: Because No template compilation in browser, Less runtime work, Startup becomes faster.
	
	✅ 2. Smaller Bundle Size: AOT removes Angular compiler (~500KB), Metadata not needed at runtime
	
	✅ 3. Better Performance: Templates converted to ɵɵelementStart(...), ɵɵtext(...), Precompiled instructions → faster rendering.
	
	✅ 4. Earlier Error Detection: Template errors caught at build time. Example: {{ user.nam }}, Caught during build, not runtime.
	
	✅ 5. Improved Security: No runtime template compilation means Lower injection attack surface, Safer production builds

🧠 3️⃣ How to Improve Initial Load Time?
	Initial load time = Download time + Parse time + Execute time + Render time, Let’s optimize each.
	
	🚀 A. Enable Production Mode: ng build --configuration production
		Enables: ✔ AOT, Minification, Tree shaking, Dead code elimination

🔹 Q4. How to optimize large tables? How to implement virtual scrolling?
=============================================================================================================
🧠 1️⃣ Why Large Tables Are Slow?
		When you render: <tr *ngFor="let item of 10000 items">
		Angular: ✔ Creates 10,000 DOM nodes, Runs change detection on all, Re-renders if data changes, Browser layout recalculates
		This causes: ❌ Slow initial render, High memory usage, Scroll lag, UI freezing

🚀 2️⃣ How to Optimize Large Tables (Before Virtual Scroll)
	✅ A. Use OnPush Change Detection: changeDetection: ChangeDetectionStrategy.OnPush, Prevents unnecessary re-checking of rows.
	
	✅ B. Use trackBy in *ngFor
			Without trackBy → Angular recreates DOM.
					trackById(index: number, item: any) {
					  return item.id;
					}
			*ngFor="let item of items; trackBy: trackById"
			Huge performance improvement.

	✅ C. Avoid Heavy Template Logic
		❌ Bad: {{ calculateTotal(item) }}
		✔ Precompute values in component instead.

	✅ D. Use Pagination (Simple Fix)
			Instead of 10,000 rows: ✔ Show 50 per page, Use server-side pagination, This reduces DOM size drastically.

🧠 3️⃣ What is Virtual Scrolling?
	Virtual scrolling means render only the visible rows in the viewport instead of the entire dataset.
	Example: If viewport shows 20 rows → only 20 rows exist in DOM.
	As user scrolls: ✔ Old rows removed, New rows rendered
	Massive performance gain.
	
🚀 4️⃣ How to Implement Virtual Scrolling (Angular CDK)
	Angular provides Virtual Scroll via: @angular/cdk/scrolling
	🔹 Step 1: Install CDK: npm install @angular/cdk
	
	🔹 Step 2: Import Module
				import { ScrollingModule } from '@angular/cdk/scrolling';

				@NgModule({
				  imports: [ScrollingModule]
				})
				export class AppModule {}

	🔹 Step 3: Use cdk-virtual-scroll-viewport
				<cdk-virtual-scroll-viewport itemSize="50" class="viewport">
					<div *cdkVirtualFor="let item of items; trackBy: trackById" class="row"> {{ item.name }} </div>
				</cdk-virtual-scroll-viewport>

	🔹 Step 4: Add Height Styling: 
			.viewport {
			  height: 500px;
			  width: 100%;
			}
		itemSize must match row height.

🧠 How It Works Internally
		Instead of: 10000 DOM nodes
		Virtual scroll keeps: ~20–40 DOM nodes
		As user scrolls: ✔ Reuses DOM elements, Updates data, Maintains scroll illusion

🚀 5️⃣ Virtual Scroll + Table Example
		For table layout:
			<cdk-virtual-scroll-viewport itemSize="48" class="viewport">
				  <table>
					<tr *cdkVirtualFor="let row of rows; trackBy: trackById">
					  <td>{{ row.id }}</td>
					  <td>{{ row.name }}</td>
					</tr>
				  </table>
			</cdk-virtual-scroll-viewport>
			
🏢 Enterprise Best Practices for Large Tables
	🔹 Combine Techniques: ✔ Virtual scroll, OnPush, trackBy, Server-side pagination, Debounce filters
	🔹 Avoid Rendering All Columns:	For 30+ columns: ✔ Lazy render hidden columns, Use column virtualization
	🔹 Use Server-Side Sorting & Filtering: Avoid sorting 50k rows on frontend.
	🔹 Avoid Complex Components in Each Cell
	Heavy nested components per row = performance killer.

⚠ Common Mistakes: ❌ Forgetting itemSize, Using heavy pipes inside rows, Not using trackBy, Using default change detection

🔹 Q5. What is Angular Universal (SSR)? What is hydration?
=============================================================================================================
🧠 1️⃣  What is Angular Universal (SSR)?
	Angular Universal is Angular’s solution for: Server-Side Rendering (SSR)
	It renders Angular applications on the server instead of in the browser.
	
	📦 Entity Angular Universal

🏗 Normal Angular (CSR – Client Side Rendering)
	Without SSR: Browser → Download JS → Execute → Render HTML
	Problems: ❌ Blank screen until JS loads, Slow First Contentful Paint (FCP), Poor SEO, Slow on low-end devices

	🚀 With Angular Universal (SSR)
		Browser → Request page
		Server → Renders HTML
		Browser → Receives ready HTML
		Then Angular bootstraps
		Now:
			✔ Immediate HTML content, Faster perceived load, SEO-friendly, Better performance metrics

🧠 2️⃣ How Angular Universal Works Internally
	When SSR is enabled:
		1️⃣ User requests /home
		2️⃣ Node.js server runs Angular app
		3️⃣ Angular renders component tree to HTML string
		4️⃣ Server sends HTML to browser
		5️⃣ Browser displays content immediately
		6️⃣ Angular JS loads and bootstraps
	
	Important: SSR only renders initial HTML. Interactivity comes after client bootstrapping.

🚀 How to Enable SSR: ng add @nguniversal/express-engine
	This: ✔ Adds server.ts, Creates Express server, Adds build:ssr scripts

🧠 3️⃣ What is Hydration?
	Hydration means: Reusing server-rendered HTML instead of recreating it on the client.
	Without hydration: Server renders HTML, Client loads JS, Client re-renders everything again ❌
	With hydration: Server renders HTML, Client attaches event listeners to existing DOM ✔, No DOM recreation.

📦 Angular Hydration: Angular 16+ supports built-in hydration.
	Enable it:
		bootstrapApplication(AppComponent, {
		  providers: [
			provideClientHydration()
		  ]
		});
		
🧠 4️⃣ Why Hydration is Important?
		Without hydration: Double rendering cost, Flicker, Performance loss
		With hydration: Faster Time to Interactive (TTI), No flicker, Reuses DOM, Better Web Vitals

🏢 When to Use SSR?
	Use SSR when: ✔ SEO is important, Public marketing site, E-commerce, Blog, Content-heavy app
	Avoid SSR when: ❌ Internal admin dashboards, Purely authenticated apps, Heavy browser-only APIs

⚖ SSR vs CSR vs Hydration
| Feature            | CSR        | SSR          | SSR + Hydration       |
| ------------------ | ---------- | ------------ | --------------------- |
| SEO                | ❌ 	      | ✅  	     | ✅     	             |
| Initial Load Speed | Slow       | Fast         | Very Fast             |
| Server Cost        | Low        | Higher       | Higher                |
| Best For           | Dashboards | Public Sites | High-performance apps |

🧠 Enterprise-Level Considerations
	✔ Use caching on server, Avoid heavy synchronous logic in SSR, Be careful with browser-only APIs (window, document)
	✔ Use TransferState to avoid duplicate API calls, Combine with lazy loading

🔥 TransferState (Important in SSR)
	Avoid double API call: Server calls API, Client loads and calls same API again
	Use TransferState to share data from server to client.

🏗 E-commerce Enterprise Architecture Example 
	SSR + Hydration: Home, Product listing, Product detail, Landing pages, Search
	CSR Only: Cart, Checkout, Order history, Admin panel, User profile

🔥 Bonus: Use TransferState
	Important for e-commerce:
		Without TransferState: Server fetches product, Client fetches product again
		With TransferState: ✔ Avoid duplicate API calls, Faster hydration

🔹 Q1. What is XSS? How Angular protects against XSS?
=============================================================================================================
🧠 1️⃣  XSS (Cross-Site Scripting) is a security vulnerability where an attacker injects malicious JavaScript into a web application, which then executes in another user’s browser.
	🚨 Example: Imagine a comment field:
		<input value="<script>alert('Hacked')</script>">
		If application renders this without sanitization: Script executes, Cookies stolen, Session hijacked, Account takeover

🧠 2️⃣ Types of XSS
	🔹 1. Stored XSS: Malicious script stored in DB.
			Example: User comment → Stored in DB → Rendered to others, Most dangerous.
	
	🔹 2. Reflected XSS
			Script injected via URL: example.com?q=<script>...</script>
	
	🔹 3. DOM-based XSS: Client-side JS manipulates DOM unsafely.

🛡 3️⃣ How Angular Protects Against XSS: 	Angular has built-in sanitization.
	By default: ✔ Angular escapes interpolation, Sanitizes HTML bindings, Sanitizes URLs, Sanitizes style values

	🔹 1. Interpolation is Safe: <div>{{ userInput }}</div>
			If userInput contains: <script>alert(1)</script>
			Angular renders it as text, not executable script.

	🔹 2. [innerHTML] is Sanitized: <div [innerHTML]="htmlContent"></div>
			Angular removes dangerous tags automatically.

	🔹 3. URL Sanitization: <a [href]="userProvidedUrl"></a>
			Angular blocks: javascript:alert(1)
			
🧠 Angular Security Contexts
| Context      | Example            |
| ------------ | ------------------ |
| HTML         | innerHTML          |
| Style        | [style.background] |
| URL          | href, src          |
| Resource URL | iframe src         |

4️⃣ When Angular Protection Can Be Bypassed
	Angular provides: DomSanitizer
	Example: this.sanitizer.bypassSecurityTrustHtml(html); This disables sanitization.
	⚠ Dangerous if used with untrusted input.

🚨 Example of Risk
	bypassSecurityTrustHtml(userInput); Now script will execute.
	Only use for: ✔ Trusted server content, Sanitized HTML

🏢 Enterprise Best Practices
	✔ Never use innerHTML with user input, Avoid bypassSecurityTrust* unless necessary
	✔ Validate input on backend, Use Content Security Policy (CSP), Use HttpOnly cookies
	✔ Use SameSite cookies, Sanitize data server-side

🔥 Angular Also Protects Against Template Injection
	Angular does NOT allow runtime template compilation in production.
	This reduces: ✔ Template injection attacks, Especially with AOT enabled.

🧠 Backend Still Required
	Angular protection is frontend-level only. Backend must: ✔ Validate inputs, Escape outputs, Prevent SQL injection, Implement CSRF protection

🔹 Q2. What is CSRF and how to prevent it?
=============================================================================================================
🧠 1️⃣  CSRF (Cross-Site Request Forgery) is an attack where a malicious website tricks a logged-in user’s browser into sending unauthorized requests to another website. The key idea: ✔ Victim is authenticated, Browser automatically sends cookies, Attacker triggers hidden request

🧨 Example Scenario
	User is logged into: https://bank.com
	Session cookie: sessionId=abc123
	Now user visits malicious site: <img src="https://bank.com/transfer?amount=10000&to=attacker">
	Browser automatically sends: Cookie: sessionId=abc123
	Bank thinks request is legitimate. Money transferred

🧠 Why CSRF Happens
	Because: Browser automatically includes cookies, Server trusts cookies blindly, No request validation
	CSRF mainly affects: Cookie-based authentication systems, Not typical for pure JWT in Authorization header.

🚀 2️⃣ How to Prevent CSRF
	There are multiple protection strategies.
	
	✅ 1️⃣ CSRF Token (Synchronizer Token Pattern) Most common protection.
		Flow:
			1️⃣ Server generates CSRF token
			2️⃣ Sends it to client
			3️⃣ Client sends token in header for every request
			4️⃣ Server verifies token
		
		If attacker sends request: They don’t have valid token
		🔹 Example
				Server response: { "csrfToken": "xyz123" }
				Client sends:
					POST /transfer
					X-CSRF-Token: xyz123
					Cookie: sessionId=abc123
				Server verifies token matches session.

	🚀 2️⃣ SameSite Cookies: Modern browsers support:
			Set-Cookie: sessionId=abc123; SameSite=Strict
			| Value  | Behavior                         |
			| ------ | -------------------------------- |
			| Strict | No cross-site requests allowed   |
			| Lax    | Limited cross-site allowed       |
			| None   | No restriction (requires Secure) |
		SameSite prevents automatic cookie sending in many CSRF cases.
	

🚀 3️⃣ Double Submit Cookie Pattern
		Client stores CSRF token in cookie and header.
		Server compares both.

🚀 4️⃣ Use Authorization Header Instead of Cookies
		JWT in header: Authorization: Bearer token
		Browsers don’t automatically attach Authorization header to cross-site requests.
		So CSRF risk is reduced.

🧠 3️⃣ How Angular Helps Prevent CSRF
	Angular provides built-in CSRF support via: HttpClientXsrfModule
	
	🔹 Enable in AppModule
			import { HttpClientXsrfModule } from '@angular/common/http';

			@NgModule({
			  imports: [
				HttpClientXsrfModule.withOptions({
				  cookieName: 'XSRF-TOKEN',
				  headerName: 'X-XSRF-TOKEN'
				})
			  ]
			})
			export class AppModule {}

	🔹 How It Works
		1️⃣ Server sets cookie: Set-Cookie: XSRF-TOKEN=xyz123
		2️⃣ Angular reads cookie
		3️⃣ Automatically adds header: X-XSRF-TOKEN: xyz123
		4️⃣ Server verifies token
		No manual coding needed.

🏢 Enterprise Best Practices
	✔ Use SameSite=Strict cookies, Use HttpOnly cookies, Enable CSRF token validation
	✔ Validate CSRF token on backend, Use HTTPS always, Avoid GET for state-changing actions
	✔ Combine with JWT or session validation

⚠ Important Clarification: CSRF ≠ XSS
| CSRF                    | XSS                      |
| ----------------------- | ------------------------ |
| Uses authenticated user | Injects malicious script |
| Exploits cookies        | Exploits DOM             |
| Cross-site request      | Script injection         |

🔹 Q3. How to secure JWT in Angular? Where should token be stored? How to refresh token automatically?
=============================================================================================================
🧠 1️⃣  JWT (JSON Web Token) is used for:
		✔ Authentication, Authorization, Stateless session management
		But JWT itself is not secure unless handled correctly.
		Security depends on: Storage method, Transmission method, Refresh strategy, Backend validation

🏗 JWT Authentication Flow
					User logs in
						 ↓
					Server returns:
					  - Access Token (short-lived)
					  - Refresh Token (long-lived)
						 ↓
					Angular stores tokens
						 ↓
					Angular sends Access Token in header
						 ↓
					Backend validates

🧠 2️⃣ Where Should JWT Be Stored?
		There are 3 common options:
		❌ Option 1: localStorage (Not Recommended for Sensitive Apps): localStorage.setItem('token', token);
				Risk: Vulnerable to XSS, JavaScript can access token, If attacker injects script → token stolen
				
		❌ Option 2: sessionStorage: Same XSS risk. Only difference: Cleared on tab close, Still vulnerable to XSS.

		✅ Option 3: HttpOnly Secure Cookies (Recommended)
			Best practice for enterprise apps.
			Server sets:
				Set-Cookie: accessToken=abc;
				  HttpOnly;
				  Secure;
				  SameSite=Strict;
			
			Benefits: 
			JavaScript cannot access token, Protected from XSS, SameSite reduces CSRF, Automatically sent with requests, Most secure option.

🧠 Recommended Enterprise Approach
	Access Token → HttpOnly Cookie (short expiry)
	Refresh Token → HttpOnly Cookie (long expiry)
	Never expose refresh token to frontend JS.

🚀 3️⃣ How to Send JWT in Angular?
	If token stored in memory/localStorage: Use HTTP interceptor.

	🔹 JWT Interceptor
			@Injectable()
			export class AuthInterceptor implements HttpInterceptor {

			  intercept(req: HttpRequest<any>, next: HttpHandler) {

				const token = this.authService.getToken();

				if (token) {
				  const cloned = req.clone({
					setHeaders: {
					  Authorization: `Bearer ${token}`
					}
				  });

				  return next.handle(cloned);
				}

				return next.handle(req);
			  }
			}
			
🔁 4️⃣ How to Refresh Token Automatically?
	Access token should: Be short-lived (e.g., 15 mins), Expire frequently
	Refresh token: Long-lived (e.g., 7 days)

	🚀 Automatic Refresh Flow
		When API returns: 401 Unauthorized
		Interceptor should:
			1️⃣ Pause outgoing requests
			2️⃣ Call refresh endpoint
			3️⃣ Update access token
			4️⃣ Retry original request

	🔹 Refresh Token Interceptor Example
			private isRefreshing = false;
			private refreshTokenSubject = new BehaviorSubject<string | null>(null);

			intercept(req: HttpRequest<any>, next: HttpHandler) {
			  return next.handle(req).pipe(
				catchError(error => {

				  if (error.status === 401 && !this.isRefreshing) {

					this.isRefreshing = true;

					return this.authService.refreshToken().pipe(
					  switchMap(newToken => {

						this.isRefreshing = false;
						this.refreshTokenSubject.next(newToken);

						const cloned = req.clone({
						  setHeaders: {
							Authorization: `Bearer ${newToken}`
						  }
						});

						return next.handle(cloned);
					  })
					);
				  }

				  return throwError(() => error);
				})
			  );
			}

🧠 Enterprise-Level Best Practices
	✔ Use HttpOnly + Secure cookies, Short expiry access tokens, Rotate refresh tokens, Implement token revocation
	✔ Validate tokens server-side, Use SameSite=Strict, Use HTTPS only, Do NOT store refresh token in localStorage

⚠ Important Security Notes
	🔹 Protect Against XSS : If token in localStorage: XSS → token stolen.
		So: ✔ Use CSP, Sanitize input, Avoid bypassSecurityTrust

	🔹 Protect Against CSRF: If using cookies: Use SameSite, Use CSRF tokens, Validate origin

🧠 When to Use Which Storage?
| Scenario                    | Recommended                      |
| --------------------------- | -------------------------------- |
| High-security app (banking) | HttpOnly Cookie                  |
| Medium security             | HttpOnly Cookie                  |
| Internal dashboard          | Possibly memory + refresh cookie |
| Public API SPA              | Header-based JWT                 |

🔹 Q1. How HttpClient works internally?
=============================================================================================================
🧠 1️⃣ Angular’s HttpClient is a high-level service used to make HTTP requests.
		It is part of: @angular/common/http
		It is built on top of: ✔ RxJS Observables, Browser XHR / Fetch API, Interceptor chain

🏗 High-Level Architecture
	When you call: this.http.get('/api/users'), It goes through:

			HttpClient
			   ↓
			Interceptor chain
			   ↓
			HttpHandler
			   ↓
			HttpBackend
			   ↓
			Browser (XHR or Fetch)
			   ↓
			Server

2️⃣ Step-by-Step Internal Flow
🔹 Step 1: Create HttpRequest Object => When you call: this.http.post('/login', body);
		Angular creates: HttpRequest<T>
		This object contains: URL, Method, Headers, Body, Params, ResponseType
		Important: ✔ HttpRequest is immutable, If modified → cloned

🔹 Step 2: Pass Through Interceptor Chain
		HttpClient does NOT directly call backend.
		Instead it passes request to: HttpHandler
		Each interceptor: intercept(req, next)
		Can: ✔ Modify request, Add headers, Handle errors, Retry request, Short-circuit request

🔹 Step 3: Final Handler → HttpBackend
		After all interceptors:
			Request reaches: HttpBackend
			Default implementation: HttpXhrBackend
			Which uses: ✔ XMLHttpRequest (older versions), Fetch API (modern Angular option)

🔹 Step 4: Browser Sends Request
		Browser handles: ✔ DNS, TCP, TLS, Network transfer

🔹 Step 5: Response Converted to Observable
		When response comes back: Angular wraps it in: Observable<HttpEvent<T>>
		Then: ✔ JSON parsed, Typed response returned

🧠 3️⃣ How Interceptors Work Internally
		Interceptors form a chain (like middleware).
		Example:
				AuthInterceptor
				   ↓
				LoggingInterceptor
				   ↓
				ErrorInterceptor
				   ↓
				Backend

		Each interceptor: return next.handle(modifiedRequest);
		Angular composes them in reverse order.

🔥 Interceptor Pattern Internally
		It uses: reduceRight() To wrap handlers like nested functions.

🧠 4️⃣ Why HttpClient Uses Observables (Not Promises)?
		Because Observables: 
			✔ Are lazy (no request until subscribed), Support cancellation, Support retry, 
			✔ Support multiple events (progress), Work well with RxJS operators
		
		Example: this.http.get('/api').pipe(retry(2), catchError(...))

🧠 5️⃣ HttpEvent Lifecycle
		If you enable progress tracking: observe: 'events'
		| Event            | Meaning           |
		| ---------------- | ----------------- |
		| Sent             | Request sent      |
		| UploadProgress   | Upload progress   |
		| ResponseHeader   | Headers received  |
		| DownloadProgress | Download progress |
		| Response         | Final response    |

6️⃣ Dependency Injection Role
	HttpClient is provided via: HttpClientModule
	DI injects: ✔ Interceptors, Backend, XSRF support

🛡 Built-in Features
	✔ Automatic JSON parsing, XSRF protection, Request cancellation, Interceptor support, Strong typing, Error handling

🏢 Enterprise-Level Behavior
	In large apps: 
		✔ Auth interceptor adds JWT
		✔ Refresh interceptor handles 401
		✔ Logging interceptor tracks metrics
		✔ Error interceptor centralizes errors
		✔ Caching interceptor stores responses

🧠 Internal Class Flow Summary
								HttpClient
								   ↓
								HttpInterceptingHandler
								   ↓
								HttpHandler chain
								   ↓
								HttpXhrBackend
								   ↓
								XMLHttpRequest

🔹 Q2. How to add global headers? How to implement request/response interceptor?
=============================================================================================================
🧠 1️⃣ In Angular, global headers are added using: HTTP Interceptors
	Because HttpClient itself is immutable — every request must be cloned before modification.

🏗 Architecture
				HttpClient
				   ↓
				AuthInterceptor (adds headers)
				   ↓
				LoggingInterceptor
				   ↓
				ErrorInterceptor
				   ↓
				Backend

🚀 2️⃣ Request Interceptor (Add Global Headers)
		Example: Add Authorization + Custom Headers
		
			import { Injectable } from '@angular/core';
			import { HttpEvent, HttpHandler, HttpInterceptor, HttpRequest } from '@angular/common/http';
			import { Observable } from 'rxjs';
	
			@Injectable()
			export class AuthInterceptor implements HttpInterceptor {

			  intercept(req: HttpRequest<any>, next: HttpHandler): Observable<HttpEvent<any>> {

				const authToken = 'sample-token'; // get from service

				const clonedRequest = req.clone({
				  setHeaders: {
					Authorization: `Bearer ${authToken}`,
					'X-App-Version': '1.0.0',
					'Content-Type': 'application/json'
				  }
				});

				return next.handle(clonedRequest);
			  }
			}

	🔹 Register Interceptor
		For module-based apps:
					providers: [
					  {
						provide: HTTP_INTERCEPTORS,
						useClass: AuthInterceptor,
						multi: true
					  }
					]
		For standalone apps:
					provideHttpClient(withInterceptors([authInterceptor]))


🚀 3️⃣ Response Interceptor
		Interceptors can also modify or handle responses.
		Example: Logging + Transform Response
	
			import { tap } from 'rxjs/operators';

			@Injectable()
			export class LoggingInterceptor implements HttpInterceptor {

			  intercept(req: HttpRequest<any>, next: HttpHandler) {

				const startTime = Date.now();

				return next.handle(req).pipe(
				  tap(event => {
					if (event.type === 4) { 
					  const duration = Date.now() - startTime;
					  console.log(`${req.url} took ${duration}ms`);
					}
				  })
				);
			  }
			}
			
🚀 4️⃣ Global Error Handling Interceptor: Most common enterprise interceptor.

				import { catchError } from 'rxjs/operators';
				import { throwError } from 'rxjs';

				@Injectable()
				export class ErrorInterceptor implements HttpInterceptor {

				  intercept(req: HttpRequest<any>, next: HttpHandler) {

					return next.handle(req).pipe(
					  catchError(error => {

						if (error.status === 401) {
						  console.log('Unauthorized');
						  // redirect to login
						}

						if (error.status === 500) {
						  console.log('Server error');
						}

						return throwError(() => error);
					  })
					);
				  }
				}
				
🧠 5️⃣ Order of Interceptors
		Interceptors execute in the order provided.
		Request flow: Auth → Logging → Error → Backend
		Response flow: Backend → Error → Logging → Auth
		It works like middleware stack.

🚀 6️⃣ Advanced: Conditional Header Injection
		Sometimes you don’t want headers for certain URLs:
			if (req.url.includes('/login')) {
			  return next.handle(req);
			}
		
🚀 7️⃣ Skip Interceptor for Specific Requests
		Use HttpBackend directly:
			constructor(private backend: HttpBackend) {
			  this.http = new HttpClient(backend);
			}

		This bypasses interceptors.

				 ↓
			Angular creates FormData

🚀 2️⃣ Angular File Upload
	🔹 HTML:

							@RequestParam("file") MultipartFile file) {

						try {

							Path uploadPath = Paths.get("uploads/");
							Files.createDirectories(uploadPath);

							Path filePath = uploadPath.resolve(file.getOriginalFilename());
							Files.write(filePath, file.getBytes());

							return ResponseEntity.ok("File uploaded successfully");

						} catch (IOException e) {
							return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR)
									.body("Upload failed");
						}
					}

📥 4️⃣ File Download (Spring Boot → Angular)

					Path filePath = Paths.get("uploads/").resolve(filename);
					Resource resource = new UrlResource(filePath.toUri());

					return ResponseEntity.ok()
							.contentType(MediaType.APPLICATION_OCTET_STREAM)
							.header(HttpHeaders.CONTENT_DISPOSITION,
									"attachment; filename=\"" + filename + "\"")
							.body(resource);

		Pagination solves: ✔ Small chunks of data, Faster queries, Better UX

2️⃣ Pagination in Spring Boot (Offset-Based)

					{
					  "content": [...],
					  "totalElements": 1000,
					  "totalPages": 100,
					  "number": 0
					}

🧠 3️⃣ Angular Pagination

			page = 0;

	🔹 Cursor Pagination (Recommended for Large Data)

	For streaming large file download:

		For real-time streaming: ✔ Use Server-Sent Events (SSE), Use WebSocket, Use chunked response

🏢 Enterprise Strategy
	For Regular UI Table: ✔ Use pagination, Use server-side filtering, Use sorting on backend
	For Infinite Scroll: ✔ Use cursor pagination, Use virtual scrolling in Angular
	For Data Export (CSV/Excel): ✔ Use streaming, Use chunked transfer encoding, Avoid loading all records in memory

🔹 Q1. What is standalone components?
=============================================================================================================
🧠 What Are Standalone Components?
	Standalone components are: Angular components that do NOT require NgModule.
	Before standalone: Component → Must be declared inside NgModule
	With standalone: Component → Self-contained, No declarations array required.

🏗 Traditional Angular Architecture
	Before Angular 14:
			@NgModule({
			  declarations: [AppComponent],
			  imports: [BrowserModule]
			})
			export class AppModule {}

		Every component had to be declared in a module.
		This created: ❌ Boilerplate, Large module dependencies, Hard-to-manage shared modules

🚀 Standalone Component (Angular 14+)
		import { Component } from '@angular/core';

		@Component({
		  selector: 'app-home',
		  standalone: true,
		  template: `<h1>Hello Standalone!</h1>`
		})
		export class HomeComponent {}
	No module required.
	
🧠 How Do Standalone Components Import Dependencies?
	Instead of NgModule imports:

				@Component({
				  standalone: true,
				  imports: [CommonModule, RouterModule],
				  template: ` <div *ngIf="true">Standalone Works</div> `
				})
		Everything required is added in imports array.

🚀 Bootstrapping Without NgModule
	Instead of AppModule: 
						import { bootstrapApplication } from '@angular/platform-browser';
						bootstrapApplication(AppComponent);

	Modern Angular uses: bootstrapApplication() instead of platformBrowserDynamic().bootstrapModule().

🧠 Routing with Standalone Components
		const routes: Routes = [
		  {
			path: 'home',
			loadComponent: () => import('./home.component').then(m => m.HomeComponent); 🔥 Lazy Loading with Standalone
		  }
		];
		
	Uses loadComponent instead of loadChildren.

	This enables: Component-level lazy loading, Smaller bundles, Cleaner routing

🧠 Why Angular Introduced Standalone?
	Goals: Reduce boilerplate, Improve tree-shaking, Improve lazy loading, Simplify mental model, Move away from heavy NgModule usage

🏢 Enterprise Benefits
	✔ Smaller bundle size, Better code splitting, Faster startup, More modular architecture, Easier micro-frontend integration

🧠 Standalone Supports Everything
	Standalone works with: ✔ Components, Pipes, Directives, Routing, Guards, Interceptors, Providers

🧠 Internal Difference
	With standalone: No NgModule injector hierarchy, Uses environment injector, Direct dependency graph
	Improves performance and tree-shaking

🔹 Q2. What is signal in Angular? Difference between signal and observable? What is dependency injection tree?
=============================================================================================================
1️⃣ A signal is a reactive value container that automatically tracks dependencies and updates UI when the value changes.

	🔹 Simple Example
					import { signal } from '@angular/core';

					count = signal(0);

					increment() {
					  this.count.update(v => v + 1);
					}

		Template:
				<p>{{ count() }}</p>
				<button (click)="increment()">+</button>

		Important: ✔ You read signal as a function → count(), UI auto-updates when value changes, No manual subscription

	🧠 Why Signals Were Introduced?
		Angular previously relied on: ✔ Zone.js, Change detection tree, Observables
		Signals provide: ✔ Fine-grained reactivity, Better performance, Less change detection overhead, No Zone.js dependency (optional)

	🧠 Signal Core APIs
		🔹 Writable Signal: count = signal(0);
		🔹 Computed Signal: double = computed(() => this.count() * 2); => Auto recalculates when dependency changes.
		🔹 Effect: effect(() => { console.log(this.count()); }); => Runs whenever signal changes.

	🚀 2️⃣ Signal vs Observable
		🔹 Observable (RxJS)
			Observables are: ✔ Stream of asynchronous data, Can emit multiple values over time, Push-based, Require subscription
			Example: this.http.get('/api').subscribe(...)

		🔹 Signal: ✔ Synchronous reactive value, Fine-grained tracking, No subscription required, Pull-based

	📊 Signal vs Observable Comparison
	| Feature         | Signal          | Observable            |
	| --------------- | --------------- | --------------------- |
	| Nature          | State container | Async stream          |
	| Subscription    | Not required    | Required              |
	| Sync/Async      | Sync            | Mostly async          |
	| Multiple values | Yes             | Yes                   |
	| Operators       | Limited         | Rich RxJS operators   |
	| Best for        | UI state        | HTTP, events, streams |

	🧠 When to Use What?
		✔ Use Signals for UI state
		✔ Use Observables for HTTP / WebSocket / streaming
		✔ Combine both if needed
			Example: toSignal(this.http.get(...))
		
2️⃣ What is Dependency Injection (DI) Tree?
	Angular uses hierarchical Dependency Injection.
		The DI Tree mirrors:
					Application
					   ↓
					Module / Standalone
					   ↓
					Component
					   ↓
					Child Component
			Each level has its own injector.

	🧠 How DI Tree Works
		When Angular resolves a dependency:
			1️⃣ Check current component injector
			2️⃣ If not found → go to parent injector
			3️⃣ Continue upward
			4️⃣ Finally → root injector
			If not found → error.

			🔹 Example
				@Injectable({ providedIn: 'root' })
				export class AuthService {}

				This means: ✔ Single instance across app, Stored in root injector

		🔹 Component-Level Provider
				@Component({
				  providers: [LocalService]
				})
			
			Now: ✔ New instance for that component subtree, Not shared globally

	🧠 Types of Injectors
		✔ Root Injector, ✔ Module Injector, ✔ Element Injector, ✔ EnvironmentInjector (Standalone era)

	🏢 Why DI Tree Is Important?
	✔ Service scoping
	✔ Lazy loaded module isolation
	✔ Memory optimization
	✔ Feature encapsulation

	Example: Lazy-loaded module gets its own injector.

	into: Optimized JavaScript instructions

	5️⃣ Generate Ivy instructions

🧠 2️⃣ What Are Structural Directives?
	Structural directives: Change the structure of the DOM by adding/removing elements. They use * syntax.
	Built-in Structural Directives
		| Directive | Purpose               |
		| --------- | --------------------- |
		| *ngIf     | Conditional rendering |
		| *ngFor    | Loop rendering        |
		| *ngSwitch | Switch rendering      |

🧠 How *ngIf Works Internally
	When you write: <div *ngIf="isVisible">Hello</div>
	Angular converts it to:
			<ng-template [ngIf]="isVisible">
			  <div>Hello</div>
			</ng-template>

	So * is syntactic sugar.

🧠 Structural vs Attribute Directives
	| Type       | What it does               |
	| ---------- | -------------------------- |
	| Structural | Add/remove DOM             |
	| Attribute  | Modify appearance/behavior |

	Example of attribute directive: <div appHighlight></div>
	
🚀 3️⃣ How to Create Custom Attribute Directive
	🔹 Example: Highlight Directive
			import { Directive, ElementRef, HostListener } from '@angular/core';

			@Directive({
			  selector: '[appHighlight]'
			})
			export class HighlightDirective {

			  constructor(private el: ElementRef) {}

			  @HostListener('mouseenter')
			  onMouseEnter() {
				this.el.nativeElement.style.backgroundColor = 'yellow';
			  }

			  @HostListener('mouseleave')
			  onMouseLeave() {
				this.el.nativeElement.style.backgroundColor = null;
			  }
			}

		Usage: <p appHighlight>Hover me</p>

🚀 4️⃣ How to Create Custom Structural Directive
		This is more advanced.
		You need: ✔ TemplateRef, ViewContainerRef
		🔹 Example: Custom *appIf Directive
							import {Directive, Input, TemplateRef, ViewContainerRef } from '@angular/core';

							@Directive({
							  selector: '[appIf]'
							})
							export class AppIfDirective {

							  constructor( private templateRef: TemplateRef<any>, private viewContainer: ViewContainerRef ) {}

							  @Input() set appIf(condition: boolean) {

								if (condition) {
								  this.viewContainer.createEmbeddedView(this.templateRef);
								} else {
								  this.viewContainer.clear();
								}
							  }
							}
			
			Usage: <div *appIf="isVisible">Custom If</div>
			
🧠 How Structural Directive Works Internally
	TemplateRef → represents template content
	ViewContainerRef → controls insertion/removal
	createEmbeddedView() → adds DOM
	clear() → removes DOM

🏢 Enterprise Use Cases for Custom Directives
	✔ Role-based UI display, Feature flag rendering, Permission-based rendering, Lazy UI rendering, DOM performance optimization

		4️⃣ Bootstrapped lazily

🏗 2️⃣ What is Micro-Frontend?
	Micro-Frontend means splitting frontend into independent deployable applications.
	Similar to microservices, but for UI. Instead of one huge Angular app you have:

				Shell App
				   ↓
				Product App
				Cart App
				Profile App

🚀 3️⃣ Micro-Frontend with Angular (Main Approaches) There are 3 main approaches:
	🔹 1️⃣ Module Federation (Recommended): Uses Webpack Module Federation.

		Architecture:
						Shell (Host)
						   ↓
						Loads remoteEntry.js
						   ↓
						Loads Remote Module
	
	🔹 Example (Shell)

					  remoteEntry: 'http://localhost:4201/remoteEntry.js',

				}

			Each remote app deploys independently.

	🔹 2️⃣ Angular Elements-Based Micro-Frontend
		Each app exports web component. Shell loads them dynamically:
			<script src="products.js"></script>
			<product-widget></product-widget>
		
		Good for: ✔ Framework-agnostic integration, Cross-framework usage
		But: ❌ Bundle duplication, Larger size

	🔹 3️⃣ iFrame-Based (Old School)
		Isolated but: ❌ Poor UX, No shared routing, Hard communication

🧠 Module Federation vs Angular Elements

	| Framework Agnostic         | ❌ No              | ✅ Yes            |
	| Performance                | Better            | Heavier          |
	| Recommended for Enterprise | ✅ Yes             | Depends          |

🏢 Enterprise Micro-Frontend Strategy
	Large org example:
				Shell App
				 ├── Auth MFE
				 ├── Product MFE
				 ├── Cart MFE
				 ├── Admin MFE

		Each team: ✔ Owns one MFE, Deploys independently, Shares UI library

🧠 Challenges in Micro-Frontend
	✔ Shared dependencies version conflict, Global state management, Routing sync, Auth sharing, Performance

🔥 Best Practices
	✔ Use Module Federation, Share Angular core as singleton, Centralize auth in shell, Use shared component library
	✔ Define clear contract between MFEs, Version control remoteEntry

	Modern alternatives: ✔ Jest, Cypress (E2E)

🏗 Basic Component Test Structure
	When you generate a component: ng generate component login

🚀 Basic Unit Test Example
	
	🔹 Component:
				@Component({
				  selector: 'app-counter',
				  template: `
					<p>{{ count }}</p>
					<button (click)="increment()">+</button>

				export class CounterComponent {

	🔹 Test:
			import { TestBed } from '@angular/core/testing';
			import { CounterComponent } from './counter.component';

			describe('CounterComponent', () => {

			  let component: CounterComponent;

			  beforeEach(() => {
				TestBed.configureTestingModule({
				  declarations: [CounterComponent]
				});

				const fixture = TestBed.createComponent(CounterComponent);
				component = fixture.componentInstance;
			  });

			  it('should increment count', () => {
				component.increment();
				expect(component.count).toBe(1);
			  });

			});
			
🧠 Testing Template Interaction
		it('should increment when button clicked', () => {
		  const fixture = TestBed.createComponent(CounterComponent);
		  fixture.detectChanges();

		  const button = fixture.nativeElement.querySelector('button');
		  button.click();

		  fixture.detectChanges();

		  const text = fixture.nativeElement.querySelector('p').textContent;
		  expect(text).toContain('1');
		});

	Here we test: ✔ Component logic, Template rendering, DOM interaction



🧠 2️⃣ Unit Test vs Integration Test
	🔹 Unit Test Tests: One component, One service, One function
		Isolated from dependencies (mocked).
		Example: spyOn(service, 'getData').and.returnValue(of(mockData));
		Characteristics:
				✔ Fast, No real HTTP calls, Uses mocks/stubs, Tests business logic

	🔹 Integration Test Tests: Multiple components together, Component + Service, Component + Routing, Component + HTTP
		Dependencies may be real or partially mocked.
		Example:
				TestBed.configureTestingModule({
				  imports: [HttpClientTestingModule]
				});
		
		Integration test ensures: ✔ Parts work together, Template + logic integration, Service integration

🔥 Comparison Table
	| Feature      | Unit Test        | Integration Test    |
	| ------------ | ---------------- | ------------------- |
	| Scope        | Single unit      | Multiple units      |
	| Speed        | Fast             | Medium              |
	| Dependencies | Mocked           | Real/Partial        |
	| Focus        | Logic            | Interaction         |
	| Example      | Component method | Component + Service |

3️⃣ Testing HTTP Calls
	Angular provides: HttpClientTestingModule, HttpTestingController
	
	🔹 Example:
				import { HttpClientTestingModule, HttpTestingController } from '@angular/common/http/testing';

				it('should fetch users', () => {

				  const httpMock = TestBed.inject(HttpTestingController);
				  const service = TestBed.inject(UserService);

				  service.getUsers().subscribe(data => {
					expect(data.length).toBe(2);
				  });

				  const req = httpMock.expectOne('/api/users');
				  req.flush([{ id: 1 }, { id: 2 }]);

				  httpMock.verify();
				});

			This avoids real HTTP calls.

🧠 4️⃣ Shallow vs Deep Testing
	🔹 Shallow Testing: Mock child components, Faster, schemas: [NO_ERRORS_SCHEMA]
	🔹 Deep Testing: Include child components, Slower but realistic.

🏢 Enterprise Best Practices
	✔ Mock external services, Test business logic separately, Avoid testing Angular internals
	✔ Use data-testid attributes, Keep tests independent, Use Jest for faster execution, Use Cypress for E2E

🧠 Testing Strategy Pyramid: Most tests should be unit tests.
        E2E (Few)
    Integration (Some)
	Unit Tests (Many)

🔹 Q2. How to mock services? How to test async code?
=============================================================================================================
🧠 1️⃣ How to Mock Services in Angular?
	When unit testing a component, you should Mock dependencies to isolate the component.
	Angular uses Dependency Injection, so mocking is easy.

🏗 Example Scenario
	🔹 Real Service:
					@Injectable()
					export class UserService {
					  getUsers() {
						return this.http.get('/api/users');
					  }
					}
	🔹 Component Using Service:
					@Component({
					  template: `<p *ngFor="let u of users">{{ u.name }}</p>`
					})
					export class UserComponent {

					  users: any[] = [];

					  constructor(private userService: UserService) {}

					  ngOnInit() {
						this.userService.getUsers()
						  .subscribe(data => this.users = data);
					  }
					}
					
🚀 2️⃣ Mocking Strategy #1: Use Jasmine Spy
		Most common method.
						describe('UserComponent', () => {
						  let mockService: any;

						  beforeEach(() => {
							mockService = jasmine.createSpyObj('UserService', ['getUsers']);
							TestBed.configureTestingModule({
							  declarations: [UserComponent],
							  providers: [
								{ provide: UserService, useValue: mockService }
							  ]
							});
						  });
						});
						
	🔹 Return Mock Data:
						it('should load users', () => {

						  mockService.getUsers.and.returnValue(of([
							{ name: 'John' },
							{ name: 'Jane' }
						  ]));

						  const fixture = TestBed.createComponent(UserComponent);
						  fixture.detectChanges();

						  expect(fixture.componentInstance.users.length).toBe(2);
						});

			Here:  Real HTTP call avoided, Service mocked, Component tested in isolation

🚀 3️⃣ Mocking Strategy #2: Use Fake Class
						class MockUserService {
						  getUsers() {
							return of([{ name: 'Mock User' }]);
						  }
						}

					Provide:

						providers: [
						  { provide: UserService, useClass: MockUserService }
						]

			Good for reusable mocks.

🚀 4️⃣ Mocking Strategy #3: HttpClientTestingModule
		For testing real service: imports: [HttpClientTestingModule]
		Use: HttpTestingController
	
🧠 5️⃣ How to Test Async Code? Async code includes: Observables, Promises, setTimeout, HTTP calls
	Angular provides 3 main approaches:
	
	🔹 1️⃣ done() (Basic Way): Used for Promise/Observable completion.
	
			it('should resolve promise', (done) => {
				  service.getData().subscribe(data => {
					expect(data).toBe('value');
					done();
				  });
			});
		Not ideal for complex scenarios.

	🔹 2️⃣ async() / waitForAsync():
			Used when template has async operations.
				it('should load async data', waitForAsync(() => {
				  fixture.detectChanges();
				  fixture.whenStable().then(() => {
					expect(component.users.length).toBe(2);
				  });

				}));

	🔹 3️⃣ fakeAsync() + tick() (Most Powerful) Best for timers and Observables.

					import { fakeAsync, tick } from '@angular/core/testing';

					it('should handle async operation', fakeAsync(() => {

					  let result: any;
					  service.getUsers().subscribe(data => {
						result = data;
					  });
					  tick(); // simulate time passing
					  expect(result.length).toBe(2);
					}));
					
🧠 fakeAsync Internals
	fakeAsync: ✔ Runs test in fake zone, Controls timers, Controls microtasks, Allows synchronous testing of async code

🔥 Testing setTimeout Example
				it('should update after timeout', fakeAsync(() => {
				  component.delayedUpdate();
				  tick(1000);
				  expect(component.value).toBe('updated');

				}));

🧠 Testing Observables
	For immediate Observables: of(data)
					No async handling needed.
	
	For delayed: of(data).pipe(delay(1000))
	Use: tick(1000)
	
🧠 Testing Promises
				it('should resolve promise', fakeAsync(() => {
				  let value;
				  service.getPromise().then(v => value = v);
				  tick(); // resolve microtask
				  expect(value).toBe('resolved');
				}));

🧠 1️⃣ TestBed is Angular’s primary testing utility. It allows you to Create a testing module environment that simulates an Angular module.
		Think of it as: Mini Angular runtime for tests

🏗 Why TestBed is Needed?
	Angular components depend on: ✔ DI (Dependency Injection), Templates, Services, Pipes, Directives, You cannot instantiate a component using new Component() directly.
	
	TestBed: ✔ Creates testing module, Resolves dependencies, Compiles template, Provides DI container


🧠 How TestBed Works Internally
	When you write:
					TestBed.configureTestingModule({
					  declarations: [MyComponent],
					  providers: [MyService]
					});

	Internally:
		1️⃣ Create testing NgModule
		2️⃣ Register providers
		3️⃣ Compile templates
		4️⃣ Create injector
		5️⃣ Instantiate component

🚀 Basic TestBed Example
			describe('MyComponent', () => {

			  beforeEach(() => {
				TestBed.configureTestingModule({
				  declarations: [MyComponent]
				});
			  });

			  it('should create component', () => {
				const fixture = TestBed.createComponent(MyComponent);
				const component = fixture.componentInstance;

				expect(component).toBeTruthy();
			  });

			});
			
🧠 Key TestBed APIs
| API                    | Purpose                   |
| ---------------------- | ------------------------- |
| configureTestingModule | Setup test module         |
| createComponent        | Create component instance |
| inject                 | Get service instance      |
| overrideProvider       | Replace dependency        |
| compileComponents      | Compile templates         |

2️⃣ How to Test HTTP Calls?
	When testing HTTP: Do NOT make real API calls, Mock them using Angular utilities
	Angular provides: HttpClientTestingModule, HttpTestingController

	🚀 Step-by-Step: Testing HTTP Call in Service
	🔹 Service Example
					@Injectable()
					export class UserService {

					  constructor(private http: HttpClient) {}

					  getUsers() {
						return this.http.get<any[]>('/api/users');
					  }
					}
	🔹 Test Setup
				import {
				  HttpClientTestingModule,
				  HttpTestingController
				} from '@angular/common/http/testing';

				describe('UserService', () => {

				  let service: UserService;
				  let httpMock: HttpTestingController;

				  beforeEach(() => {
					TestBed.configureTestingModule({
					  imports: [HttpClientTestingModule],
					  providers: [UserService]
					});

					service = TestBed.inject(UserService);
					httpMock = TestBed.inject(HttpTestingController);
				  });
	🔹 Test the HTTP Call
				it('should fetch users', () => {

				  const mockUsers = [
					{ id: 1, name: 'John' },
					{ id: 2, name: 'Jane' }
				  ];

				  service.getUsers().subscribe(users => {
					expect(users.length).toBe(2);
					expect(users).toEqual(mockUsers);
				  });

				  const req = httpMock.expectOne('/api/users');

				  expect(req.request.method).toBe('GET');

				  req.flush(mockUsers);

				  httpMock.verify();

				});
🧠 What Happens Internally?
	1️⃣ HttpClient call intercepted
	2️⃣ HttpTestingController captures request
	3️⃣ You assert request details
	4️⃣ You manually return mock response using flush()
	5️⃣ No real network call happens
	
🧠Important HttpTestingController Methods
| Method      | Purpose                    |
| ----------- | -------------------------- |
| expectOne() | Expect single request      |
| match()     | Match multiple requests    |
| verify()    | Ensure no pending requests |
| flush()     | Return mock response       |
| error()     | Simulate error             |

🚀 Testing Error Scenario
						it('should handle error', () => {

						  service.getUsers().subscribe({
							error: error => {
							  expect(error.status).toBe(500);
							}
						  });

						  const req = httpMock.expectOne('/api/users');

						  req.flush('Error', { status: 500, statusText: 'Server Error' });

						});
🧠 Why Use HttpClientTestingModule?
✔ Fast, Deterministic, No real backend dependency, Full request control

🔹 Q4. What is Jasmine and Karma? What is Cypress?
=============================================================================================================
🔹 What is Jasmine?
	Jasmine is a JavaScript testing framework used to write unit tests. It provides:
	describe() → group of test cases
	it() → individual test case
	expect() → assertion
	beforeEach() → setup logic

Example (Angular component test)
				describe('LoginComponent', () => {
				  let component: LoginComponent;

				  beforeEach(() => {
					component = new LoginComponent();
				  });

				  it('should validate username', () => {
					component.username = 'munna';
					expect(component.isValid()).toBeTrue();
				  });
				});

Key Features: 
	BDD style (Behavior Driven Development)
	Assertions & matchers
	Spies (mock functions)
	Async testing support

	👉 Jasmine only writes tests. It does NOT run them in browser.

🔹 What is Karma?
	Karma is a test runner. It:
		Launches real browsers (Chrome, Firefox)
		Executes Jasmine tests
		Shows results
		Supports headless mode (CI/CD friendly)

How they work together
	Jasmine → Writes tests
	Karma → Runs tests in browser
	Angular Default Setup

When you create Angular app: ng new my-app It comes with: Jasmine, Karma, TestBed

🔹 What is Cypress?
	Cypress is an End-to-End (E2E) testing framework. It tests: Full user flows, UI + API interaction, Real browser behavior
	Instead of testing small functions, Cypress tests the application like a real user.

Example: Cypress Test
			describe('Login Flow', () => {
			  it('should login successfully', () => {
				cy.visit('/login');
				cy.get('#username').type('munna');
				cy.get('#password').type('1234');
				cy.get('button').click();
				cy.url().should('include', '/dashboard');
			  });
			});

| CI/CD          | Yes               | Yes                   |

🔹 Q5. How to measure test coverage?
=============================================================================================================
🔹 What is Test Coverage?
	Test coverage measures: How much of your source code is executed when tests run.
	It is expressed as a percentage.

Example:
	Total lines: 100
	Lines covered by tests: 80
	Coverage: 80%
	
🔹 Types of Coverage (Interview Important)
| Type               | Meaning                     |
| ------------------ | --------------------------- |
| Line Coverage      | % of lines executed         |
| Branch Coverage    | % of if/else paths executed |
| Function Coverage  | % of functions called       |
| Statement Coverage | % of statements executed    |

👉 Branch coverage is more important than line coverage.

Example:
	if (age > 18) {
		  allow();
		} else {
		  deny();
		}
	If you only test age > 18 → Line coverage = 100%, Branch coverage = 50%
	That’s why branch coverage matters.

🔹 How to Measure Coverage in Angular (Jasmine + Karma)
	Angular uses Istanbul under the hood.
		Step 1: Run coverage command: ng test --code-coverage
		Step 2: Open report It generates: /coverage/index.html Open in browser 
			→ you’ll see: 
					% coverage summary, File-wise breakdown
					Highlighted uncovered lines (in red)
			Example Coverage Output
						Statements   : 85%
						Branches     : 70%
						Functions    : 90%
						Lines        : 84%

🔹 How to Enforce Minimum Coverage (Enterprise Setup)
	In karma.conf.js:
				coverageReporter: {
				  type: 'html',
				  check: {
					global: {
					  statements: 80,
					  branches: 70,
					  functions: 80,
					  lines: 80
					}
				  }
				}

		If coverage drops below threshold → CI fails.
		Very common in enterprise projects.


🔹 CI/CD Integration
	In enterprise projects:
		Coverage report uploaded to: SonarQube, Jenkins, GitHub Actions
		Pull request fails if coverage decreases

🔹 Q1. How to handle 1 lakh records in table?
=============================================================================================================
✅ 1. Backend Strategy (Spring Boot)
	❌ Wrong Way:  SELECT * FROM orders; Then return 100,000 records.
	✅ Correct Way → Pagination + Sorting, Use Pageable in Spring Boot
				@GetMapping("/orders")
				public Page<Order> getOrders(Pageable pageable) {
					return orderRepository.findAll(pageable);
				}

		Repository: Page<Order> findAll(Pageable pageable);
		Client Request: /orders?page=0&size=50&sort=createdDate,desc
		
		Now: Only 50 records come at a time, DB load is controlled, Memory optimized
		🔥 Use Indexing in Database
		If filtering/sorting: CREATE INDEX idx_created_date ON orders(created_date);
		Without index → pagination will still be slow.

✅ 2. Frontend Strategy (Angular)
	Even if you paginate backend, you must handle UI efficiently.
	
	🔹 Option 1: Server-side Pagination (Recommended)
		Flow:
			User changes page
				↓
			Call backend with page & size
				↓
			Render only 50 rows

			This is best for large datasets.

	🔹 Option 2: Virtual Scrolling (Very Important)
		Instead of rendering 100,000 rows, render only visible rows in viewport.
		Angular CDK Virtual Scroll
		Install: npm install @angular/cdk
		Example:
				<cdk-virtual-scroll-viewport itemSize="50" class="viewport">
				  <div *cdkVirtualFor="let item of items"> {{item.name}} </div>
				</cdk-virtual-scroll-viewport>

		👉 Only visible rows are rendered in DOM
		👉 Huge performance improvement

🔥 Difference: Pagination vs Virtual Scroll
| Feature     | Pagination  | Virtual Scroll         |
| ----------- | ----------- | ---------------------- |
| Data loaded | Page-wise   | Can be full or partial |
| DOM size    | Small       | Small                  |
| Best for    | APIs        | Large UI list          |
| UX          | Page change | Smooth scroll          |

🔥 3. Advanced: Infinite Scrolling
	Like: Instagram, LinkedIn, Amazon, Load next 50 when user scrolls near bottom.
		Scroll near bottom
		   ↓
		Call API with next page
		   ↓
		Append to list
		
🔥 4. For 1 Lakh+ Records (Enterprise Architecture)
	For very large data: Use Filtering before loading, Don’t show all records.
	Example: Date range filter, Status filter, Search first
	
	✔ Use Cursor-based Pagination (Better than Offset)
		Instead of: ?page=2000
		Use: ?lastId=938472
		Better performance in big tables.

🔥 5. Real Enterprise Architecture (Angular + Spring Boot)
						User filters data
							 ↓
						Backend API (Pageable + Indexed Query)
							 ↓
						Returns 50 records
							 ↓
						Angular renders with virtual scroll
							 ↓
						Lazy load next batch

🚫 What NOT to Do
	❌ Don’t fetch 1 lakh records at once, Don’t use client-side pagination for big data
	❌ Don’t sort in frontend, Don’t remove DB indexes

🔹 Q2. How to fix memory leak issue?
=============================================================================================================
🔹 What is Memory Leak?
	A memory leak happens when: Objects are no longer needed but are still referenced, so GC cannot remove them.
	Over time: Memory increases 📈
	Performance drops, Eventually → OutOfMemoryError / browser crash

🟢 PART 1 — Fix Memory Leak in Angular
	Most common reason in Angular: Unsubscribed Observables
	❌ Problem Example:
					ngOnInit() {
					  this.userService.getUsers().subscribe(data => {
						this.users = data;
					  });
					}

			If component is destroyed, subscription remains active → memory leak.

✅ Fix 1 — Use takeUntil Pattern (Best Practice)
							private destroy$ = new Subject<void>();

							ngOnInit() {
							  this.userService.getUsers()
								.pipe(takeUntil(this.destroy$))
								.subscribe(data => this.users = data);
							}

							ngOnDestroy() {
							  this.destroy$.next();
							  this.destroy$.complete();
							}
							
✅ Fix 2 — Use Async Pipe (Even Better)
						<div *ngFor="let user of users$ | async">
						  {{ user.name }}
						</div>

				Async pipe auto-unsubscribes.

🔥 Angular Memory Leak Checklist
	Unsubscribed Observables, setInterval not cleared, Event listeners not removed, Large arrays stored in services, Detached DOM nodes

🔎 How to Detect Angular Memory Leak
	Use: Chrome DevTools → Memory tab, Take heap snapshot, Navigate multiple times, Compare heap growth

🔹 Q4. API called multiple times — how to fix?
=============================================================================================================
1️⃣ If Angular Is Calling API Multiple Times
🔍 Case 1 — Multiple Subscriptions
		❌ Problem
				ngOnInit() {
				  this.userService.getUsers().subscribe();
				  this.userService.getUsers().subscribe();
				}
			Each subscribe = new API call.

		✅ Fix — Share the Observable
					users$ = this.userService.getUsers().pipe(shareReplay(1));
				Now: One HTTP call, Multiple subscribers reuse response

🔍 Case 2 — Calling API in Template
		❌ Problem: <div *ngFor="let user of getUsers()">
			Every change detection → API called again.

		✅ Fix: call API once in ngOnInit() and store in variable.

🔍 Case 3 — Button Double Click
		❌ Problem: User clicks submit multiple times.

		✅ Fix: Disable button: <button [disabled]="isLoading">Submit</button> Or use: exhaustMap()
		
🔍 Case 4 — Route Re-render
		If component reloads due to route change → API fires again.
		Fix: Use route reuse strategy , Cache result in service

🔍 Case 5 — Interceptor Loop: If token refresh interceptor calls same API again without condition → infinite loop.
	Fix: Add URL check:
				if (req.url.includes('/refresh')) {
				  return next.handle(req);
				}
				
🔥 Angular Best Practices to Avoid Multiple Calls
	Use shareReplay(1), Use async pipe, Avoid function calls in template, Use take(1) for one-time API, Use debounceTime() for search

🔹 Q5. How to debug change detection issue?
=============================================================================================================
🔹 What is Change Detection?
	Angular checks: Has any data changed? If yes → update DOM. By default, Angular uses: Default Change Detection Strategy
	Based on Zone.js Runs on: Click, HTTP response, setTimeout, Promise, Input change

🚨 Common Change Detection Problems
	UI not updating, Too many re-renders, API called multiple times, Performance slow, ExpressionChangedAfterItHasBeenCheckedError

🔎 Step-by-Step Debugging Strategy
	🟢 1️⃣ Detect Excessive Change Detection (Performance Issue)
		Enable Angular DevTools
		Use: Angular DevTools
		It shows: Change detection cycles, Component re-render count, Performance flame graph
		What to Look For: Component re-rendering too many times, Large tree re-checking frequently

	🟢 2️⃣ Console Debug Trick
		Inside component:
			ngDoCheck() {
			  console.log('Change detection triggered');
			}
		If console floods → excessive detection.

	🟢 3️⃣ Common Root Causes
			🔴 Problem 1 — Function Call in Template
					❌ Wrong: <div>{{ calculateTotal() }}</div>, Every change detection → function runs again.
			
					✅ Fix: Compute once: total = this.calculateTotal();
			
			🔴 Problem 2 — Large Object Mutation
					❌ Wrong (Mutation): this.user.name = 'Munna'; OnPush may not detect this.
					✅ Fix (Immutable Update): this.user = { ...this.user, name: 'Munna' };
			
			🔴 Problem 3 — Not Using OnPush: Default strategy checks entire tree.
					✅ Use OnPush
								@Component({
								  changeDetection: ChangeDetectionStrategy.OnPush
								})

						Now Angular checks only when: Input reference changes, Event triggered, Observable emits, Huge performance improvement.

			🔥 OnPush vs Default
			| Feature               | Default | OnPush |
			| --------------------- | ------- | ------ |
			| Checks entire tree    | Yes     | No     |
			| Performance           | Slower  | Faster |
			| Requires immutability | No      | Yes    |

	🟢 4️⃣ Manual Change Detection Debug
		Inject: constructor(private cd: ChangeDetectorRef) {}
		Trigger manually: this.cd.detectChanges();	Or: this.cd.markForCheck();
		Useful when: Updating data outside Angular zone, Using third-party libraries
	
	🟢 5️⃣ Zone Issues
		If using: setTimeout(() => {}, 0); Or third-party JS library Angular may not detect change.
		Fix: 
			constructor(private ngZone: NgZone) {}
			this.ngZone.run(() => {
			   this.value = 100;
			});
			
	🟢 6️⃣ Use trackBy (Very Important)
		Without trackBy: <div *ngFor="let item of items">, Angular re-renders entire list.
		✅ Fix
			<div *ngFor="let item of items; trackBy: trackById">
			trackById(index: number, item: any) {
			  return item.id;
			}
		Big performance boost for large tables.

🔥 Real Production Debug Flow
						UI slow?
						   ↓

🔹 Q6. How to implement role-based dynamic menu?
=============================================================================================================
🎯 Requirement different users see different menus:
| Role    | Menu                       |
| ------- | -------------------------- |
| ADMIN   | Dashboard, Users, Settings |
| USER    | Dashboard, Profile         |
| AUDITOR | Dashboard, Reports         |

Menu should: Be dynamic, Hide unauthorized routes, Secure backend APIs, Support future roles easily

🏗️ Architecture Overview
			User logs in
				 ↓
			Backend returns JWT + roles
				 ↓
			Frontend stores roles
				 ↓
			Menu built dynamically
				 ↓
			Route Guard prevents unauthorized access
			
🟢 Step 1 — Backend (Spring Boot)
	After login, return roles in JWT:
	Example JWT payload: { "sub": "munna", "roles": ["ADMIN", "USER"] }
	
	In Spring Security:
			@PreAuthorize("hasRole('ADMIN')")
			@GetMapping("/users")
			public List<User> getUsers() { }

		👉 Backend must always enforce security.
			Frontend is just for UI control.

🟢 Step 2 — Define Menu Configuration (Angular)
	Create a central config:
							export const MENU_CONFIG = [
							  {
								label: 'Dashboard',
								route: '/dashboard',
								roles: ['ADMIN', 'USER', 'AUDITOR']
							  },
							  {
								label: 'Users',
								route: '/users',
								roles: ['ADMIN']
							  },
							  {
								label: 'Reports',
								route: '/reports',
								roles: ['AUDITOR']
							  }
							];
							
🟢 Step 3 — Filter Menu Based on Role
				getMenuByRole(userRoles: string[]) {
				  return MENU_CONFIG.filter(menu => menu.roles.some(role => userRoles.includes(role)) );
				}
			Call after login.

🟢 Step 4 — Render Menu
					<ul>
					  <li *ngFor="let item of menu">
						<a [routerLink]="item.route">{{ item.label }}</a>
					  </li>
					</ul>

				Now menu is dynamic.

🟢 Step 5 — Add Route Guard (Very Important)
		Even if menu hidden, user can type URL manually.
		Create Auth Guard:
							canActivate(route: ActivatedRouteSnapshot): boolean {
							  const roles = route.data['roles'];
							  return roles.some(r => this.authService.hasRole(r));
							}

		Route config: Now route also protected.
						{
						  path: 'users',
						  component: UsersComponent,
						  canActivate: [AuthGuard],
						  data: { roles: ['ADMIN'] }
						}

		

🔥 Enterprise-Level Enhancement
	Instead of hardcoding menu in frontend, store menu in database.
	Example DB table: id label	route	role
	Backend API: GET /api/menu
	Returns menu based on logged-in user.
	Frontend just renders what backend sends.
	
	✔ Fully dynamic, No redeploy for menu change, Works for microservices

🔐 Security Reminder
	Frontend hiding menu ≠ security.
	Backend must enforce: @PreAuthorize("hasAuthority('ROLE_ADMIN')")
	Never trust frontend.

🟢 Handling Multiple Roles
	If user has: ["ADMIN", "AUDITOR"]
	Menu shows union of both roles.
	Already handled via: some()

🔥 Bonus: Hierarchical Menu (Sidebar Tree)
	Structure: Filter recursively.
				{
				  label: 'Admin',
				  roles: ['ADMIN'],
				  children: [
					 { label: 'Users', route: '/users', roles: ['ADMIN'] }
				  ]
				}

🔹 Q7. How to handle offline mode?
=============================================================================================================
🎯 What is Offline Mode?
	App should work even when internet is not available. That means: Detect offline, Show cached data, Store user actions locally, Sync when back online

🏗️ High-Level Architecture
				User Offline
					 ↓
				Use Local Storage / IndexedDB
					 ↓
				Store user actions locally
					 ↓
				Network restored
					 ↓
				Sync pending requests to backend
				
🟢 Step 1 — Detect Online / Offline (Angular)
	Use browser API: window.navigator.onLine Or listen to events:
		ngOnInit() {
		  window.addEventListener('online', () => console.log('Online'));
		  window.addEventListener('offline', () => console.log('Offline'));
		}
		
		Better: Create an OnlineStatusService.

🟢 Step 2 — Show Offline Banner: Good UX practice.
		<div *ngIf="!isOnline" class="offline-banner">
		  You are offline
		</div>

🟢 Step 3 — Cache Data Locally
	Use: LocalStorage (small data), SessionStorage, IndexedDB (large structured data) ✅ recommended
	
	🔹 Best Option → IndexedDB
		For Angular apps, use: IndexedDB Or library: NgRx Data (for state-based caching) Or simple library like idb.
		Example (basic idea):
				saveDataLocally(data) {
				  localStorage.setItem('users', JSON.stringify(data));
				}

		Load if offline:
			if (!navigator.onLine) {
			  return JSON.parse(localStorage.getItem('users'));
			}
		
🟢 Step 4 — Queue API Requests (Very Important)
	When offline: Do NOT call backend, Store request locally
	Example structure:
				{
				  "method": "POST",
				  "url": "/orders",
				  "body": {...}
				}

		Store in IndexedDB.

🟢 Step 5 — Sync When Back Online
	When online event fires:
			window.addEventListener('online', () => {
			   this.syncPendingRequests();
			});

	Process queue:
			for (let req of pendingRequests) {
			   http.post(req.url, req.body);
			}
		After success → remove from local storage.

🔥 Enterprise Pattern: Service Worker (PWA)
	Use: Angular Service Worker
	Enable PWA: ng add @angular/pwa
	It: Caches static assets, Caches API responses, Works offline automatically
	Generated config: ngsw-config.json
	Example:
				{
				  "dataGroups": [
					{
					  "name": "api-cache",
					  "urls": ["/api/**"],
					  "cacheConfig": {
						"maxSize": 100,
						"maxAge": "1d",
						"strategy": "freshness"
					  }
					}
				  ]
				}
	
🟢 Backend Considerations (Spring Boot)
	When syncing: Avoid duplicate inserts, Use idempotency keys, Handle conflict resolution
	Example: POST /orders
	Idempotency-Key: 12345
	
🔥 Conflict Handling Strategy
	If: Same record updated offline and online
	Options: Last write wins, Version-based update, Manual conflict resolution, Enterprise apps usually use versioning.

🟢 Real Production Offline Strategy
| Feature           | Approach            |
| ----------------- | ------------------- |
| Static assets     | Service Worker      |
| API GET cache     | IndexedDB           |
| API POST queue    | Local request queue |
| Sync logic        | Background sync     |
| Conflict handling | Version control     |

🚀 Example Real Scenario
	Delivery app: Delivery agent marks parcel delivered (offline), Data stored locally, Internet comes back
	All pending deliveries synced to backend

🔹 Q8. How to implement multi-language (i18n)?
=============================================================================================================
🌍 What is i18n?
	i18n = Internationalization (18 letters between i and n 😉)
	It allows: Multiple languages, Dynamic language switching
	Locale-based formatting (date, currency, number)

🟢 PART 1 — Angular i18n
	Angular supports 2 major approaches:
	| Approach              | Best For                             |
	| --------------------- | ------------------------------------ |
	| Built-in Angular i18n | Static language at build time        |
	| ngx-translate         | Dynamic runtime language switching ✅ |
	
	In enterprise apps, we usually use: ngx-translate

	✅ Option 1 — Using ngx-translate (Recommended)
		Step 1 — Install: npm install @ngx-translate/core @ngx-translate/http-loader
		Step 2 — Create Translation Files: 
					assets/i18n/en.json
									{
									  "TITLE": "Welcome",
									  "LOGIN": "Login"
									}

					assets/i18n/fr.json

									{
									  "TITLE": "Bienvenue",
									  "LOGIN": "Connexion"
									}

		Step 3 — Configure Module
						TranslateModule.forRoot({
						  loader: {
							provide: TranslateLoader,
							useFactory: httpLoaderFactory,
							deps: [HttpClient]
						  }
						});
		
		Step 4 — Use in Template
					<h1>{{ 'TITLE' | translate }}</h1>
					<button>{{ 'LOGIN' | translate }}</button>
					
		
		Step 5 — Switch Language Dynamically
							constructor(private translate: TranslateService) {}

							switchLang(lang: string) {
							  this.translate.use(lang);
							}

			Now language changes instantly — no reload.

🟢 Persist Selected Language
	localStorage.setItem('lang', lang);
	Load on app start: this.translate.use(localStorage.getItem('lang') || 'en');
	
🟢 PART 2 — Angular Built-in i18n (Build Time)
	Angular official tool: Angular i18n	
	Steps: 
		Mark text with i18n
		Extract messages
		Create translation file (XLF)
		Build per language

	Example: <h1 i18n="@@welcomeTitle">Welcome</h1>
	Build: ng build --localize
	⚠️ Limitation: One build per language, Not dynamic switching

🟢 PART 3 — Spring Boot i18n (Backend)
	For validation messages or API responses.
	Step 1 — Create Message Files
			messages.properties
			messages_fr.properties

	
	Step 2 — Configure MessageSource
							@Bean
							public MessageSource messageSource() {
								ReloadableResourceBundleMessageSource messageSource = new ReloadableResourceBundleMessageSource();
								messageSource.setBasename("classpath:messages");
								messageSource.setDefaultEncoding("UTF-8");
								return messageSource;
							}
							
	Step 3 — Use in Controller
					@Autowired
					private MessageSource messageSource;

					public String getMessage(Locale locale) {
					   return messageSource.getMessage("welcome", null, locale);
					}

				Spring automatically detects Accept-Language header.

🌍 Locale Formatting (Date, Currency)
	In Angular:
				{{ amount | currency:'INR' }}
				{{ today | date }}

		Or set locale: registerLocaleData(localeFr);
		
🔥 Enterprise-Level Strategy
	Best practice:
			Frontend:
			- Use ngx-translate for UI
			- Store language in user profile

			Backend:
			- Use Accept-Language header
			- Return localized messages

	Flow:
		User selects French
			 ↓
		Frontend sends Accept-Language: fr
			 ↓
		Backend returns localized response
		
🔐 Multi-Tenant SaaS Case
	Sometimes: Language stored per tenant, Loaded dynamically from DB, Translations editable in admin panel
	Advanced approach: Store translations in database, Cache them in Redis

app/

			core/
			 ├── services/
			 ├── guards/

		shared/

		 ├── pipes/

		❌ Bad Structure (Type-based)
		components/
		services/
		models/
		Hard to scale.

		✅ Good Structure (Feature-based)
		features/
		 ├── users/
		 │    ├── components/
		 │    ├── pages/
		 │    ├── services/
		 │    ├── users-routing.module.ts
		 │    └── users.module.ts

		Each feature is self-contained.

		{
		  path: 'users',
		  loadChildren: () => import('./features/users/users.module').then(m => m.UsersModule)
		}

	Loads only when needed.
	Improves: Initial load time, Performance

Smart Component (Container)
   ↓
Dumb Components (Presentation)

Smart: API calls, Business logic
Dumb: Input/Output, UI only
OnPush strategy This improves reusability and testing.

🟢 6️⃣ State Management Strategy
	For small app: Services + RxJS
	For large enterprise: NgRx / Signal-based store
	Example: NgRx
	Structure:
			features/users/store/
			 ├── users.actions.ts
			 ├── users.reducer.ts
			 ├── users.effects.ts
			 └── users.selectors.ts
			 
🟢 7️⃣ Environment-Based Config
		environments/
		 ├── environment.ts
		 ├── environment.prod.ts

		Never hardcode API URLs.

🟢 8️⃣ Layout Module
	For: Sidebar, Header, Footer
	layout/
	 ├── sidebar/
	 ├── header/
	 └── layout.module.ts
	 
🟢 9️⃣ Folder Naming Best Practices: kebab-case for folders
	feature-name.module.ts
	feature-name-routing.module.ts
	feature-name.service.ts
	Consistency is key.

🔥 Large Enterprise Example (Realistic Structure)
		Each feature: Lazy loaded, Own routing, Own components, Own services, Own state

🔐 Micro-Frontend Ready Structure
	If planning future scaling: Each feature module should be isolated, Avoid tight coupling, Clear API contracts
	Use Module Federation (if needed later)

🔹 Q2. What is differential loading?
=============================================================================================================
🎯 Differential loading means Angular builds two different JavaScript bundles — one for modern browsers and one for legacy browsers.
	So: Modern browsers → get modern JS (smaller, faster)
		Old browsers → get compatible ES5 code

🧠 Why Was It Needed?
	Older browsers (like IE11) did not support modern JavaScript (ES6+).
	Example:
		Modern JS: const greet = (name) => `Hello ${name}`;
				   Older browsers can’t understand arrow functions (=>).
		
		So Angular would: 
			Compile one version → ES2015 (modern)
			Compile another version → ES5 (legacy)

🏗️ Build Output Example (Angular 8–11)
	After: ng build --prod
	You would see:
		main-es2015.js   → Modern browsers
		main-es5.js      → Legacy browsers
		polyfills-es2015.js
		polyfills-es5.js

		And in index.html: 
			<script type="module" src="main-es2015.js"></script>
			<script nomodule src="main-es5.js"></script>
			
🔍 How Browser Chooses?
	Modern browser supports: type="module"
	So it loads: main-es2015.js
	Older browser doesn’t understand modules → ignores that script → loads: nomodule Very clever optimization.

📦 Benefits of Differential Loading
	Benefit
	Smaller bundles	Modern JS is smaller
	Faster parsing	ES2015 loads faster
	Better performance	Less polyfills
	Backward compatibility	Old browsers still supported
⚠️ Important: Is It Still Used?

Starting from: Angular 12+ Angular dropped support for IE11.
	So: 👉 Differential loading is no longer needed in modern Angular versions.
		Now Angular builds only modern bundles.

🏗️ Modern Angular Build (Angular 15+)
	Angular now uses: Modern ES2020+
	esbuild (faster bundling)
	No ES5 bundles, No dual builds, Much simpler and faster build.

🔥 Related Concept: Polyfills
	Even with differential loading, Angular included:
	Core-js
	Zone.js
	Polyfills add missing browser features.
	Configured in:
		polyfills.ts

🔹 Q1. Why Angular over React? When not to use Angular?
=============================================================================================================
🆚 Angular vs React (High-Level View)
| Feature                | Angular                  | React                   |
| ---------------------- | ------------------------ | ----------------------- |
| Type                   | Full Framework           | UI Library              |
| Language               | TypeScript (default)     | JavaScript / TypeScript |
| Structure              | Opinionated              | Flexible                |
| Built-in Features      | Routing, DI, Forms, HTTP | Need external libraries |
| Learning Curve         | Steeper                  | Easier initially        |
| Enterprise Suitability | Very Strong              | Depends on architecture |

🟢 Why Choose Angular?
1️⃣ Full-Fledged Framework
	Angular provides everything out of the box: Routing, Dependency Injection, HTTP Client, Form handling, Interceptors, Guards, CLI tooling
	Official framework: Angular
	
	React is maintained by: React , React focuses mainly on UI layer.

2️⃣ Strong Architecture for Enterprise
	Angular enforces: Modular structure, Feature-based design, Dependency injection, Clear project structure
	This is useful when: 20+ developers working, Large monolithic app, Banking / ERP systems

3️⃣ TypeScript by Default
	Angular is written in TypeScript.
	Benefits: Better IntelliSense, Compile-time safety, Maintainable large codebase
	React can use TypeScript, but not mandatory.

4️⃣ Dependency Injection (DI)
	Angular has built-in powerful DI system.
	Example: constructor(private userService: UserService) {}
	Better for: Testability, Maintainability, Service management

5️⃣ Two-Way Binding: <input [(ngModel)]="name">
	Simplifies form-heavy apps.

6️⃣ CLI & Build Tooling
	Angular CLI: ng generate component, ng generate module, ng build Standardized structure.

🟢 When Angular Is a Better Choice
✔ Large enterprise applications
✔ Long-term maintainable systems
✔ Strong backend integration (Spring Boot)
✔ Teams needing strict structure
✔ Complex form-heavy applications

🔴 When NOT to Use Angular

	❌ 1️⃣ Small Landing Pages: Angular bundle size is bigger.
		If you need: Small marketing page, SEO-first static site, React / Next.js may be better.
	
	❌ 2️⃣ Highly Custom UI Animation Apps: React ecosystem has more UI/animation flexibility.

	❌ 3️⃣ Very Small Team or Prototype: Angular has: Steeper learning curve, More boilerplate, For quick MVP → React might be faster.
	
	❌ 4️⃣ Micro-Frontend Ecosystem Already in React: If company stack already React-based → don’t mix unnecessarily.

🔥 Real Enterprise Perspective
	Angular is strong when:
		Structure > Flexibility
		Stability > Experimentation
		Enterprise Governance > Rapid Prototyping

	React is strong when: 
		Flexibility > Opinionated structure
		Rapid UI development
		Heavy ecosystem usage

🔹 Q2. SPA vs MPA?
=============================================================================================================
🧠 What is SPA?
🔹 SPA = Single Page Application
	A web app that loads one HTML page and dynamically updates content using JavaScript.
	Example frameworks: Angular, React

	🔹 How SPA Works: No full page reload, Only data (JSON) is fetched from backend.

🧠 What is MPA?
🔹 MPA = Multi Page Application
	A traditional web app where: Every click → new HTML page from server, Full page reload happens
	Examples: Traditional Spring MVC apps, PHP websites, Server-rendered JSP apps

📊 SPA vs MPA Comparison
| Feature            | SPA             | MPA               |
| ------------------ | --------------- | ----------------- |
| Page Reload        | ❌ No            | ✅ Yes             |
| Speed (after load) | Fast            | Slower            |
| Initial Load       | Heavier         | Lighter           |
| SEO                | Needs SSR       | Better by default |
| Backend            | Mostly REST API | Server-rendered   |
| UX                 | App-like        | Traditional       |
| Complexity         | Higher          | Simpler           |
