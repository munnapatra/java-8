🔥 1️⃣ Core JPA & Hibernate Fundamentals (Advanced Level)

🔹 Q1. What is the difference between JPA and Hibernate?
======================================================================================================
📘 What is JPA? Oracle Corporation introduced JPA (Java Persistence API) as part of Java EE.
	It is a specification 
		Defines interfaces like: EntityManager, EntityTransaction, PersistenceContext
		Defines annotations: @Entity, @Id, @OneToMany
		Does NOT provide implementation
		Needs a provider (like Hibernate)
		Think of JPA like an interface in Java.
	
🔥 What is Hibernate?
	Hibernate ORM is: A full-featured ORM framework, Implements JPA specification, Provides additional features beyond JPA
	Can work: With JPA, Without JPA (native Hibernate API)
	Think of Hibernate as a concrete class implementing JPA.

🔍 Deep Comparison (Senior Level)
| Feature             | JPA                                         | Hibernate                            |
| ------------------- | ------------------------------------------- | ------------------------------------ |
| Type                | Specification                               | Implementation                       |
| Package             | `javax.persistence` / `jakarta.persistence` | `org.hibernate`                      |
| Who defines it?     | Java standard                               | Red Hat                              |
| Can run standalone? | ❌ No                              	        | ✅ Yes                       	       |
| Caching support     | Defines concept                             | Implements 1st & 2nd level cache     |
| Query language      | JPQL                                        | HQL (extends JPQL)                   |
| Extra features      | Limited                                     | Filters, Interceptors, Multi-tenancy |

🚀 Example Difference in Code
	Using JPA (portable)
		@PersistenceContext
		private EntityManager entityManager;
		entityManager.persist(user);

	Using Hibernate native API
		Session session = sessionFactory.openSession();
		session.save(user);

	If tomorrow you replace Hibernate with: EclipseLink, Apache OpenJPA, Your JPA-based code will still work (because it depends on specification).

🔥 Advanced Interview-Level Points
	1️⃣ Hibernate existed before JPA: 
		Hibernate was created first. Later JPA specification was designed based on Hibernate concepts.
	2️⃣ Hibernate Superset of JPA: 
		All JPA features are supported by Hibernate.
		But Hibernate has extra features like: StatelessSession, NaturalId, Filters, Multi-tenancy, Custom types

	3️⃣ EntityManager vs Session
		| JPA                | Hibernate     |
		| ------------------ | ------------- |
		| EntityManager      | Session       |
		| PersistenceContext | Session cache |

	4️⃣ In Spring Boot
		When you use: spring-boot-starter-data-jpa
		Under the hood: JPA API is used, Hibernate is default implementation

🔹 Q2. What is the role of EntityManager? Difference between EntityManager and Session?
======================================================================================================
🔷 Role of EntityManager
	EntityManager is the core JPA interface responsible for managing:
		Entity lifecycle (persist, merge, remove, detach), Persistence context, Transactions (in non-Spring environments), JPQL execution
		First-level cache management, It is defined by the JPA specification (now under jakarta.persistence).

🎯 In Simple Words
	EntityManager manages the persistence context and acts as a bridge between your Java objects and the database.
	It ensures: Objects are tracked, Changes are automatically detected (dirty checking), SQL is generated at flush time

🔁 Entity Lifecycle Management
	EntityManager handles entity states: Transient, Managed, Detached, Removed
	Example:
		entityManager.persist(user);  // becomes managed
		entityManager.remove(user);   // marked for deletion
		entityManager.detach(user);   // becomes detached

🧠 Persistence Context
		Each EntityManager has one persistence context (first-level cache).
		It ensures: Only one instance of entity per ID, Automatic dirty checking, Write-behind mechanism
		Example:
			User u1 = entityManager.find(User.class, 1);
			User u2 = entityManager.find(User.class, 1);
			System.out.println(u1 == u2); // true
			Because of first-level cache.

🔷 What is Session? Session belongs to: Hibernate ORM, It is Hibernate’s native API for ORM operations. It existed before JPA.

🔥 Difference Between EntityManager and Session
| Feature             | EntityManager         | Session                                 |
| ------------------- | --------------------- | --------------------------------------- |
| Belongs to          | JPA Specification     | Hibernate                               |
| Package             | `jakarta.persistence` | `org.hibernate`                         |
| Portability         | Yes                   | No                                      |
| Implementation      | Interface only        | Concrete class                          |
| Extra features      | Limited               | Rich (filters, stateless session, etc.) |
| Underlying relation | Wrapper over Session  | Core engine                             |

🔎 Internal Relationship
	When using Hibernate as JPA provider:
		EntityManager → internally delegates to → Session
		You can even unwrap: Session session = entityManager.unwrap(Session.class);

🚀 Functional Differences
	1️⃣ save() vs persist()
		persist() → JPA, does not return ID
		save() → Hibernate, returns generated ID

	2️⃣ get() vs find()
		find() → JPA
		get() → Hibernate
		Behavior almost same, but: load() (Hibernate only) returns proxy

	3️⃣ Flush Modes
		EntityManager: FlushModeType.AUTO, FlushModeType.COMMIT
		Session: FlushMode.AUTO, FlushMode.COMMIT, FlushMode.MANUAL (Hibernate-specific)
		Hibernate provides more control.
	
	4️⃣ StatelessSession (Hibernate Only)
		Hibernate has: StatelessSession
		No: First-level cache, Dirty checking, Persistence context
		Used for bulk processing. JPA does NOT have this.

🔥 Real Production Insight
	In Spring Boot:
		When you use: spring-boot-starter-data-jpa
			You interact with EntityManager, Hibernate Session runs underneath
			If performance tuning is needed: You may unwrap Session, Use batch processing, Use StatelessSession
	
	⚠️ Interview Trap Questions
		❓ Is EntityManager thread-safe? ➡️ No
		❓ Is Session thread-safe? ➡️ No
		❓ Is SessionFactory thread-safe? ➡️ Yes

🔹 Q3. What is Persistence Context? What are the states of an entity in JPA?
======================================================================================================
🔷 What is Persistence Context?
	Persistence Context is a set of managed entity instances maintained by the EntityManager (or Hibernate Session).
	It acts as: 
		First-level cache, Change tracker (dirty checking), Identity map (one object per DB row per context)
	Each EntityManager has exactly one persistence context.

🧠 Simple Definition (Interview Style)
	Persistence Context is a logical container where Hibernate/JPA keeps track of entity objects and synchronizes their state with the database.

🔥 Internally It Works Like This
					Database Row  ↔  Entity Object
										↑
								Persistence Context


			When you load an entity: User user = entityManager.find(User.class, 1);
			Entity is loaded from DB, Stored inside persistence context, Becomes Managed
			If you update it: user.setName("Munna"); Hibernate does NOT execute update immediately.
			At flush time, it: Compares current state with original snapshot Generates SQL automatically (Dirty Checking)

🔥 Key Characteristics
	1️⃣ First-Level Cache
		If you fetch same entity twice:
			User u1 = entityManager.find(User.class, 1);
			User u2 = entityManager.find(User.class, 1);
			u1 == u2 → TRUE
		Because persistence context ensures single object per ID.

	2️⃣ Automatic Dirty Checking
			No need to call update: user.setEmail("new@mail.com");
			Hibernate detects change during flush and executes SQL.

	3️⃣ Write-Behind Mechanism
			SQL is executed: On flush, On commit, Before query (if flush mode AUTO)

🔷 Types of Persistence Context
	1️⃣ Transaction-scoped (Default): Lives only during transaction, Most common (Spring Boot default)
	2️⃣ Extended Persistence Context: Lives beyond transaction, Used in stateful apps (rare in microservices)

🔥 States of an Entity in JPA, There are 4 entity states:
	1️⃣ Transient (New State): 
		Not associated with persistence context, Not stored in DB, No ID in DB
		Example: User user = new User();

	2️⃣ Managed (Persistent State)
		Associated with persistence context, Tracked by Hibernate, Changes automatically synchronized
		Example:
				entityManager.persist(user);
			OR
				User user = entityManager.find(User.class, 1);

	3️⃣ Detached
		Was managed earlier, Now removed from persistence context, Changes are NOT tracked
		Happens when:
					entityManager.detach(user);
					entityManager.clear();
					entityManager.close();
		To reattach:
				entityManager.merge(user);

	4️⃣ Removed
		Marked for deletion, SQL DELETE executed at flush
		entityManager.remove(user);

🔥 State Transition Diagram (Important for Interview)

	Transient --persist()--> Managed --remove()--> Removed
		 ↑                       |
		 |                       |
		 ←-------detach()--------|
     
		Managed --detach()--> Detached
		Detached --merge()--> Managed

🔥 Senior-Level Insights
	1️⃣ Persistence Context != Database
		Just because entity is managed does NOT mean DB is updated immediately.
		Update happens at: flush, commit
	
	2️⃣ Persistence Context is NOT Shared Across Threads
		EntityManager is NOT thread-safe. Each transaction has its own persistence context.

	3️⃣ Memory Issue in Large Batch
		If you insert 1 million records without clearing: entityManager.persist(obj);
		All objects stay in persistence context → OutOfMemoryError
		
		Solution:
			if (i % 50 == 0) {
				entityManager.flush();
				entityManager.clear();
			}

🔹 Q4. Difference between persist() and merge()? What happens if we call merge() on a managed entity?
======================================================================================================
🔷 Difference Between persist() and merge()
	persist() makes a transient entity managed.
	merge() copies the state of a detached entity into a managed entity.

🔥 1️⃣ persist()
		Defined by JPA (EntityManager.persist())
		✔ Used For: New (Transient) entities
		✔ Behavior: 
			Makes entity Managed, Schedules INSERT
			Does NOT return anything, Fails if entity already exists

		Example:
			User user = new User();
			entityManager.persist(user);
		
		State transition: Transient → Managed
		⚠ Important Points: INSERT executed at flush/commit
			If ID already exists → EntityExistsException
			If entity is Detached → exception

🔥 2️⃣ merge()
		Defined by JPA (EntityManager.merge())
		✔ Used For: Detached entities, Updating existing records
		✔ Behavior: 
			Returns a new managed entity, Copies state from given object
			Does NOT attach the passed object, May trigger SELECT before UPDATE

		Example:
			User detachedUser = new User();
			detachedUser.setId(1L);
			detachedUser.setName("Munna");
			User managedUser = entityManager.merge(detachedUser);
		
		State transition: Detached → (Copy State) → Managed
		⚠ Important: managedUser is managed, detachedUser remains detached
	
	🔍 Internal Working of merge()
		Internally Hibernate does: 
			Check if entity exists in persistence context
			If not: Fetch from DB (SELECT), Copy all field values, Return managed instance, Update executed at flush

🔥 Side-by-Side Comparison
	| Feature              | persist() | merge()               |
	| -------------------- | --------- | --------------------- |
	| Works on             | Transient | Detached              |
	| Returns value        | No        | Yes (Managed copy)    |
	| If entity exists     | Exception | Updates               |
	| If entity is managed | No effect | Returns same instance |
	| May trigger SELECT   | No        | Yes                   |
	| Used for             | Insert    | Update                |

🔥 What Happens If We Call merge() on a Managed Entity?
	
	If entity is already Managed:
			User user = entityManager.find(User.class, 1);
			User result = entityManager.merge(user);

	What happens: Hibernate detects entity is already managed, No new object created, Returns same instance
	No additional SELECT executed, No copying needed
	So: result == user  → TRUE
	
🔥 Very Important Senior Insight
	Many developers misuse merge like this: user = entityManager.merge(user);
	If entity is already managed: This assignment is unnecessary
	If entity is detached: This assignment is necessary

🔥 Production Scenario Example
	Scenario:
		Transaction 1: 
			User user = entityManager.find(User.class, 1);
			Session closed → user becomes Detached
			Modify outside transaction: user.setName("Updated");
		
		Transaction 2: 
			entityManager.merge(user);
			Hibernate: Loads entity from DB, Copies new values, Executes UPDATE at flush

🔥 Performance Consideration
	merge() may trigger unnecessary SELECT.
	If you are sure entity is managed: Do NOT use merge
	If updating many records: Prefer bulk JPQL update

🔹 Q5. What is dirty checking? How does it work internally?
======================================================================================================
🔷 Dirty Checking is the mechanism by which Hibernate automatically detects changes made to managed entities and synchronizes them with the database at flush time. You don’t call update() manually. Hibernate detects changes automatically.

✅ Simple Example:
					@Transactional
					public void updateUser() {
						User user = entityManager.find(User.class, 1);
						user.setName("Munna");
					}
		You never call save() or update().
		Still Hibernate generates: UPDATE user SET name = 'Munna' WHERE id = 1;
		This happens because of dirty checking.

🔥 When Does Dirty Checking Happen?
	Dirty checking runs during: flush(), Transaction commit, Before executing JPQL (if FlushMode.AUTO)

🔷 How Dirty Checking Works Internally
	🧠 Step 1: Snapshot Creation
			When entity becomes Managed, Hibernate: 
				Loads entity from DB, Creates a snapshot (copy of field values), Stores snapshot inside persistence context
			
			Internally stored in:
						PersistenceContext
						  → EntityEntry
							   → loadedState (snapshot array)
							   
	🧠 Step 2: Field Modification
			You modify entity: user.setEmail("new@mail.com"); No SQL executed yet.

	🧠 Step 3: Flush Time Comparison
			At flush: Hibernate iterates over all managed entities
			Compares: Current state, Snapshot state, If different → Entity marked as DIRTY, Generates UPDATE SQL

🔥 Internal Algorithm (Simplified)
	Pseudo-process:
		for each managedEntity:
			if currentState != snapshotState:
				generate UPDATE SQL
		
🔷 Where Snapshot is Stored?
	Inside Hibernate’s: Session, PersistenceContext, EntityEntry, loadedState[]
	Each entity has:
			EntityEntry {
			   status
			   loadedState[]
			   version
			}
	
🔥 How Hibernate Compares Fields? Two approaches:
	1️⃣ Reflection-based comparison (default)
		Compare each field, Slightly slower
		
	2️⃣ Bytecode Enhancement (Optimized)
		Hibernate enhances entity bytecode to: Track field-level changes, Maintain dirty flag, Avoid full comparison
		More efficient.
			
🔥 Example With Versioning
	If entity has:
		@Version
		private int version;
	
	During dirty check: Version incremented, Prevents lost updates, Implements optimistic locking

🔥 Important Rules
	✔ Dirty Checking Works Only For: Managed entities, Within active persistence context
	❌ Does NOT Work For: Detached entities, Transient entities, StatelessSession

🔥 Performance Impact
	If you load 10,000 entities: List<User> users = repository.findAll();
	At flush: Hibernate checks all 10,000 entities for changes.
	Can impact performance.

🔥 How to Optimize?
	1️⃣ Use @Transactional(readOnly = true): Disables dirty checking in Spring.
	
	2️⃣ Use DTO instead of entity: If only reading data.
	
	3️⃣ Use StatelessSession (Hibernate-specific): No dirty checking.

🔥 Real Production Issue Example
	Scenario: Loaded 50k records, Updated 10, Hibernate checked all 50k at flush, CPU spike
	Solution: Clear persistence context periodically, Use batch updates, Use JPQL bulk update

🔹 Q6. What is flush? When does flush happen? Difference between flush() and commit()?
======================================================================================================
🔷 Flush is the process of synchronizing the persistence context (in-memory changes) with the database by executing SQL statements. Important: Flush does NOT commit the transaction. Flush only sends SQL to DB. Data may still be rolled back.

🧠 Simple Understanding
		Entity Changes (Memory)
				↓
			flush()
				↓
		   SQL Executed
				↓
		   commit()
				↓
		   Transaction Permanently Saved
		   
🔥 Example
		@Transactional
		public void updateUser() {
			User user = entityManager.find(User.class, 1);
			user.setName("Munna");

			entityManager.flush();  // SQL executed here
		}
	
		At flush(): UPDATE user SET name='Munna' WHERE id=1;
		But until commit(): Changes are NOT permanent, Can still rollback

🔷 When Does Flush Happen? Flush happens automatically or manually.
	
	✅ 1️⃣ Automatic Flush (Default)
			FlushModeType.AUTO (default)
			Flush occurs: Before transaction commit, Before executing JPQL/Criteria query, When explicitly calling flush()
			Example: Flush Before Query
				user.setName("Munna");
				entityManager.createQuery("SELECT u FROM User u").getResultList();
			
			Hibernate flushes BEFORE query to maintain consistency.

	✅ 2️⃣ On Transaction Commit
			When transaction commits: transaction.commit();
			Hibernate flushes automatically before commit.

	✅ 3️⃣ Manual Flush
			entityManager.flush();
			Used in: Batch processing, Force early SQL execution

🔷 Flush Modes
	JPA Flush Modes: FlushModeType.AUTO, FlushModeType.COMMIT
	AUTO (default): Flush before queries + commit
	COMMIT: Flush only at commit, No flush before query

	Hibernate-Specific Flush Modes: AUTO, COMMIT, MANUAL (very important for performance)
	MANUAL: No automatic flush, Only flush when explicitly called

🔥 What Happens Internally During Flush?
	At flush: 
		Dirty checking runs, Detect modified entities, Generate SQL, Execute SQL
		Update snapshots, Synchronize persistence context

	Flush processes: INSERT, UPDATE, DELETE, Collection changes

🔷 Difference Between flush() and commit()
	.flush() synchronizes persistence context with database.
	commit() permanently saves transaction and ends it.

	🔥 Detailed Comparison
		| Feature                     | flush() | commit()        |
		| --------------------------- | ------- | --------------- |
		| Executes SQL                | Yes     | Yes (via flush) |
		| Ends transaction            | No      | Yes             |
		| Can rollback after?         | Yes     | No              |
		| Closes persistence context? | No      | Usually yes     |
		| Triggers dirty checking?    | Yes     | Yes (via flush) |

	🔥 Key Important Concept
		commit() internally calls flush() first.
		So: commit()
			   → flush()
			   → DB commit
		   
🔥 Real Scenario Example
		@Transactional
		public void example() {
			user.setName("Munna");
			entityManager.flush();
			throw new RuntimeException();
		}

	Result: UPDATE executed, But transaction rolled back, DB unchanged
	Why? Because flush ≠ commit

🔥 Production Use Case of flush()
	1️⃣ Batch Insert Optimization
			for (int i = 0; i < 10000; i++) {
				entityManager.persist(user);
				if (i % 50 == 0) {
					entityManager.flush();
					entityManager.clear();
				}
			}
		Prevents memory overflow.

	2️⃣ Getting DB Constraint Error Early
		If unique constraint exists:
				entityManager.persist(user);
				entityManager.flush(); // exception thrown here
		Without flush: Error appears only at commit

🔹 Q7. What is detach() and clear()? When would you use it?
======================================================================================================
🔷 detach() removes an entity from the persistence context, making it no longer managed by the EntityManager.
	After detaching: Hibernate stops tracking changes, Dirty checking stops, No automatic update will happen

🔥 Simple Example
	User user = entityManager.find(User.class, 1); // Managed
	entityManager.detach(user); // Now Detached
	user.setName("Munna"); // No update will happen
	Even if transaction commits → DB will NOT update.

🔷 Entity State Transition
	Before: Managed → (detach()) → Detached
	Detached entity: Exists in DB, Exists in memory, But not tracked

🔥 An entity becomes detached when: 
	entityManager.detach(entity)
	entityManager.clear() → detaches all entities
	entityManager.close(): Transaction ends (in transaction-scoped persistence context)

🔥 Difference Between detach() and clear()
	detach(entity)	Removes only that entity
	clear()	Removes all managed entities
	
🔷 When Would You Use detach()?
	
	✅ 1️⃣ Prevent Unwanted Updates; If you want to modify object in memory but NOT persist it:
			User user = entityManager.find(User.class, 1);
			entityManager.detach(user);
			user.setName("Temporary Change"); // No DB update

	✅ 2️⃣ Reduce Memory Usage (Large Batch Processing)
			When processing large datasets:
				for (User user : users) {
					process(user);
					entityManager.detach(user);
				}
			
			Prevents persistence context from growing too large.
			Otherwise: All entities remain managed, Dirty checking runs on all, Memory increases, CPU increases

	✅ 3️⃣ Improve Performance
			If you are only reading data: Load entity, Detach immediately, Avoid dirty checking overhead
	
	✅ 4️⃣ Send Entity to Another Layer
			In layered architecture: Load entity, Detach it, Send to another thread/system, Prevent accidental DB update
		
🔥 Important: Re-attaching Detached Entity
	Detached entity changes are NOT saved automatically.
	To save: entityManager.merge(user);
	merge() copies detached state into a managed entity.

🔥 Real Production Scenario
	We once had: 50,000 entities loaded, Only 10 needed update, But dirty checking ran on all 50,000, CPU spike

	Solution: Detach unnecessary entities Or clear persistence context periodically

🔹 Q8. What is contains() method?
======================================================================================================
🔷 contains() is a method of EntityManager. boolean contains(Object entity)
	It checks whether the given entity instance is currently managed by the persistence context.

🔥 What Does It Actually Mean?
	If entityManager.contains(entity) returns: ✅ true → entity is in Managed state
	❌ false → entity is Detached or Transient

	🔥 Example 1 – Managed Entity
			User user = entityManager.find(User.class, 1);
			boolean result = entityManager.contains(user);
			System.out.println(result); // true
		
		Why: Because find() loads entity into persistence context → Managed.

	🔥 Example 2 – Detached Entity
			User user = entityManager.find(User.class, 1);
			entityManager.detach(user);
			boolean result = entityManager.contains(user);
			System.out.println(result); // false
			
			Because it is no longer tracked.

	🔥 Example 3 – Transient Entity
			User user = new User();
			boolean result = entityManager.contains(user);
			System.out.println(result); // false
			
			It was never managed.

🔥 Important Clarification
	contains() checks: 
		✔ Is this exact object instance managed?
		❌ It does NOT check if the entity exists in the database.

	Example:
			User user = new User();
			user.setId(1L);
		Even if ID exists in DB: entityManager.contains(user); // false
		Because this object is not managed.

🔥 What Happens After merge()?
		User detachedUser = new User();
		detachedUser.setId(1L);
		User managedUser = entityManager.merge(detachedUser);
		entityManager.contains(detachedUser); // false
		entityManager.contains(managedUser);  // true

	Very important interview point: merge() returns managed copy, Original object remains detached

🔥 How Hibernate Checks Internally
	Internally: PersistenceContext maintains a map like: EntityKey → EntityInstance
	contains() checks whether: The exact instance exists in that map, And status is Managed

🔥 When Would You Use contains()? Rare in normal code, but useful when:

	1️⃣ Defensive Programming
			if (!entityManager.contains(entity)) {
				entity = entityManager.merge(entity);
			}
	
	2️⃣ Debugging Lifecycle Issues
		To verify: Is entity managed? Why update not happening?

	3️⃣ Avoid IllegalStateException
			Before calling remove:
				if (!entityManager.contains(user)) {
					user = entityManager.merge(user);
				}
				entityManager.remove(user);

			Because remove() requires managed entity.

🔥 Interview Trap Questions
	Does contains() check database existence? ➡️ No.
	Does contains() trigger DB query? ➡️ No.
	Does contains() work across different EntityManagers? ➡️ No. Each persistence context is independent.
	Is contains() same as find()? ➡️ No. find() loads from DB; contains() just checks context.

🔥 2️⃣ Entity Lifecycle & Internal Working
======================================================================================================
🔹 Q1. How does Hibernate track changes in entities?
======================================================================================================
🔷 Hibernate tracks changes using a mechanism called Dirty Checking. When an entity becomes managed, Hibernate stores a snapshot of its state. At flush time, it compares the current state with the snapshot and generates SQL if differences are detected.

🔷 Step-by-Step Internal Working
	When an entity is loaded: User user = entityManager.find(User.class, 1);
	Hibernate does: Loads data from DB, Creates entity object, Stores it in Persistence Context
	Takes a snapshot of field values

	🔥 Where Snapshot is Stored? Inside Hibernate’s internal structures:

			Session
			   → PersistenceContext
					→ EntityEntry
						 → loadedState[]

		Each managed entity has: Current object, Snapshot array (loadedState), Status, Version (if any)
		🔷 Example
			Initial DB: id = 1, name = "John"
			Snapshot stored: loadedState = ["John"]
			Now you modify: user.setName("Munna");
			No SQL executed yet.

		🔷 What Happens During Flush?
			At flush: Hibernate iterates over all managed entities
			Compares: Current field values with Snapshot values
			If different → Entity marked DIRTY, Generates UPDATE SQL
			Example SQL: UPDATE user SET name='Munna' WHERE id=1;
			

🔷 Important Points
	✔ Dirty checking happens: On flush(), Before commit, Before query (AUTO mode)
	❌ It does NOT happen: Immediately after setter, For detached entities, In StatelessSession

🔥 How Hibernate Compares Fields? Two main approaches:
	1️⃣ Reflection-Based (Default)
		Hibernate: Uses reflection, Compares each field, Slightly slower for large entity sets

	2️⃣ Bytecode Enhancement (Optimized)
		Hibernate enhances entity class at runtime or build-time.
		Instead of comparing all fields, it: 
			Tracks field-level modifications
			Maintains dirty flags
			Knows exactly which property changed
			Much more efficient.

🔷 Example With Versioning
	If entity has:
		@Version
		private int version;
		
	When dirty detected: Version incremented
	Optimistic locking enforced, Prevents lost updates

🔥 Performance Consideration
	If 50,000 entities are managed: At flush: Hibernate checks all 50,000, Even if only 5 changed Can cause CPU spike.

🔥 How to Optimize?
	1️⃣ Use readOnly transactions
				@Transactional(readOnly = true)
				Disables dirty checking in Spring.
	
	2️⃣ Use DTO instead of entity for read-only queries
	3️⃣ Clear persistence context periodically
			entityManager.flush();
			entityManager.clear();
	4️⃣ Use bulk update
			entityManager.createQuery(
			  "UPDATE User u SET u.status='INACTIVE'"
			).executeUpdate();
		Bypasses dirty checking.

🔥 Real Production Issue Example
	Problem: Batch job loading 1M records, Dirty checking for all, High memory + CPU
	Solution: Use StatelessSession, Flush and clear every 50 records

🔥 Interview Trap Questions
❓ Does Hibernate use equals() for dirty checking? ➡️ No, it compares field values.
❓ Does dirty checking work for detached entity? ➡️ No.
❓ When exactly does comparison happen? ➡️ At flush.
❓ Is dirty checking same as flush? ➡️ No. Dirty checking is part of flush.

🔹 Q2. What happens internally when save() is called? What is the difference between save() and persist()?
======================================================================================================
🔷 save() belongs to Hibernate’s native API (Session).
	Session session = sessionFactory.openSession();
	session.save(user);
	Now let’s see what happens internally.

🔥 Step-by-Step Internal Flow of save()
	1️⃣ Check Entity State
		Hibernate checks: Is entity transient? Does it already have identifier? Is it already in persistence context?

	2️⃣ Identifier Generation
		If ID is generated (e.g., @GeneratedValue), Hibernate: Determines ID strategy, Generates ID (depending on strategy)
		Examples:
			IDENTITY → Immediate DB insert to get ID
			SEQUENCE → Fetch next sequence value
			TABLE → Read from ID table
	
	3️⃣ Add to Persistence Context
		Hibernate: Stores entity in Session, Creates EntityEntry, Takes snapshot for dirty checking, Now entity becomes Managed

	4️⃣ Schedule Insert
		Hibernate does NOT always execute INSERT immediately.
		It: Adds INSERT action to ActionQueue, Executes at flush time
		Exception: With GenerationType.IDENTITY, INSERT happens immediately because DB must generate ID.
	
	5️⃣ Return Generated ID
		save() returns the generated identifier: Long id = (Long) session.save(user);
		This is one major difference from persist().

🔷 persist() belongs to JPA (EntityManager). entityManager.persist(user);
🔥 What Happens Internally with persist()?
	1️⃣ Checks entity state
	2️⃣ Registers entity in persistence context
	3️⃣ Schedules INSERT
	4️⃣ Does NOT return ID
	5️⃣ Follows JPA specification rules
		Important: 
			INSERT may happen at flush
			With IDENTITY, insert may happen early

🔥 Difference Between save() and persist()
	✅ Short Interview Answer
		save() is Hibernate-specific and returns the generated ID.
		persist() is JPA-standard and does not return the ID.
		persist() strictly follows JPA lifecycle rules.

🔥 Detailed Comparison
| Feature                   | save()           | persist()                   |
| ------------------------- | ---------------- | --------------------------- |
| Belongs to                | Hibernate        | JPA                         |
| Returns ID                | Yes              | No                          |
| Works outside transaction | Yes              | No (JPA spec says needs tx) |
| On detached entity        | Saves new row    | Throws exception            |
| IDENTITY strategy         | Immediate insert | Immediate insert            |
| Portability               | No               | Yes                         |

🔥 Very Important Behavioral Difference
	1️⃣ Detached Entity Case
			User user = new User();
			user.setId(1L);	
		If record with ID 1 exists:
			save(): session.save(user);
				May cause: Duplicate insert, Constraint violation, Because Hibernate treats it as new entity.
			persist(): entityManager.persist(user);
				Throws: EntityExistsException, Because JPA enforces strict lifecycle rules.

🔥 Real Production Insight
	In modern Spring Boot apps: You usually use persist() indirectly
	Spring Data JPA uses JPA API, Hibernate runs underneath
	
	Best practice:
		✔ Use JPA API (persist)
		❌ Avoid Hibernate-specific save() unless necessary

🔥 ID Generation Strategy Impact
	🔹 SEQUENCE Strategy
		save(): Gets sequence value, Schedules insert
		persist(): Same behavior

	🔹 IDENTITY Strategy
		Both: Must execute INSERT immediately To get generated ID
		That’s why sometimes you see early SQL execution.

🔥 Interview Trap Questions
❓ Does save() always execute INSERT immediately? ➡️ No (except IDENTITY).
❓ Does persist() return ID? ➡️ No.
❓ Can persist() work without transaction? ➡️ According to spec, no.
❓ Can save() work without transaction? ➡️ Yes (Hibernate allows it).

🔹 Q3. What happens if two sessions modify the same entity?
======================================================================================================
🔷 Scenario: Two Sessions Modify Same Entity
	Suppose we have a User entity:
				@Entity
				public class User {
					@Id
					private Long id;

					private String name;
				}
		And record in DB: id = 1, name = "John"
		
🔥 Case 1: Without Versioning (No @Version)
			Session 1: 
						User user1 = session1.get(User.class, 1);
						user1.setName("Munna");
			
			Session 2:
						User user2 = session2.get(User.class, 1);
						user2.setName("Patra");

			Now: Session 1 commits → DB becomes "Munna"
			Session 2 commits → DB becomes "Patra"

		⚠ Problem: Lost Update, Session 1's change is overwritten. Hibernate does NOT prevent this automatically.

		🔥 Why Does This Happen?
			Because: Each session has its own Persistence Context, Each session has its own snapshot
			No concurrency control without locking
				Session1 → PC1
				Session2 → PC2
			They don’t know about each other.

🔥 Case 2: With Optimistic Locking (@Version)
					@Entity
					public class User {

						@Id
						private Long id;

						private String name;

						@Version
						private int version;
					}

			DB: id = 1, name = "John", version = 1

			Session 1: Loads version = 1, Updates name 
			On commit: UPDATE user SET name='Munna', version=2 WHERE id=1 AND version=1; Success.
			DB: name = "Munna", version = 2
			
			Session 2: Loaded version = 1 earlier
			Tries to update: UPDATE user SET name='Patra', version=2 WHERE id=1 AND version=1;
			But version is now 2. No rows updated. Hibernate throws: OptimisticLockException
			🔥 Result: ✔ Lost update prevented, Exception thrown, Application can retry

🔥 Case 3: Pessimistic Locking
		Using: entityManager.find(User.class, 1, LockModeType.PESSIMISTIC_WRITE);
		What happens? DB lock acquired (SELECT FOR UPDATE)
		Second transaction waits, No concurrent modification possible

🔥 Summary of Behaviors
			 No locking => Lost update
	 Optimistic locking => Exception thrown
	Pessimistic locking	=> DB-level blocking
	
🔥 Important Senior-Level Points
	1️⃣ Hibernate itself does NOT prevent concurrency issues
		Unless: You use @Version Or explicit locking

	2️⃣ Optimistic Locking is Default Best Practice
		✔ No DB blocking, Scales well, Good for microservices

	3️⃣ Pessimistic Locking is Heavy
		✔ DB row lock, ❌ Reduces concurrency, ❌ Risk of deadlocks

🔥 Real Production Scenario
	Bank balance example: Without versioning: Two withdrawals at same time, Incorrect final balance
	With optimistic locking: Second transaction fails, Retry logic applied

🔥 Interview Trap Questions
❓ Does Hibernate automatically prevent lost update? ➡️ No.
❓ Does @Transactional prevent concurrent modification? ➡️ No.
❓ What happens if version field is missing? ➡️ Last commit wins.
❓ Which is better: optimistic or pessimistic? ➡️ Depends on use case.

🔥 3️⃣ Fetching Strategies (Very Important for 8+ Years)
======================================================================================================
🔹 Q1. Difference between EAGER and LAZY? Why is EAGER dangerous?
======================================================================================================
🔷 Fetching strategy defines when related entities are loaded from the database.
	In JPA:
		@OneToMany(fetch = FetchType.LAZY)
		@ManyToOne(fetch = FetchType.EAGER)

	Two types: EAGER, LAZY

🔷 1️⃣ EAGER Fetching
		Related entity is loaded immediately along with parent entity.
		Example:
				@Entity
				public class Order {

					@ManyToOne(fetch = FetchType.EAGER)
					private Customer customer;
				}

			When you fetch Order: Order order = entityManager.find(Order.class, 1);
			Hibernate also loads customer immediately.

🔷 2️⃣ LAZY Fetching
		Related entity is loaded only when accessed.

			@Entity
			public class Order {

				@ManyToOne(fetch = FetchType.LAZY)
				private Customer customer;
			}

		Now: Order order = entityManager.find(Order.class, 1);
		Customer is NOT loaded yet.
		When accessed: order.getCustomer().getName();
		Then Hibernate fires query.

🔥 Internal Working
	LAZY Uses Proxy
	Hibernate creates: Proxy object Or bytecode-enhanced class
	When you access property: Hibernate triggers SELECT

🔥 Difference Between EAGER and LAZY
	| Feature               | EAGER           | LAZY                        |
	| --------------------- | --------------- | --------------------------- |
	| When loaded           | Immediately     | On demand                   |
	| Performance           | Heavy           | Optimized                   |
	| Risk                  | Query explosion | LazyInitializationException |
	| Default for ManyToOne | EAGER           |                             |
	| Default for OneToMany |                 | LAZY                        |

🔥 Why is EAGER Dangerous?
	
	1️⃣ Query Explosion (N+1 Problem)
		Imagine:
				@Entity
				public class User {

					@OneToMany(fetch = FetchType.EAGER)
					private List<Order> orders;
				}

		Now fetch 100 users: List<User> users = userRepository.findAll();
		Hibernate might execute:
			1 query → fetch users
				100 queries → fetch orders
			= 101 queries

		This is N+1 problem.

	2️⃣ Cartesian Product Explosion
		If multiple EAGER associations:
			User → Orders (EAGER)
			User → Roles (EAGER)
		Hibernate may generate huge JOIN query: User × Orders × Roles
		Result: Duplicate rows, Memory explosion, Slow performance

	3️⃣ Unnecessary Data Loading
		Even if you don't need related entity: User user = repository.findById(1);
		EAGER still loads everything. Waste of: Memory, Network bandwidth, CPU

	4️⃣ Hard to Control
		EAGER is fixed at mapping level. You cannot disable it per query easily.
		But LAZY can be controlled using: JOIN FETCH, EntityGraph, DTO projection
	
	5️⃣ Recursive Relationships Can Break App
		Example:
			User → Orders (EAGER)
			Order → User (EAGER)
		Can cause: Infinite loading, StackOverflow, Huge query

🔥 Real Production Example
	We had: Order entity with EAGER relationships
		findAll() call 2000 records 15 joins 2GB heap usage DB CPU spike
	
	Solution: Changed everything to LAZY, Used DTO projection, Performance improved drastically.

🔥 Best Practice (Senior Level)
	✔ Default all relationships to LAZY
	✔ Use JOIN FETCH when needed
	✔ Avoid EAGER unless absolutely required
	✔ Use DTO for read-only APIs

🔥 Interview Trap Questions
❓ Is LAZY always safe? ➡️ No. Can cause LazyInitializationException.
❓ Why ManyToOne default is EAGER? ➡️ JPA spec design decision (but bad in practice).
❓ Can we override EAGER per query? ➡️ Yes, using JPQL join fetch.
❓ Does EAGER guarantee single query? ➡️ No. It may still trigger multiple queries.

🔹 Q2. What is LazyInitializationException? How to solve LazyInitializationException properly?
======================================================================================================
🔷 LazyInitializationException occurs when you try to access a LAZY-loaded association outside of an active persistence context (i.e., outside a transaction or after the session is closed).
	It is thrown by: Hibernate ORM

🔥 Why Does It Happen?
	Because LAZY relationships are loaded using proxies. When you access a lazy field: order.getCustomer().getName();
	Hibernate needs: Active Session, Open Persistence Context, If session is closed → exception thrown.

🔥 Example Scenario

		Entity layer:	@Entity
						public class Order {

							@ManyToOne(fetch = FetchType.LAZY)
							private Customer customer;
						}
		
		Service Layer:
						@Transactional
						public Order getOrder(Long id) {
							return orderRepository.findById(id).get();
						}

			Transaction ends here.

		Controller Layer:
							Order order = service.getOrder(1L);
							order.getCustomer().getName();  // ❌ Exception here

				Why? Transaction already committed, Persistence context closed, Proxy cannot initialize

🔥 What Internally Happens?
	1️⃣ Order loaded → customer is proxy
	2️⃣ Transaction ends → session closed
	3️⃣ Access customer → Hibernate tries DB call
	4️⃣ No session → throws: LazyInitializationException: could not initialize proxy

🔥 How to Solve LazyInitializationException Properly
	✅ 1️⃣ Use JOIN FETCH (Best Practice)
		Fetch required associations in query:
			@Query("SELECT o FROM Order o JOIN FETCH o.customer WHERE o.id = :id")
			Order findOrderWithCustomer(Long id);

		Now: Order + Customer loaded together
		No lazy initialization later, Clean solution
		✔ Recommended approach

	✅ 2️⃣ Use EntityGraph (Clean & Flexible)
			@EntityGraph(attributePaths = {"customer"})
			Optional<Order> findById(Long id);

			Very clean in Spring Data JPA. ✔ Preferred in modern applications

	✅ 3️⃣ Use DTO Projection (Best for APIs)
			@Query("SELECT new com.example.OrderDTO(o.id, c.name) 
					FROM Order o 
					JOIN o.customer c 
					WHERE o.id = :id")

		Advantages: No entity graph issues, No lazy loading problems, Better performance
		✔ Best for REST APIs

⚠️ 4️⃣ Open Session In View (Not Recommended)
		Spring Boot default: spring.jpa.open-in-view=true
		What it does: Keeps session open till view rendering
		Problem: DB access in controller, Hidden performance issues, Long transactions, Hard to debug
		Most senior teams disable it.

❌ 5️⃣ Changing to EAGER (Wrong Fix)
		Some developers change: fetch = FetchType.EAGER
		Bad idea: Causes N+1, Memory explosion, Performance degradation, Never use EAGER as solution.

🔥 Real Production Example
	Problem: REST API returning 1000 orders, Each order had LAZY associations
	OSIV enabled, DB hammered during JSON serialization
	
	Fix: Disabled OSIV, Used DTO projection, Reduced queries from 2000 → 2

🔥 Best Practice Summary
| Solution        | Recommended? |
| --------------- | ------------ |
| JOIN FETCH      | ✅ Yes        |
| EntityGraph     | ✅ Yes        |
| DTO Projection  | ✅ Best       |
| OSIV            | ⚠ Avoid      |
| Change to EAGER | ❌ No         |

🔥 Interview Trap Questions
❓ Why does LAZY work inside transaction but not outside? ➡️ Because persistence context is active.
❓ Does EAGER fix LazyInitializationException? ➡️ Yes, but creates worse performance issues.
❓ What is Open Session In View? ➡️ Keeps session open across request lifecycle.
❓ How to avoid LazyInitializationException in microservices? ➡️ Use DTO projection.

🔹 Q3. Difference between: JOIN, JOIN FETCH, LEFT JOIN FETCH
======================================================================================================
🔷 First: These Are JPQL Constructs
	Used inside JPQL queries: @Query("SELECT o FROM Order o JOIN o.customer c")
	They look similar to SQL JOIN — but behavior is different in JPA/Hibernate.

🔥 1️⃣ JOIN (Normal JOIN)
	Example:
			@Query("SELECT o FROM Order o JOIN o.customer c")
			List<Order> findOrders();
			
	What it does: 
		✔ Creates SQL JOIN, Allows filtering based on joined entity
		❌ Does NOT fetch the association into persistence context

	Meaning: customer is still LAZY (if defined LAZY)
	Accessing order.getCustomer() later may trigger additional query

	🔎 Important Point
		JOIN is used for: Filtering, Where clause conditions, Selecting specific fields
		It does NOT override fetch strategy.

🔥 2️⃣ JOIN FETCH
	Example:
			@Query("SELECT o FROM Order o JOIN FETCH o.customer")
			List<Order> findOrders();
			
	What it does:
		✔ Creates SQL JOIN, Loads associated entity immediately, Overrides LAZY for this query
		✔ Prevents LazyInitializationException, Prevents N+1 problem
	
	Now: order.getCustomer().getName(); // No extra query Because customer is already loaded.

🔥 3️⃣ LEFT JOIN FETCH
	Example:
			@Query("SELECT o FROM Order o LEFT JOIN FETCH o.customer")
			List<Order> findOrders();
			
	What it does:
		✔ Fetches association, Includes records even if association is NULL, Safe for optional relationships

🔥 Difference Between JOIN and JOIN FETCH
| Feature                       | JOIN | JOIN FETCH |
| ----------------------------- | ---- | ---------- |
| SQL JOIN generated            | Yes  | Yes        |
| Fetch association             | No   | Yes        |
| Overrides LAZY                | No   | Yes        |
| Prevents N+1                  | No   | Yes        |
| Returns association in result | No   | Yes        |

🔥 Difference Between JOIN FETCH and LEFT JOIN FETCH
| Feature                        | JOIN FETCH | LEFT JOIN FETCH |
| ------------------------------ | ---------- | --------------- |
| Type of join                   | INNER JOIN | LEFT OUTER JOIN |
| Null associations included?    | No         | Yes             |
| Safe for optional relationship | No         | Yes             |

🔥 Example Scenario
	Suppose:
			@Entity
			public class Order {

				@ManyToOne(fetch = FetchType.LAZY)
				private Customer customer;
			}
	Case 1: JOIN
			SELECT o FROM Order o JOIN o.customer c
			Filters only orders with customers, Customer NOT fetched
			Access later → additional query
	
	Case 2: JOIN FETCH
			SELECT o FROM Order o JOIN FETCH o.customer
			Orders with customers, Customer loaded, No extra query

	Case 3: LEFT JOIN FETCH
			SELECT o FROM Order o LEFT JOIN FETCH o.customer
			All orders Even if customer is null, Customer loaded if exists

🔥 Important Senior-Level Points
	1️⃣ JOIN FETCH Cannot Be Used in Count Query, Not allowed.
			SELECT COUNT(o) FROM Order o JOIN FETCH o.customer  ❌
	
	2️⃣ Multiple JOIN FETCH on Collections Causes Cartesian Explosion
		Example:
				SELECT u FROM User u
				JOIN FETCH u.orders
				JOIN FETCH u.roles
		If: 5 orders, 3 roles, Result rows = 15 rows, Huge duplication.
	
	3️⃣ Only One Collection Fetch Allowed (Hibernate Limitation)
		Fetching multiple @OneToMany collections with JOIN FETCH causes:
			cannot simultaneously fetch multiple bags

🔥 Real Production Scenario
	We had: SELECT u FROM User u JOIN FETCH u.orders
	For 1000 users: Orders loaded in single query
	No N+1, Performance improved drastically
	But when added second collection: Query became huge, Memory spike
	Switched to DTO approach

🔥 Best Practice
	✔ Default relationships as LAZY
	✔ Use JOIN FETCH selectively
	✔ Use LEFT JOIN FETCH for optional relations
	✔ Use DTO for complex APIs
	✔ Avoid multiple collection fetches

🔹 Q4. What is EntityGraph?
======================================================================================================
🔷 EntityGraph is a JPA feature that allows you to dynamically specify which associations should be fetched eagerly for a particular query. It overrides the default fetch strategy (LAZY/EAGER) at runtime. It belongs to JPA specification (jakarta.persistence), not Hibernate-specific.

🔥 Why Was EntityGraph Introduced?
	Problems with: Hardcoding fetch = EAGER, Writing complex JOIN FETCH queries, Query duplication, Poor flexibility
	EntityGraph gives:
		✔ Clean, Reusable, Dynamic fetch control

🔷 Basic Example
	Suppose:
			@Entity
			public class Order {

				@Id
				private Long id;

				@ManyToOne(fetch = FetchType.LAZY)
				private Customer customer;

				@OneToMany(fetch = FetchType.LAZY)
				private List<OrderItem> items;
			}

		By default: customer → LAZY, items → LAZY

🔥 Using EntityGraph in Spring Data JPA
			@EntityGraph(attributePaths = {"customer", "items"})
			Optional<Order> findById(Long id);
		
		Now when calling: orderRepository.findById(1L);
		Hibernate generates:
				SELECT o.*, c.*, i.*
				FROM orders o
				LEFT JOIN customer c
				LEFT JOIN order_item i
				WHERE o.id = ?
		Both associations fetched eagerly for that query.

🔷 Two Types of EntityGraph
	1️⃣ Fetch Graph: Only specified attributes are eagerly fetched. Unspecified attributes remain LAZY.
		@EntityGraph(type = EntityGraph.EntityGraphType.FETCH)

	2️⃣ Load Graph: Specified attributes fetched eagerly. Unspecified attributes follow their default fetch type.
		@EntityGraph(type = EntityGraph.EntityGraphType.LOAD)

🔥 Named EntityGraph (Reusable)
	Defined at entity level:
						@NamedEntityGraph(
							name = "Order.detail",
							attributeNodes = {
								@NamedAttributeNode("customer"),
								@NamedAttributeNode("items")
							}
						)
						@Entity
						public class Order {
						}

		Then used:
			@EntityGraph(value = "Order.detail")
			Optional<Order> findById(Long id);
			
🔥 EntityGraph vs JOIN FETCH
| Feature                    | EntityGraph | JOIN FETCH               |
| -------------------------- | ----------- | ------------------------ |
| Dynamic control            | Yes         | Hardcoded in query       |
| Reusable                   | Yes         | No                       |
| Clean syntax               | Yes         | Complex for large graphs |
| Works with derived queries | Yes         | No                       |
| Flexibility                | High        | Medium                   |

🔥 When to Use EntityGraph?
	✔ REST APIs
	✔ Avoid LazyInitializationException
	✔ Avoid N+1
	✔ Clean repository methods
	✔ When same query needs different fetch plans

🔥 Real Production Example
	We had: findById()
	Used in: Admin API → needed items + customer
	Mobile API → needed only customer
	
	Using JOIN FETCH: Required two different queries
	Using EntityGraph: Same method, Different graphs, Cleaner design

🔥 Advanced: Dynamic EntityGraph Programmatically
		EntityGraph<Order> graph = entityManager.createEntityGraph(Order.class);
		graph.addAttributeNodes("customer");
		Map<String, Object> hints = new HashMap<>();
		hints.put("jakarta.persistence.fetchgraph", graph);
		entityManager.find(Order.class, 1L, hints);
		
		Used in complex service layers.

🔥 Common Interview Trap Questions
❓ Does EntityGraph replace LAZY/EAGER mapping? ➡️ No, it overrides per query.
❓ Does it prevent N+1? ➡️ Yes, when used correctly.
❓ Can we fetch multiple collections? ➡️ Yes, but same Cartesian explosion issue applies.
❓ Is EntityGraph Hibernate-specific? ➡️ No, JPA standard.

🔹 Q5. Difference between FetchType and FetchMode?
======================================================================================================
🔷 Short Answer (Interview Style)
	FetchType decides when an association is fetched.
	FetchMode decides how it is fetched (SQL strategy).

🔷 1️⃣ FetchType (JPA Standard) Belongs to JPA (jakarta.persistence.FetchType)
		@OneToMany(fetch = FetchType.LAZY)
		@ManyToOne(fetch = FetchType.EAGER)
		Two options: LAZY, EAGER

	🔥 What FetchType Controls
		It controls: Timing of loading
		EAGER => Load immediately with parent
		LAZY =>	Load only when accessed
	
🔷 2️⃣ FetchMode (Hibernate Specific)
	Belongs to: org.hibernate.annotations.FetchMode
	Used with:
			@Fetch(FetchMode.JOIN)
			@Fetch(FetchMode.SELECT)
			@Fetch(FetchMode.SUBSELECT)
	
	🔥 What FetchMode Controls
	It controls: SQL strategy used to fetch associations
	It decides: JOIN query? Separate SELECT? Subselect strategy?

	🔥 FetchMode Options
	🔹 1️⃣ FetchMode.SELECT (Default): Separate query for association.
		Example:
			SELECT * FROM orders;
			SELECT * FROM order_items WHERE order_id = 1;
			SELECT * FROM order_items WHERE order_id = 2;

		⚠ Can cause N+1 problem.

	🔹 2️⃣ FetchMode.JOIN: Fetch using SQL JOIN.
			SELECT o.*, i.*
			FROM orders o
			LEFT JOIN order_items i ON ...
		
		✔ Single query, ⚠ Cartesian product risk

	🔹 3️⃣ FetchMode.SUBSELECT
			Hibernate first fetches parent IDs: SELECT id FROM orders;
			Then: SELECT * FROM order_items WHERE order_id IN (subselect result);
		
		✔ Avoids N+1, ✔ No Cartesian explosion ✔ Good for collections

🔥 Side-by-Side Comparison
| Feature    | FetchType      | FetchMode                 |
| ---------- | -------------- | ------------------------- |
| Defined in | JPA            | Hibernate                 |
| Controls   | When to fetch  | How to fetch              |
| Options    | LAZY / EAGER   | SELECT / JOIN / SUBSELECT |
| Standard   | Yes            | No                        |
| Applied at | Entity mapping | Hibernate mapping         |

🔥 Important Relationship Between Them
	Even if: fetch = FetchType.EAGER
	Hibernate may still use: FetchMode.SELECT
	Meaning: It loads immediately, But using separate queries
	So: FetchType ≠ SQL JOIN
	
	🔥 Example to Understand Clearly
				@OneToMany(fetch = FetchType.EAGER)
				@Fetch(FetchMode.SELECT)
				private List<OrderItem> items;

		What happens? Items loaded immediately, But using separate SELECT queries, May cause N+1

🔥 Real Production Insight	
	We had: @OneToMany(fetch = FetchType.LAZY) And N+1 problem.
	Instead of changing to EAGER: We used: @Fetch(FetchMode.SUBSELECT)
	Result: One subselect query, Performance improved, No Cartesian explosion

🔥 Best Practice
	✔ Default everything to LAZY
	✔ Use JOIN FETCH in query
	✔ Use EntityGraph
	✔ Use SUBSELECT carefully
	✔ Avoid EAGER at mapping level

🔥 Interview Trap Questions
❓ Does FetchType.EAGER mean JOIN always? ➡️ No.
❓ Can FetchMode override FetchType? ➡️ FetchType decides timing, FetchMode decides SQL strategy.
❓ Which one is portable? ➡️ FetchType only.
❓ Can FetchMode fix LazyInitializationException? ➡️ No, because that’s timing issue.

🔹 Q7. What is batch fetching? What is subselect fetching?
======================================================================================================
🔷 Batch fetching is a Hibernate optimization that loads multiple lazy associations in batches instead of one-by-one.
	It helps reduce the N+1 problem without using JOIN FETCH.

🔷 How Batch Fetching Works
		Hibernate groups multiple lazy loads into a single query.
		Instead of:
				SELECT * FROM customer WHERE id = 1;
				SELECT * FROM customer WHERE id = 2;
				SELECT * FROM customer WHERE id = 3;

		Hibernate does: SELECT * FROM customer WHERE id IN (1,2,3,4,5);
		
🔷 How to Enable Batch Fetching
	1️⃣ Global Configuration: hibernate.default_batch_fetch_size=10
	
	2️⃣ Per Association
			@BatchSize(size = 10)
			@ManyToOne(fetch = FetchType.LAZY)
			private Customer customer;

		🔥 What Happens Internally
			If batch size = 10
			Hibernate: Detects multiple uninitialized proxies, Collects up to 10 IDs, Executes single IN query

🔥 When to Use Batch Fetching?
	✔ When you load multiple entities, When using LAZY, When you want to avoid JOIN FETCH
	✔ When fetching collections would cause Cartesian explosion

🔥 Batch Fetching vs JOIN FETCH
	| Feature               | Batch Fetch | JOIN FETCH                   |
	| --------------------- | ----------- | ---------------------------- |
	| Query Count           | Reduced     | 1                            |
	| Cartesian Risk        | No          | Yes                          |
	| Memory Usage          | Controlled  | High if large                |
	| Works for collections | Yes         | Limited (multiple bag issue) |

🔷 2️⃣ What is Subselect Fetching?
	Subselect fetching is a Hibernate strategy where Hibernate fetches related collections using a single subquery based on previously loaded parent entities.

	🔥 How It Works
		Suppose:
				@Entity
				public class Order {

					@OneToMany(fetch = FetchType.LAZY)
					private List<OrderItem> items;
				}
		
		You load:List<Order> orders = orderRepository.findAll();
		Hibernate runs: SELECT * FROM orders;
		Now when accessing first order’s items: orders.get(0).getItems();
		Instead of N separate queries, 
			Hibernate runs: SELECT * FROM order_items WHERE order_id IN ( SELECT id FROM orders );
		Single subselect query.

🔷 How to Enable Subselect Fetching
			@OneToMany(fetch = FetchType.LAZY)
			@Fetch(FetchMode.SUBSELECT)
			private List<OrderItem> items;
		
		This is Hibernate-specific.

🔥 Batch Fetch vs Subselect Fetch
	| Feature       | Batch Fetch           | Subselect Fetch          |
	| ------------- | --------------------- | ------------------------ |
	| SQL Style     | IN clause             | Subquery                 |
	| Works for     | ToOne & collections   | Mostly collections       |
	| Query Trigger | When lazy accessed    | When first lazy accessed |
	| Good for      | Mixed access patterns | Bulk loading parents     |

🔥 Example Comparison
	Assume 100 Orders loaded. Batch Fetch (size=10)
	Hibernate runs: 1 query → orders, 10 queries → items (10 at a time)
	Total ~11 queries

	Subselect Fetch
		Hibernate runs: 1 query → orders, 1 query → all items for those orders
		Total 2 queries

🔥 When to Use Which?
	Use Batch Fetching When: Random access pattern, Large dataset, Mixed associations

	Use Subselect When: You load all parents first, Then access collections, Want minimum query count

🔥 Real Production Scenario
	We had: 500 orders, Each order had items
	Using default SELECT → 501 queries
	Enabled SUBSELECT → 2 queries
	Performance improved drastically, But: When orders loaded partially → SUBSELECT not ideal.

🔥 Important Senior-Level Insight
	Neither batch nor subselect: Fix bad entity design, Replace DTO projection for APIs
	They are optimizations, not architecture solutions.

🔥 4️⃣ Caching (Must Know for Senior)
======================================================================================================
🔹 Q1. What is first-level cache? Can we disable it? Scope of first-level cache?
======================================================================================================
✅ First-level cache is the Session-level cache maintained by Hibernate.
	It is enabled by default , It is mandatory, It is associated with a single Session, Also called Persistence Context
	When you load an entity, Hibernate stores it inside the session cache. If you try to load the same entity again within the same session, Hibernate returns it from memory instead of hitting the database.

🔎 Internal Working
	Session session = sessionFactory.openSession();
	Employee emp1 = session.get(Employee.class, 1);  // DB hit
	Employee emp2 = session.get(Employee.class, 1);  // No DB hit (from cache)

	What happens internally?
		session.get() is called, Hibernate checks Persistence Context (First-Level Cache)
		If entity exists → return from cache
		If not → Query DB → Store entity in cache → Return entity
		Internally, it is basically a: Map<EntityKey, Entity>
		Where: EntityKey = (EntityName + PrimaryKey), Value = Managed entity object

🎯 Scope of First-Level Cache
| Property     | Value                             |
| ------------ | --------------------------------- |
| Scope        | **Per Session**                   |
| Lifetime     | Until session is closed           |
| Shared?      | ❌ No                              |
| Default?     | ✅ Yes                             |
| Thread-safe? | ❌ No (Session is not thread-safe) |

🧠 Why is First-Level Cache Important?
	1️⃣ Prevents Multiple DB Hits: Reduces unnecessary database queries.
	2️⃣ Maintains Object Identity (Very Important)
		Within one session:
			Employee e1 = session.get(Employee.class, 1);
			Employee e2 = session.get(Employee.class, 1);
			System.out.println(e1 == e2);  // TRUE
		
		Hibernate guarantees: Same row → Same Java object inside session
		This ensures consistency and proper dirty checking.

🔄 What Gets Stored in First-Level Cache?
	Entities loaded via: get(), load(), HQL, Criteria API, Native Query (mapped entities)
	Also stores: Snapshots (for dirty checking), Proxies, Collection wrappers

❓ Can We Disable First-Level Cache?
	🚫 No. You cannot disable it.
	It is: Mandatory, Core part of Hibernate architecture
	Required for: Dirty checking, Transactional write-behind, Object identity

🧹 But You Can Control It
	1️⃣ Clear the Cache: session.clear(); Removes all managed entities from persistence context.
	
	2️⃣ Evict Specific Entity: session.evict(emp); Removes only that entity.
	
	3️⃣ Detach Entity (JPA way): entityManager.detach(emp);
	4️⃣ Flush and Clear (Batch Processing), session.flush();, session.clear();
		Used in bulk insert/update to prevent memory overflow.

⚠️ Important Interview Trick Question
Q: If I open two sessions, will they share first-level cache?
	Session s1 = sessionFactory.openSession();
	Session s2 = sessionFactory.openSession();
	❌ No. Each session has its own independent first-level cache.
	
======================================================================================================
🔹 Q2. What is second-level cache? How to configure it? When should we avoid second-level cache?
======================================================================================================
✅ Second-level cache (L2 cache) is a cache shared across multiple sessions.
	It is associated with SessionFactory, It is optional, It must be explicitly enabled, It survives across sessions
	It reduces database load across transactions

🔎 Difference from First-Level Cache
| Feature     | First-Level Cache | Second-Level Cache    |
| ----------- | ----------------- | --------------------- |
| Scope       | Session           | SessionFactory        |
| Shared      | ❌ No              | ✅ Yes                 |
| Default     | Enabled           | Disabled              |
| Lifetime    | Session           | Application lifecycle |
| Can disable | ❌ No              | ✅ Yes                 |

🏗 Architecture Flow
						Application
						   ↓
						Session
						   ↓
						First-Level Cache
						   ↓ (if not found)
						Second-Level Cache
						   ↓ (if not found)
						Database

			Flow: Check First-Level Cache, If miss → Check Second-Level Cache, If miss → Query DB → Store in both caches

📦 What Can Be Cached?
	Hibernate can cache: Entities, Collections, Queries (Query Cache), Natural IDs

🧠 How Second-Level Cache Works Internally
	Each entity class has a separate cache region.
	Internally something like: Region: Employee
		  Key: 1 → Employee object data
		  Key: 2 → Employee object data

	Hibernate stores: Entity state (not actual object reference), Serializable data
	When another session loads the same entity: It reconstructs the entity from cached state

⚙️ How to Configure Second-Level Cache
	You need: Cache Provider, Enable L2 cache, Mark entities cacheable

	🔹 Step 1: Add Cache Provider
			Common providers: Ehcache, Hazelcast, Infinispan, Redis (via custom integration)
			Most common: Ehcache
			Example (Spring Boot + Maven):
						<dependency>
							<groupId>org.ehcache</groupId>
							<artifactId>ehcache</artifactId>
						</dependency>

	🔹 Step 2: Enable L2 Cache in Configuration: application.properties
			spring.jpa.properties.hibernate.cache.use_second_level_cache=true
			spring.jpa.properties.hibernate.cache.region.factory_class=org.hibernate.cache.jcache.JCacheRegionFactory
			spring.jpa.properties.hibernate.javax.cache.provider=org.ehcache.jsr107.EhcacheCachingProvider

	🔹 Step 3: Mark Entity as Cacheable
						@Entity
						@Cacheable
						@org.hibernate.annotations.Cache(
							usage = CacheConcurrencyStrategy.READ_WRITE
						)
						public class Employee {
						}
						
🔐 Cache Concurrency Strategies (VERY IMPORTANT)
| Strategy             | Use Case                               |
| -------------------- | -------------------------------------- |
| READ_ONLY            | Static data (countries, roles)         |
| READ_WRITE           | Frequently read + occasionally updated |
| NONSTRICT_READ_WRITE | Tolerate stale data                    |
| TRANSACTIONAL        | Full XA transaction support            |

	Example
	@Cache(usage = CacheConcurrencyStrategy.READ_ONLY)
	
📌 Query Cache (Separate)
		Enable: spring.jpa.properties.hibernate.cache.use_query_cache=true
		And use: query.setHint("org.hibernate.cacheable", true);
	
	⚠️ Query cache stores IDs, not full entities.

🚫 When Should We Avoid Second-Level Cache?
	
		❌ 1. Highly Volatile Data
			Example: Stock prices, Real-time order status, Payment transactions
			Frequent updates cause: Constant cache invalidation, Cache thrashing, Performance degradation
		
		❌ 2. Write-Heavy Systems
			If updates >> reads: Cache overhead becomes higher than DB cost, Locking overhead increases (READ_WRITE strategy)
		
		❌ 3. Large Dataset with Low Reuse
			If: Data rarely accessed twice, No repeated reads, Cache becomes memory waste.

		❌ 4. Distributed Microservices Without Shared Cache
			If each microservice has: Independent DB, No distributed cache, Then L2 cache gives little benefit.

		❌ 5. Strict Consistency Required
			If system requires: 100% real-time accuracy, No stale reads allowed, Then L2 cache might not be suitable.

🔥 Real Production Example
	Good candidate for L2 cache: Country list, Currency, User roles, Product categories, Configuration tables
	Bad candidate: Shopping cart, Payment status, Live bidding system, High-frequency trading data

🧠 Advanced Interview Insight
	Second-level cache is: 
		Not a replacement for database, Not always beneficial, Should be used selectively, Must be benchmarked
	
	Golden rule: Use L2 cache only when read-to-write ratio is high.

🎯 Memory & Cluster Consideration
	In cluster: Use distributed cache (Hazelcast / Redis)
	Otherwise each node has its own local cache → inconsistency
	
======================================================================================================
🔹 Q3. Difference between: READ_ONLY, NONSTRICT_READ_WRITE, READ_WRITE, TRANSACTIONAL
======================================================================================================
1️⃣ READ_ONLY
	Entity is never updated, No locking, No invalidation, Simple cache retrieval.
	Behavior: If you try to update → Hibernate throws exception.
		@Cache(usage = CacheConcurrencyStrategy.READ_ONLY)

	Best For: Country, Currency, Roles, Static reference tables
	🚀 Performance: Fastest strategy, No synchronization overhead
	❌ When NOT to use: If entity can change even once

2️⃣ NONSTRICT_READ_WRITE
	Allows updates, No strict locking, Cache invalidated after transaction commit, Temporary stale data possible
	Behavior: Update DB,  Cache entry removed (not locked), Small window where stale data may be read
		@Cache(usage = CacheConcurrencyStrategy.NONSTRICT_READ_WRITE)

	Best For: Data updated occasionally, Small risk of stale data acceptable
	Example: Product description, User profile info
	⚠️ Important: There is no lock during update. So: Thread A updates, Thread B may still read old cached value
	
	🚀 Performance: Faster than READ_WRITE, No locking overhead

3️⃣ READ_WRITE (Most Common in Enterprise)
	Uses soft locking mechanism, Prevents stale data, Strong consistency
	@Cache(usage = CacheConcurrencyStrategy.READ_WRITE)

	🔹 Internal Mechanism (Very Important)
		When update happens: Hibernate puts a soft lock in cache, Other transactions cannot read stale data
		After commit → cache updated, Lock removed, This prevents dirty reads.

	Best For: Moderate read-write systems, Financial systems, Enterprise apps
	Example: Employee, Orders, Customer

	Performance: Slower than NONSTRICT, Slight locking overhead

4️⃣ TRANSACTIONAL (Advanced)
	Full transactional consistency, Requires JTA + XA capable cache, Works with distributed transactions
	@Cache(usage = CacheConcurrencyStrategy.TRANSACTIONAL)
	
	🔹 Requires: JTA environment, Transaction-aware cache provider (Infinispan, JBoss Cache)
	Best For: Banking systems, Distributed systems, XA transactions

	Performance: Slowest, Heavy transaction coordination

🧠 Deep Comparison (Senior-Level)
| Feature             | READ_ONLY | NONSTRICT | READ_WRITE | TRANSACTIONAL    |
| ------------------- | --------- | --------- | ---------- | ---------------- |
| Updates Allowed     | ❌         | ✅       | ✅        | ✅               |
| Locking             | ❌         | ❌       | Soft Lock  | Real Transaction |
| Stale Data Possible | ❌         | ✅       | ❌         | ❌              |
| Cluster Safe        | Yes        | Risky     | Yes        | Yes              |
| Best Performance    | ⭐⭐⭐⭐  | ⭐⭐⭐   | ⭐⭐       | ⭐              |

🔥 Internal Update Flow Comparison
	READ_ONLY: Read → Cache → Done, Update → Exception
	NONSTRICT_READ_WRITE: Update DB, Remove cache entry, Next read → DB hit, Possible stale window.
	READ_WRITE: Put soft lock, Update DB, Update cache, Remove lock
	TRANSACTIONAL: Cache participates in transaction, Commit/rollback synchronized
	
🔥Interview-Ready Summary
	READ_ONLY is fastest but supports no updates.
	NONSTRICT_READ_WRITE allows updates but may serve stale data.
	READ_WRITE uses soft locking for strong consistency and is most common.
	TRANSACTIONAL provides full ACID support in JTA environments but has highest overhead.

🔹 Q4. What is Query Cache?
======================================================================================================
✅ Query Cache stores the result set of a query (mainly the list of entity IDs), so that Hibernate can skip executing the same query again.
	⚠️ Important: Query Cache does NOT store full entity data. It stores only identifiers (primary keys) and scalar values.

🔎 How It Works Internally
	When you execute:
						List<Employee> list = session
							.createQuery("from Employee where department = :dept")
							.setParameter("dept", "IT")
							.setHint("org.hibernate.cacheable", true)
							.getResultList();

	First Time: Query hits DB, Result IDs are stored in Query Cache, Entities are stored in Second-Level Cache (if enabled)
	Example stored:
			Query: from Employee where department='IT'
			Result IDs: [1, 5, 9, 15]

	Second Time: Hibernate checks Query Cache, Gets IDs from cache, Loads entities from Second-Level Cache
			If entity not in L2 → DB hit

	🏗 Architecture Flow
				Session
				  ↓
				First-Level Cache
				  ↓
				Query Cache (IDs only)
				  ↓
				Second-Level Cache (entity data)
				  ↓
				Database
				
⚙️ How to Enable Query Cache: Query cache depends on Second-Level Cache. You cannot use Query Cache alone.
	Step 1: Enable Second-Level Cache: spring.jpa.properties.hibernate.cache.use_second_level_cache=true
	
	Step 2: Enable Query Cache: spring.jpa.properties.hibernate.cache.use_query_cache=true
	
	Step 3: Mark Query as Cacheable: query.setHint("org.hibernate.cacheable", true);
			OR (Hibernate API): query.setCacheable(true);
			
📦 What Query Cache Stores
| Type              | Stored? |
| ----------------- | ------- |
| Entity IDs        | ✅ Yes   |
| Scalar results    | ✅ Yes   |
| Full entity state | ❌ No    |
| Collections       | ❌ No    |

🧠 Important Interview Point
	Query Cache works best when: Same query runs multiple times, Underlying data does not change frequently
		Second-Level Cache is enabled

🔥 Query Cache vs Second-Level Cache
| Feature                   | Query Cache       | Second-Level Cache |
| ------------------------- | ----------------- | ------------------ |
| Stores                    | Query result IDs  | Entity state       |
| Scope                     | SessionFactory    | SessionFactory     |
| Must enable L2 first?     | ✅ Yes   	        | —                  |
| Affected by data changes? | Yes (invalidated) | Yes                |


🔥 5️⃣ Transactions & Concurrency
======================================================================================================
🔹 Q1. Explain optimistic vs pessimistic locking ?
======================================================================================================
1️⃣ Optimistic Locking
	🔸 Core idea: Assume conflicts are rare. Don’t lock data while reading; check for conflicts at update time.
		✔ How it works: 
			Add a version field to the entity, Every update checks the version, If version changed → fail the update
			
			@Entity
			public class Product {
				@Id
				private Long id;
				private int quantity;

				@Version
				private int version;
			}
			
		✔ Pros : High performance, No database locks, Scales well, Ideal for read-heavy systems
		✔ Cons : ❌ Update may fail, ❌ Requires retry logic
		✔ Best use cases: High-read, low-write systems, Web apps, REST APIs, Microservices, User-driven updates
		
2️⃣ Pessimistic Locking
	🔸 Core idea: Assume conflicts are likely. Lock data upfront so no one else can modify it.
		✔ How it works:
			DB acquires a lock when data is read, Other transactions must wait.
			
			@Lock(LockModeType.PESSIMISTIC_WRITE)
			@Query("select p from Product p where p.id = :id")
			Product findForUpdate(@Param("id") Long id);
		
		✔ Pros : Strong consistency, No update conflicts, Predictable behavior
		✔ Cons : ❌ Lower concurrency, ❌ Risk of deadlocks, ❌ Slower under load
		✔ Best use cases: High-contention updates, Financial transactions, Inventory reservations, Critical sections

======================================================================================================
🔹 Q2. Difference between LockModeType options?
======================================================================================================
✅ Why Do We Need Lock Modes?
	Problem scenario: Two users update the same record at the same time:
		T1 reads salary = 10000
		T2 reads salary = 10000
		T1 updates to 12000
		T2 updates to 15000
		
		👉 Final salary = 15000, T1’s update lost (Lost Update Problem)
		Lock modes help prevent this.

🔥 LockModeType Options (JPA)
| Category    | Lock Mode                   |
| ----------- | --------------------------- |
| Optimistic  | OPTIMISTIC                  |
| Optimistic  | OPTIMISTIC_FORCE_INCREMENT  |
| Pessimistic | PESSIMISTIC_READ            |
| Pessimistic | PESSIMISTIC_WRITE           |
| Pessimistic | PESSIMISTIC_FORCE_INCREMENT |
| Legacy      | READ                        |
| Legacy      | WRITE                       |

🟢 1️⃣ OPTIMISTIC: 
		Uses @Version field, No DB lock, Version checked at commit time, Prevents lost updates
		entityManager.find(Employee.class, 1, LockModeType.OPTIMISTIC);
		How it works: 
				Read entity, Another transaction updates it
				On commit → version mismatch, Throws OptimisticLockException
			
		Best for: Read-heavy systems, Web applications, Low conflict probability

🟢 2️⃣ OPTIMISTIC_FORCE_INCREMENT
		Same as OPTIMISTIC, but: Increments version even if entity not modified.
		Useful when: You want to signal logical update, Prevent concurrent modifications

🔴 3️⃣ PESSIMISTIC_READ
		Places shared lock at DB level, Other transactions can read, But cannot update
			entityManager.find(Employee.class, 1, LockModeType.PESSIMISTIC_READ);
			
		SQL Generated (Example MySQL): SELECT * FROM employee WHERE id=1 LOCK IN SHARE MODE;
		
		Best for: Preventing concurrent updates, When you expect updates but want safe read

🔴 4️⃣ PESSIMISTIC_WRITE (Most Important)
		Places exclusive DB lock, Blocks both read (depending on isolation) and write Uses SELECT ... FOR UPDATE
			entityManager.find(Employee.class, 1, LockModeType.PESSIMISTIC_WRITE);
		SQL Example: SELECT * FROM employee WHERE id=1 FOR UPDATE;
		
		Behavior: Other transactions must wait, Prevents dirty read, Prevents lost update
		Best for: Banking systems, Inventory systems, Payment processing

🔴 5️⃣ PESSIMISTIC_FORCE_INCREMENT
	Same as PESSIMISTIC_WRITE, Also increments version column
	Used when: Both pessimistic locking and version increment required
	Rarely used in real projects.

⚪ 6️⃣ READ (Legacy): Equivalent to OPTIMISTIC (JPA 1.0 legacy).

⚪ 7️⃣ WRITE (Legacy): Equivalent to OPTIMISTIC_FORCE_INCREMENT.

🧠 Optimistic vs Pessimistic (Critical Interview Comparison)
| Feature          | Optimistic | Pessimistic           |
| ---------------- | ---------- | --------------------- |
| DB Lock          | ❌ No       | ✅ Yes                 |
| Version Required | ✅ Yes      | Optional              |
| Performance      | Better     | Slower                |
| Deadlock Risk    | Low        | High                  |
| Best For         | Read-heavy | High conflict systems |

🔹 Q3. How does Hibernate integrate with Spring transactions?
======================================================================================================
✅ Big Picture: Who Manages the Transaction?
	In modern apps:
		Spring → Manages transactions
		Hibernate → Participates in the transaction
		Database → Executes commit/rollback
	
	Hibernate does NOT manage transactions directly when used with Spring. Spring does.

🏗 Architecture Flow
						Controller
						   ↓
						Service (@Transactional)
						   ↓
						Spring Transaction Manager
						   ↓
						Hibernate Session (EntityManager)
						   ↓
						JDBC Connection
						   ↓
						Database
						
🔥 Core Components Involved
	1️⃣ @Transactional
		From: org.springframework.transaction.annotation.Transactional
		Applied on:
					@Service
					public class EmployeeService {

						@Transactional
						public void updateSalary(Long id) {
							Employee emp = repo.findById(id).get();
							emp.setSalary(100000);
						}
					}
					
	2️⃣ PlatformTransactionManager
		Spring uses: JpaTransactionManager When using JPA/Hibernate.
		If using plain Hibernate (no JPA), then: HibernateTransactionManager
		
🧠 What Happens Internally?
	When a @Transactional method is called:
		Step 1: Spring AOP Proxy Intercepts Call, Spring creates a proxy around your service.
				When method is called: Proxy checks if transaction exists, If not → starts new transaction

		Step 2: Transaction Manager Starts Transaction
				JpaTransactionManager: 
					Gets JDBC connection, Disables auto-commit, Binds connection to current thread
					Creates/obtains Hibernate Session, Binds Session to thread
					ThreadLocal is used internally.

		Step 3: Hibernate Joins Transaction
				Hibernate Session: Uses same JDBC connection, First-level cache created, Tracks dirty entities
		
		Step 4: Commit Phase
				When method completes successfully: 
					Spring calls EntityManager.flush(), Hibernate generates SQL, DB commit happens
					If exception occurs: Spring triggers rollback, Hibernate clears session

	🔎 Important Internal Mechanism		
		Spring uses: TransactionSynchronizationManager
		To bind: JDBC Connection, Hibernate Session, Transaction status, To the current thread.
		That’s why: Transactions are thread-bound, Sessions are thread-bound

🔥 What About OpenSessionInView?
	In Spring Boot: spring.jpa.open-in-view=true
	This keeps Hibernate session open until view rendering completes.
	Used to avoid: LazyInitializationException
	But in microservices, usually disabled.

	⚡ How Hibernate Flush Works With Spring
		By default: FlushMode.AUTO
		Hibernate flushes: Before commit, Before query execution (if needed)
		Spring triggers flush at commit time.

🔐 Propagation & Isolation
	Spring handles:
				@Transactional(
					propagation = Propagation.REQUIRED,
					isolation = Isolation.REPEATABLE_READ
				)
		Hibernate simply works within the transaction boundary provided by Spring.

🔥 Nested Transaction Example
			@Transactional
			public void methodA() {
				methodB();
			}

			@Transactional(propagation = REQUIRES_NEW)
			public void methodB() {
			}

		Spring: Suspends outer transaction, Starts new one, Commits/rolls back independently
		Hibernate Session behavior depends on transaction manager configuration.

🧠 Real Interview Deep Insight
Q: Does Hibernate manage transactions in Spring Boot?

Correct Answer: No. Spring manages transactions using PlatformTransactionManager. Hibernate participates in the transaction using the same JDBC connection bound to the current thread.

🚨 Common Production Issue
	❌ Calling @Transactional method internally
			public void outer() {
				inner();  // Transaction NOT applied
			}

			@Transactional
			public void inner() {
			}

		Because: Proxy not used, Self-invocation bypasses Spring AOP
		Fix: Move to another service, Or use AspectJ mode

⚙️ Configuration in Spring Boot
	Usually auto-configured. But behind the scenes:
		spring.jpa.properties.hibernate.transaction.coordinator_class=jta

	If using JTA. Otherwise default local transactions.

🔹 Q4. How to avoid stale object state exception?
======================================================================================================
✅ What is StaleObjectStateException?
	It occurs when: Two transactions modify the same entity
		Entity has @Version
	
	Version in DB ≠ Version in memory at commit time
	Hibernate detects: “Someone else modified this row before you committed.”

🔎 Example Scenario:
					@Entity
					public class Employee {

						@Id
						private Long id;

						@Version
						private Long version;

						private double salary;
					}

	Timeline:
		T1 reads Employee(id=1, version=1)
		T2 reads Employee(id=1, version=1)

		T1 updates salary → version becomes 2
		T2 tries to update salary → fails

		T2 gets: org.hibernate.StaleObjectStateException
		
🔥 Why It Happens Internally
	Hibernate generates update like:
		UPDATE employee SET salary=?, version=? WHERE id=? AND version=?
	If version doesn’t match: 0 rows updated, Hibernate throws exception

🎯 How to Avoid StaleObjectStateException: There are multiple strategies depending on your system design.
	
	🟢 1️⃣ Implement Retry Mechanism (Most Common Solution): Best practice in web apps.

				public void updateWithRetry(Long id) {
					int retries = 3;
					while (retries > 0) {
						try {
							updateEmployee(id);
							return;
						} catch (ObjectOptimisticLockingFailureException e) {
							retries--;
						}
					}
					throw new RuntimeException("Failed after retries");
				}
		
		Why? Because optimistic locking is expected to fail sometimes.

	🟢 2️⃣ Use PESSIMISTIC Locking (When Conflict Is High)
			If conflict probability is high:
				entityManager.find(
					Employee.class,
					1L,
					LockModeType.PESSIMISTIC_WRITE
				);
		This prevents concurrent modification instead of detecting it later.
		Use when: Inventory systems, Banking systems, High contention rows

	🟢 3️⃣ Keep Transactions Short
		Long transactions increase probability of version conflict.
		❌ Bad:
			@Transactional
			public void longProcess() {
				Employee emp = repo.findById(id).get();
				Thread.sleep(5000); // risky
				emp.setSalary(100000);
			}
		Long DB-held entity → higher conflict chance.

	🟢 4️⃣ Don’t Manually Change Version Field
		Never do this: emp.setVersion(10L); // WRONG
		Hibernate manages version automatically.

	🟢 5️⃣ Always Reload Latest Entity Before Update (DTO Pattern)
		Common mistake:
			Frontend sends old object: repo.save(oldDetachedObject);
			Better approach:
				Employee managed = repo.findById(dto.getId()).get();
				managed.setSalary(dto.getSalary());
			Always update managed entity, not detached stale object.

	🟢 6️⃣ Handle Merge Carefully
		Using: entityManager.merge(detachedEntity);
		If detached entity has stale version → exception.
		Ensure: Entity is freshly fetched Or version matches DB
	
	🟢 7️⃣ Avoid Concurrent Batch Updates Without Locking
		In batch jobs: Use pagination, Use pessimistic locks, Or disable versioning if not needed

🔥 Real Production Scenario
	We had: Order status update API, Two users updated same order simultaneously, UI showed generic 500 error
	Fix: Caught OptimisticLockException
	Returned proper business message: "This record was modified by another user. Please refresh and try again."
	Better UX.

🧠 When NOT to Avoid It
	Important insight: StaleObjectStateException is not a bug.
	It is: A safety mechanism.
	If you disable versioning, you risk lost updates.

🔴 Dangerous “Solution” (Don’t Do This)
	Removing @Version annotation to avoid exception.
	That causes silent data overwrite.
	Never do this in real systems.

⚖️ Optimistic vs Pessimistic Strategy
| Situation        | Recommendation     |
| ---------------- | ------------------ |
| Low conflict     | Optimistic + retry |
| High conflict    | Pessimistic write  |
| Financial system | Pessimistic        |
| Web CRUD app     | Optimistic         |

🔥 6️⃣ Performance Optimization (High Probability)
======================================================================================================
🔹 Q1. How to improve Hibernate performance?
======================================================================================================
🚀 1️⃣ Solve N+1 Problem (Most Common Issue)
		Problem: List<Employee> list = repo.findAll();
		If Employee has LAZY @OneToMany, accessing it triggers extra queries.
		1 query → fetch employees, N queries → fetch collections
		
		✅ Solution: JOIN FETCH
			@Query("select e from Employee e join fetch e.department")
			List<Employee> findAllWithDept();
		Or use:
			@EntityGraph(attributePaths = {"department"})
			
		✅ Alternative: Batch Fetching
			hibernate.default_batch_fetch_size=20
			This reduces N+1 to batched queries.

🚀 2️⃣ Use Proper Fetch Strategy
	❌ Avoid EAGER by default: EAGER loads data always — even when not needed.
		@ManyToOne(fetch = FetchType.LAZY), Use LAZY everywhere unless required.

🚀 3️⃣ Enable JDBC Batching (Huge Improvement)
	For bulk inserts/updates:
		spring.jpa.properties.hibernate.jdbc.batch_size=50
		spring.jpa.properties.hibernate.order_inserts=true
		spring.jpa.properties.hibernate.order_updates=true

	Without batching: 1000 inserts → 1000 round trips
	With batching: 1000 inserts → 20 round trips
	
	⚠️ Important For batching to work: Use GenerationType.SEQUENCE, Avoid IDENTITY (breaks batching)

🚀 4️⃣ Use StatelessSession for Bulk Operations
	If doing massive insert/update: StatelessSession session = sessionFactory.openStatelessSession();
	It: Skips first-level cache, No dirty checking, Faster for batch jobs

🚀 5️⃣ Optimize Flush Mode
	Default: FlushMode.AUTO, Hibernate flushes before query execution sometimes.
	For read-only: @Transaction(readOnly = true) Or: entityManager.setFlushMode(FlushModeType.COMMIT);
	Reduces unnecessary flush.

🚀 6️⃣ Use Second-Level Cache Carefully
	Use L2 cache for: Lookup tables, Read-heavy entities, Reference data
	Avoid for: Highly volatile data, Write-heavy systems

🚀 7️⃣ Use Query Projections Instead of Full Entities
	Instead of: select e from Employee e
	Use: select new com.dto.EmployeeDTO(e.id, e.name) from Employee e
	Or Spring projection:
			interface EmployeeView {
				String getName();
			}

	Reduces: Memory usage, Dirty checking, Unnecessary columns fetch

🚀 8️⃣ Avoid Open Session In View (OSIV) in High Load
	spring.jpa.open-in-view=false
	OSIV can: Keep connection open longer, Increase DB connection usage, Better to fetch inside service layer.

🚀 9️⃣ Index Database Properly
	Hibernate cannot fix: Missing DB indexes, Poor query plans
	Always: Index foreign keys, Index frequently filtered columns

🚀 🔟 Monitor SQL
	Enable:
		hibernate.show_sql=true
		hibernate.format_sql=true
		hibernate.generate_statistics=true
	
	Or use: p6spy, datasource-proxy, APM tools, Most performance issues are visible in SQL logs.

🚀 1️⃣1️⃣ Avoid Large First-Level Cache
	In batch processing:
			for (int i = 0; i < 10000; i++) {
				session.persist(emp);
				if (i % 50 == 0) {
					session.flush();
					session.clear();
				}
			}

	Prevents: Memory leak, Huge persistence context

🚀 1️⃣2️⃣ Use Proper Pagination
	Never do: repo.findAll();
	Instead: PageRequest.of(page, size);
	Avoid loading huge datasets into memory.

🚀 1️⃣3️⃣ Use Database-Level Optimizations
	Connection pool tuning (HikariCP), Proper isolation level, Query plan optimization
	Hibernate performance = 50% ORM + 50% DB tuning.

🔹 Q2. What is JDBC batching?
======================================================================================================
✅ JDBC batching is a technique where multiple SQL statements are grouped together and sent to the database in a single network round trip.
	Instead of executing: Insert 1 → DB call, Insert 2 → DB call, Insert 3 → DB call
	We send: Insert 1, Insert 2, Insert 3, → Single DB round trip
	This reduces: Network overhead, DB round trips, Execution time

🔥 Without Batching
	If you insert 1000 records: 1000 network calls, 1000 statement executions, Slow ❌

🚀 With Batching
	If batch size = 50: 1000 records / 50 = 20 DB calls, Much faster ✅

🧠 How It Works at JDBC Level
	Using plain JDBC:
		PreparedStatement ps = connection.prepareStatement(sql);
		for (int i = 0; i < 100; i++) {
				ps.setString(1, "Name" + i);
				ps.addBatch();
			}
		ps.executeBatch();

		addBatch() → Adds SQL to buffer
		executeBatch() → Executes all at once

🔥 How Hibernate Uses JDBC Batching
	Hibernate can automatically batch: INSERT, UPDATE, DELETE, When configured properly.

⚙️ How to Enable in Spring Boot
	spring.jpa.properties.hibernate.jdbc.batch_size=50
	spring.jpa.properties.hibernate.order_inserts=true
	spring.jpa.properties.hibernate.order_updates=true
	
🧠 Important: Why order_inserts?
	Hibernate groups similar SQL statements together.
	Example:
		Without ordering:  Cannot batch properly.
				Insert Employee
				Insert Department
				Insert Employee
				Insert Department

		With ordering: Now batching works.
				Insert Employee
				Insert Employee
				Insert Department
				Insert Department

🚨 Critical Interview Point
	❌ Batching DOES NOT work with IDENTITY strategy
	If using: @GeneratedValue(strategy = GenerationType.IDENTITY)
	Hibernate must execute insert immediately to get generated ID.
	That breaks batching.

	✅ Use SEQUENCE instead
		@GeneratedValue(strategy = GenerationType.SEQUENCE)
		Sequence allows pre-allocation of IDs → batching works.

		🔥 Example in Hibernate
		for (int i = 0; i < 1000; i++) {
			Employee emp = new Employee();
			emp.setName("Emp" + i);
			entityManager.persist(emp);
		}
	With batch_size=50: Hibernate sends inserts in groups of 50.
	⚠️ Important: Flush & Clear in Large Batches, Hibernate stores all entities in first-level cache.
	So do this:
			for (int i = 0; i < 10000; i++) {
				entityManager.persist(emp);

				if (i % 50 == 0) {
					entityManager.flush();
					entityManager.clear();
				}
			}

		Prevents: OutOfMemoryError, Huge persistence context

🧠 When to Use JDBC Batching
	✔ Bulk inserts, Bulk updates, ETL jobs, Data migration, Batch processing

🚫 When Not Useful
	❌ Single record insert, Highly dynamic SQL, Small datasets

🔥 Real Production Case
	We had: 10,000 record insert, Took 18 seconds
	After enabling batching (size=50): Reduced to 2.5 seconds, Huge performance improvement.

🧠 Difference: JDBC Batching vs Hibernate Batch Fetching
| Feature  | JDBC Batching         | Batch Fetching           |
| -------- | --------------------- | ------------------------ |
| Works on | INSERT/UPDATE         | SELECT                   |
| Purpose  | Reduce DB round trips | Reduce N+1               |
| Config   | jdbc.batch_size       | default_batch_fetch_size |

🚀 Advanced Tip: Versioned Data
	If entity has @Version:
		Add: hibernate.jdbc.batch_versioned_data=true
	Otherwise versioned entities won’t batch updates.

🔹 Q5. Difference between: saveAll(), saveAllAndFlush()
======================================================================================================
✅ 1️⃣ saveAll()
	What it does: Persists a collection of entities, Does NOT immediately flush to DB
	SQL execution happens at: Transaction commit Or when flush is triggered automatically
	
	Example: repo.saveAll(employeeList);
	
	Behind the scenes: 
		Calls EntityManager.persist() or merge(), Entities become managed, Stored in First-Level Cache
		SQL not necessarily executed immediately
	
	When SQL Executes? If inside:
			@Transactional
			public void insert() {
				repo.saveAll(list);
				// SQL may not run yet
			}
	SQL executes at: Transaction commit time
	Because Hibernate uses: FlushMode.AUTO
	
✅ 2️⃣ saveAllAndFlush()
	What it does: Calls saveAll() Then immediately calls flush()
	Equivalent to:
			repo.saveAll(list);
			repo.flush();
	Example: repo.saveAllAndFlush(employeeList);
	This: Persists entities, Immediately executes SQL, Synchronizes persistence context with DB

🔥 Core Difference
| Feature           | saveAll()         | saveAllAndFlush()               |
| ----------------- | ----------------- | ------------------------------- |
| Persists entities | ✅                 | ✅                               |
| Immediate DB hit  | ❌                 | ✅                               |
| Flush triggered   | At commit         | Immediately                     |
| Use case          | Normal batch save | When DB sync required instantly |

🔥 When Should You Use saveAllAndFlush()?
	✅ 1. When You Need DB Constraint Validation Immediately
		Example: Unique constraint violation, Foreign key validation
		If you use only saveAll(): Exception may occur at commit time
		If you use saveAllAndFlush(): Exception thrown immediately

	✅ 2. When You Need DB-Generated Values Immediately
		Example: Trigger-generated values, DB-calculated columns

❌ When NOT to Use saveAllAndFlush()
	During bulk insert: repo.saveAllAndFlush(hugeList);
	This: Forces immediate flush, Breaks batching, Reduces performance
	Better: repo.saveAll(list); Let transaction commit handle flush

🔹 Q6. Why bulk update via JPQL is faster?
======================================================================================================
✅ What Is Bulk Update via JPQL?
	Example:
		@Modifying
		@Query("UPDATE Employee e SET e.salary = e.salary + 1000 WHERE e.department.id = :deptId")
		int increaseSalary(@Param("deptId") Long deptId);
		
	This is a bulk update query.
	Hibernate translates it directly into: UPDATE employee SET salary = salary + 1000 WHERE department_id = ?
	Single SQL statement.

🔥 Why Is It Faster?
	Because it bypasses the ORM lifecycle and persistence context. Let’s compare.

🟥 Normal Update (Entity-Based)
	Example: List<Employee> list = repo.findByDepartmentId(10L);
		for (Employee e : list) {
			e.setSalary(e.getSalary() + 1000);
		}
	What Happens Internally? 
		SELECT query loads all employees, Entities stored in First-Level Cache
		Hibernate creates snapshots for dirty checking
		On commit → compares snapshots
		Generates N UPDATE statements
		If 1000 rows:
			1 SELECT
			1000 UPDATE statements
		
		Also: Memory consumption increases, Dirty checking overhead, More round trips, Slow ❌

🟢 Bulk JPQL Update
			@Modifying
			@Query("UPDATE Employee e SET e.salary = e.salary + 1000 WHERE e.department.id = :deptId")

	What Happens Internally?
		Hibernate sends single SQL UPDATE, DB updates all matching rows, No entities loaded, No dirty checking
		No first-level cache usage
		If 1000 rows: 1 UPDATE statement, Fast ✅

🔥 Core Reason It’s Faster
| Factor                   | Normal Update | JPQL Bulk Update |
| ------------------------ | ------------- | ---------------- |
| Entities loaded?         | Yes           | No               |
| Dirty checking?          | Yes           | No               |
| Memory usage             | High          | Low              |
| Number of SQL statements | N             | 1                |
| Round trips              | Many          | 1                |

🚨 Important Side Effect (Interview Trap)
	After bulk update: Persistence context becomes inconsistent.
	Because: Entities in memory are NOT updated, DB is updated directly
	Example: 
		employeeRepository.bulkUpdate(...);
		Employee emp = entityManager.find(Employee.class, 1L);
	
	If entity was already loaded before bulk update → stale value.

✅ Best Practice After Bulk Update
	Call: entityManager.clear(); Or use: @Modifying(clearAutomatically = true)
	Example:
			@Modifying(clearAutomatically = true)
			@Query("UPDATE Employee e SET e.salary = e.salary + 1000")

🧠 When Should You Use Bulk Update?
	✔ Large dataset, No need for entity listeners, No cascading required, No per-entity business logic

❌ When NOT to Use Bulk Update
	If you rely on: @PreUpdate, @PostUpdate, Business logic inside entity, Cascade updates
	Version-based optimistic locking (unless handled properly)
	Bulk update does NOT trigger entity lifecycle callbacks.

🔹 Q7. How to handle large result sets efficiently?
======================================================================================================
🚨 The Problem
	If you do: List<Employee> list = repo.findAll();
	Hibernate: Loads all rows, Stores them in First-Level Cache, Creates snapshots, Keeps them managed
	For 500k rows → 💥 Huge memory usage.

✅ 1️⃣ Use Pagination (Most Common Solution)
		Instead of loading everything:
			Page<Employee> page = repo.findAll(PageRequest.of(pageNo, pageSize));
		Example: PageRequest.of(0, 100);
		This: Loads only 100 rows at a time, Reduces memory footprint, Improves responsiveness
		🔥 For APIs
			Always implement: limit, offset, page, size
		Never return full dataset in REST API.

✅ 2️⃣ Use Slice Instead of Page (Better Performance)
	If total count not needed: Slice<Employee> slice = repo.findAll(PageRequest.of(0, 100));
	Why better? Page runs extra: SELECT COUNT(*), Slice does not. Less DB load.

✅ 3️⃣ Use Streaming (Best for Processing Large Data)
	Spring Data JPA supports streaming:

		@Query("SELECT e FROM Employee e")
		Stream<Employee> streamAll();

	Usage:
		@Transactional(readOnly = true)
		public void process() {
			try (Stream<Employee> stream = repo.streamAll()) {
				stream.forEach(this::processEmployee);
			}
		}
		
	Why this is good: Results fetched gradually, Doesn’t load entire dataset at once, Lower memory footprint
	⚠ Must keep transaction open.

✅ 4️⃣ Use ScrollableResults (Hibernate Specific)
	Session session = entityManager.unwrap(Session.class);
	ScrollableResults results = session.createQuery("FROM Employee").scroll(ScrollMode.FORWARD_ONLY);
	while (results.next()) {
		Employee emp = (Employee) results.get(0);
		process(emp);

		session.clear(); // Prevent memory buildup
	}
	Best for: Batch processing, Data migration, ETL

✅ 5️⃣ Use Fetch Size (Very Important)
	Tell JDBC to fetch rows in batches: spring.jpa.properties.hibernate.jdbc.fetch_size=100
	Or: query.setHint("org.hibernate.fetchSize", 100);
	This avoids: Loading entire result set into memory, Improves DB cursor handling

✅ 6️⃣ Use DTO Projections (Huge Memory Saver)
	Instead of: SELECT e FROM Employee e
	Use: SELECT new com.dto.EmployeeDTO(e.id, e.name) FROM Employee e
	Benefits: No entity management, No dirty checking, Less memory, Faster, Best for read-only operations.

✅ 7️⃣ Clear Persistence Context Periodically
	If processing in loop: 
		int count = 0;
		for (Employee emp : largeList) {
			process(emp);
			count++;
			if (count % 100 == 0) {
				entityManager.clear();
			}
		}
	Prevents First-Level Cache growth.

✅ 8️⃣ Use Native Query with Streaming (Extreme Case)
	For millions of rows:
		entityManager
			.createNativeQuery("SELECT id, name FROM employee")
			.unwrap(org.hibernate.query.Query.class)
			.setFetchSize(100);
	Or even use plain JDBC for heavy ETL jobs.

🔥 What NOT To Do
	❌ findAll() for huge tables, Return large list in REST response, Keep transaction open too long
	❌ Forget to clear persistence context, Use EAGER fetch

🧠 Real Production Scenario
	We had: 2 million records export API
	Used findAll(); Memory jumped to 5GB, App crashed
	Fix: Switched to streaming, Set fetch size = 500, Wrote directly to CSV while streaming
	Memory stayed stable.

🚀 Best Strategy by Use Case
| Scenario            | Recommended               |
| ------------------- | ------------------------- |
| REST API pagination | Page / Slice              |
| Data processing job | Streaming                 |
| Batch ETL           | ScrollableResults         |
| Report generation   | DTO projection            |
| Millions of rows    | Native query + fetch size |

🔥 7️⃣ Advanced Mapping
======================================================================================================
🔹 Q1. Difference between: @OneToOne, @OneToMany, @ManyToOne, @ManyToMany
======================================================================================================
🔵 1️⃣ @OneToOne: One entity instance is associated with exactly one instance of another entity.
		Example: User ↔ Profile, Person ↔ Passport
		
		📌 Example
				@Entity
				public class User {

					@Id
					private Long id;

					@OneToOne
					@JoinColumn(name = "profile_id")
					private Profile profile;
				}
	🗄 Database Structure
		USER      			PROFILE
		-----      			--------
		id (PK)   			id (PK)
		profile_id (FK)

	🔹 Key Points: Usually implemented using foreign key
		Can be: Unidirectional, Bidirectional
		One side is owning (has @JoinColumn), Other side uses mappedBy
		⚠ Performance Note: Default fetch = EAGER (dangerous)
		Always prefer: @OneToOne(fetch = FetchType.LAZY)

🟢 2️⃣ @ManyToOne: Many entities refer to one entity.
		Example: Many Employees → One Department, Many Orders → One Customer
		
		📌 Example
				@Entity
				public class Employee {

					@ManyToOne
					@JoinColumn(name = "department_id")
					private Department department;
				}

	🗄 Database Structure
		EMPLOYEE               DEPARTMENT
		---------             -----------
		id (PK)               id (PK)
		department_id (FK)

	🔹 Key Points: Foreign key stored on "many" side, Most common relationship
		Default fetch = EAGER (in JPA)
		Best practice: @ManyToOne(fetch = FetchType.LAZY)

🟡 3️⃣ @OneToMany: One entity has multiple related entities.
		Example: Department → Many Employees, Order → Many OrderItems
		
		📌 Example (Bidirectional)
					@Entity
					public class Department {

						@OneToMany(mappedBy = "department")
						private List<Employee> employees;
					}
		
		🗄 Database Structure: Foreign key still stored in Employee table:

			EMPLOYEE
			---------
			id
			department_id (FK)

		🔹 Important
			@OneToMany is usually inverse side.
			Owning side is: @ManyToOne
			
		⚠ Performance Warning
				@OneToMany can cause: N+1 problem, Huge collections in memory
				Prefer: LAZY loading, Pagination on child side

🔴 4️⃣ @ManyToMany: Many entities relate to many entities.
		Example: Students ↔ Courses, Users ↔ Roles
		
		📌 Example
				@Entity
				public class Student {

					@ManyToMany
					@JoinTable(
						name = "student_course",
						joinColumns = @JoinColumn(name = "student_id"),
						inverseJoinColumns = @JoinColumn(name = "course_id")
					)
					private Set<Course> courses;
				}
				
		🗄 Database Structure
			STUDENT    COURSE     STUDENT_COURSE
			--------   --------   ---------------
			id          id        student_id (FK)
								  course_id (FK)

		🔹 Key Points: Requires join table, No foreign key in main tables, One side is owning (defines @JoinTable)
		⚠ Performance & Design Warning
				Many-to-many can: Cause complex joins, Be hard to maintain, Create cascading issues
				Best practice: Often replace with explicit entity: StudentCourse
				If relationship has extra fields (like enrollment date).

🔥 Quick Comparison Table
| Annotation  | Cardinality | Foreign Key Location | Join Table Required? |
| ----------- | ----------- | -------------------- | -------------------- |
| @OneToOne   | 1 ↔ 1       | One side             | ❌   	              |
| @ManyToOne  | N → 1       | Many side            | ❌   	              |
| @OneToMany  | 1 → N       | Stored in Many side  | ❌  	              |
| @ManyToMany | N ↔ N       | Join table           | ✅   	              |

🔥 Real Production Advice
	Prefer: @ManyToOne + @OneToMany, Explicit join entity instead of ManyToMany, LAZY fetch always
	Avoid: EAGER fetch on collections, Huge bidirectional relationships without need

======================================================================================================
🔹 Q2. Owning side vs inverse side?
======================================================================================================
✅ In a bidirectional relationship, JPA requires:
	One side must be the Owning Side
	The other side becomes the Inverse Side

🔥 Core Rule (Very Important)
	✅The Owning Side is the side that controls the relationship in the database.
	It is the side that: Contains @JoinColumn Or defines @JoinTable, Updates the foreign key in DB
	
	✅The Inverse Side: Uses mappedBy, Does NOT update the relationship in DB

🧠 Example 1: @OneToMany / @ManyToOne
	Employee (Many side)
				@Entity
				public class Employee {

					@ManyToOne
					@JoinColumn(name = "department_id")
					private Department department;
				}
	
	Department (One side)
				@Entity
				public class Department {

					@OneToMany(mappedBy = "department")
					private List<Employee> employees;
				}
	🔎 Who is Owning Side? 👉 Employee (ManyToOne side)
		Because it has: @JoinColumn, It holds the foreign key.

	🔎 Who is Inverse Side? 👉 Department
		Because it has: mappedBy = "department"
		
	🔥 Very Important Behavior
		If you do: department.getEmployees().add(emp);
		But you don’t set: emp.setDepartment(department);
		👉 Relationship will NOT be saved in DB. Because only Owning Side updates the foreign key.
	
	🧠 Golden Rule: Always update both sides in bidirectional mapping.
		Example helper method:
					public void addEmployee(Employee emp) {
						employees.add(emp);
						emp.setDepartment(this);
					}
					
🧠 Example 2: @OneToOne
		User (Owning Side):
					@OneToOne
					@JoinColumn(name = "profile_id")
					private Profile profile;
					
		Profile (Inverse Side):
					@OneToOne(mappedBy = "profile")
					private User user;

		Owning side = side with @JoinColumn.

🧠 Example 3: @ManyToMany
		Student (Owning Side):
						@ManyToMany
						@JoinTable(
							name = "student_course",
							joinColumns = @JoinColumn(name = "student_id"),
							inverseJoinColumns = @JoinColumn(name = "course_id")
						)
						private Set<Course> courses;
		
		
		Course (Inverse Side):
						@ManyToMany(mappedBy = "courses")
						private Set<Student> students;

		Owning side = side with @JoinTable.

🔥 Why Does JPA Need This Concept?
	To avoid ambiguity. If both sides tried to update relationship: Conflict, Duplicate updates, Inconsistent foreign keys
	So JPA says: Only one side controls DB relationship.

📊 Quick Comparison Table
| Feature                   | Owning Side | Inverse Side |
| ------------------------- | ----------- | ------------ |
| Has `@JoinColumn`         | ✅ Yes 	  | ❌ No 	     |
| Has `mappedBy`            | ❌ No 	  | ✅ Yes 	     |
| Updates foreign key       | ✅ Yes	  | ❌ No  	     |
| Required in bidirectional | Yes  	      | Yes   	     |

🔹 Q3. What is cascading?
======================================================================================================
✅ Cascading means an operation performed on a parent entity is automatically applied to its related child entities.
	Instead of manually saving/updating/deleting child entities, Hibernate does it automatically.

🧠 Why Do We Need Cascading?
	Without cascading:
		Department dept = new Department();
		Employee emp = new Employee();
		dept.getEmployees().add(emp);
		entityManager.persist(dept);
	
		👉 Only Department is saved. Employee is NOT saved (unless cascade is configured).

	With cascading:
			@OneToMany(mappedBy = "department", cascade = CascadeType.PERSIST)
			private List<Employee> employees;

	Now: entityManager.persist(dept);
	👉 Both Department and Employee are saved.

🔥 Cascade Types in JPA
	| Cascade Type | Meaning                             |
	| ------------ | ----------------------------------- |
	| PERSIST      | Save child when parent saved        |
	| MERGE        | Update child when parent updated    |
	| REMOVE       | Delete child when parent deleted    |
	| REFRESH      | Refresh child when parent refreshed |
	| DETACH       | Detach child when parent detached   |
	| ALL          | All of the above                    |

	🟢 1️⃣ CascadeType.PERSIST: When: entityManager.persist(parent); Children are also persisted.
	🟢 2️⃣ CascadeType.MERGE: When: entityManager.merge(parent); Children are also merged (updated).
	🔴 3️⃣ CascadeType.REMOVE: When: entityManager.remove(parent); Children are automatically deleted.
		⚠ Be careful — this can delete a lot of data.
	
	🟢 4️⃣ CascadeType.REFRESH; When: entityManager.refresh(parent); Children are refreshed from DB.
	🟢 5️⃣ CascadeType.DETACH: When: entityManager.detach(parent); Children are detached too.
	🟢 6️⃣ CascadeType.ALL: 
		Shortcut for:
			cascade = {
				PERSIST, MERGE, REMOVE, REFRESH, DETACH
			}

	🔥 Example: Parent–Child Relationship
					@Entity
					public class Order {

						@OneToMany(mappedBy = "order", cascade = CascadeType.ALL)
						private List<OrderItem> items;
					}

			Now: entityManager.persist(order); Automatically saves: Order, All OrderItems
	🚨 Important: Cascade ≠ Relationship Ownership
		Cascade and owning side are different concepts.
		Owning side → controls DB foreign key, Cascade → controls operation propagation, They are independent.

🔥 Real Production Advice: Use cascade carefully
	Good use case: Order → OrderItems, Child depends completely on parent.
	Dangerous use case: Employee → Department, If you use: cascade = CascadeType.REMOVE
	Deleting employee might delete department. 💥 Disaster.

✅ What is orphanRemoval?
	orphanRemoval = true means: If a child entity is removed from the parent’s collection, Hibernate will automatically delete that child from the database.
	It works when: Parent-child relationship exists, Child depends fully on parent, Child cannot exist independently

	🔥 Example: Parent: Order, Child: OrderItem
				@Entity
				public class Order {

					@OneToMany(mappedBy = "order",
							   cascade = CascadeType.ALL,
							   orphanRemoval = true)
					private List<OrderItem> items = new ArrayList<>();
				}
				
	🧠 What Happens?
		If you do: order.getItems().remove(item);
		Hibernate will: DELETE FROM order_item WHERE id = ?
		Even if you don’t explicitly call: entityManager.remove(item);
	
	🔥 Why Is It Called "Orphan"?
		Because the child becomes: No longer referenced by parent
		It becomes an “orphan”. And Hibernate deletes it.

	🧠 Example to Understand Difference
		Case 1: orphanRemoval = true
				order.getItems().remove(item);
			
			Result: Only that item deleted
		
		Case 2: CascadeType.REMOVE
				entityManager.remove(order);
				
			Result: Order deleted, All order items deleted

	🔥 Important Rule
			orphanRemoval works only on: @OneToMany, @OneToOne
			Not on: @ManyToMany Because ManyToMany children are shared.

📊 Summary Table
| Action on Parent | Without Cascade     | With Cascade     |
| ---------------- | ------------------- | ---------------- |
| persist()        | Only parent saved   | Children saved   |
| merge()          | Only parent merged  | Children merged  |
| remove()         | Only parent deleted | Children deleted |

🔹 Q4. Embedded vs Embeddable?
======================================================================================================
✅ Big Picture
| Annotation    | Used On | Purpose                               |
| ------------- | ------- | ------------------------------------- |
| `@Embeddable` | Class   | Defines a reusable value type         |
| `@Embedded`   | Field   | Embeds that value type into an entity |

👉 One defines the type, The other uses the type

🟢 1️⃣ @Embeddable: Marks a class as a value object that can be embedded inside an entity.
	It: Does NOT have its own table, Has no identity, Shares parent table, Cannot exist independently

	📌 Example:
				@Embeddable
				public class Address {

					private String street;
					private String city;
					private String zipCode;
				}

		This class: Is NOT an entity, Has no @Id, No separate table

🟢 2️⃣ @Embedded: Used inside an entity to embed an @Embeddable class.
	
	📌 Example:
				@Entity
				public class Employee {

					@Id
					private Long id;
					private String name;

					@Embedded
					private Address address;
				}

	🗄 Database Structure: No separate Address table. Columns are merged into parent table.
		EMPLOYEE
		---------
		id
		name
		street
		city
		zip_code
	
	
🔥 Multiple Embedded Objects
	If same embeddable used twice:
					@Embedded
					private Address homeAddress;

					@Embedded
					private Address officeAddress;

	Problem: Column name conflict
	
	✅ Solution: @AttributeOverrides
			@Embedded
			@AttributeOverrides({
				@AttributeOverride(name="street", column=@Column(name="home_street")),
				@AttributeOverride(name="city", column=@Column(name="home_city"))
			})
			private Address homeAddress;

🔥 Real-World Use Cases: Good candidates for @Embeddable: 
		Address, Audit fields (createdAt, updatedAt), Money (amount + currency), GeoLocation (lat + long)

🧠 Example: Audit Fields (Common Pattern)
		@Embeddable
		public class Audit {
			private LocalDateTime createdAt;
			private LocalDateTime updatedAt;
		}
	
	Used in multiple entities:
		@Embedded
		private Audit audit;
	Reusable and clean.

🔹 Q5. Inheritance mapping strategies: SINGLE_TABLE, JOINED, TABLE_PER_CLASS
======================================================================================================
🔥 When you have an object-oriented hierarchy like:
			Vehicle
			 ├── Car
			 ├── Bike
			 └── Truck
	Hibernate must decide: How do we map this inheritance hierarchy into relational tables?
	JPA provides 3 strategies: SINGLE_TABLE, JOINED, TABLE_PER_CLASS

🟢 1️⃣ SINGLE_TABLE (Default Strategy)
	@Inheritance(strategy = InheritanceType.SINGLE_TABLE)
	@DiscriminatorColumn(name = "vehicle_type")

	✅ How It Works: Entire hierarchy stored in one single table.
	🗄 Database Structure
		VEHICLE
		---------
		id (PK)
		vehicle_type   ← discriminator
		model
		speed
		car_doors
		bike_type
		truck_capacity
	All subclass fields are in same table.
	Unused columns = NULL.

	🔎 Example
			@Entity
			@Inheritance(strategy = InheritanceType.SINGLE_TABLE)
			@DiscriminatorColumn(name = "type")
			public abstract class Vehicle { }

			@Entity
			@DiscriminatorValue("CAR")
			public class Car extends Vehicle {
				private int doors;
			}

	🔥 Advantages: Best performance (no joins), Simple queries, Fast polymorphic queries, No table joins required
	❌ Disadvantages: Many nullable columns, Schema can become messy, Harder to enforce NOT NULL constraints

	🚀 Best For: Large hierarchy, Performance-critical systems, When subclasses share many fields

2️⃣ JOINED: @Inheritance(strategy = InheritanceType.JOINED)
	
	✅ How It Works: One table for parent, Separate table for each subclass, Joined using primary key
	
	🗄 Database Structure
		VEHICLE    CAR                        BIKE
		--------- ---------                  ---------
		id (PK)   id (PK, FK → VEHICLE.id)   id (PK, FK → VEHICLE.id)
		model     doors                      bike_type
		speed

	🔎 Query Example
		When loading Car: SELECT * FROM VEHICLE v JOIN CAR c ON v.id = c.id
	
	🔥 Advantages: Clean schema, No nullable columns, Better normalization, Proper DB constraints possible
	❌ Disadvantages: Join required, Slightly slower reads, More complex SQL
	
	🚀 Best For: Clean database design required, Moderate hierarchy depth, Enterprise systems

🔴 3️⃣ TABLE_PER_CLASS: @Inheritance(strategy = InheritanceType.TABLE_PER_CLASS)
	
	✅ How It Works: Each subclass has its own table, Parent has no dedicated table
	
	🗄 Database Structure: No VEHICLE table.
		CAR         BIKE
		---------   ---------
		id          id
		model       model
		speed       speed
		doors       bike_type

	🔎 Polymorphic Query: SELECT v FROM Vehicle v
	Hibernate generates: SELECT ... FROM CAR UNION SELECT ... FROM BIKE
	
	🔥 Advantages: No joins, Fully independent tables
	❌ Disadvantages: UNION queries (slow), Duplicate columns, Hard to maintain, ID generation can be tricky

	🚀 Best For: Rare use cases, Independent subclasses, Small hierarchies

📊 Performance Comparison
| Strategy        | Insert | Read         | Schema Cleanliness | Best Use               |
| --------------- | ------ | ------------ | ------------------ | ---------------------- |
| SINGLE_TABLE    | Fast   | Fastest      | Low                | Performance-heavy apps |
| JOINED          | Medium | Medium       | High               | Enterprise systems     |
| TABLE_PER_CLASS | Medium | Slow (UNION) | Medium             | Rare cases             |

🔹 Q6. Composite primary key: @IdClass, @EmbeddedId
======================================================================================================
🧠 What is a Composite Primary Key?
	Example table:
			ORDER_ITEM
			------------
			order_id   (PK)
			product_id (PK)
			quantity
			price

	Here: Primary key = (order_id, product_id), Single @Id is not enough.

🟢 1️⃣ @IdClass
	✅ Concept: You define a separate key class and reference its fields in entity using multiple @Id
	🔹 Step 1: Create Key Class
				import java.io.Serializable;
				import java.util.Objects;
				public class OrderItemId implements Serializable {
					private Long orderId;
					private Long productId;

					public OrderItemId() {}

					public OrderItemId(Long orderId, Long productId) {
						this.orderId = orderId;
						this.productId = productId;
					}

					// equals() and hashCode() mandatory
				}

		⚠ Must: Implement Serializable, Have default constructor, Override equals() and hashCode()

	🔹 Step 2: Use in Entity
						@Entity
						@IdClass(OrderItemId.class)
						public class OrderItem {

							@Id
							private Long orderId;

							@Id
							private Long productId;

							private int quantity;
						}
				🗄 Database
				ORDER_ITEM
				-----------
				order_id (PK)
				product_id (PK)
				quantity

	🔥 Characteristics: Key fields defined twice (in key class + entity)
		Slight duplication, Simpler JPQL usage
		Example query:
			orderItemRepository.findByOrderId(1L);
			
🟡 2️⃣ @EmbeddedId
	✅ Concept: Composite key is treated as a single embedded object.

	🔹 Step 1: Create Embeddable Key
					@Embeddable
					public class OrderItemId implements Serializable {

						private Long orderId;
						private Long productId;

						public OrderItemId() {}

						public OrderItemId(Long orderId, Long productId) {
							this.orderId = orderId;
							this.productId = productId;
						}

						// equals() and hashCode()
					}

	🔹 Step 2: Use in Entity
				@Entity
				public class OrderItem {

					@EmbeddedId
					private OrderItemId id;

					private int quantity;
				}
		🔎 Accessing Fields: orderItem.getId().getOrderId();
		
📊 Side-by-Side Comparison
| Feature                | @IdClass       | @EmbeddedId        |
| ---------------------- | -------------- | ------------------ |
| Key defined            | Separate class | Embeddable class   |
| Fields in entity       | Multiple @Id   | Single @EmbeddedId |
| Duplication            | Yes            | No                 |
| JPQL usage             | Simpler        | Slightly nested    |
| Object-oriented design | Less clean     | More clean         |
| Recommended            | Legacy systems | Modern design      |

🔥 Important Differences
	With @IdClass: Entity has direct access: orderItem.getOrderId();
	With @EmbeddedId You must use: orderItem.getId().getOrderId();
	
🧠 Advanced: Relationship in Composite Key
	If key contains foreign keys:
	Example:
		ENROLLMENT
		------------
		student_id (PK, FK)
		course_id  (PK, FK)

	With @EmbeddedId you can combine with: @MapsId
	Example:
		@ManyToOne
		@MapsId("studentId")
		private Student student;

	This is cleaner with @EmbeddedId.

🔥 Which One Should You Use?
	Modern Recommendation: 👉 Prefer @EmbeddedId 
		Because: Cleaner, Better encapsulation, Supports complex mappings easily, Works well with @MapsId

🔥 8️⃣ JPQL & Criteria API
======================================================================================================
🔹 Q1. Difference between JPQL and SQL?
======================================================================================================
✅ 1️⃣ SQL (Structured Query Language) is: Database-specific query language used to query tables directly.
	It works with: Tables, Columns, Database schema
		Example (MySQL): SELECT id, name , FROM employee WHERE department_id = 10;
		
✅ 2️⃣ JPQL (Java Persistence Query Language) is: Object-oriented query language used in JPA to query entities instead of tables. It works with: Entities, Fields, Relationships

	Example: SELECT e FROM Employee e WHERE e.department.id = 10
	Notice: Uses entity name Employee, Uses field department, No table names

🔥 Core Difference
| Feature             | SQL          | JPQL        |
| ------------------- | ------------ | ----------- |
| Works on            | Tables       | Entities    |
| Uses                | Column names | Field names |
| Database dependent? | Yes          | No          |
| Object-oriented?    | No           | Yes         |
| Portable?           | No           | Yes         |

🔥 Important Internal Behavior
	JPQL is converted into SQL by Hibernate.
	Flow: JPQL → Hibernate → SQL → Database , So JPQL is abstraction over SQL.

🟢 Advantages of JPQL
	✔ Database independent, works with entity relationships, Automatic join handling
	✔ Type-safe with Criteria API, Cleaner domain modeling
	
	Example: SELECT e FROM Employee e JOIN e.department d WHERE d.name = 'IT'
	Hibernate generates proper JOIN SQL.

🔴 Advantages of SQL (Native Query)
	✔ Full DB power, Use DB-specific functions, Better for complex reporting
	✔ Better for performance tuning, Use CTE, Window functions, etc.
	
	Example: SELECT e.*, RANK() OVER (ORDER BY salary DESC) FROM employee e;
	JPQL does not support window functions.

🔥 Performance Difference
	JPQL: Slight overhead (converted to SQL), Usually negligible
	SQL: Direct execution, No ORM translation
	In most cases performance difference is minimal.

🧠 When to Use What?
	Use JPQL when: 
		✔ CRUD operations, Working with entity relationships
		✔Business logic layer, Want DB independence
		
	Use Native SQL when: 
		✔ Complex joins, Aggregation-heavy queries, DB-specific features 
		✔ Performance-critical reporting, Bulk operations

🔹 Q2. What is Criteria API? When should we prefer Criteria?
======================================================================================================
✅ Criteria API is a Type-safe, programmatic way to build queries in JPA using Java code instead of JPQL strings.
	Instead of writing: @Query("SELECT e FROM Employee e WHERE e.salary > :salary")
	You build the query dynamically using Java objects.
	
	🔎 Basic Example
		JPQL Version: SELECT e FROM Employee e WHERE e.salary > 50000
		
		Criteria API Version:
			CriteriaBuilder cb = entityManager.getCriteriaBuilder();
			CriteriaQuery<Employee> cq = cb.createQuery(Employee.class);
			Root<Employee> root = cq.from(Employee.class);
			cq.select(root)
			.where(cb.gt(root.get("salary"), 50000));

			List<Employee> result = entityManager.createQuery(cq).getResultList();

		More verbose, but dynamic and type-safe.

🧠 Why Was Criteria API Introduced?
	JPQL has limitations:
		❌ String-based (error-prone), Hard to build dynamically, Refactoring breaks silently
		❌ No compile-time validation

	Criteria API solves these problems.

🔥 Key Features
| Feature                | Criteria API |
| ---------------------- | ------------ |
| Type-safe              | ✅ Yes        |
| Dynamic query building | ✅ Yes        |
| Refactor-safe          | ✅ Yes        |
| Verbose                | ❌ Yes        |
| Readable               | ❌ Less       |

🟢 When Should We Prefer Criteria API?
	✅ 1️⃣ Dynamic Search Filters (Most Common Use Case)
			Example: Search API with optional filters
				GET /employees?name=John&minSalary=50000&department=IT
			You don’t know in advance which parameters will be present.
			
			Criteria example:
					List<Predicate> predicates = new ArrayList<>();
					if (name != null) {
						predicates.add(cb.equal(root.get("name"), name));
					}
					if (minSalary != null) {
						predicates.add(cb.gt(root.get("salary"), minSalary));
					}

					cq.where(predicates.toArray(new Predicate[0]));

			Perfect for dynamic conditions.

	✅ 2️⃣ Complex Search Screens
			Enterprise systems with: 10–20 filters, Multiple joins, Sorting, Pagination, Criteria shines here.

	✅ 3️⃣ When Using Specifications (Spring Data)
			Spring Data JPA uses Criteria internally:
				public class EmployeeSpecification {

					public static Specification<Employee> hasSalaryGreaterThan(Double salary) {
						return (root, query, cb) ->
							cb.gt(root.get("salary"), salary);
					}
				}

			Then: repo.findAll(EmployeeSpecification.hasSalaryGreaterThan(50000));
			Very powerful for modular filtering.

🔴 When NOT to Use Criteria
	❌ Simple queries, Static queries, Readability important, Very complex analytics queries
	For simple query: SELECT e FROM Employee e WHERE e.name = :name
	JPQL is cleaner.

🧠 Criteria vs JPQL Comparison
| Feature       | JPQL     | Criteria  |
| ------------- | -------- | --------- |
| Syntax        | String   | Java API  |
| Type-safe     | ❌ No     | ✅ Yes     |
| Dynamic       | ❌ Hard   | ✅ Easy    |
| Readability   | ✅ Better | ❌ Verbose |
| Refactor-safe | ❌ No     | ✅ Yes     |

🔥 Criteria + Pagination Example
		TypedQuery<Employee> query = entityManager.createQuery(cq);
		query.setFirstResult(0);
		query.setMaxResults(10);
		
	Works well with large datasets.

🧠 Advanced: Criteria with Join
		Join<Employee, Department> dept = root.join("department");
		cq.where(cb.equal(dept.get("name"), "IT"));
	
	Hibernate generates proper SQL join.

🚨 Important Note
	Criteria API: Still converted into SQL, No performance difference from JPQL, Just safer and dynamic

🔥 Real Production Scenario
	We had: Employee search screen, 15 optional filters, JPQL string concatenation was messy
	Switched to: Spring Data Specifications (Criteria internally)
	Code became: Modular, Clean, Reusable, Maintainable

🔹 Q3. What is TypedQuery and NamedQuery?
======================================================================================================
✅ 1️⃣  TypedQuery<T> is: A type-safe query interface that returns results of a specific type.
		Instead of returning raw Object or List<Object[]>, it ensures compile-time type safety.
	
	🔎 Example (Without TypedQuery)
			Query query = entityManager.createQuery("SELECT e FROM Employee e");
			List list = query.getResultList();

	Problem: Returns raw List, Needs casting, Risk of ClassCastException

	🔎 Example (With TypedQuery)
			TypedQuery<Employee> query = entityManager.createQuery( "SELECT e FROM Employee e", Employee.class );
			List<Employee> employees = query.getResultList();

		Now: ✔ Compile-time type safety, No casting needed, Cleaner code

🔥 When Do We Use TypedQuery?
	Whenever: Query returns entity, Query returns DTO, You know return type
	Example with DTO:
		TypedQuery<EmployeeDTO> query = entityManager.createQuery(
							"SELECT new com.dto.EmployeeDTO(e.id, e.name) FROM Employee e",
							EmployeeDTO.class
						);

✅ 2️⃣  @NamedQuery is: A statically defined JPQL query attached to an entity and validated at startup.
		Defined once, reused multiple times.

		🔎 Example
				@Entity
				@NamedQuery(
					name = "Employee.findByDepartment",
					query = "SELECT e FROM Employee e WHERE e.department.id = :deptId"
				)
				public class Employee {
				}
				
		🔎 Usage
			TypedQuery<Employee> query = entityManager.createNamedQuery( "Employee.findByDepartment", Employee.class );
			query.setParameter("deptId", 10L);
			List<Employee> result = query.getResultList();

🔥 Why Use NamedQuery?
	1️⃣ Pre-validated at startup
		If JPQL is wrong: Application fails during startup, Not at runtime, Better safety.
	
	2️⃣ Performance
		Named queries are: Parsed once, Cached by Hibernate, Slightly faster for repeated use

	3️⃣ Centralized Query Management
		Keeps queries near entity definition.

📊 TypedQuery vs NamedQuery
| Feature    | TypedQuery          | NamedQuery                      |
| ---------- | ------------------- | ------------------------------- |
| Purpose    | Type-safe execution | Predefined query                |
| Defined    | At runtime          | At compile/startup              |
| Reusable   | No                  | Yes                             |
| Validation | Runtime             | Startup                         |
| Type-safe  | Yes                 | Yes (when used with TypedQuery) |

🔥 Important Clarification
	They are NOT alternatives. They solve different problems.
	NamedQuery defines the query
	TypedQuery ensures type safety
	They can be used together.

🧠 Real Production Advice
	Use: 
		NamedQuery for frequently used static queries
		TypedQuery for dynamic or simple JPQL
		Criteria for dynamic filtering
		Native query for heavy reporting

🔹 Q4. Difference between: NativeQuery , JPQL
======================================================================================================
✅ 1️⃣ JPQL (Java Persistence Query Language) is: An object-oriented query language that works with JPA entities and their fields. It is database-independent.

	🔎 Example (JPQL)
		@Query("SELECT e FROM Employee e WHERE e.department.id = :deptId")
		List<Employee> findByDepartment(Long deptId);
	
	It works with: Entity name → Employee, Field name → department, Not table/column names
	Hibernate converts JPQL → SQL.

✅ 2️⃣ Native Query is: A raw SQL query executed directly against the database.
	It works with: Table names, Column names, DB-specific syntax

	🔎 Example (Native Query)
		@Query(
		  value = "SELECT * FROM employee WHERE department_id = :deptId",
		  nativeQuery = true
		)
		List<Employee> findByDepartmentNative(Long deptId);
	This is pure SQL.

🔥 Core Difference
| Feature                | JPQL         | Native Query   |
| ---------------------- | -----------  | -------------- |
| Works on               | Entities     | Tables         |
| Uses                   | Field names  | Column names   |
| Database independent   | ✅ Yes       | ❌ No          |
| ORM aware              | ✅ Yes       | ❌ Limited     |
| Supports relationships | ✅ Yes       | ❌ Manual      |
| DB-specific features   | ❌ Limited   | ✅ Full support|
| Portability            | High         | Low            |

🧠 How They Work Internally
	JPQL Flow: JPQL → Hibernate → SQL → DB, Hibernate generates optimized SQL.
	Native Query Flow; SQL → DB, Hibernate sends it directly to database.

🟢 Advantages of JPQL
	✔ Database independent, Works with entity relationships, Automatic joins
	✔ Polymorphic queries, Cleaner code

	Example: SELECT e FROM Employee e JOIN e.department d WHERE d.name = 'IT'
	No need to write JOIN manually.

🔴 Advantages of Native Query
	✔ Use DB-specific features, Use window functions, Use CTE (WITH clause)
	✔ Use advanced aggregations, Complex reporting queries, Performance tuning
	
	Example (MySQL window function):
		SELECT name, RANK() OVER (ORDER BY salary DESC) FROM employee;
		
	JPQL does NOT support window functions.

🔥 Performance Comparison
	Is Native always faster? ❌ Not necessarily.
	Because: JPQL also becomes SQL.
	Performance depends on generated SQL.
	Native is faster only when: Using advanced DB features, Optimized complex queries, Avoiding ORM overhead

🧠 Entity Lifecycle Difference
	JPQL: Fully managed by Hibernate, Triggers entity lifecycle events, Works with cache
	Native Query: Bypasses some ORM features, May skip cache, Manual mapping required sometimes

🔥 Mapping Differences
	JPQL:
		List<Employee> list = entityManager
			.createQuery("SELECT e FROM Employee e", Employee.class)
			.getResultList();

	Native:
		List<Employee> list = entityManager
			.createNativeQuery("SELECT * FROM employee", Employee.class)
			.getResultList();

	For DTO in native: @SqlResultSetMapping(...)
	More complex.

🚨 Common Interview Trap
	Q: Is JPQL faster than Native?
	Answer: Both ultimately execute SQL in database. Native gives more control but JPQL is sufficient for most business queries.

🧠 When to Use JPQL?	
	✔ CRUD operations, Relationship navigation, Polymorphic queries
	✔ Business logic queries, Database portability required

🧠 When to Use Native?
	✔ Complex reporting, Heavy aggregation, DB-specific functions
	✔ Window functions, Stored procedures, Performance-critical queries

🔹 Q6. What is constructor expression?
======================================================================================================
✅ A constructor expression in JPQL allows you to: Directly map query results into a DTO (Data Transfer Object) using its constructor. Instead of returning full entities, you return only required fields.

🔥 Why Do We Need It?
	If you write: SELECT e FROM Employee e
	Hibernate: Loads full entity, Tracks it in persistence context, Creates snapshot for dirty checking, Consumes more memory. Sometimes you only need: id, name. Not the whole entity.
	Constructor expression solves this.

	🔎 Example
	Step 1️⃣ Create DTO
					package com.dto;

					public class EmployeeDTO {

						private Long id;
						private String name;

						public EmployeeDTO(Long id, String name) {
							this.id = id;
							this.name = name;
						}

						// getters
					}
					
	Step 2️⃣ Use Constructor Expression in JPQL
				SELECT new com.dto.EmployeeDTO(e.id, e.name) FROM Employee e

		In Spring Data:
			@Query("SELECT new com.dto.EmployeeDTO(e.id, e.name) FROM Employee e")
			List<EmployeeDTO> findAllEmployeeDTO();
			
🧠 What Happens Internally?
	Hibernate: Executes SQL, Fetches selected columns, Calls DTO constructor, Returns list of DTO objects
	Does NOT store in persistence context

🔥 Important Rules
	✔ Must use fully qualified class name, Constructor must match parameter types and order
	✔ Constructor must be public, Works only in JPQL (not native directly)

🧠 Constructor Expression vs Entity Fetch
| Feature                       | Entity Query          | Constructor Expression |
| ----------------------------- | --------------------- | ---------------------- |
| Returns                       | Managed Entity        | DTO                    |
| Stored in persistence context | ✅ Yes       	        | ❌ No        	         |
| Dirty checking                | ✅ Yes 	            | ❌ No      	         |
| Memory usage                  | Higher                | Lower                  |
| Performance                   | Slower for large sets | Faster                 |

🔥 Performance Advantage
	Instead of: SELECT e FROM Employee e
	Better: SELECT new com.dto.EmployeeDTO(e.id, e.name) FROM Employee e
	Benefits: Fetch only required columns, No entity tracking, Lower memory usage
	Better performance for large result sets

🧠 Example with Join:
			SELECT new com.dto.EmployeeDeptDTO( e.name,  d.name) FROM Employee e JOIN e.department d
	Very useful for reporting.

🚨 Common Interview Trap
	Q: Does constructor expression create entity? ❌ No. It creates a new DTO object.
	Q: Can we update DTO and save back? ❌ No. DTO is not managed.

🧠 Constructor Expression vs Interface Projection (Spring Data)
	Spring Data also supports:
			public interface EmployeeView {
				String getName();
			}
	
	But constructor expression is: More explicit, More flexible, Works in plain JPA
	
🚀 When Should You Use Constructor Expression?
	✔ Read-only queries, Reporting, Large datasets
	✔ Avoid unnecessary entity loading, Microservices returning API DTO

❌ When NOT to Use: If you need managed entity, If you plan to update the entity, If business logic depends on entity lifecycle

🔹 Q7. How to implement dynamic search with Criteria API?
======================================================================================================
🔥 Dynamic search means: Build queries at runtime based on optional filters.
	Example API: GET /employees?name=John&minSalary=50000&department=IT
	Some filters may be present, some not.

🎯 Scenario
	We have:
				@Entity
				public class Employee {

					@Id
					private Long id;

					private String name;

					private Double salary;

					@ManyToOne
					private Department department;
				}
				
✅ Step 1️⃣ Create Search DTO
				public class EmployeeSearchCriteria {

					private String name;
					private Double minSalary;
					private Double maxSalary;
					private String departmentName;

					// getters & setters
				}
				
✅ Step 2️⃣ Implement Dynamic Criteria Query
				public List<Employee> search(EmployeeSearchCriteria criteria) {

					CriteriaBuilder cb = entityManager.getCriteriaBuilder();
					CriteriaQuery<Employee> cq = cb.createQuery(Employee.class);
					Root<Employee> root = cq.from(Employee.class);

					List<Predicate> predicates = new ArrayList<>();

					// name filter
					if (criteria.getName() != null) {
						predicates.add(
							cb.like(
								cb.lower(root.get("name")),
								"%" + criteria.getName().toLowerCase() + "%"
							)
						);
					}

					// min salary
					if (criteria.getMinSalary() != null) {
						predicates.add(
							cb.greaterThanOrEqualTo(
								root.get("salary"),
								criteria.getMinSalary()
							)
						);
					}

					// max salary
					if (criteria.getMaxSalary() != null) {
						predicates.add(
							cb.lessThanOrEqualTo(
								root.get("salary"),
								criteria.getMaxSalary()
							)
						);
					}

					// department join
					if (criteria.getDepartmentName() != null) {
						Join<Employee, Department> dept = root.join("department");
						predicates.add(
							cb.equal(dept.get("name"), criteria.getDepartmentName())
						);
					}

					cq.where(predicates.toArray(new Predicate[0]));

					return entityManager.createQuery(cq).getResultList();
				}
🔥 How It Works
	Create CriteriaBuilder
	Create CriteriaQuery
	Define Root
	Add predicates conditionally
	Apply them using where()

🧠 Why This Is Powerful
	Without Criteria:
		You would need: String concatenation, Risky dynamic JPQL, Hard-to-maintain query logic
	
	With Criteria: Type-safe, Clean, Modular, Easy to extend

✅ Adding Pagination
		TypedQuery<Employee> query = entityManager.createQuery(cq);
		query.setFirstResult(page * size);
		query.setMaxResults(size);
		return query.getResultList();
		
✅ Adding Sorting
		cq.orderBy(cb.asc(root.get("salary")));
		Or dynamic sorting:
		if (sortField != null) {
			cq.orderBy(cb.asc(root.get(sortField)));
		}

🔥 Real-World Better Approach: Spring Data Specification
Instead of writing Criteria manually:
				public class EmployeeSpecification {

					public static Specification<Employee> hasName(String name) {
						return (root, query, cb) ->
							name == null ? null :
							cb.like(cb.lower(root.get("name")), "%" + name.toLowerCase() + "%");
					}
				}
		Then:
				repo.findAll(
					Specification.where(hasName(criteria.getName()))
						.and(hasSalaryGreaterThan(criteria.getMinSalary()))
				);
		Cleaner for large systems.

📊 Why Not JPQL?
	JPQL would require:
				String query = "SELECT e FROM Employee e WHERE 1=1";
				if (name != null) {
					query += " AND e.name LIKE :name";
				}

Problems: Hard to maintain, Error-prone, No compile-time validation

🧠 Advanced Tips
	Use cb.conjunction() for clean base condition
	Predicate predicate = cb.conjunction();
	Use criteriaQuery.distinct(true) when using joins
	Prevents duplicate results.

🚨 Common Mistakes
	❌ Forgetting transaction for streaming, Not handling null properly
	❌ Using wrong field name string, Not clearing persistence context in large result sets

🔥 9️⃣ Hibernate Internals (Senior Level)
======================================================================================================
🔹 Q1. How does Hibernate generate SQL?
======================================================================================================
🎯 Big Picture Hibernate generates SQL through multiple layers:

				Entity Mapping
				   ↓
				JPQL / Criteria / Entity Operation
				   ↓
				Hibernate Query Translator
				   ↓
				SQL AST (Abstract Syntax Tree)
				   ↓
				Dialect-specific SQL
				   ↓
				JDBC
				   ↓
				Database

	🧠 1️⃣ Mapping Metadata Creation (Startup Phase)
		When application starts Hibernate reads: @Entity, @Table, @Column, @OneToMany, @JoinColumn, etc.
		It builds: Entity metadata model
		Internally stored in: EntityPersister, Metamodel, PersistentClass
		This metadata contains: Table name, Column mappings, Relationships, ID generation strategy, Versioning info

	🟢 2️⃣ When You Call persist(), merge(), remove()
		Example: entityManager.persist(employee);
		Hibernate: Adds entity to persistence context, Marks it as "INSERT required", SQL is NOT generated immediately (usually)

	🟢 3️⃣ Flush Time (Important)
		SQL is generated during: Transaction commit, Explicit flush(), Sometimes before query execution (FlushMode.AUTO)
		At flush time Hibernate performs:
			Step 1: Dirty Checking: Compares entity snapshot with current state, Detects changed fields
			Step 2: Action Queue Creation
					Hibernate builds: InsertAction, UpdateAction, DeleteAction
					Stored in: ActionQueue
					
	🟢 4️⃣ SQL Generation Phase
		Hibernate now generates SQL using: EntityPersister
		For example: SingleTableEntityPersister, JoinedSubclassEntityPersister
		It builds SQL like: INSERT INTO employee (name, salary, department_id) VALUES (?, ?, ?)

	🟢 5️⃣ Dialect Translation
		Hibernate uses Dialect to generate database-specific SQL.
		Examples: MySQLDialect, PostgreSQLDialect, OracleDialect
		Example difference: 
			MySQL pagination: LIMIT 10 OFFSET 20
			Oracle pagination: ROWNUM <= ?
		Hibernate adapts automatically.

	🟢 6️⃣ PreparedStatement Creation
		Hibernate: Creates PreparedStatement, Binds parameters, Executes via JDBC

	🔥 Example: JPQL to SQL
		JPQL: SELECT e FROM Employee e WHERE e.salary > 50000
		Process: JPQL parsed into HQL AST, Semantic validation, SQL AST created
		SQL generated: SELECT e.id, e.name, e.salary FROM employee e WHERE e.salary > ?
	
🟢 Criteria API Flow
	Criteria API: cb.gt(root.get("salary"), 50000);
	Internally: Builds query tree, Converts to SQL AST, Same SQL generation pipeline

🟢 Native Query Flow
	Native query: entityManager.createNativeQuery("SELECT * FROM employee");
	Hibernate: Skips translation, Directly executes SQL, Maps result to entity

🔥 Important: Insert Example with SEQUENCE
	If using: @GeneratedValue(strategy = SEQUENCE)
	Hibernate: Fetches next sequence value, Assigns ID, Generates INSERT SQL

🔥 Important: Insert Example with IDENTITY
	If using: @GeneratedValue(strategy = IDENTITY)
	Hibernate: Executes INSERT immediately, Retrieves generated key, Stores it in entity
	That’s why IDENTITY breaks batching.

🧠 How Hibernate Handles Relationships
	Example:
			@ManyToOne
			@JoinColumn(name="department_id")

	Hibernate knows: Foreign key column, Join condition, Nullability, Cascade rules
	So generated SQL includes: department_id = ?
	
🟢 Query Plan Caching
	Hibernate caches: Parsed JPQL, Generated SQL, Query plans, So repeated queries are faster.

🔥 Real Production Insight
	SQL generation performance depends on: 
		Flush frequency, Batch size, Fetch strategy, Dirty checking overhead, Dialect configuration

📊 Summary of SQL Generation Triggers
| Operation   | When SQL Generated |
| ----------- | ------------------ |
| persist()   | At flush           |
| merge()     | At flush           |
| remove()    | At flush           |
| JPQL select | Immediately        |
| Bulk update | Immediately        |
| flush()     | Immediately        |

🔹 Q2. What is Dialect?
======================================================================================================
✅ A Dialect is a Hibernate class that defines how SQL should be generated for a specific database.
	Because different databases have different SQL syntax, Hibernate uses Dialect to generate database-specific SQL.

🧠 Why Do We Need Dialect?
	Different databases support SQL differently. Hibernate handles aame logic with different syntax using Dialect.

🔥 How Dialect Works: When Hibernate generates SQL:
			JPQL / Criteria
				 ↓
			Hibernate SQL Generator
				 ↓
			Dialect
				 ↓
			Database-specific SQL

	Dialect tells Hibernate:
		How to generate pagination
		How to handle sequences
		How to handle identity columns
		Which functions are supported
		Data type mapping

📌 Common Dialects Hibernate provides built-in dialect classes.
	MySQL: org.hibernate.dialect.MySQLDialect, org.hibernate.dialect.MySQL8Dialect
	PostgreSQL: org.hibernate.dialect.PostgreSQLDialect
	Oracle: org.hibernate.dialect.OracleDialect
	SQL Server: org.hibernate.dialect.SQLServerDialect
	
⚙️ Configuration Example
	In Spring Boot: spring.jpa.database-platform=org.hibernate.dialect.MySQL8Dialect
	Usually auto-detected.

🔥 What Dialect Controls Internally
	1️⃣ Pagination SQL
	2️⃣ ID Generation Strategy
	3️⃣ Data Type Mapping
	4️⃣ Function Support
	5️⃣ Locking Syntax

🔥 Real Production Example
	We migrated: MySQL → PostgreSQL
	Only change required: spring.jpa.database-platform=org.hibernate.dialect.PostgreSQLDialect
	Hibernate automatically adjusted: Pagination, Data types, ID generation, Functions

🚨 What Happens If Wrong Dialect Is Used?
	Problems like: SQL syntax errors, Pagination not working, Wrong ID generation, Incorrect data types
	Example: Using OracleDialect with MySQL → failure.

| Functions              | Yes             |
| Transaction management | No              |

🔹 Q3. What is SessionFactory? Is SessionFactory thread-safe? Is Session thread-safe?
======================================================================================================
✅ 1️⃣ SessionFactory is: A heavyweight, thread-safe factory object responsible for creating Hibernate Session instances.
	It is created once at application startup.

🧠 What Does It Contain?
	When Hibernate starts, it builds: Entity metadata, Mapping information, SQL generation logic, Second-level cache
	Query plan cache, Connection configuration
	All of this is stored inside SessionFactory.

🏗 Lifecycle
			Application Startup
					↓
			Build SessionFactory (expensive)
					↓
			Reuse throughout application lifetime
					↓
			Destroy on shutdown
			
		🔎 Example
		SessionFactory sessionFactory =  new Configuration().configure().buildSessionFactory();
		In Spring Boot, this is auto-configured.

🔥 Is SessionFactory Thread-Safe?
	✅ YES. You can safely: Share across threads, Use in multiple requests, Keep as singleton
	Why? Because: It is immutable after creation, It does not hold entity state, It only creates sessions

🟢 2️⃣ What is Session?
	Session is: A lightweight, single-threaded object that represents a unit of work with the database.
	It is: Created from SessionFactory, Used per request or per transaction, Holds First-Level Cache (Persistence Context)

	🔎 Example
	Session session = sessionFactory.openSession();
	In Spring: Managed automatically, Bound to transaction

🧠 What Does Session Contain?
	Persistence Context (First-Level Cache), Managed entities, Dirty checking snapshots, Transaction state
	JDBC connection reference

🔥 Is Session Thread-Safe? 👉 ❌ NO.
	Session is NOT thread-safe.
	Why? Because: It stores entity state, It maintains persistence context, It is mutable, It tracks dirty checking
	If shared across threads: Race conditions, Data corruption, Unexpected behavior

🧠 Real-World Explanation
	SessionFactory Like a factory building machines. Safe to use by everyone.
	Session Like a personal workspace. Only one person should use it at a time.
	
📊 Quick Comparison
| Feature         | SessionFactory  | Session                |
| --------------- | --------------- | ---------------------- |
| Purpose         | Creates Session | Performs DB operations |
| Heavyweight?    | Yes             | No                     |
| Thread-safe?    | ✅ Yes           | ❌ No                   |
| Contains cache? | Second-level    | First-level            |
| Lifecycle       | App-wide        | Per transaction        |

🔹 Q4. What is proxy in Hibernate? How lazy loading works internally?
======================================================================================================
🔥 A Hibernate proxy is a dynamically generated subclass of your entity class that Hibernate uses to enable lazy loading.
	When you fetch an entity lazily, Hibernate does not immediately load the full object from the database.
	Instead, it returns a proxy object that:
		Has only the identifier (primary key) populated
		Defers loading the remaining fields until they are accessed
		Intercepts method calls to trigger database fetch
		Internally, Hibernate uses: Bytecode enhancement or CGLIB / ByteBuddy (depending on configuration)

	Example:
		@Entity
		public class Order {
			@Id
			private Long id;

			@ManyToOne(fetch = FetchType.LAZY)
			private Customer customer;
		}

	When you load Order, Hibernate does: Order order = session.get(Order.class, 1L);
	If customer is LAZY: Hibernate does NOT immediately fetch Customer
	It creates a Customer proxy When order.getCustomer().getName() is called → DB query executes
	
🔥How Lazy Loading Works Internally
	
	Step 1: Entity Retrieval
			When you call: session.get(Order.class, 1L);
			Hibernate: Executes SQL to load Order
			Sees @ManyToOne(fetch = LAZY)
			Instead of loading Customer, creates a proxy object
			At this moment:
						Order {
						   id = 1
						   customer = Customer$HibernateProxy (only ID set)
						}

	Step 2: Proxy Creation
			Hibernate generates a subclass like:
				class Customer$HibernateProxy extends Customer {
					LazyInitializer initializer;
				}
			This proxy: Overrides getter methods, Contains a LazyInitializer
			Knows: Entity name, Identifier, Session reference

	Step 3: Method Interception
			When you call: order.getCustomer().getName();
			Flow: getCustomer() returns proxy, getName() is intercepted
			Proxy checks: "Is entity initialized?"
				If NOT initialized:
					Executes SQL: select * from customer where id = ?
					Loads entity from DB, Replaces proxy internal target, Returns actual value
			After first access → entity becomes fully initialized.

🔥Key Internal Components
	Internally Hibernate uses: HibernateProxy, LazyInitializer, PersistenceContext, Session
	Lazy loading only works if: The Session is open, The entity is attached, 
	
🔥What Happens If Session is Closed?
	If you try: order.getCustomer().getName();
	After session closed: 
		You get: LazyInitializationException
		Because: Proxy tries to fetch data, But session is unavailable

🔥Difference Between get() and load() (Proxy Related)
	session.get(Customer.class, 1L); Immediately hits DB, Returns fully initialized object
	session.load(Customer.class, 1L); Returns proxy, DB hit only when accessed
		
🔥How Lazy Works for Collections?
	Example:
		@OneToMany(mappedBy="order", fetch=FetchType.LAZY)
		private List<OrderItem> items;

	Hibernate uses: PersistentBag, PersistentSet, PersistentList, These are wrapper implementations.
	When you call: order.getItems().size();
	Hibernate: Intercepts collection access, Fires SQL, Loads collection

🔥Important Interview Points (8+ Years Level)
	1️⃣ Proxy requires: Non-final class, Non-final methods, Default constructor
	2️⃣ Lazy loading uses: Proxy for single-valued associations, Collection wrapper for collections
	3️⃣ How to check if initialized? Hibernate.isInitialized(order.getCustomer());
	4️⃣ How to force initialize? Hibernate.initialize(order.getCustomer());
	5️⃣ How to avoid LazyInitializationException properly?
		Correct way: Use JOIN FETCH, Use DTO projection, Use EntityGraph, Use transactional service layer
		Wrong way: OpenSessionInView blindly, Changing everything to EAGER
		
🔥Visual Flow (Internal Lifecycle)
	DB Row → Hibernate → Proxy Object → Method Call → Interceptor → SQL → Real Entity → Return Value

🔥Performance Implication
	Lazy loading: Reduces unnecessary joins, Prevents loading large object graphs, Can cause N+1 if misused

🔹 Q5. What is Bytecode instrumentation?
======================================================================================================
✅ Bytecode Instrumentation (or Enhancement) is: The process of modifying entity classes at runtime or build time to add extra behavior without changing source code. Hibernate enhances your entity classes to support: Lazy loading, Dirty checking, Change tracking, Performance optimizations

🧠 Why Does Hibernate Need It?
	Normally, Java objects are simple POJOs.
	But Hibernate needs to: Detect when a field changes, Load lazy attributes on demand, Track modifications efficiently
	Avoid full object snapshot comparisons
	To do this efficiently, Hibernate modifies the class bytecode.

🔎 What Does Hibernate Add?
	Example:
		Original setter:
				public void setName(String name) {
					this.name = name;
				}

		After enhancement, Hibernate may transform it to:
				public void setName(String name) {
					markDirty("name");
					this.name = name;
				}

		Now Hibernate can track exactly which field changed.

🎯 What Problems Does It Solve?
	1️⃣ Efficient Dirty Checking
	2️⃣ Lazy Loading Without Proxies (Field-Level Lazy)
	3️⃣ Performance Improvements: Reduced memory usage, Reduced snapshot overhead, Faster flush, Better scalability

🧠 Types of Bytecode Enhancement: Hibernate supports:
	1️⃣ Runtime Enhancement: Happens at JVM startup, Uses instrumentation agent
	2️⃣ Build-Time Enhancement (Recommended): Happens during compilation, Using Maven/Gradle plugin

🔧 Maven Example
			<plugin>
			  <groupId>org.hibernate.orm.tooling</groupId>
			  <artifactId>hibernate-enhance-maven-plugin</artifactId>
			  <version>6.x.x</version>
			  <executions>
				<execution>
				  <configuration>
					<enableLazyInitialization>true</enableLazyInitialization>
					<enableDirtyTracking>true</enableDirtyTracking>
				  </configuration>
				</execution>
			  </executions>
			</plugin>
			
🔥 Without Bytecode Enhancement
	Hibernate relies on: Proxies, Snapshot comparison, Reflection, Less efficient.

📊 With vs Without Enhancement
| Feature        | Without Enhancement | With Enhancement     |
| -------------- | ------------------- | -------------------- |
| Dirty checking | Snapshot comparison | Field-level tracking |
| Lazy loading   | Proxy-based         | Field-based          |
| Performance    | Slower              | Faster               |
| Memory usage   | Higher              | Lower                |

🔹 Q6. How does Hibernate manage connection pooling?
======================================================================================================
🔥Hibernate itself does not manage production-grade connection pooling. It integrates with a connection pool (like HikariCP or a DataSource). Hibernate only borrows and returns connections via its ConnectionProvider abstraction.

1️⃣ Basic Flow – How Hibernate Gets a Connection
	When you call: session.beginTransaction();
	Internally: 
		Session asks JdbcCoordinator
		JdbcCoordinator asks LogicalConnection
		LogicalConnection asks ConnectionProvider
		ConnectionProvider borrows a connection from the pool
		Connection is returned after transaction ends

		Architecture:
					Session
					   ↓
					JdbcCoordinator
					   ↓
					LogicalConnection
					   ↓
					ConnectionProvider
					   ↓
					Connection Pool (Hikari / C3P0 / etc.)
					   ↓
					Database
					
2️⃣ Does Hibernate Have Its Own Pool?
	Yes — but only for development/testing.
	Hibernate provides a built-in pool, enabled via: hibernate.connection.pool_size=10

	⚠️ This pool: Is very basic, Has no advanced features, Not production-ready, Not recommended
	For production, always use a real pool.

3️⃣ Production Connection Pool Options
	Hibernate integrates with: 
		HikariCP (default in Spring Boot), C3P0, Apache DBCP, Application server pool (Tomcat, WildFly, etc.)
	
	Any custom DataSource: In Spring Boot, Hibernate does NOT manage pooling at all.
	Instead: Spring Boot → HikariCP → Hibernate → DB
	
4️⃣ Example – Spring Boot (Most Common Case)
	Default setup:
		spring.datasource.url=jdbc:mysql://localhost:3306/test
		spring.datasource.username=root
		spring.datasource.password=pass

	Spring Boot auto-configures: 
		HikariCP DataSource
		Hibernate uses that DataSource
		Hibernate just calls dataSource.getConnection()
		Hibernate never sees pool internals.

5️⃣ How Connection Lifecycle Works
	Connection Acquisition
		Depends on: hibernate.connection.handling_mode
		Common strategy (default in modern Hibernate): DELAYED_ACQUISITION_AND_RELEASE_AFTER_TRANSACTION
		Meaning: Connection acquired lazily (when first SQL runs), Released after transaction completes

	Step-by-Step During Transaction
					@Transactional
					public void saveOrder() {
					   repository.save(order);
					}

		Flow: 
			Transaction begins -> First SQL executes -> Hibernate requests connection
			Pool gives connection -> SQL runs -> Transaction commits -> Hibernate closes logical connection
			Physical connection returned to pool

6️⃣ Important Internal Interfaces
	Hibernate uses:
		🔹 ConnectionProvider
				Responsible for: Obtaining connections, Releasing connections
		🔹 MultiTenantConnectionProvider (if multitenancy)
		🔹 JdbcServices

7️⃣ Connection Release Modes
	Older Hibernate versions used: hibernate.connection.release_mode

Options:
| Mode              | Meaning                            |
| ----------------- | ---------------------------------- |
| after_statement   | Release after each SQL (rare)      |
| after_transaction | Release after transaction (common) |
| on_close          | Release when session closes        |

Modern Hibernate prefers handling_mode instead.

8️⃣ What Happens Under Heavy Load?
	Let’s say: Pool size = 10, 100 concurrent requests
	Behavior: First 10 get connections, Remaining 90 wait in pool queue
	If wait exceeds connectionTimeout
	Pool throws: SQLTransientConnectionException
	Hibernate simply propagates the exception.

9️⃣ Important Interview-Level Concepts
🔥 1. Why Pooling Is Needed?
	Without pooling: Every request opens TCP connection, Authentication handshake, DB resource allocation, Very expensive

	Pooling: Reuses connections, Reduces latency, Improves throughput

🔥 2. How to Size Pool?
	Rule of thumb: connections ≈ (CPU cores * 2) + disk count
	But depends on: DB capacity, Query latency, Blocking time
	Too many connections → DB thrashing
	Too few → Thread waiting

🔥 3. How Hibernate Avoids Holding Connection Too Long
	Because of: Delayed acquisition, Early release
	Hibernate: Does not grab connection when Session opens, Only when first SQL executes
	Releases ASAP after transaction
	This improves scalability.

🔥 4. OpenSessionInView Problem
	If OSIV is enabled: Connection may stay open for entire HTTP request, Can exhaust pool, Dangerous in high load systems

🔟 Connection Pool Monitoring
	With HikariCP you can monitor: Active connections, Idle connections, Threads waiting, Connection timeout
	In Spring Boot: /actuator/metrics/hikaricp.connections.active
	
1️⃣1️⃣ Multi-Tenancy Case
	Hibernate uses: MultiTenantConnectionProvider
	Depending on strategy:
		DATABASE → Different DB per tenant
		SCHEMA → Same DB, different schema
		DISCRIMINATOR → Same table, tenant column
	Connection provider switches connection accordingly.

1️⃣2️⃣ Common Interview Questions
Q: Does Hibernate open connection per session? No. Session ≠ Physical connection. Connection is acquired lazily.

Q: Does Hibernate manage pool size? Only if using built-in pool.
	In real apps, pool size is managed by Hikari/C3P0/etc.
	
Q: What happens if you forget to close session? Connection may not return to pool → leak.

Q: How to detect connection leak? Hikari has: leakDetectionThreshold

🔹 Q7. How does Hibernate integrate with JTA?
======================================================================================================
1️⃣ What is JTA?
	Jakarta EE
	Java Transaction API
	JTA is a specification that allows:
		Distributed transactions (multiple databases)
		Transactions across DB + JMS + other resources
		Two-phase commit (2PC)
		Container-managed transactions

Hibernate integrates with JTA to allow:
	Global transactions, XA transactions, Application server–managed transactions

2️⃣ When Do We Use Hibernate + JTA?
	| Scenario              | Example                            |
	| --------------------- | ---------------------------------- |
	| Multiple databases    | MySQL + Oracle in same transaction |
	| DB + JMS              | Save order + send message          |
	| Microservices with XA | Rare but possible                  |
	| App server managed tx | WildFly, WebLogic                  |

	If you're using only one DB → use RESOURCE_LOCAL.
	If distributed/global transaction → use JTA.

3️⃣ How Hibernate Integrates Internally
	Hibernate does NOT implement transactions itself in JTA mode.
	Instead it:
		Looks up the JTA TransactionManager
		Registers as a synchronization participant
		Flushes automatically before commit
		Delegates commit/rollback to JTA
	
	Flow:
			Application starts transaction (via container/Spring)
					↓
			JTA TransactionManager begins global transaction
					↓
			Hibernate joins transaction
					↓
			Hibernate registers Synchronization callback
					↓
			On commit:
			   - Hibernate flush()
			   - 2PC prepare phase
			   - Commit
 
🔥 Important Properties
	| Property                                      | Purpose                                  |
	| --------------------------------------------- | ---------------------------------------- |
	| `transaction-type=JTA`                        | Enables JTA mode                         |
	| `jta-data-source`                             | XA datasource                            |
	| `hibernate.transaction.coordinator_class=jta` | Use JTA coordinator                      |
	| `hibernate.transaction.jta.platform`          | Tells Hibernate how to talk to container |

5️⃣ JTA Transaction Flow (Under the Hood)
	When transaction begins: UserTransaction.begin();
	Hibernate: Gets current JTA transaction, Associates Session with it, Registers Synchronization
				public void beforeCompletion() {
					session.flush();
				}

				public void afterCompletion(int status) {
					if (rollback) clear session
				}
	Hibernate flush happens in beforeCompletion().

6️⃣ Two-Phase Commit (2PC)
	If multiple XA resources exist:
	Phase 1: Prepare: All resources vote commit or rollback
	Phase 2: If all vote YES → commit, If any vote NO → rollback, Hibernate participates through JDBC XA driver.

7️⃣ Hibernate + Spring Boot + JTA
	If using: Spring Framework
	You configure:
				@Bean
				public PlatformTransactionManager transactionManager() {
					return new JtaTransactionManager();
				}

		Or with Atomikos/Narayana dependency.

	Spring handles: Transaction boundary, JTA manager lookup, Propagation, Hibernate just joins.

8️⃣ RESOURCE_LOCAL vs JTA
| Feature                  | RESOURCE_LOCAL | JTA         |
| ------------------------ | -------------- | ----------- |
| Who manages transaction? | Application    | Container   |
| Distributed transactions | ❌              | ✅           |
| Two-phase commit         | ❌              | ✅           |
| Microservice use         | Common         | Rare        |
| App server required      | No             | Usually Yes |

9️⃣ Important Classes Involved
	Internally Hibernate uses: JtaTransactionCoordinator, JtaPlatform, Synchronization, TransactionManager, UserTransaction

🔟 Common Interview Questions
Q1: Does Hibernate start JTA transactions? No. Container/Spring starts it.
Q2: When does flush happen? Before commit in beforeCompletion().
Q3: What happens if rollback occurs? Session is cleared and entities become detached.
Q4: Can we use @Transactional with JTA? Yes — if Spring is configured with JTA manager.

1️⃣1️⃣ Real-World Example
	E-commerce:
		Save Order → DB1
		Save Payment → DB2
		Send Kafka message
	
	All must succeed or rollback. JTA ensures atomicity across all.

1️⃣2️⃣ When NOT to Use JTA
	Avoid JTA if: Single DB, Microservices with REST calls, High performance critical systems (2PC overhead)
	Cloud-native architecture (prefer Saga)

1️⃣3️⃣ Summary
	Hibernate + JTA: Hibernate becomes transaction participant, JTA manages global transaction, Flush happens before commit, Supports distributed XA transactions, Uses 2PC, Needs JTA platform configuration

				Your Repository Interface

			if (entityInformation.isNew(entity)) {

🔹 Q2. How is transaction started in Spring?
======================================================================================================
1️⃣ Short Answer: In Spring, a transaction is started by a proxy created by Spring AOP when a method annotated with @Transactional is invoked. The proxy delegates transaction handling to a PlatformTransactionManager, which Creates a transaction, Binds resources (like Connection / EntityManager) to the current thread, Commits or rolls back based on outcome

2️⃣ What Triggers Transaction Start?
			@Transactional
			public void createOrder() {
				orderRepository.save(order);
			}
	When this method is called: Spring does NOT start transaction inside your method.
	A proxy starts it before your method executes.

3️⃣ Internal Flow (Step by Step)
🔹 Step 1: Spring Creates Proxy
			When Spring sees: @EnableTransactionManagement
			It registers: TransactionInterceptor, ProxyTransactionManagementConfiguration
			Spring wraps your bean with: Transactional Proxy
			
🔹 Step 2: Method Is Called
			Call chain:
				Controller
				   ↓
				Service Proxy
				   ↓
				TransactionInterceptor
				
🔹 Step 3: TransactionInterceptor Executes
			TransactionInterceptor does:
					invoke() {
						txManager.getTransaction()
						try {
							method.invoke()
							txManager.commit()
						} catch(Exception e) {
							txManager.rollback()
						}
					}
			This is where transaction actually starts.

4️⃣ Who Actually Starts the Transaction?
	The real work is done by: PlatformTransactionManager
	Common implementations:
		DataSourceTransactionManager → JDBC
		JpaTransactionManager → JPA/Hibernate
		JtaTransactionManager → JTA

5️⃣ Example: JPA Transaction Flow
	If using JPA: @Transactional
	Internally: 
			TransactionInterceptor
				↓
			JpaTransactionManager.getTransaction()
				↓
			EntityManagerFactory.createEntityManager()
				↓
			entityManager.getTransaction().begin()

	Hibernate then: Session.beginTransaction(), Connection auto-commit = false

6️⃣ What Happens Internally in JDBC Case?
	If using: DataSourceTransactionManager
	Steps: Get connection from DataSource, Set autoCommit = false, Bind connection to ThreadLocal, Execute method
	commit() or rollback()

7️⃣ Thread Binding (Very Important)
	Spring uses: TransactionSynchronizationManager
	It binds: Connection / EntityManager → Current Thread
	That’s why: Same connection reused, First-level cache works, Lazy loading works

8️⃣ Visual Internal Flow
					Method Call
					   ↓
					Proxy
					   ↓
					TransactionInterceptor
					   ↓
					PlatformTransactionManager
					   ↓
					Transaction Begin
					   ↓
					Business Logic
					   ↓
					Commit / Rollback
					
9️⃣ What If Exception Occurs?
	Default behavior:
	| Exception Type    | Action                     |
	| ----------------- | -------------------------- |
	| RuntimeException  | Rollback                   |
	| Checked Exception | Commit (unless configured) |

	You can override: @Transactional(rollbackFor = Exception.class)

🔟 Important Interview Trap
❓ What if internal method calls another transactional method?
		public void outer() {
			inner(); // NO proxy
		}
	Transaction will NOT start for inner().
	Because: Proxy works only for external method calls.
	This is called: Self-invocation problem
	
1️⃣1️⃣ When Does Transaction Actually Begin?
	Transaction begins at: txManager.getTransaction()
	Which internally calls: doBegin()
	For example in DataSourceTransactionManager: connection.setAutoCommit(false);
	That is the exact point transaction starts.

1️⃣2️⃣ How Spring Knows Which Manager to Use?
	Spring Boot auto-configures:
		If JPA present → JpaTransactionManager
		If only JDBC → DataSourceTransactionManager
		If JTA configured → JtaTransactionManager
	Based on classpath detection.

1️⃣3️⃣ Deep Internal Components Involved
	@Transactional
	TransactionAttributeSource
	TransactionInterceptor
	PlatformTransactionManager
	TransactionSynchronizationManager
	AOP Proxy (JDK or CGLIB)

🔹 Q3. Difference between: @Transactional(readOnly = true), @Transactional
======================================================================================================
| Feature               | `@Transactional` | `@Transactional(readOnly = true)` |
| --------------------- | ---------------- | --------------------------------- |
| Transaction created?  | ✅ Yes            | ✅ Yes            	           |
| Can modify data?      | ✅ Yes            | ⚠️ Should not  	               |
| Flush on commit?      | ✅ Yes            | ❌ Usually skipped   	           |
| Dirty checking?       | ✅ Yes            | ⚠️ Disabled/optimized  	       |
| DB optimization hint? | ❌ No             | ✅ Yes          	               |

2️⃣ What Happens Internally?
	Spring uses: Spring Framework, Hibernate
🔹 Case 1: @Transactional Flow:
								TransactionInterceptor
								   ↓
								PlatformTransactionManager
								   ↓
								Begin transaction
								   ↓
								Hibernate Session (FlushMode.AUTO)
								   ↓
								Dirty checking enabled
								   ↓
								Commit → flush() → SQL update/insert

			Default behavior: FlushMode.AUTO, Changes detected, SQL executed at commit

🔹 Case 2: @Transactional(readOnly = true) Flow:
								TransactionInterceptor
								   ↓
								PlatformTransactionManager
								   ↓
								Begin transaction
								   ↓
								Hibernate Session FlushMode.MANUAL
								   ↓
								No automatic flush
								   ↓
								Commit → no update SQL

			Spring does: session.setHibernateFlushMode(MANUAL);
			Meaning: No dirty checking flush, No auto update, Better performance for reads

3️⃣ Does readOnly Prevent Updates?
	NO. This will still work:
			@Transactional(readOnly = true)
			public void test() {
				user.setName("New");
			}

		But: Hibernate will not flush automatically, No update SQL will be executed (unless manually flushed)
		If you manually call: entityManager.flush();
		It WILL update. So: It is an optimization hint, not a restriction.

4️⃣ JDBC-Level Difference
	If using: DataSourceTransactionManager
	Spring calls: connection.setReadOnly(true);
	Some databases optimize: MySQL, PostgreSQL, Oracle (limited effect)
	It may: Avoid locks, Route to replica (in some setups)
	Improve execution plan

5️⃣ Performance Impact
	For read-heavy systems:
		@Transactional(readOnly = true)
		public List<User> findAll()
	
	Benefits: Skips dirty checking, Skips flush, Better performance, Lower memory overhead
	Especially useful in: Reporting services, Read APIs, Search endpoints

6️⃣ Dirty Checking Difference
	Normal Transaction
		Hibernate tracks: Original state, Current state
		On commit: Compare, Generate update SQL, readOnly Transaction
		
	Hibernate: Does NOT track changes aggressively, FlushMode.MANUAL, Reduces overhead.

7️⃣ Isolation & Locking
	Both support: @Transactional(isolation = Isolation.SERIALIZABLE)
	readOnly does NOT change isolation level automatically.

8️⃣ Real Production Usage Pattern
	Service Layer Pattern:
			@Service
			public class UserService {

				@Transactional(readOnly = true)
				public User getUser(Long id) {}

				@Transactional
				public void updateUser(User user) {}
			}

		Best practice: Read methods → readOnly = true, Write methods → default

9️⃣ What Happens in JPA Transaction Manager?
	If using: JpaTransactionManager
	Spring: Sets EntityManager to read-only mode, Changes flush mode to MANUAL, Skips dirty synchronization

🔟 Interview Trap Question
	❓ If I update entity inside readOnly transaction, will it fail?
		No. It won't throw error. But changes won't be flushed unless explicitly flushed.

1️⃣1️⃣ Advanced Detail (Very Important)
	In Spring 5+: readOnly = true also: TransactionDefinition.isReadOnly()
	Which affects: TransactionSynchronizationManager, ORM optimizations, JDBC hinting

1️⃣2️⃣ Quick Internal Summary
| Layer          | @Transactional | readOnly = true   |
| -------------- | -------------- | ----------------- |
| Spring AOP     | Same           | Same              |
| Tx Manager     | Same           | Same              |
| JDBC           | Normal         | setReadOnly(true) |
| Hibernate      | FlushMode.AUTO | FlushMode.MANUAL  |
| Dirty checking | Enabled        | Reduced           |
| Auto flush     | Yes            | No                |

🔹 Q4. How propagation works with Hibernate?
======================================================================================================
1️⃣ Propagation is implemented by: Spring Framework
	Hibernate (Hibernate) only works with: The Session, The JDBC Connection, The active transaction
	Spring decides: Create new transaction? Join existing? Suspend current?
	Hibernate just uses the connection bound to the thread.

2️⃣ How Spring Makes Propagation Work
	Core component: TransactionInterceptor And: PlatformTransactionManager
	When a @Transactional method is called:
									Proxy
									   ↓
									TransactionInterceptor
									   ↓
									getTransaction()
									   ↓
									Based on propagation → decide behavior

3️⃣ The Key Mechanism: Thread Binding
	Spring uses: TransactionSynchronizationManager
	It binds: 
			Connection / EntityManager / Hibernate Session
					↓
			ThreadLocal
	
	Propagation works by checking: Is there already a transaction bound to this thread?

4️⃣ Propagation.REQUIRED
	Behavior:
		| If transaction exists | Join it |
		| If none exists | Create new |
		
	Flow:
		Outer method → REQUIRED
			 ↓
		Transaction created
			 ↓
		Inner method → REQUIRED
			 ↓
		Joins same transaction

	Hibernate: Same Session, Same Connection, Same persistence context, One commit

5️⃣ Propagation.REQUIRES_NEW
	Behavior:
		| If transaction exists | Suspend it |
		| Create new | Always |
	
	Flow:
		Outer tx (T1)
		   ↓
		Call inner REQUIRES_NEW
		   ↓
		Suspend T1
		   ↓
		Create T2
		   ↓
		Commit T2
		   ↓
		Resume T1
	
	Hibernate behavior: New EntityManager, New Session, New Connection, Separate commit
	Very important: Even if outer transaction rolls back, inner transaction remains committed.

6️⃣ Propagation.NESTED
	Behavior: 
			Uses savepoints, Works only with JDBC (DataSourceTransactionManager)
			Not supported by JTA, Hibernate supports via JDBC savepoints
	
	Flow:
		Outer tx
		   ↓
		Create savepoint
		   ↓
		Inner fails
		   ↓
		Rollback to savepoint
		   ↓
		Outer continues

	Hibernate still uses: Same Session, Same Connection, But rollback to savepoint

7️⃣ Propagation.SUPPORTS
	Behavior:
		| If tx exists | Join |
		| If none | Run without transaction |

	Hibernate behavior: If no transaction → auto-commit mode, No rollback capability

8️⃣ Propagation.NOT_SUPPORTED
	Behavior: Suspend current transaction, Execute non-transactionally
	Hibernate: Connection in auto-commit mode, No transaction boundaries

9️⃣ Propagation.NEVER
	If transaction exists → throw exception.

🔟 Propagation.MANDATORY
	If no transaction exists → throw exception.

1️⃣1️⃣ Deep Internal Flow Example
	Example:
			@Transactional
			public void outer() {
				userRepository.save(user);
				inner();
			}

			@Transactional(propagation = REQUIRES_NEW)
			public void inner() {
				orderRepository.save(order);
			}

Internal:
	Call outer() → Transaction T1 created → Hibernate Session S1 → Call inner() → Suspend T1 → Create T2
	→ Hibernate Session S2 → Commit T2 → Resume T1 → Commit T1
	
	Two different sessions.

1️⃣2️⃣ How Hibernate Detects Existing Transaction
	Hibernate checks: session.getTransaction().isActive()
	But Spring manages the lifecycle.
	Hibernate only reacts to: begin(), commit(), rollback()

1️⃣3️⃣ Very Important Interview Point
	Propagation works because: Spring binds the transactional resources to the current thread.
	If the method is called within same bean: this.inner();
	No propagation change happens.
	Because: Proxy is bypassed.
	This is called: Self-invocation problem

1️⃣4️⃣ Visual Summary
	Thread → TransactionSynchronizationManager → Connection / Session bound 
	→ Propagation decides: - Join? - Suspend? - New?
	Hibernate just uses whatever session is currently bound.

🔹 Q5. How isolation works with Hibernate?
======================================================================================================
1️⃣ Isolation controls how transactions interact when running concurrently. It prevents problems like: Dirty reads, Non-repeatable reads, Phantom reads.
	Isolation levels are defined by: Spring Framework, Hibernate
	Actually defined in: java.sql.Connection
	And implemented by the database (MySQL, PostgreSQL, Oracle, etc.).

2️⃣ Where Is Isolation Set in Spring + Hibernate?
	Example:
		@Transactional(isolation = Isolation.REPEATABLE_READ)
		public void updateUser() {}
		What Happens Internally?

	Flow:
		Proxy
		  ↓
		TransactionInterceptor
		  ↓
		PlatformTransactionManager
		  ↓
		connection.setTransactionIsolation(...)
		  ↓
		Hibernate Session uses that connection

	Hibernate DOES NOT manage isolation itself. It simply uses the same JDBC connection.

3️⃣ How Spring Applies Isolation
	Inside DataSourceTransactionManager: 
		connection.setTransactionIsolation(Connection.TRANSACTION_REPEATABLE_READ);
	Then: connection.setAutoCommit(false);
	Transaction begins. Hibernate then executes queries using that connection.

4️⃣ Isolation Levels Explained (With Hibernate Impact)
	🔹 READ_UNCOMMITTED: @Transactional(isolation = Isolation.READ_UNCOMMITTED)
		Allows: Dirty reads, Non-repeatable reads, Phantom reads
		Hibernate impact: Can read uncommitted changes from other transactions
		Rarely supported (e.g., PostgreSQL upgrades to READ_COMMITTED)
		Rarely used in production.

	🔹 READ_COMMITTED (Most Common)
		Default in: PostgreSQL, Oracle, SQL Server
		Prevents: Dirty reads
		Allows: Non-repeatable reads, Phantom reads
		Hibernate behavior: Each SELECT sees only committed data
		Same query may return different result inside same transaction
		Example:
			T1: SELECT salary → 1000
			T2: UPDATE salary → 2000 COMMIT
			T1: SELECT salary → 2000

		Non-repeatable read occurs.

	🔹 REPEATABLE_READ
		Default in: MySQL InnoDB
		Prevents: Dirty reads, Non-repeatable reads
		May allow: Phantom reads (depends on DB)
		Hibernate impact: Same row read twice → same value, Snapshot-based reads (MVCC)
		Better consistency for business logic.

	🔹 SERIALIZABLE
		Strongest isolation.
		Prevents: Dirty reads, Non-repeatable reads, Phantom reads
		Effect: DB behaves as if transactions are sequential, Can cause heavy locking, Low concurrency
		Hibernate just executes SQL — DB handles locking.

5️⃣ Important: Hibernate First-Level Cache Interaction
	Hibernate has: Session (Persistence Context)
	If you read same entity twice inside same transaction:
		User u1 = em.find(User.class, 1);
		User u2 = em.find(User.class, 1);
	
	Hibernate returns cached entity.
	So: Even at READ_COMMITTED, you may NOT see non-repeatable read because Hibernate caches it.
	To test real isolation:
		You must clear session: entityManager.clear();
	This is a common interview trap.

6️⃣ Phantom Reads with Hibernate
	Example:  List<User> users1 = repo.findByAge(30);
	Another transaction inserts new User age 30.
	Then: entityManager.clear();
		List<User> users2 = repo.findByAge(30);
	
	At:
		READ_COMMITTED → new row visible
		REPEATABLE_READ → depends on DB
		SERIALIZABLE → not visible
	Hibernate does not control this — database does.

7️⃣ How JPA Transaction Manager Applies Isolation
	If using: JpaTransactionManager It delegates isolation setting to JDBC.
	If using: JtaTransactionManager Isolation usually controlled by:
		Application server, Database default, JTA may ignore isolation configuration.

8️⃣ Real Internal Flow
			@Transactional(isolation = REPEATABLE_READ)
					↓
			Spring AOP Proxy
					↓
			TransactionInterceptor
					↓
			PlatformTransactionManager
					↓
			Connection.setTransactionIsolation()
					↓
			Hibernate Session uses that connection
					↓
			DB enforces isolation rules
			
9️⃣ Important Production Insight
	Isolation ≠ Locking strategy.
	Hibernate also supports: Pessimistic locking (@Lock(PESSIMISTIC_WRITE)), Optimistic locking (@Version)
	Isolation works at transaction level.
	Locking works at query/entity level.
	They are separate mechanisms.

🔟 Interview-Level Comparison
| Level            | Dirty Read | Non-Repeatable | Phantom | Performance |
| ---------------- | ---------- | -------------- | ------- | ----------- |
| READ_UNCOMMITTED | ✅          | ✅              | ✅       | Fast        |
| READ_COMMITTED   | ❌          | ✅              | ✅       | Good        |
| REPEATABLE_READ  | ❌          | ❌              | ⚠️      | Moderate    |
| SERIALIZABLE     | ❌          | ❌              | ❌       | Slow        |

1️⃣1️⃣ Very Important Interview Trap
Does Hibernate isolation guarantee consistency?
	No. Isolation is enforced by the database. Hibernate simply executes SQL within the configured transaction isolation level.

🔹 Q6. How to handle multiple data sources?
======================================================================================================
1️⃣ When Do We Need Multiple Data Sources?
	Common scenarios:
		✅ Two different databases (MySQL + PostgreSQL)
		✅ Read DB + Write DB
		✅ Microservice accessing 2 schemas
		✅ Legacy DB + New DB
		✅ Multi-tenant architecture

2️⃣ High-Level Architecture
	Using: Spring Framework, Spring Boot, Hibernate
	Architecture:
				DataSource1
				   ↓
				EntityManagerFactory1
				   ↓
				TransactionManager1
				   ↓
				Repository1

				DataSource2
				   ↓
				EntityManagerFactory2
				   ↓
				TransactionManager2
				   ↓
				Repository2

	Each database must have its own: DataSource, EntityManagerFactory, TransactionManager, Repository package

3️⃣ Step-by-Step Configuration (Two Databases)
	Assume: 
		Primary DB → user_db
		Secondary DB → order_db

	Step 1️⃣ Configure application.yml
									spring:
									  datasource:
										primary:
										  url: jdbc:mysql://localhost:3306/user_db
										  username: root
										  password: root
										secondary:
										  url: jdbc:mysql://localhost:3306/order_db
										  username: root
										  password: root
										  
	Step 2️⃣ Primary DataSource Configuration
						@Configuration
						@EnableTransactionManagement
						@EnableJpaRepositories(
							basePackages = "com.example.user",
							entityManagerFactoryRef = "primaryEntityManagerFactory",
							transactionManagerRef = "primaryTransactionManager"
						)
						public class PrimaryDbConfig {

							@Bean
							@Primary
							@ConfigurationProperties("spring.datasource.primary")
							public DataSource primaryDataSource() {
								return DataSourceBuilder.create().build();
							}

							@Bean
							@Primary
							public LocalContainerEntityManagerFactoryBean primaryEntityManagerFactory(
									EntityManagerFactoryBuilder builder) {
								return builder
										.dataSource(primaryDataSource())
										.packages("com.example.user.entity")
										.persistenceUnit("primary")
										.build();
							}

							@Bean
							@Primary
							public PlatformTransactionManager primaryTransactionManager(
									EntityManagerFactory emf) {
								return new JpaTransactionManager(emf);
							}
						}
						
	Step 3️⃣ Secondary Configuration: Same pattern, different packages:

					@EnableJpaRepositories(
						basePackages = "com.example.order",
						entityManagerFactoryRef = "secondaryEntityManagerFactory",
						transactionManagerRef = "secondaryTransactionManager"
					)

4️⃣ How Spring Decides Which DB to Use?
	Based on:
			@EnableJpaRepositories(
				basePackages = "...",
				transactionManagerRef = "..."
			)
	Repositories in different packages use different transaction managers.

5️⃣ Using @Transactional With Multiple DBs
	If you want to specify explicitly:
				@Transactional("secondaryTransactionManager")
				public void saveOrder() {}

	Otherwise: Default → Primary

6️⃣ Problem: Distributed Transactions?
	If both DBs must commit together:
		Save user → DB1
		Save order → DB2
	You need: JTA, XA transactions
	Use: JtaTransactionManager
	Otherwise: One may commit, other may fail.

7️⃣ Option 2: AbstractRoutingDataSource (Dynamic Routing)
	Used for: Read/Write splitting, Tenant-based routing
	Spring provides: AbstractRoutingDataSource
	Example:
			public class RoutingDataSource extends AbstractRoutingDataSource {

				@Override
				protected Object determineCurrentLookupKey() {
					return DataSourceContext.getCurrent();
				}
			}

	Then dynamically switch: DataSourceContext.set("READ_DB");
	This works using ThreadLocal.

8️⃣ Multi-Tenancy in Hibernate (Advanced)
	Hibernate supports:
		DATABASE strategy
		SCHEMA strategy
		DISCRIMINATOR strategy
	
	Example:
			hibernate.multiTenancy=DATABASE
			hibernate.tenant_identifier_resolver=...
	Hibernate decides DB at runtime.

9️⃣ Common Real-World Patterns
	🔹 Read / Write Splitting
		Primary → Write
		Replica → Read
		Route based on readOnly transaction
	
	🔹 Reporting DB: Heavy queries on separate DB
	🔹 Microservices: Usually: One DB per service, Avoid multiple data sources

🔟 Common Interview Pitfalls

❓ What happens if you use one @Transactional for two different data sources?
	Answer: Only one transaction manager will manage it unless using JTA.
	Without JTA: Transactions are independent.

1️⃣1️⃣ Real Production Advice
	✔ Use multiple data sources only if necessary
	✔ Avoid distributed transactions, Prefer eventual consistency (Saga)
	✔ Keep one DB per microservice, Use JTA only when absolutely required

	Multiple EntityManagerFactories
		  ↓
	Multiple TransactionManagers
		  ↓
	Repositories mapped to each
		  ↓

🔹 Q7. How to implement soft delete?
======================================================================================================
🔥 Soft delete means:
	❌ Don’t physically delete the row
	✅ Mark it as deleted
	✅ Automatically exclude it from queries

	In real enterprise systems, soft delete is used for: 
		Audit compliance, Data recovery, Business history tracking, Legal requirements

1️⃣ Basic Concept
	Instead of: DELETE FROM users WHERE id = 1;
	We do: UPDATE users SET deleted = true WHERE id = 1;
	
2️⃣ Recommended Approach (Best Practice with Hibernate)
	Hibernate provides: @SQLDelete, @Where
	This is the cleanest solution.
	
	✅ Step 1: Add a Column
				@Entity
				@Table(name = "users")
				@SQLDelete(sql = "UPDATE users SET deleted = true WHERE id = ?")
				@Where(clause = "deleted = false")
				public class User {

					@Id
					private Long id;

					private String name;

					private boolean deleted = false;
				}
		
		🔥 What Happens Internally?
			When you call: userRepository.deleteById(1L);
			Hibernate executes: UPDATE users SET deleted = true WHERE id = ?
			Instead of DELETE. And all queries automatically include: WHERE deleted = false
			
3️⃣ How It Works Internally
	@SQLDelete: Overrides default delete SQL.
	Instead of: DELETE FROM users WHERE id = ? Hibernate uses custom SQL.
	@Where Adds static filter condition to every SELECT.
	So: userRepository.findAll();
	Becomes: SELECT * FROM users WHERE deleted = false
	Automatically.

4️⃣ Limitations of @Where
	⚠️ It is static. It cannot: Dynamically include deleted records
	Be turned off at runtime If you need flexibility → use Filters.

5️⃣ Advanced Approach: Hibernate Filters (Dynamic)
	If you want: Admin → see deleted, User → don’t see deleted
	Use Hibernate filter.
	
	Step 1️⃣ Define Filter
				@FilterDef(name = "deletedFilter",
					parameters = @ParamDef(name = "isDeleted", type = Boolean.class))
				@Filter(name = "deletedFilter", condition = "deleted = :isDeleted")

	Step 2️⃣ Enable Filter
				Session session = entityManager.unwrap(Session.class);
				session.enableFilter("deletedFilter")
					   .setParameter("isDeleted", false);

			Now it’s dynamic.

6️⃣ Alternative Simple Approach (Manual)
	You can avoid Hibernate-specific annotations and:
		Add deleted field, Override delete method, Add condition in queries manually, But that is error-prone.

7️⃣ Soft Delete with Spring Data JPA
	Override delete:

			@Override
			@Transactional
			public void delete(User entity) {
				entity.setDeleted(true);
				save(entity);
			}
		
		But: ❌ Doesn’t protect direct delete queries, Risky in large systems

8️⃣ Adding Audit Information (Real Production Pattern)
	Better design:
		private boolean deleted;
		private LocalDateTime deletedAt;
		private String deletedBy;

		And custom SQL:
			@SQLDelete(sql = 
			  "UPDATE users SET deleted = true, deleted_at = now() WHERE id = ?")
			  
9️⃣ Handling Unique Constraints
	Problem: If you soft delete a user with email: email = test@gmail.com
	And try to create another user with same email → unique constraint fails.
	Solutions:
		1️⃣ Include deleted column in unique constraint
		2️⃣ Use partial index (PostgreSQL)
		
	Example:
		CREATE UNIQUE INDEX unique_email ON users(email) WHERE deleted = false;
		
🔟 Handling Relationships (Important)
	If using: @OneToMany(mappedBy = "user")
	Soft delete does NOT cascade automatically.
	You must: Either implement soft delete for children Or use DB constraint carefully

1️⃣1️⃣ Performance Considerations
	Soft delete causes: Table size growth, Slower queries, Need index on deleted column
	Always add index: CREATE INDEX idx_deleted ON users(deleted);
	
1️⃣2️⃣ Real Enterprise Pattern
	Large systems often combine: Soft delete, Archival tables, Scheduled cleanup jobs, Audit logging
	Sometimes: Active table, Deleted table, Archive table

🔹 Q8. How to implement auditing?
======================================================================================================
🔥 Auditing means automatically tracking:
	✅ Who created the record
	✅ When it was created
	✅ Who modified it
	✅ When it was modified

In enterprise apps, auditing is mandatory for: Compliance, Debugging, Security, Business tracking


1️⃣ Basic Auditing (Created/Updated Fields)
	✅ Step 1: Create Base Audit Class
							@MappedSuperclass
							@EntityListeners(AuditingEntityListener.class)
							public abstract class BaseAuditEntity {

								@CreatedDate
								@Column(updatable = false)
								private LocalDateTime createdAt;

								@LastModifiedDate
								private LocalDateTime updatedAt;

								@CreatedBy
								@Column(updatable = false)
								private String createdBy;

								@LastModifiedBy
								private String updatedBy;
							}

	✅ Step 2: Enable Auditing
							@Configuration
							@EnableJpaAuditing
							public class AuditConfig {
							}

	✅ Step 3: Implement AuditorAware
		Spring needs to know who is the current user.

				@Component
				public class AuditorProvider implements AuditorAware<String> {

					@Override
					public Optional<String> getCurrentAuditor() {
						return Optional.of(SecurityContextHolder
								.getContext()
								.getAuthentication()
								.getName());
					}
				}

			Now auditing works automatically.

2️⃣ What Happens Internally?
	When you save an entity: userRepository.save(user);
	Spring Data JPA: 
		Detects @CreatedDate, @LastModifiedDate, Calls AuditingEntityListener, Sets fields before persist/update, Hibernate persists values, No manual coding required.

3️⃣ Full Change History (Audit Table)
	Basic auditing only tracks timestamps.
	If you need: Full history, Every change stored, Versioned entities
	Use:
		🔥 Hibernate Envers

		Step 1️⃣ Add Dependency
							<dependency>
								<groupId>org.hibernate</groupId>
								<artifactId>hibernate-envers</artifactId>
							</dependency>

		Step 2️⃣ Add @Audited
							@Entity
							@Audited
							public class User {
								@Id
								private Long id;
								private String name;
							}
			
				What Happens? Hibernate automatically creates: user, user_AUD
				Every insert/update/delete creates a new row in user_AUD.

Example:
| id | name   | REV | REVTYPE |
| -- | ------ | --- | ------- |
| 1  | John   | 1   | ADD     |
| 1  | Johnny | 2   | MOD     |
| 1  | null   | 3   | DEL     |

4️⃣ How Envers Works Internally
	When transaction commits:
		Hibernate detects entity change, Envers listener captures change
		Inserts revision entry, Maintains revision metadata table
		It hooks into Hibernate event system.

5️⃣ Retrieving Audit History
	AuditReader reader = AuditReaderFactory.get(entityManager);
	User oldUser = reader.find(User.class, 1L, 1);
	Fetch entity at revision 1.

6️⃣ Custom Revision Entity
	If you want: Store who changed, Store IP address, Store timestamp
	Create custom revision entity:
							@Entity
							@RevisionEntity(CustomRevisionListener.class)
							public class CustomRevisionEntity extends DefaultRevisionEntity {

								private String username;
							}
							
7️⃣ Alternative Approach (Manual Audit Table)
	Another way: Create separate audit table
	Use @PrePersist, @PreUpdate
	
	Example:
		@PreUpdate
		public void onUpdate() {
			this.updatedAt = LocalDateTime.now();
		}
	But this won’t track history versions.

8️⃣ Comparing Approaches
| Feature              | Spring Auditing | Envers  |
| -------------------- | --------------- | --------|
| createdAt            | ✅   	         | ✅      |
| updatedAt            | ✅ 	         | ✅      |
| Full history         | ❌  	         | ✅      |
| Separate audit table | ❌  	         | ✅      |
| Versioning           | ❌   	         | ✅      |
| Automatic            | ✅   	         | ✅      |

9️⃣ Real Production Strategy
	Most companies use: Basic auditing for normal services, Envers for critical domains (finance, payments)
	Event sourcing for high-scale systems

🔟 Important Production Concerns
	1️⃣ Performance: Envers doubles write operations.
	2️⃣ Table Growth: Audit tables grow fast.
	3️⃣ Indexing Required: Add index on REV and entity ID.

🔹 Q1. Production issue due to N+1 — how did you fix?
======================================================================================================
✅ Situation: We had a performance issue in production where an API response time increased from 200ms to 8–10 seconds after data volume grew. The endpoint was returning a list of Orders with associated Customer and OrderItems.

🔍 Problem Identified
	Logs showed:
		1 query to fetch orders
		+ N queries to fetch customers
		+ N queries to fetch order items

	For 500 orders: 1 + 500 + 500 = 1001 queries
		Classic N+1 problem caused by LAZY relationships being accessed inside a loop.

🛠 Root Cause: Entity mapping:
				@OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
				private List<OrderItem> items;

				@ManyToOne(fetch = FetchType.LAZY)
				private Customer customer;

	And in service:
				orders.forEach(o -> {
					o.getCustomer().getName();
					o.getItems().size();
				});
			Each getter triggered a separate SQL query.

🚀 Fix Applied
	🔥 1️⃣ Used JOIN FETCH: 
						@Query("""
						   select o from Order o
						   join fetch o.customer
						   left join fetch o.items
						""")
						List<Order> findAllWithDetails();

			This reduced: 1001 queries → 1 query
		
	🔥 2️⃣ Used EntityGraph (Cleaner Option)
						@EntityGraph(attributePaths = {"customer", "items"})
						List<Order> findAll();

				More maintainable than writing JPQL.

	🔥 3️⃣ Used Batch Fetching (For Large Collections)
			In some cases where join fetch caused Cartesian product explosion:
				hibernate.default_batch_fetch_size=50
				This changed behavior from: 1 + 500 queries To: 1 + 10 queries (batch of 50)

	🔥 4️⃣ Used DTO Projection (Best for APIs)
			Instead of fetching entities:
									@Query("""
									   select new com.dto.OrderDTO(o.id, c.name)
									   from Order o
									   join o.customer c
									""")
				Avoided loading unnecessary relationships. This gave best performance improvement.

📊 Result
	Response time dropped from 8s → 250ms
	DB load reduced significantly
	CPU utilization dropped
	Production stabilized

🔹 Q2. Memory leak due to Hibernate — what happened?
======================================================================================================
We faced a memory leak issue in production caused by Hibernate’s first-level cache growing during a long-running batch transaction. Since Hibernate stores all managed entities in the persistence context for dirty checking, processing a large dataset without clearing the session caused heap memory to grow continuously, eventually leading to OutOfMemoryError. We resolved it by implementing batch processing with periodic flush() and clear(), enabling JDBC batching, and in some cases using StatelessSession for bulk operations. This significantly reduced memory usage and stabilized the application.

🔹 Q3. High DB load because of improper fetch — how resolved?
======================================================================================================
We faced high DB load in production due to improper fetching, which caused an N+1 query problem. A list API was loading parent entities, and lazy associations were being accessed inside a loop, triggering hundreds of additional queries.

We identified it through SQL logs and APM monitoring, then resolved it by using JOIN FETCH and @EntityGraph to fetch required associations in a single query. For large datasets, we applied batch fetching and switched to DTO projections to avoid loading unnecessary entities. This significantly reduced query count and brought DB load back to normal.

🔹 Q4. Deadlock issue — how debugged?
======================================================================================================
In production, we faced intermittent transaction failures with deadlock exceptions. Logs showed database deadlock errors during concurrent updates on the same set of tables.

First, we enabled SQL logging and analyzed database deadlock logs (InnoDB/PG logs) to identify which queries and transactions were involved. We found that two services were updating the same tables but in different order, causing circular locking.
	To fix it, we:
		Standardized the order of updates across transactions
		Reduced transaction scope (kept transactions short)
		Added proper indexing to avoid full table scans and excessive locking
		In some cases, switched to optimistic locking to reduce row-level lock contention
		After deploying the fix, deadlocks stopped and DB contention reduced significantly.

	Step 5: Fix Native Queries
			Some SQL parsing stricter in Hibernate 6.

			Critical areas to test: 
				Lazy loading, Transactions, Custom queries, Batch processing, Second-level cache, Pagination
			⚠️ Common Migration Issues
			ClassNotFoundException (javax → jakarta)

			Custom types failing
			Dialect mismatch

🔥 1️⃣2️⃣ Trick / Trap Questions
======================================================================================================
🔹 Q1. Can we make entity final?
======================================================================================================
Technically, you should not make a Hibernate/JPA entity final.
Hibernate creates proxies (usually via subclassing using CGLIB/ByteBuddy) for features like lazy loading. If the entity class is final, Hibernate cannot create a proxy subclass, and lazy loading will fail.

So: ❌ final entity → breaks proxying
✅ Non-final entity → allows lazy loading and enhancement
In Hibernate 6 with bytecode enhancement, some cases may work, but as a best practice, entities should not be final.

======================================================================================================
🔹 Q2. Can we use primitive type for version field?
======================================================================================================
Yes, you can use a primitive type (like int or long) for the @Version field in Hibernate/JPA.

@Version
private int version;

However, using a wrapper type (Integer / Long) is generally preferred because:
	Wrapper allows null before the entity is first persisted
	Hibernate can better detect a new (unsaved) entity
	Primitive defaults to 0, which may create ambiguity
	So while primitives work, wrapper types are considered safer and more flexible in production systems.

🔹 Q3. Why equals() and hashCode() are important in entity?

equals() and hashCode() are important in JPA/Hibernate entities 
because entities are often stored in collections like Set, used in caching, and managed inside the persistence context.

If they are not implemented properly:
	You may get duplicate entities in Set
	Entity comparisons may fail across sessions
	Hibernate may behave unexpectedly with detached/merged entities
	
Best practice:
	Base equals() and hashCode() on a stable business key or immutable identifier
	Avoid using generated ID before it is assigned
	Improper implementation can lead to subtle bugs in collections and persistence behavior.

🔹 Q4. Can we use Lombok @Data on entity?
======================================================================================================
No, it’s not recommended to use Lombok @Data on JPA/Hibernate entities.
	
	@Data generates: equals(), hashCode(), toString(), All getters/setters
	
	Problems: 
		equals()/hashCode() may include all fields (including lazy associations) → can cause performance issues and unexpected behavior.
		toString() may trigger lazy loading.
		Can lead to infinite recursion in bidirectional relationships.

	Better approach:
		Use specific Lombok annotations like: @Getter, @Setter, @NoArgsConstructor
		And manually control equals() and hashCode().
	So: Avoid @Data on entities in production systems.

🔹 Q5. What happens if we remove default constructor?
======================================================================================================
If we remove the default (no-argument) constructor from a JPA/Hibernate entity, Hibernate will fail at runtime.
	Reason:
		Hibernate creates entity instances using reflection and requires a no-arg constructor (at least protected or public). 
	
	Without it, you’ll get an exception like: org.hibernate.InstantiationException: No default constructor
	JPA specification also mandates that every entity must have a no-arg constructor.
	So: Default constructor is mandatory for entities.

🔹 Q6. Can we inject Spring bean inside entity?
======================================================================================================
No, we should not inject a Spring bean inside an entity.

Reason: 
	Entities are managed by Hibernate/JPA, not by the Spring container.
	Since Spring does not control entity lifecycle, dependency injection (@Autowired) will not work reliably.

	Also, entities should be pure domain objects (POJOs) without business/service dependencies.
	Best practice:
		Keep entities simple (state + mapping only)
		Put business logic in Service layer where Spring beans can be injected
	So: Do not inject Spring beans into entities.

🔹 Q7. Why bidirectional mapping may cause StackOverflow?
======================================================================================================
Bidirectional mapping can cause StackOverflowError because of infinite recursion.

Example:
	class Order {
		@OneToMany(mappedBy = "order")
		List<OrderItem> items;
	}

	class OrderItem {
		@ManyToOne
		Order order;
	}

If toString(), equals(), hashCode(), or JSON serialization is called:
	Order → calls items
	OrderItem → calls order
	Order → calls items again
	… and it continues infinitely

This recursive loop causes StackOverflowError.
	Common fixes: 
		Exclude one side in toString()
		Use @JsonIgnore in REST APIs
		Avoid including relationships in equals()/hashCode()

🔹 Q8. What happens if two @OneToMany are eager?
======================================================================================================
If two @OneToMany associations are marked as EAGER, Hibernate will try to fetch both collections immediately
which can cause:
	Large JOIN queries
	Cartesian product explosion
	Duplicate rows
	Huge result sets
	High memory usage and poor performance

In many cases, Hibernate may even throw: MultipleBagFetchException
if both are List (bags) and fetched via join.

Best practice:
	Keep @OneToMany as LAZY
	Fetch selectively using JOIN FETCH or EntityGraph

So: Multiple EAGER collections can severely impact performance and should be avoided.

🔹 Q9. Can Hibernate work without JPA?
======================================================================================================
Yes, Hibernate can work without JPA.
	Hibernate is a standalone ORM framework and existed before JPA. You can use its native API like:
	Session session = sessionFactory.openSession();
	without using EntityManager or JPA annotations.
	However, in modern applications, Hibernate is usually used as a JPA provider, because JPA provides a standard API and better portability.

So: Hibernate can work independently, but today it’s commonly used through JPA.

🔹 Q10. Difference between getOne() and findById()?
======================================================================================================
findById() 
	immediately hits the database and returns an Optional<T>.
	If the entity is not found, it returns Optional.empty().

getOne() (deprecated in newer Spring Data versions, replaced by getReferenceById()) 
	returns a lazy proxy and does NOT hit the database immediately.
	The database query is executed only when you access a field.

If the entity doesn’t exist:
	findById() → no exception, just empty
	getOne() / getReferenceById() → throws EntityNotFoundException when accessed

So:
	findById() → eager fetch
	getOne() → lazy proxy reference
