🔹 Q1. What is the difference between process and thread? Why is Java considered a multi-threaded language?
=============================================================================================================
🔹 Process: An independent program in execution with its own memory space and system resources.
🔹 Thread: A thread is a lightweight execution unit inside a process that shares the process’s memory and resources.
	
	🔧 Code Perspective
		JVM Process
		 ├── Main Thread
		 ├── GC Thread
		 ├── JIT Compiler Thread
		 ├── HTTP Worker Threads
		All these threads share heap memory but have separate stacks.

🎯Why Is Java Considered a Multi-Threaded Language?
	🔹1: Built-in Thread Support: Thread, Runnable, Callable, Executor Framework, ForkJoinPool, CompletableFuture
	🔹2: JVM Itself Is Multi-Threaded: Main thread, Garbage Collector thread, JIT compiler thread, Signal handler threads
	🔹3: Shared Memory Model: Fast inter-thread communication, High-performance parallelism
			With built-in synchronization: synchronized, volatile, Locks, Atomic variables
	🔹4: Platform Independence: Java threads map to native OS threads
			So Java programs: Scale across CPU cores, Use true parallelism, Behave consistently across platforms

🔹 Q2. What are the benefits and drawbacks of multithreading?
=============================================================================================================
✅ Benefits of Multithreading
	1️⃣ Better CPU Utilization & Parallelism
		Use multiple CPU cores simultaneously, instead of wasting idle CPU time.
		👉 Example: One thread waits for DB, Another processes data, Another handles user requests

	2️⃣ Improved Application Responsiveness
		Time-consuming tasks can run in background, keeping the main flow responsive.
		✔ Users don’t feel the delay

	3️⃣ Higher Throughput for Server Applications
		Web servers handle thousands of concurrent requests using thread pools.
			1 JVM Process
			 └── 200+ Worker Threads
		✔ More requests per second, Better scalability

	4️⃣ Efficient Resource Sharing => Threads share Heap memory, Open files, Network connections
		This makes inter-task communication faster than inter-process communication.
			synchronized(cache) {
				cache.put(key, value);
			}
		✔ Less overhead compared to processes

	5️⃣ Better Design for Asynchronous Tasks
		Multithreading naturally fits: Producer–Consumer, Event-driven systems, Non-blocking workflows
			BlockingQueue<Task> queue = new LinkedBlockingQueue<>();

	6️⃣ JVM-Level Optimization
		The JVM is optimized for concurrency: Adaptive thread scheduling, Optimized locks, Lock elision & biased locking (earlier versions)
		✔ Performance tuned by JVM itself

❌ Drawbacks of Multithreading: 
	1️⃣ Increased Complexity: Harder to write, Harder to reason about, Harder to debug, Bugs are often non-deterministic.
	
	2️⃣ Concurrency Bugs (Very Dangerous)
		| Bug Type          | Description                     |
		| ----------------- | ------------------------------- |
		| Race condition    | Incorrect results due to timing |
		| Deadlock          | Threads wait forever            |
		| Livelock          | Threads active but no progress  |
		| Starvation        | Thread never gets CPU           |
		| Visibility issues | Stale data due to CPU cache     |

	3️⃣ Performance Overhead => Threads introduce overhead: Context switching, Lock contention, Cache invalidation
		⚠ Too many threads can degrade performance.

	4️⃣ Difficult Debugging & Testing
		Problems may: Appear only in production, Disappear during debugging, Depend on CPU timing, Hard to reproduce consistently

	5️⃣ Risk of Data Corruption => Since threads share memory: One bad thread can corrupt shared state, Crash entire JVM
			sharedList.add(item); // unsafe without synchronization

	6⃣ Memory Issues: ThreadLocal leaks in thread pools, Stack memory per thread, Increased GC pressure, Too many threads = OutOfMemoryError

🔹 Q3. Explain Thread vs Runnable. Which one do you prefer and why?
=============================================================================================================
1️⃣ Thread vs Runnable — Core Difference
	| Aspect           | Thread                      | Runnable             |
	| ---------------- | --------------------------- | -------------------- |
	| Type             | Class                       | Functional interface |
	| Inheritance      | Cannot extend another class | Can extend any class |
	| Responsibility   | Task + execution            | Task only            |
	| Reusability      | Poor                        | High                 |
	| Design           | Tight coupling              | Loose coupling       |
	| Executor support | ❌ Not directly   	         | ✅ Preferred  	    |
	| Lambda support   | ❌                	         | ✅        	        |
	| Testability      | Low                         | High                 |

2️⃣ Why Runnable Is Better (Senior Perspective)
	✅ 1. Supports Inheritance
	✅ 2. Separation of Concerns (Very Important)
		Runnable separates: What to run → Runnable, How to run → Thread / Executor, This is clean OOP design.

	✅ 3. Works with Executor Framework: Executors accept Runnable, not Thread.
	✅ 4. Better Reusability => Same Runnable can be executed by: Multiple threads, Different executors, Async frameworks
	✅ 5. Lambda Friendly (Java 8+)
	✅ 6. Easier Testing: Runnable logic can be unit tested without threads, Mocked and reused

3️⃣ When Would You Use Thread?
	Rare, but valid cases: Override thread behavior, Customize thread lifecycle, Create specialized thread types
		Example:
				class LoggingThread extends Thread {
					public void run() {
						log();
						super.run();
					}
				}																
		Even then, many teams avoid this.

🔹 Q4. Difference between start() and run() methods.
=============================================================================================================
1️⃣ Core Difference
	✅ start(): Creates a new thread, Tells the JVM to schedule the thread, Internally calls run() in a new call stack
	✅ run(): Contains the task logic, Executes like a normal method call, Runs in the current thread

2️⃣ What Happens Internally?
	▶ When start() is called: 
		JVM creates a new native thread, Allocates a new call stack, Registers thread with OS scheduler, JVM invokes run() asynchronously
			main thread stack
			new thread stack → run()

	▶ When run() is called: No new thread, No new stack, Just a normal method call
			main thread stack → run()

3️⃣ Why Can’t We Call start() Twice? Reason: A thread’s lifecycle can start only once, JVM does not allow reusing terminated threads
		Thread t = new Thread();
		t.start();
		t.start(); // ❌ IllegalThreadStateException

🔹 Q5. What are daemon threads? Give real-world use cases.
=============================================================================================================
1️⃣ A daemon thread is a background service thread in Java that runs in the background and provides supporting services to user threads, Does not prevent JVM from exiting. When all user (non-daemon) threads finish, the JVM terminates automatically, even if daemon threads are still running.

2️⃣ How to Create a Daemon Thread
		Thread t = new Thread(() -> { while (true) { System.out.println("Daemon running..."); }});
		t.setDaemon(true); // must be before start() otherwise throws IllegalThreadStateException.
		t.start();

3️⃣ JVM Behavior with Daemon Threads
	User Threads running → JVM alive  
	Only Daemon Threads running → JVM exits
	Daemon threads: Are abruptly stopped, Do not execute finally blocks reliably, Should not manage critical resources
	
4️⃣ User Thread vs Daemon Thread (Comparison)	
	| Aspect          | User Thread           | Daemon Thread             |
	| --------------- | --------------------- | ------------------------- |
	| Purpose         | Application logic     | Background support        |
	| JVM exit        | Keeps JVM alive       | JVM ignores it            |
	| Lifecycle       | Controlled by app     | Controlled by JVM         |
	| Reliability     | Guaranteed completion | May stop anytime          |
	| Resource safety | Safer                 | Unsafe for critical tasks |

5️⃣ Real-World Use Cases (Very Important)
	🔹 1. Garbage Collector (GC): 	GC runs as daemon threads. ✔ JVM cleans memory, Stops automatically when app exits
	🔹 2. JIT Compiler Threads: Compiles bytecode to native code, Improves performance at runtime, Runs in background
	🔹 3. Background Monitoring & Cleanup Tasks
			Examples: Cache cleanup, Session expiration, Health checks, Metrics collection

				ScheduledExecutorService scheduler = Executors.newSingleThreadScheduledExecutor(r -> {
					Thread t = new Thread(r);
					t.setDaemon(true);
					return t;
				});

	🔹 4. Logging Framework Internals
			Some async loggers use daemon threads for: Flushing logs, Buffer management
			⚠ Logs may be lost if JVM exits suddenly.

	🔹 5. Timer Threads: Timer timer = new Timer(true); // daemon timer
			Used for: Periodic background tasks, Cleanup jobs
	🔹 6. Framework Background Threads
			Frameworks like: Spring, Netty, Kafka clients
			Use daemon threads for: Heartbeats, Watchdogs, Housekeeping

6️⃣ When NOT to Use Daemon Threads
	❌ For: File writing, Database updates, Payment processing, Critical business workflows
		Because: JVM can terminate them anytime, No guarantee of completion

🔹 Q6. How does JVM decide thread scheduling? Is thread scheduling platform dependent?
=============================================================================================================
1️⃣ JVM does not directly schedule threads. It delegates scheduling to the underlying operating system.
	▶  The JVM creates Java threads, Maps them to native OS threads
		Sets thread properties like: Priority, Daemon status, Hands control to the OS scheduler
		👉 After that, OS decides: Which thread runs, For how long, On which CPU core

	▶ Modern JVMs rely on preemptive multitasking, meaning a running thread can be paused by OS
      CPU time is shared via time slicing, Threads don’t cooperate voluntarily
	  This prevents one thread from monopolizing the CPU.
		
2️⃣ What Factors Influence Thread Scheduling?
	Even though JVM doesn’t schedule directly, these factors matter:
		✔ Thread Priority
				Thread t = new Thread(task);
				t.setPriority(Thread.MAX_PRIORITY);
			JVM passes priority to OS, OS may honor it, No guarantee
			⚠ Priority is a hint, not a command.

		✔ Thread State : Only threads in RUNNABLE state are eligible. Blocked / waiting threads are ignored.
		✔ CPU Availability: Single-core → time slicing, Multi-core → true parallel execution
		✔ OS Scheduling Algorithm
			Examples: Linux → Completely Fair Scheduler (CFS), Windows → Priority-based scheduler

3️⃣ Is Thread Scheduling Platform Dependent?
	✅ YES — JVM relies on native OS threads, Different OS use different scheduling algorithms
        	Priority handling varies across platforms

🔹 Q7. What is context switching? Explain green threads vs native threads.
=============================================================================================================
1️⃣  Context switching is the process of saving the state of one running thread and restoring the state of another thread,
   so the CPU can switch execution between them.

	🔧 What Gets Saved & Restored?
		When a context switch happens, the system saves: CPU registers, Program counter, Stack pointer, Thread state
		Then loads the state of the next thread.
			Thread A running
			   ↓ (context switch)
			Thread A state saved
			Thread B state restored
			Thread B running

	🔹 Why Context Switching Is Needed?
		Because: CPU cores are limited, Threads are more than CPU cores, OS must share CPU time fairly
				Even on single-core systems, context switching creates the illusion of parallelism.

	🔹 Cost of Context Switching (Very Important): Context switching is expensive:
		CPU cycles wasted, Cache invalidation, Pipeline flush
		⚠ Too many threads → too many context switches → performance degradation

2️⃣ Green Threads vs Native Threads: This is about who manages the threads — JVM or OS.
	🟢 Green Threads: 
		🔹 Green threads are user-level threads managed entirely by the JVM, not by the operating system.
			JVM schedules threads, OS sees only ONE process
		
		🔹 Characteristics: 
			JVM handles scheduling, OS is unaware of individual threads
			No true parallelism on multi-core CPUs
			Context switching is faster (no kernel involvement)
		
		🔹 Pros: ✔ Low context switching cost, Fast thread creation, Portable
		🔹 Cons: ❌ No true parallelism, One blocking call blocks all threads, Poor scalability
		🔹 Example Early Java versions on Solaris supported green threads.


	🔵 Native Threads
		🔹 Native threads are OS-level threads. Each Java thread maps to a native OS thread.
			Java Thread ↔ OS Thread
		
		🔹 Characteristics
			OS scheduler controls execution, True parallelism on multi-core CPUs
			Blocking in one thread doesn’t block others, Higher context switching cost
		
		🔹 Pros: ✔ Uses multiple CPU cores, Better scalability, Better I/O handling
		🔹 Cons: ❌ Higher overhead, OS-dependent behavior, More context switching cost

4️⃣ What Does Modern Java Use?
	✅ Native Threads ONLY: Since Java 1.3+, Java uses native threads on all major platforms.
		👉 This is why: Java scales well on multi-core CPUs, Servers handle thousands of requests
		GC, JIT, application threads run independently

🔹 Q8. What is thread priority? Does it really work?
=============================================================================================================
1️⃣  Thread priority is an integer hint (1–10) that suggests to the thread scheduler how important a thread is relative to others.
	In Java:
			Thread.MIN_PRIORITY  = 1
			Thread.NORM_PRIORITY = 5
			Thread.MAX_PRIORITY  = 10

			Thread t = new Thread(task);
			t.setPriority(Thread.MAX_PRIORITY);

2️⃣ What Is the Purpose of Thread Priority?
	Higher priority → more CPU time
	Lower priority → less CPU time
	👉 But this is only a suggestion, not a guarantee.

3️⃣ Does Thread Priority Really Work?
	🔹 Honest Senior Answer: Not reliably.
	🔹 Correct Technical Answer: Thread priority behavior is platform dependent and should not be relied upon for correctness.

4️⃣ Why Thread Priority Is Unreliable
	🔸 1. JVM Delegates Scheduling to OS and its depends on OS scheduling algorithm, OS priority mapping, CPU availability

	🔸 2. Platform-Dependent Behavior:
			| OS      | Priority Handling           |
			| ------- | --------------------------- |
			| Linux   | Often ignored or normalized |
			| Windows | More likely to respect      |
			| macOS   | Limited effect              |
		👉 Same Java code → different behavior on different OSes.

	🔸 3. No Execution Order Guarantee
			✔ Output is non-deterministic, ❌ Order No guarantee

	🔸 4. Risk of Starvation: 
			High-priority threads can starve lower-priority threads
			causing: Performance degradation, Unresponsive systems, This is one reason JVMs often ignore or minimize priority effects.

5️⃣ When Thread Priority Can Help (Rare Cases)
	✔ Best-effort background tasks, Non-critical monitoring threads, CPU-heavy background computations
	Example: 
		Thread monitoring = new Thread(task);
		monitoring.setPriority(Thread.MIN_PRIORITY);
		monitoring.setDaemon(true);
	Even then → don’t depend on it.

6️⃣ What Should Be Used Instead (Senior Best Practice)
	Instead of thread priority use ExecutorService, Thread pools, Proper task queues, Back-pressure mechanisms, Concurrency utilities
	Example: ExecutorService executor = Executors.newFixedThreadPool(4);
	This gives: Predictable behavior, Better scalability, OS-agnostic execution

🔹 Q9. Explain thread lifecycle with states. Difference between NEW, RUNNABLE, BLOCKED, WAITING, TIMED_WAITING, TERMINATED.
=============================================================================================================
1️⃣ Thread Lifecycle Diagram (Mental Model)
	NEW
	 │ start()
	 ▼
	RUNNABLE  <───────────────┐
	 │   │                    │
	 │   │ lock acquired      │ notify/timeout
	 │   ▼                    │
	BLOCKED              WAITING / TIMED_WAITING
	 │                         │
	 │ lock released           │
	 └──────────────► RUNNABLE
				 │
				 ▼
			TERMINATED

2️⃣ Thread States Explained (With Causes & Examples)
	🔵 1. NEW: Thread object is created, start() has not been called, No system resources allocated, Cannot be scheduled
			Thread t = new Thread(() -> {});
			
	🟢 2. RUNNABLE: Thread is ready to run or running, JVM has handed thread to OS scheduler
			t.start(); // enters RUNNABLE
			Important: Java does NOT distinguish between: Ready and Running, Both are RUNNABLE.
			Tip: “RUNNABLE means eligible to run, not necessarily running.”

	🔴 3. BLOCKED: Thread is waiting to acquire a monitor lock, Happens only with synchronized
			synchronized(lock) {
				// another thread holding lock
			}
			Causes: Thread tries to enter synchronized block/method, Lock held by another thread
			Key Difference: BLOCKED → waiting for lock, WAITING → waiting for signal

	🟡 4. WAITING: Thread waits indefinitely for another thread’s action
			Causes:
					obj.wait();
					thread.join();
					LockSupport.park();
			How It Leaves WAITING: notify(), notifyAll(), Target thread finishes (for join)

	🟠 5. TIMED_WAITING: Thread waits for a specified time and After Timeout Thread returns to RUNNABLE automatically
			Causes:
					Thread.sleep(1000);
					obj.wait(1000);
					thread.join(1000);
					lock.tryLock(1, TimeUnit.SECONDS);

	⚫ 6. TERMINATED: Thread execution is complete, run() method exits
			Important: Cannot be restarted, Calling start() again → IllegalThreadStateException

4️⃣ When Does a Thread Enter the BLOCKED State?
	A thread is trying to acquire an intrinsic (monitor) lock that is already held by another thread.
	👉 BLOCKED = waiting for a monitor lock
	🔹 The Only Cause of BLOCKED State
		✔ Attempting to enter a synchronized block or method, Lock is already owned by another thread
				synchronized (lock) {
					// critical section
				}
		If another thread already holds lock, the current thread becomes BLOCKED.

5️⃣ BLOCKED vs WAITING (Very Common Trap)
	| Aspect        | BLOCKED          | WAITING           |
	| ------------- | ---------------- | ----------------- |
	| Reason        | Lock unavailable | Signal required   |
	| Triggered by  | synchronized     | wait(), join()    |
	| Timeout       | ❌ No   	       | ❌ No   	       |
	| Controlled by | JVM monitor      | Application logic |

6️⃣ WAITING vs TIMED_WAITING
	| Aspect   | WAITING    | TIMED_WAITING    |
	| -------- | ---------- | ---------------- |
	| Duration | Indefinite | Fixed time       |
	| Wake-up  | notify     | timeout / notify |
	| Example  | wait()     | sleep(1000)      |

🔹 Q10. Can a thread move from WAITING to RUNNABLE without notify? What happens if notify is called before wait?
=============================================================================================================
1️⃣ Can a Thread Move from WAITING → RUNNABLE Without notify()?
	✅ YES — A thread in WAITING state can move back to RUNNABLE without notify() in the following cases:
	🔹 1. Thread Interruption: If a waiting thread is interrupted:
				synchronized (lock) {
					lock.wait();   // WAITING
				}
				thread.interrupt();

			➡ The thread: Wakes up, Throws InterruptedException, Moves to RUNNABLE, No notify() involved
	
	🔹 2. Spurious Wakeups: The JVM is allowed to wake up a waiting thread without any notify.
			This is rare, but legal per Java spec. 👉 That’s why this is mandatory:
				synchronized (lock) {
					while (!condition) {
						lock.wait();
					}
				}
			❌ Using if is wrong, ✅ while rechecks the condition safely

	🔹 3. Target Thread Completion (join): If a thread is waiting on join():
			t.join();  // WAITING
			When thread t finishes: Waiting thread wakes up, Moves to RUNNABLE, No notify is required
	❌ What Does NOT Wake a WAITING Thread: sleep(), Lock release, Thread priority, CPU availability


2️⃣ What Happens If notify() Is Called Before wait()?
	🔹 Short Answer: The notification is lost.
	🔍 Explanation: Consider this code:
			synchronized (lock) {
				lock.notify(); // nobody is waiting
			}
		Later:
			synchronized (lock) {
				lock.wait();   // waits forever
			}
		What happens? No thread was waiting at notify time, JVM does not store notifications, The waiting thread blocks indefinitely
		➡ Dead wait situation
		🔥 Key Rule (Must Say in Interview): wait/notify has no memory — notify does not queue signals.

3️⃣ Why This Design Exists
	Java’s wait/notify is low-level and signal-based, not condition-based.
	It assumes: 
		The programmer controls the condition, Notifications are not persistent,
		That’s why higher-level constructs exist:
				BlockingQueue, CountDownLatch, Semaphore, Condition

4️⃣ Correct Pattern to Avoid Lost Notifications
	✅ Always use a condition check
			synchronized (lock) {
				while (!condition) {
					lock.wait();
				}
			}
		And:
			synchronized (lock) {
				condition = true;
				lock.notifyAll();
			}

5️⃣ notify() vs notifyAll() in This Context
	| Method      | Risk                         |
	| ----------- | ---------------------------- |
	| notify()    | Can wake wrong thread        |
	| notifyAll() | Safer, avoids missed signals |
👉 In real systems, notifyAll is usually safer.

🔹 Q11. How does JVM clean up a terminated thread? What happens if a thread throws an exception?
=============================================================================================================
1️⃣ How Does JVM Clean Up a Terminated Thread?
	🔹 What Is a Terminated Thread? 
		A thread enters TERMINATED state when: run() method finishes normally, or An uncaught exception escapes run()
		Once terminated: The thread can never be restarted, It becomes eligible for garbage collection
		🔧 What JVM Cleans Up (Internally): When a thread terminates, the JVM:
			✅ 1. Releases Thread Stack Memory
					Each thread has its own stack, Stack frames are discarded immediately
					This memory is returned to the OS / JVM
			
			✅ 2. Releases Native OS Thread: Java threads map to native OS threads, JVM informs OS to clean up native thread resources
			✅ 3. Releases Locks Automatically
					If thread exits a synchronized block or method:
						JVM automatically releases the monitor lock
									synchronized (lock) {
										throw new RuntimeException();
									} // lock released automatically
						✔ No deadlock due to terminated thread

			✅ 4. Clears Thread-Local Stack Data: Local variables, Method frames, Program counter

❌ What JVM Does NOT Clean Automatically
	⚠ ThreadLocal values: Stored in ThreadLocalMap inside the thread
		If thread lives long (e.g., thread pool), values may cause memory leaks
		Must be cleaned manually using remove(): threadLocal.remove();
	🧠 Important GC Point: The Thread object itself is garbage collected only if no references remain
	Termination ≠ immediate GC

🎯 Interview Line : When a thread terminates, the JVM releases its stack memory, native thread resources, and any acquired intrinsic locks, and the Thread object becomes eligible for garbage collection if no references remain.

2️⃣ What Happens If a Thread Throws an Exception?
	This depends on whether the exception is caught or uncaught.
	🔹 Case 1: Exception Is Caught Inside run()
						public void run() {
							try {
								int x = 10 / 0;
							} catch (Exception e) {
								System.out.println("Handled");
							}
						}
			✔ Thread continues execution, Thread terminates normally after run() completes, No impact on other threads
	
	🔹 Case 2: Uncaught Runtime Exception (Most Important)
						public void run() {
							int x = 10 / 0; // ArithmeticException
						}
			What happens? Exception propagates out of run(), JVM invokes UncaughtExceptionHandler, Thread terminates immediately
			Other threads continue normally, ❗ JVM does NOT crash because of one thread

			▶  Default Behavior: If no handler is set:
					JVM prints stack trace to stderr, Thread dies silently after logging
			
			▶  Custom UncaughtExceptionHandler (Senior-Level)
							Thread.setDefaultUncaughtExceptionHandler((t, e) -> {
								System.out.println("Thread " + t.getName() + " failed: " + e);
							});

					Or per thread:
							t.setUncaughtExceptionHandler((thread, ex) -> {
								log.error("Thread crashed", ex);
							});

					✔ Used in production systems for: Logging, Alerts, Restarting tasks,
	🔹 Case 3: Checked Exception in run()
			❌ Not allowed directly
				public void run() throws IOException { // compile-time error
				}
			✔ Must be handled inside run()

3️⃣ What About Locks & finally Blocks?
	✔ Locks: JVM releases intrinsic locks automatically

	✔ finally Blocks
			try {
				risky();
			} finally {
				cleanup();
			}
		✔ finally executes before thread termination, ❌ Except in extreme cases like System.exit()

4️⃣ What Happens to Other Threads?
	| Scenario                     | Impact                                      |
	| ---------------------------- | ------------------------------------------- |
	| One thread throws exception  | Other threads unaffected                    |
	| Main thread throws exception | JVM exits **only if no other user threads** |
	| Daemon thread crashes        | Ignored by JVM                              |

🔹 Q12. What is synchronization and why is it needed? What problem does synchronization solve?
=============================================================================================================
1️⃣  Synchronization is a mechanism in Java that controls access to shared mutable resources so that only one thread executes a critical section at a time, while also ensuring memory visibility between threads.
▶ In Java, synchronization is achieved using:
	synchronized keyword, Intrinsic (monitor) locks, Locks in java.util.concurrent

2️⃣ Why Is Synchronization Needed? 
	▶ Because multiple threads share memory.
		Without synchronization: 
			Threads may read stale data, Updates may overwrite each other, Program behavior becomes non-deterministic

3️⃣ What Problem Does Synchronization Solve?
	Synchronization primarily solves three core concurrency problems:
	🔴 Problem 1: Race Condition (Atomicity Issue)
		❌ Without Synchronization
				class Counter {
					int count = 0;

					void increment() {
						count++; // read → modify → write
					}
				}

		If two threads execute count++ concurrently: Both read the same value, Both write back the same result, One update is lost
		This is a race condition.
		✅ With Synchronization
			synchronized void increment() {
				count++;
			}
		✔ Ensures only one thread updates count at a time, Makes count++ atomic

	🔴 Problem 2: Memory Visibility (Stale Data)
		❌ Without Synchronization
				boolean running = true;
				void stop() {
					running = false;
				}
				void run() {
					while (running) {
						// may loop forever
					}
				}
		Why? Each thread may cache running, Update by one thread may not be visible to others

		✅ With Synchronization
				synchronized void stop() {
					running = false;
				}

				synchronized void run() {
					while (running) {}
				}

			✔ Synchronization enforces happens-before relationship, Changes become visible across threads

	🔴 Problem 3: Instruction Reordering
		Without synchronization: JVM & CPU may reorder instructions, Leads to partially initialized objects being visible
		✅ Synchronization prevents unsafe reordering.

4️⃣ What Synchronization Guarantees (Very Important)
	| Guarantee        | Explanation                             |
	| ---------------- | --------------------------------------- |
	| Mutual exclusion | Only one thread enters critical section |
	| Atomicity        | Compound operations execute as one      |
	| Visibility       | Writes by one thread visible to others  |
	| Ordering         | Prevents harmful reordering             |

5️⃣ What Synchronization Does NOT Guarantee
	❌ No fairness guarantee, No deadlock prevention, No performance guarantee

6️⃣ Real-World Example (Senior-Level)
	In a Spring Boot application: Multiple HTTP requests update the same in-memory cache
	Without synchronization → corrupted cache
	With synchronization → consistent state

🔹 Q13. Difference between synchronized method vs synchronized block. Why is synchronized block preferred?
=============================================================================================================
1️⃣ synchronized Method vs synchronized Block
	🔹 Synchronized Method: When you mark a method as synchronized, the entire method body is locked.
				class Counter {
					public synchronized void increment() {
						count++;
						log();          // also synchronized
						validate();     // also synchronized
					}
				}
			Lock acquired on: this (for instance method), ClassName.class (for static method)

	🔹 Synchronized Block: Only a specific section of code is synchronized.
				class Counter {
					public void increment() {
						// non-critical code
						validate();

						synchronized (this) {
							count++;
						}

						// non-critical code
						log();
					}
				}
			Lock acquired on: Explicit object (this, or any other lock object)
			
2️⃣ Key Differences (Interview Table)
	| Aspect      | Synchronized Method    | Synchronized Block    |
	| ----------- | ---------------------- | --------------------- |
	| Lock scope  | Entire method          | Only critical section |
	| Flexibility | Low                    | High                  |
	| Performance | Lower (coarse-grained) | Better (fine-grained) |
	| Lock object | Implicit               | Explicit              |
	| Readability | Simpler                | Slightly more verbose |
	| Control     | Less                   | More                  |
	| Best for    | Simple logic           | Complex logic         |

3️⃣ Why Is Synchronized Block Preferred?
	🔥 Reason 1: Fine-Grained Locking (Most Important): 
		Synchronized blocks allow locking only the critical section, not the entire method.
			synchronized (this) {
				updateSharedState();
			}
		✔ Less lock contention, Better concurrency, Higher throughput

	🔥 Reason 2: Better Performance
		Synchronized methods: Hold the lock longer, Block other threads unnecessarily
		Synchronized blocks: Reduce time spent holding locks, Improve scalability in multi-threaded apps
	
	🔥 Reason 3: Explicit Lock Object: 	private final Object lock = new Object();
			synchronized (lock) {
				// critical section
			}
		✔ Avoids exposing intrinsic lock (this), Prevents accidental external locking, Cleaner encapsulation

	🔥 Reason 4: Avoids Over-Synchronization
			public synchronized void process() {
				validate();
				callRemoteService();  // ❌ slow but locked
				updateCache();
			}
			Bad design. 
			
		✔Using block: Lock only where required
			public void process() {
				validate();
				callRemoteService(); // no lock
				synchronized (this) {
					updateCache();
				}
			}

	🔥 Reason 5: Multiple Locks in Same Method
			synchronized (lockA) {
				updateA();
			}

			synchronized (lockB) {
				updateB();
			}
		✔ Impossible with synchronized method, Enables better concurrency

4️⃣ When Is Synchronized Method Acceptable?
	✔ Very small methods, Simple getters/setters, No performance concern, Single shared resource
	Example:
		public synchronized int getCount() {
			return count;
		}

5️⃣ Common Interview Traps (Avoid These)
	❌ “Synchronized method is faster”, “Block is always better”, “Method uses different lock”
	✔ Correct understanding matters.

🔹 Q15. What happens if multiple threads synchronize on different objects? Is synchronization reentrant?
=============================================================================================================
1️⃣ What Happens If Multiple Threads Synchronize on Different Objects?
	🔹 Short, Correct Answer: They do NOT block each other.
	Synchronization in Java is per lock object, not per method or per code.
	🔍 Why? Each object in Java has its own intrinsic (monitor) lock.
		If threads synchronize on different objects, they acquire different locks, so they can execute in parallel.
		🧪 Code Example
					class Test {
						private final Object lock1 = new Object();
						private final Object lock2 = new Object();

						public void method1() {
							synchronized (lock1) {
								System.out.println("Thread in lock1");
								sleep();
							}
						}

						public void method2() {
							synchronized (lock2) {
								System.out.println("Thread in lock2");
								sleep();
							}
						}

						private void sleep() {
							try { Thread.sleep(2000); } catch (Exception ignored) {}
						}
					}

		If: Thread A calls method1(), Thread B calls method2() 👉 Both threads run concurrently because: lock1 ≠ lock2
		
🔥 Interview Line (Important): “Synchronization only blocks threads competing for the same lock object.”
⚠ Common Trap
			synchronized (new Object()) {
				// useless synchronization
			}
	❌ Every thread gets a new lock, No mutual exclusion, Synchronization becomes meaningless

2️⃣ Is Synchronization Reentrant? ✅ YES — Java synchronization is reentrant.
	🔹 What Does Reentrant Mean? A thread that already holds a lock can re-acquire the same lock again without getting blocked.
		The JVM keeps a lock hold count for each thread.
		🧪 Code Example (Very Important)
					class ReentrantDemo {

						public synchronized void methodA() {
							System.out.println("Inside methodA");
							methodB(); // re-acquires same lock
						}

						public synchronized void methodB() {
							System.out.println("Inside methodB");
						}
					}
		✔ Same thread, Same object lock (this), No deadlock, Works smoothly
	🔍 What Happens Internally?
		Thread acquires lock → hold count = 1
		Calls another synchronized method
		Lock re-acquired → hold count = 2
		Exits methodB() → count = 1
		Exits methodA() → count = 0 → lock released
		❌ If Synchronization Were NOT Reentrant: This code would deadlock — but it doesn’t. That’s why reentrancy is essential.

3️⃣ ReentrantLock vs synchronized (Quick Note)
	| Feature    | synchronized | ReentrantLock |
	| ---------- | ------------ | ------------- |
	| Reentrant  | ✅ Yes 	    | ✅ Yes  	    |
	| Lock count | JVM-managed  | Explicit  	|
	| Try lock   | ❌     	    | ✅    	    |

🔹 Q16. Can a constructor be synchronized? Can we synchronize on null?
=============================================================================================================
1️⃣  ❌ No, a constructor cannot be synchronized.
		class Test {
			public synchronized Test() { } // ❌ compile-time error
		}
	❓ Why Not? Because synchronization requires a lock, and: Instance synchronization uses this as the lock
	During constructor execution, the object is not fully constructed, this is not safely available as a lock yet
	👉 Locking an object that isn’t fully initialized would break JVM safety guarantees.

🔹 JVM Design Reason (Senior Insight)
	Constructors are used to create the object, Synchronization is used to protect shared state
	Before construction completes, the object: Should not be shared, Should not be locked
	That’s why Java forbids synchronized constructors at language level.

✅ How to Achieve Constructor-like Synchronization (Correct Way)
	If you need thread safety during initialization:
	✔ 1: Use synchronized block inside constructor (rare but allowed)
				class Test {
					private static final Object LOCK = new Object();

					public Test() {
						synchronized (LOCK) {
							// thread-safe initialization
						}
					}
				}
	✔ 2 (Better): Use factory methods
				class Test {
					private Test() {}

					public static synchronized Test create() {
						return new Test();
					}
				}

2️⃣ Can We Synchronize on null? ❌ No — synchronizing on null causes a runtime exception.
	Object lock = null;
		synchronized (lock) {   // ❌
			// code
		}
	💥 What Happens? ➡ JVM throws NullPointerException immediately.
	❓ Why? Because: synchronized requires a real object, JVM needs the object’s monitor, null has no monitor
		So the JVM cannot acquire a lock.
	🔹 Important Clarification: This is not a compile-time error — it fails at runtime.

🔹 Q17. What happens if an exception occurs inside synchronized block?
=============================================================================================================
✅ The lock is automatically released when the thread exits the synchronized block, even if it exits due to an exception.
	This is guaranteed by the JVM.
	1️⃣ Simple Example : 	Object lock = new Object();
		synchronized (lock) {
			System.out.println("Lock acquired");
			int x = 10 / 0;   // ArithmeticException
		}

		🔍 What Happens Step by Step? 
			Thread acquires the monitor lock on lock, Exception occurs, JVM executes monitorexit automatically, Lock is released
			Exception propagates up the call stack
		✔ No deadlock, Other threads can acquire the lock

	2️⃣ Why Lock Is Always Released (JVM Guarantee)
		At bytecode level, synchronized uses:
			monitorenter
			   // critical section
			monitorexit
		The JVM ensures monitorexit runs: On normal exit, On exceptional exit
		👉 Similar to how finally works.

	3️⃣ With try–catch–finally (Common Interview Confusion)
			synchronized (lock) {
				try {
					riskyOperation();
				} catch (Exception e) {
					handle(e);
				}
			}
		or even:

			synchronized (lock) {
				riskyOperation();
			}
		✔ In both cases, the lock is released.

	4️⃣ What If the Exception Is Uncaught?
			synchronized (lock) {
				throw new RuntimeException("Boom");
			}
		
		Lock is released: Thread terminates, JVM may invoke UncaughtExceptionHandler, Other threads continue normally
		👉 Lock release is independent of exception handling.

	5️⃣ Important Clarification: What Is Not Released Automatically
		⚠ Only intrinsic (monitor) locks are auto-released.
		❌ With ReentrantLock
					lock.lock();
					doSomething();
					throw new RuntimeException(); // ❌ lock NOT released automatically
				You must unlock explicitly:

					lock.lock();
					try {
						doSomething();
					} finally {
						lock.unlock();
					}
				This contrast is something interviewers love.

	6️⃣ Can an Exception Cause Deadlock? 	❌ Not with synchronized alone, because:
		JVM always releases the monitor 
		✔ Deadlocks come from: Multiple locks, Circular wait, Bad lock ordering

🔹 Q18. Does synchronized guarantee fairness? What is intrinsic lock / monitor lock?
=============================================================================================================
1️⃣ ❌ No — synchronized does NOT guarantee fairness
	🔹 What “fairness” means? Threads acquire the lock in the order they requested it, No thread is starved indefinitely
	🔹 Why synchronized Is Not Fair: 
		synchronized uses intrinsic (monitor) locks
		JVM does not maintain a FIFO queue for threads waiting on a monitor
		When a lock is released, any waiting thread may acquire it
		Scheduling is influenced by: OS thread scheduler, JVM optimizations, Timing and CPU availability
		👉 Result: No ordering guarantee
		🧪 Example (Starvation Possible)
			synchronized (lock) {
				// critical section
			}
		If: Thread A releases the lock, Threads B, C, D are waiting
		➡ JVM may allow any of them to enter next, ➡ Thread B could starve if unlucky
		🔥 Interview Line: “synchronized provides mutual exclusion and visibility, but it does not guarantee fairness or ordering of lock acquisition.”
	
	🔹 Why Java Designed It This Way (Senior Insight)
		Fair locking: Requires maintaining queues, Adds overhead, Reduces throughput
		synchronized is optimized for: Performance, Low contention, Fast uncontended paths
		✅ If You Need Fairness
			Use ReentrantLock with fairness enabled:
				Lock lock = new ReentrantLock(true); // fair lock
				⚠ Even fair locks: Reduce throughput, Still depend on OS scheduling

2️⃣ What Is an Intrinsic Lock / Monitor Lock?
	🔹 An intrinsic lock (also called a monitor lock) is the built-in lock that every Java object has, which is used by the synchronized keyword. 👉 Every object in Java has exactly one monitor

	🔹 How It Works
		synchronized (obj) {
			// critical section
		}
		Thread tries to acquire obj’s monitor, If free → thread enters, If held → thread becomes BLOCKED
		On exit → monitor is released automatically

	🔹 Types of Intrinsic Locks
		| Synchronization Type         | Lock Used         |
		| ---------------------------- | ----------------- |
		| Instance synchronized method | `this`            |
		| Static synchronized method   | `ClassName.class` |
		| Synchronized block           | Specified object  |

	🔹 Monitor Responsibilities
		A monitor provides: Mutual exclusion (only one thread inside), Memory visibility guarantees, wait/notify mechanism
		That’s why: wait(), notify(), notifyAll() are in Object
		They operate on the object’s monitor
		🔍 JVM Internals (Senior-Level Detail): Monitor metadata stored in object header (Mark Word)
		Implemented using: Biased locks, Lightweight locks (CAS), Heavyweight OS monitors (under contention)

3️⃣ Intrinsic Lock vs Explicit Lock (Quick Contrast)
	| Aspect       | Intrinsic Lock (`synchronized`) | Explicit Lock (`ReentrantLock`) |
	| ------------ | ------------------------------- | ------------------------------- |
	| Fairness     | ❌ No             	             | ✅ Optional      	           |
	| Timeout      | ❌ No               	         | ✅ Yes                  	       |
	| Try lock     | ❌ No                    	     | ✅ Yes                 	       |
	| Auto release | ✅ Yes                	         | ❌ Manual                	   |
	| wait/notify  | Object methods                  | Condition objects               |

🔹 Q19. How does JVM implement synchronized internally?
=============================================================================================================
1️⃣ High-Level Answer (Start Like This in Interview)
“JVM implements synchronized using monitor locks associated with every object. At bytecode level it uses monitorenter and monitorexit, and internally it optimizes locking through biased locks, lightweight locks, and heavyweight (OS) monitors depending on contention.”

2️⃣ Every Object Has a Monitor 
	In Java: Every object is associated with a monitor (intrinsic lock)
		synchronized means acquire the object’s monitor
				synchronized (obj) {
					// critical section
				}
		👉 Lock = monitor of obj

3️⃣ Bytecode Level: monitorenter / monitorexit
	When you compile synchronized code, the JVM inserts bytecode instructions:
		monitorenter
		  // critical section
		monitorexit
	For synchronized methods: JVM implicitly adds these instructions at method entry & exit
	✔ JVM guarantees monitorexit is executed,Even if an exception occurs

4️⃣ Where Is Lock Information Stored? (Important)
	🔹 Object Header (Mark Word): Each object has a header containing a Mark Word, which stores:
		Lock state, Thread ID (in some modes), Hash code, GC info
		This is where synchronization metadata lives.

5️⃣ Lock States & Optimizations (Most Important Part)
	JVM does NOT always use OS-level locks. It uses adaptive locking based on contention.
	🔥 Lock States (Evolution)
				No Lock
				  ↓
				Biased Lock
				  ↓
				Lightweight Lock
				  ↓
				Heavyweight Lock (OS Monitor)

6️⃣ Biased Locking (Fast Path – No Contention)
	🔹 What It Is: Lock is biased toward one thread, No atomic instructions needed, No context switching
	🔹 When Used: Same thread repeatedly enters synchronized block, Single-threaded access pattern
	✔ Very fast ❌ Breaks when another thread tries to acquire lock
	Note: Biased locking is disabled/removed in newer Java versions, but conceptually important.

7️⃣ Lightweight Lock (CAS-Based)
	🔹 What Happens: JVM uses CAS (Compare-And-Swap), Thread tries to acquire lock without blocking, Uses spin locking
		✔ No OS involvement, ✔ Efficient under low contention

8️⃣ Heavyweight Lock (Monitor / OS Mutex)
	🔹 When It Happens: High contention, CAS/spin fails repeatedly
	🔹 What JVM Does: Lock is inflated, Thread is blocked, OS scheduler manages the lock
	❌ Context switching, ❌ Kernel involvement ❌ Expensive

9️⃣ Lock Inflation & Deflation
	JVM inflates lock when contention increases, JVM may deflate lock later if contention drops
	Happens dynamically at runtime, This is why modern synchronized is much faster than older Java versions.

🔟 How Reentrancy Is Implemented
	JVM tracks: Owning thread, Lock hold count, Same thread can re-enter the lock: Hold count increments, Lock released only when count reaches zero

1️⃣1️⃣ Memory Visibility Guarantees: synchronized provides happens-before guarantees:
	On monitorenter: Thread reads fresh values from main memory
	On monitorexit: Thread flushes changes to main memory
	✔ Solves visibility, Solves ordering, Solves atomicity

1️⃣2️⃣ What synchronized Does NOT Do
	❌ No fairness guarantee, ❌ No timeout, ❌ No deadlock prevention
	That’s why ReentrantLock exists.

1️⃣3️⃣ Why synchronized Became Fast (Senior Insight)
	Before Java 6: Always heavyweight locks, Very slow, After Java 6+: Biased locking, Thin locks, Lock coarsening, Lock elimination
	👉 Modern synchronized is highly optimized and often good enough.

🔹 Q20. Why are wait(), notify(), notifyAll() in Object class, not Thread? Why must wait/notify be called inside synchronized block?
=============================================================================================================
1️⃣  Because wait/notify/notifyAll work on object monitors (locks), not on threads.
	🔍 The Core Idea: Coordination Is About Shared Objects
		Threads do not wait on threads — threads wait on shared resources (objects).
			synchronized (queue) {
				while (queue.isEmpty()) {
					queue.wait();   // waiting on queue’s monitor
				}
			}
		Here: The object (queue) is the coordination point, Multiple threads can wait on the same object, The lock + condition belongs to the object, 👉 That’s why these methods live in Object.

	🔹 What a Monitor Actually Provides: Every object’s monitor provides:
		Mutual exclusion (via synchronized), A wait set (threads waiting on this object), Notification mechanism (notify/notifyAll)
		Since these are object-level features, the methods belong to Object.
	
	❌ Why Putting Them in Thread Would Be Wrong
			If they were in Thread:
				Which thread would you notify?
				How would multiple threads wait on the same condition?
				How would you associate a condition with a shared resource?
				It breaks the shared-resource coordination model.

2️⃣ Why Must wait() / notify() Be Called Inside a synchronized Block?
	🔹 Because the calling thread must own the object’s monitor.
		If not → JVM throws IllegalMonitorStateException.
	
3️⃣ What Happens Internally (This Is the Key)
	▶ When wait() is called: Thread must already hold the object’s monitor
		JVM: Releases the monitor, Moves thread to the object’s wait set, Puts thread in WAITING or TIMED_WAITING
			synchronized (lock) {
				lock.wait(); // releases lock + waits atomically
			}
		This atomic release + wait is crucial.

	▶ When notify() / notifyAll() is called
		Thread must hold the same monitor
		JVM: Moves waiting thread(s) from wait set → entry set, They compete to re-acquire the lock
		Lock is released only when synchronized block exits

4️⃣ Why Synchronization Is Mandatory (Critical Reason)
	🔥 To Prevent Race Conditions & Lost Notifications
		Without synchronization: 
			A thread could call notify() before another thread actually starts wait()
			Notifications could be lost, State could be inconsistent
		Synchronization ensures:
			Visibility of shared state, Atomicity of “check condition → wait”, Correct coordination

5️⃣ What Happens If You Don’t Use synchronized?
	lock.wait();   // ❌
	lock.notify(); // ❌
	➡ JVM throws: IllegalMonitorStateException
	Because: Thread does not own the monitor, JVM cannot safely release/acquire the lock

6️⃣ Correct Usage Pattern (Must Know)
	✅ Always use wait in a loop
		synchronized (lock) {
			while (!condition) {
				lock.wait();
			}
		}
	And notify after state change:
		synchronized (lock) {
			condition = true;
			lock.notifyAll();
		}
	This handles: Spurious wakeups, Missed signals, Visibility issues

🔹 Q21. Difference between wait() and sleep(). What happens internally when wait() is called? Why should wait always be used inside a loop?
=============================================================================================================
1️⃣ Difference Between wait() and sleep()
	| Aspect                | `wait()`                       | `sleep()`               |
	| --------------------- | ------------------------------ | ----------------------- |
	| Class                 | `Object`                       | `Thread`                |
	| Purpose               | Inter-thread communication     | Pause execution         |
	| Releases lock         | ✅ Yes              	         | ❌ No          	       |
	| Requires synchronized | ✅ Yes               	         | ❌ No           	       |
	| Wakes up by           | notify / notifyAll / interrupt | Time expiry / interrupt |
	| Thread state          | WAITING / TIMED_WAITING        | TIMED_WAITING           |
	| Coordination          | Yes                            | No                      |

	🔍 Code Comparison
	sleep() example
		synchronized (lock) {
			Thread.sleep(1000); // holds lock
		}
	❌ Other threads cannot enter synchronized block, ✔ Used for timing, delays

	wait() example
		synchronized (lock) {
			lock.wait(); // releases lock
		}
	✔ Other threads can acquire lock, ✔ Used for coordination

2️⃣ What Happens Internally When wait() Is Called?
	🔹 Preconditions: Thread must own the object’s monitor, Otherwise → IllegalMonitorStateException
	🔧 Internal Steps (Very Important)
			When a thread calls wait():
				Thread verifies ownership of the monitor, JVM releases the monitor lock, Thread is placed into the object’s wait set
				Thread state becomes: WAITING (wait), TIMED_WAITING (wait(timeout))
				Thread stays dormant until: notify(), notifyAll(), interrupt, spurious wakeup
					RUNNABLE
					   ↓ wait()
					releases lock
					   ↓
					WAITING (wait set)
	
	🔹 On notify / notifyAll: 
		Thread moves from wait set → entry set, Thread must re-acquire the same lock
		Only then it continues execution, ⚠ notify does NOT release the lock immediately
		Lock is released when notifier exits synchronized block.

3️⃣ Why Must wait() Always Be Used Inside a Loop?
	🔥 This is a MUST-KNOW correctness rule.
	❌ Problem with if
			synchronized (lock) {
				if (!condition) {
					lock.wait();
				}
				// unsafe
			}
		This code is broken.

	🔴 Reason 1: Spurious Wakeups: 	JVM is allowed to wake a waiting thread without notify.
				If that happens: if condition is not rechecked, Thread continues with invalid state

	🔴 Reason 2: Lost / Wrong Notification
				With multiple waiting threads: notify() may wake wrong thread, Condition may still be false

	🔴 Reason 3: Race Conditions
				Between: notify, re-acquiring lock, State may change again.

	✅ Correct Pattern: while
		synchronized (lock) {
			while (!condition) {
				lock.wait();
			}
			// safe to proceed
		}
		✔ Condition is rechecked, Handles spurious wakeups, Handles missed notifications, Thread-safe and correct

🔹 Q22. Difference between notify() vs notifyAll(). When should notifyAll be preferred? What happens if notify is called without any waiting thread?
=============================================================================================================
1️⃣ Difference Between notify() and notifyAll()
	| Aspect           | `notify()`                       | `notifyAll()`              |
	| ---------------- | -------------------------------- | -------------------------- |
	| Wakes up         | **One arbitrary waiting thread** | **All waiting threads**    |
	| Thread selection | JVM chooses (no order guarantee) | All are moved to entry set |
	| Risk             | Can wake wrong thread            | Safer                      |
	| Performance      | Slightly better                  | More overhead              |
	| Correctness      | Risky                            | Correct in most cases      |

🔍 Important Detail
	notify() wakes one thread from the object’s wait set
	notifyAll() wakes all waiting threads
	Woken threads still must re-acquire the lock before running
	⚠ Neither method releases the lock immediately — that happens only when the synchronized block exits.

2️⃣ When Should notifyAll() Be Preferred?
	Prefer notifyAll() whenever multiple conditions or multiple types of threads may be waiting on the same lock.
	🔹 Real Reasons (Very Important)
	✅ 1. Multiple Conditions on Same Lock
			Example: Producer threads waiting for space, Consumer threads waiting for data
				Using notify(): JVM may wake wrong type of thread, Condition still false → thread waits again
					Risk of livelock / dead wait
					
				notifyAll(): All threads wake, Only threads with true condition proceed, Others go back to waiting safely

	✅ 2. Avoiding Lost Signals
			With notify(): Wrong thread may be notified, Correct thread remains waiting indefinitely
			With notifyAll(): No missed signals, Correctness over performance
	
	✅ 3. Simpler & Safer Code
			notifyAll() + while(condition) is the recommended pattern.
				synchronized (lock) {
					while (!condition) {
						lock.wait();
					}
				}

3️⃣ Performance Concern with notifyAll()
	Yes: Wakes all threads, Context switching overhead
	But: Correctness > performance, JVMs handle this efficiently, Over-optimizing with notify() causes bugs

4️⃣ What Happens If notify() Is Called When No Thread Is Waiting?
	Nothing happens — the notification is lost.
	🔍 Explanation
			synchronized (lock) {
				lock.notify(); // nobody waiting
			}
		Later:
			synchronized (lock) {
				lock.wait();   // waits forever
			}

		Why? Java does not queue notifications, notify has no memory, If no thread is waiting, signal is dropped
		⚠ Result: Potential dead wait, Common concurrency bug

5️⃣ Correct Pattern to Avoid This Bug
	Always use: Condition variable, Loop, notifyAll
		synchronized (lock) {
			while (!condition) {
				lock.wait();
			}
		}
		synchronized (lock) {
			condition = true;
			lock.notifyAll();
		}

🔹 Q23. What is spurious wakeup?
=============================================================================================================
👉A spurious wakeup occurs when a thread wakes up from wait() without: notify(), notifyAll(), timeout, interrupt
 It’s rare, but explicitly allowed by the Java specification.
	❓ Why Do Spurious Wakeups Exist?
		Because: JVM implementations use low-level OS primitives, For performance reasons, For safety under complex scheduling scenarios
		The JVM is allowed to wake a thread at any time.

	🔥 Key Rule: You must assume wait() can return at any time.

👉Why Spurious Wakeups Are Dangerous
	❌ Broken Code (Using if)
		synchronized (lock) {
			if (!condition) {
				lock.wait(); // may wake spuriously
			}
			// ❌ condition may still be false
		}
	If a spurious wakeup happens: Condition is not rechecked, Thread proceeds incorrectly, Data corruption / bugs

	✅ Correct Code (Using while)
		synchronized (lock) {
			while (!condition) {
				lock.wait();
			}
			// ✅ safe
		}

	✔ Rechecks condition, Handles spurious wakeups, Handles wrong notifications

👉Relationship Between notify & Spurious Wakeups
	Even without spurious wakeups:
		notify() may wake the wrong thread, That thread must recheck the condition, This is another reason while is mandatory.

🔹 Q24. What problem does volatile solve? Does volatile guarantee atomicity? Can volatile replace synchronization?
		Difference between volatile and synchronized.
=============================================================================================================
volatile solves the memory visibility problem, not atomicity.
	🔍 The Actual Problem: Visibility
			In multithreaded programs: 
				Each thread may cache variables in CPU registers / core-local caches
				Updates made by one thread may not be visible to others immediately
							class Worker {
								boolean running = true;

								void stop() {
									running = false;
								}

								void work() {
									while (running) {
										// may loop forever!
									}
								}
							}
				Even after stop() is called: work() thread may keep seeing cached running = true

		✅ How volatile Fixes This
			volatile boolean running = true;
			volatile guarantees: Reads always come from main memory, Writes are immediately flushed to main memory
			👉 All threads see the latest value.
			🔥 Interview Line : “volatile ensures visibility of changes across threads by preventing caching and reordering.”

2️⃣ Does volatile Guarantee Atomicity?
	❌ NO — volatile does NOT guarantee atomicity: This is one of the most important points.
	❌ Example (Broken Even with volatile)
		volatile int count = 0;
		count++; // NOT atomic
	
	Why? count++ is actually: Read count, Increment, Write back, Multiple threads can interleave these steps → lost updates.

	✅ What volatile is Atomic For: ✔ Single read, ✔ Single write
		volatile int x = 10; // atomic write
		int y = x;           // atomic read
		But not compound operations.

3️⃣ Can volatile Replace Synchronization?
	Sometimes — but only in very limited cases.
	✅ When volatile Is Enough
		Use volatile when: Only one thread writes, Other threads only read, No compound operations, No invariants to protect
		Example: volatile boolean shutdown = false;
		✔ Safe, Efficient, No locking overhead
		❌ When volatile Is NOT Enough: 
			Multiple writers, Increment/decrement, Check-then-act logic, Protecting invariants, Multiple shared variables
			Example:
				if (initialized) {   // check
					useResource();   // act
				}
			Needs synchronization.
		🔥 Senior Insight: 	“If you need mutual exclusion, volatile is the wrong tool.”

4️⃣ Difference Between volatile and synchronized
	| Aspect            | `volatile`   | `synchronized`      |
	| ----------------- | -------------| ------------------- |
	| Solves visibility | ✅ Yes       | ✅ Yes    	         |
	| Solves atomicity  | ❌ No        | ✅ Yes       	     |
	| Mutual exclusion  | ❌ No        | ✅ Yes        	     |
	| Thread blocking   | ❌ No        | ✅ Yes         	 |
	| Performance       | Very fast    | Slower (lock-based) |
	| Use case          | State flags  | Critical sections   |
	| Fairness          | ❌ No        | ❌ No       	     |
	| Deadlock risk     | ❌ No        | ✅ Yes    	         |

	🔹 Memory Semantics Difference
		volatile: Prevents instruction reordering, Establishes happens-before on read/write
		synchronized: Prevents reordering, Establishes happens-before on lock acquire/release, Also provides mutual exclusion

5️⃣ Visual Mental Model (Senior-Level)
	volatile → visibility + ordering
	synchronized → visibility + ordering + atomicity

6️⃣ Common Interview Traps (Avoid These ❌)
	❌ “volatile makes code thread-safe”, “volatile is lighter synchronized”, “volatile can replace locks everywhere”
	✔ Each tool has a specific purpose.

🔹 Q25. How volatile works internally (CPU cache + memory barriers)? Why volatile int i++ is not thread safe?
=============================================================================================================
1️⃣ How volatile Works Internally (CPU Cache + Memory Barriers)
	🔹 The Core Problem: CPU Caches
		Modern CPUs don’t read/write variables directly from RAM.
		Each core has: Registers, L1 / L2 / L3 caches
		So in multithreading: volatile boolean flag = true;
		Thread A may update flag in its CPU cache, Thread B may keep reading a stale cached value
		👉 This is the visibility problem.

2️⃣ What volatile Actually Does
	When you mark a variable as volatile, the JVM enforces special memory semantics using memory barriers (fences).
	🔥 Two Guarantees Provided by volatile: Visibility, Ordering (no dangerous reordering)

3️⃣ Memory Barriers (Key Concept)
	A memory barrier is a CPU instruction that restricts how memory operations can be reordered and cached.
	🔹 Volatile Write (volatileVar = x)
		JVM inserts a Store Barrier: Flushes the value to main memory, Invalidates cached copies in other cores
		Prevents previous writes from being reordered after the volatile write
			[Normal writes]
				↓
			StoreStore Barrier
				↓
			volatile write
				↓
			StoreLoad Barrier

	🔹 Volatile Read (x = volatileVar)
		JVM inserts a Load Barrier: Forces read from main memory, Prevents subsequent reads from being reordered before the volatile read
			volatile read
				↓
			LoadLoad Barrier
				↓
			[Normal reads]

4️⃣ Happens-Before Guarantee (Very Important)
	The Java Memory Model defines: A write to a volatile variable happens-before every subsequent read of that same variable.
	This is why volatile ensures visibility + ordering.

5️⃣ What Volatile Does NOT Do: 
	❌ Does NOT lock, Does NOT block threads, Does NOT provide mutual exclusion, Does NOT make compound operations atomic
	This leads us to the next question 👇

6️⃣ Why volatile int i++ Is NOT Thread Safe
	🔹 The Myth Many think:
		volatile int i; i++; is thread-safe because i is volatile.
		❌ This is wrong.

	🔍 What i++ Actually Means: i++ is equivalent to:
		int temp = i;   // volatile read
		temp = temp + 1;
		i = temp;       // volatile write

	That’s 3 separate operations.

🔥 The Problem: Lost Updates
		Consider two threads:
				volatile int i = 0;
				Thread A reads i = 0, increments to 1
				Thread B reads i = 0, increments to 1
				Both write back 1.
		👉 Final value = 1, not 2, 	👉 One update is lost
		Even though: Each read/write is visible, Operations are ordered, There is no mutual exclusion.
		
		8️⃣ Correct Ways to Fix i++
			✅ Option 1: synchronized
					synchronized void increment() {
						i++;
					}

			✅ Option 2: AtomicInteger (Best Practice)
					AtomicInteger i = new AtomicInteger(0);
					i.incrementAndGet();
					Uses CAS (Compare-And-Swap) + memory barriers.

			✅ Option 3: LongAdder (High Contention)
					LongAdder adder = new LongAdder();
					adder.increment();

		9️⃣ When Volatile IS the Right Tool: ✔ State flags, Shutdown signals, Configuration reload flags, One writer, many readers
		Example: volatile boolean shutdown;

✅ Yes, volatile can be used with object references
	volatile MyObject obj; But here’s the critical catch 👇
	🔥 What volatile Guarantees for Objects
		volatile on an object does NOT make the object thread-safe.
		It guarantees only that: The reference is visible to all threads
		Updates to the reference are immediately visible, The reference is safely published
		👉 It does NOT protect the object’s internal state.

		🧪 Example: Safe Reference Visibility, Unsafe State
		class Holder {

	volatile Config config;
	config = new Config(...); // safe publication

❌ When volatile is NOT enough
		❌ Mutating object fields, Multiple threads modifying same object, Protecting invariants

🔹 Q27. Explain heap, stack, registers, CPU cache in multithreading.
=============================================================================================================
1️⃣ Heap (Shared Memory): 
	Heap is a shared memory area which Stores: Objects, Instance variables, Static variables, Managed by Garbage Collector
		class User {
			int age;
		}
		User u = new User(); // object on heap

	🔹 Heap in Multithreading: ✔ Shared across all threads, ❌ Not thread-safe by default
		u.age = 30; // Thread A
		System.out.println(u.age); // Thread B
		Without synchronization: Thread B may see stale value Or partial state (unsafe publication)
	🔥 Key Interview Line: “Heap is shared among threads, which is why synchronization is required.”

2️⃣ Stack (Thread-Local Memory)
	Each thread has its own stack. Stack contains: Method call frames, Local variables, References to heap objects
		void process() {
			int x = 10;     // stack
			User u = new User(); // reference on stack, object on heap
		}
	
	🔹 Stack in Multithreading: ✔ Stack is thread-confined, ✔ No synchronization needed, ✔ Fast access
		int counter = 0; // local variable → thread-safe
	🔥 Key Interview Line: “Local variables live on the stack and are inherently thread-safe.”

3️⃣ CPU Registers (Per Thread, Ultra Fast)
	Smallest, fastest memory, Located inside CPU core
	Hold: Temporary variables, Instruction operands, Loop counters, JVM & JIT may keep variables in registers.
	
	🔹 Registers in Multithreading: ✔ Private to a CPU core / thread, ❌ Other threads cannot see register values
		while (running) { } // running may stay in register!
		This is why visibility problems happen.
	🔥 Key Interview Line: “Threads may read variables from registers instead of main memory.”

4️⃣  CPU Cache (Root Cause of Visibility Bugs)
	Faster than RAM, slower than registers
	Levels: L1 (per core), L2 (per core), L3 (shared)
	Each core may cache memory independently.
	
	🔹 CPU Cache in Multithreading
		Thread A: sharedFlag = false;
		Thread B: while (sharedFlag) { }
		Problem: Thread B may keep reading cached true, Never sees update by Thread A

	🔥 This Is the Visibility Problem, Solved by: volatile, synchronized, Locks, Atomic variables

5️⃣  How Java Fixes This (JMM Rules)
	🔹 synchronized: Flushes thread’s writes to heap on exit, Invalidates other threads’ caches on entry
	🔹 volatile: Forces reads/writes to main memory, Inserts memory barriers
	🔹 Happens-Before: Defines when writes become visible to other threads

7️⃣ Putting It All Together (Very Important)
	| Memory Area | Shared?       | Thread-Safe? | Used For                |
	| ----------- | --------------| ------------ | ------------------------|
	| Heap        | ✅ Yes        | ❌ No    	 | Objects, shared state   |
	| Stack       | ❌ No         | ✅ Yes     	 | Local variables         |
	| Registers   | ❌ No         | ✅ Yes     	 | Fast execution          |
	| CPU Cache   | ❌ (per core) | ❌ No      	 | Performance optimization|

8️⃣ Why Multithreading Bugs Happen (Senior Insight)
	Because: 
		Threads don’t read from heap every time
		CPU caches + registers cause stale reads
		JVM allows reordering for performance
	👉 Java gives tools to restore correctness, but you must use them correctly.

🔹 Q28. What are memory barriers / fences? Explain happens-before rules.
=============================================================================================================
1️⃣  Memory barriers (or fences) are low-level CPU instructions that restrict reordering of memory operations and control visibility of reads/writes across threads.
	👉 They tell the CPU and compiler: “Do NOT move memory operations across this point.”

2️⃣ Why Memory Barriers Exist
	Modern systems optimize aggressively: CPUs reorder instructions, Compilers reorder instructions, CPU cores cache values locally
	These optimizations can break multithreaded correctness.
	Memory barriers restore order and visibility when required.

3️⃣ Types of Memory Barriers (Conceptual)
	You’ll hear these names in JVM discussions:
		🔹 1. LoadLoad Barrier: Prevents loads after the barrier from moving before it
				Load A, LoadLoad, Load B   // cannot move before Load A

		🔹 2. StoreStore Barrier: Prevents stores after the barrier from moving before it
				Store A, StoreStore, Store B  // cannot move before Store A
		
		🔹 3. LoadStore Barrier: Prevents stores after the barrier from moving before earlier loads
		🔹 4. StoreLoad Barrier (Strongest & Most Expensive): Prevents loads after the barrier from moving before earlier stores
				Store A, StoreLoad, Load B  // cannot move before Store A
			⚠ This is the most expensive fence and is critical for volatile.

4️⃣ How Java Uses Memory Barriers: Java does not expose memory barriers directly, but inserts them under the hood for:
	✔ volatile
			Volatile write → StoreStore + StoreLoad barriers
			Volatile read → LoadLoad + LoadStore barriers

	✔ synchronized
			Lock acquire → Load barriers
			Lock release → Store barriers
	
	✔ Atomic classes (CAS): Use volatile semantics + CPU fences

5️⃣ What Is the Happens-Before Relationship?
	🔹 Definition : If action A happens-before action B, then all effects of A are visible to B, and A is ordered before B.
		Happens-before is a guarantee, not a timing statement.

6️⃣ Core Happens-Before Rules (Must Know)
	These rules explain when visibility is guaranteed.
		🔥 1. Program Order Rule: Within a single thread, actions happen in program order.
				x = 10;
				y = 20;
				✔ x = 10 happens-before y = 20 (in same thread)

		🔥 2. Monitor Lock Rule (synchronized): An unlock on a monitor happens-before every subsequent lock on the same monitor
				synchronized (lock) {
					shared = 42;
				} // unlock

				synchronized (lock) {
					System.out.println(shared); // guaranteed 42
				}
				✔ Visibility guaranteed, ✔ Ordering guaranteed

		🔥 3. Volatile Variable Rule: A write to a volatile variable happens-before every subsequent read of that variable
				volatile boolean ready = false;
				Thread A: data = 100; ready = true;
				Thread B: while (!ready) {} System.out.println(data); // guaranteed 100
				✔ This is the classic safe publication pattern.

		🔥 4. Thread Start Rule: Calling thread.start() happens-before any action in the started thread
				t.start();
				// everything before start() visible to t

		🔥 5. Thread Termination Rule: All actions in a thread happen-before another thread detects termination via: join()
				isAlive() == false

				t.join();
				// all t’s writes visible here

		🔥 6. Transitivity Rule
				If: A happens-before B, B happens-before C Then: A happens-before C
				This allows chaining guarantees.

7️⃣ Why Happens-Before Matters (Senior Insight)
	Without happens-before: Threads may see stale values, Objects may appear partially constructed, Bugs become non-deterministic
	Happens-before defines what is legal, not what is fast.

8️⃣ Happens-Before vs Synchronization (Clarification)
	| Concept        | Purpose                    |
	| -------------- | -------------------------- |
	| Memory barrier | Low-level CPU constraint   |
	| Happens-before | High-level JMM rule        |
	| synchronized   | Establishes happens-before |
	| volatile       | Establishes happens-before |

🔹 Q29. How does synchronization affect memory visibility? Why double-checked locking was broken earlier?
=============================================================================================================
1️⃣ Synchronization establishes a happens-before relationship that guarantees memory visibility between threads.
	🔍 What Problem Exists Without Synchronization?
		Threads may: Read values from CPU registers / caches, See stale data, Observe partially constructed objects
		Because the JVM and CPU are free to: Cache variables, Reorder instructions
	
	🔑 What Synchronization Guarantees
		✅ 1. Visibility: Writes by one thread become visible to other threads
		✅ 2. Ordering: Instructions are not reordered across synchronization boundaries

	🔬 What Happens Internally (Very Important)
		▶ On lock acquisition (monitorenter): Thread invalidates its local caches, Reads fresh values from main memory
		▶ On lock release (monitorexit): Thread flushes all writes to main memory
			🧪 Example
						int shared = 0;

						synchronized (lock) {
							shared = 42;
						} // unlock → writes flushed to main memory

						synchronized (lock) {
							System.out.println(shared); // guaranteed to see 42
						}
			This works because: Unlock happens-before subsequent lock on the same monitor

	🎯 Interview Line :	“Synchronization provides memory visibility by flushing writes on lock release and invalidating caches on lock acquisition.”

2️⃣ Why Was Double-Checked Locking Broken Earlier?
	This is where deep JVM knowledge shows.

3️⃣ What Is Double-Checked Locking?
class Singleton {
    private static Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) {              // 1st check
            synchronized (Singleton.class) {
                if (instance == null) {      // 2nd check
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}


Intent: Avoid synchronization after initialization, Improve performance

4️⃣ Why It Was Broken (Before Java 5)
	🔥 Root Cause: Instruction Reordering + Visibility
	Object creation is not atomic.
	This line: instance = new Singleton(); can be reordered internally as: Allocate memory, Assign reference to instance, Run constructor

⚠ Steps 2 and 3 could be reordered.

🧪 Broken Scenario (Very Important)
		Thread A
		instance = new Singleton();

		Reordering occurs: instance points to memory, Constructor not finished yet
		
		Thread B
		if (instance != null) {
			return instance; // sees partially constructed object
		}


		👉 Thread B observes: instance != null, But object fields are not initialized
		This caused: Random crashes, Weird null values, Impossible-looking bugs
	
	🎯 Key Insight: Synchronization alone did NOT prevent reordering before Java 5.

5️⃣ Why Java 5+ Fixed It
	Java 5 introduced: A stronger Java Memory Model, clear happens-before guarantees, Proper volatile semantics

6️⃣ Correct Double-Checked Locking (Java 5+)
		class Singleton {
			private static volatile Singleton instance;

			public static Singleton getInstance() {
				if (instance == null) {
					synchronized (Singleton.class) {
						if (instance == null) {
							instance = new Singleton();
						}
					}
				}
				return instance;
			}
		}
	🔑 Why volatile Fixes It: volatile guarantees: No instruction reordering, Safe publication, Visibility of fully constructed object
	Specifically: Constructor must complete before reference is visible

7️⃣ Why synchronized Alone Was Not Enough
	Before Java 5: JVM allowed reordering across synchronization boundaries, Memory model was weak and underspecified
	After Java 5: Lock release + acquire creates strong happens-before, Volatile adds StoreLoad memory barriers

8️⃣ Modern Best Practice (Senior Tip)
	Instead of double-checked locking:
		✅ Use Initialization-on-Demand Holder Idiom
		class Singleton {
			private Singleton() {}

			private static class Holder {
				static final Singleton INSTANCE = new Singleton();
			}

			public static Singleton getInstance() {
				return Holder.INSTANCE;
			}
		}

		✔ Thread-safe, ✔ Lazy, ✔ No explicit synchronization, ✔ Clean and fast

🔹 Q30. How does volatile affect instruction reordering? What is out-of-order execution?
=============================================================================================================
1️⃣ What Is Instruction Reordering?
	Instruction reordering is an optimization where the compiler, JVM, or CPU changes the execution order of instructions to improve performance, as long as single-threaded semantics are preserved.

	Key point: Reordering is legal within a thread, but dangerous across threads.

	🧪 Simple Example
					a = 1;
					b = 2;
			The JVM/CPU may reorder this to:
					b = 2;
					a = 1;
			✔ Same result in one thread, ❌ Can break visibility in multi-threading

2️⃣ What Is Out-of-Order Execution?
	Out-of-order execution is a CPU-level optimization where the processor executes instructions as soon as their operands are ready, rather than strictly following program order.
	This is done to: Keep CPU pipelines busy, Reduce stalls, Improve throughput
	
	🔍 Important Distinction
		| Term                   | Level                 |
		| ---------------------- | --------------------- |
		| Instruction reordering | Compiler / JVM / CPU  |
		| Out-of-order execution | CPU microarchitecture |

	👉 Out-of-order execution is one cause of instruction reordering.

	🧠 Example (CPU Perspective)
					x = y + 1;   // depends on y
					a = 5;       // independent
			
			CPU may execute: 
					a = 5
					x = y + 1
			Even if written in opposite order.

3️⃣ Why Reordering Is a Problem in Multithreading
	Consider this classic pattern:
			int data;
			boolean ready;

			Thread A: data = 42; ready = true;
			Thread B: if (ready) {System.out.println(data);}
		
		❌ Without proper memory barriers: Reordering may happen:
			ready = true;
			data = 42;

			Thread B sees: ready == true, but data == 0
		👉 Visibility + ordering bug

4️⃣ How Does volatile Affect Instruction Reordering?
	🔥 Core Answer: volatile prevents specific types of instruction reordering by inserting memory barriers around volatile reads and writes.

5️⃣ Memory Barriers Inserted by volatile
	🔹 Volatile Write: volatileVar = value;
		JVM inserts: 
			StoreStore barrier → prevents previous writes from moving after
			StoreLoad barrier → prevents subsequent reads from moving before
		
		Guarantee: All writes before the volatile write become visible before the volatile write.

	🔹 Volatile Read: value = volatileVar;
		JVM inserts:
			LoadLoad barrier → prevents later reads from moving before
			LoadStore barrier → prevents later writes from moving before
		
		Guarantee: All reads after the volatile read see effects after the volatile read.

6️⃣ Happens-Before Rule (Volatile) : 
	A write to a volatile variable happens-before every subsequent read of that same variable.
	This is the formal JMM rule that volatile enforces.

7️⃣ Visual Timeline (Senior-Level)
	Without volatile (broken)
		Thread A: data = 42   ready = true
		Thread B: if (ready) print(data) // may print 0

	With volatile (correct)
		Thread A: data = 42  → volatile write ready = true
		Thread B: volatile read ready → print(data) // guaranteed 42

8️⃣ What volatile Does NOT Prevent
	❌ Does not prevent: All reordering everywhere, Race conditions, Lost updates, Non-atomic compound operations
			volatile int i;
			i++; // still unsafe

9️⃣ Why Out-of-Order Execution Still Exists with volatile
	Important clarification: 
		CPUs still execute out-of-order internally
		Memory barriers ensure results appear ordered to other threads
	👉 Volatile controls visibility and ordering, not CPU internals.

🔹 Q31. How was it fixed in Java 5+? What is safe publication?
=============================================================================================================
1️⃣ Java 5 fixed concurrency issues by introducing a stronger Java Memory Model that clearly defined happens-before rules and strengthened volatile semantics. This ensured proper memory visibility, prevented illegal instruction reordering, and made patterns like double-checked locking safe when used with volatile. Safe publication refers to the practice of making an object visible to other threads only after it is fully constructed, ensuring that all threads see a consistent and initialized state. This can be achieved using volatile references, synchronization, final fields, static initialization, or thread start semantics.

2️⃣ What Is Safe Publication?
🔹 Ensuring that an object is made visible to other threads only after it is fully constructed, so that all threads see a consistent and initialized state.
	Ways to Achieve Safe Publication (Must Know)
	✅ 1. Using volatile Reference: 
		volatile Holder holder; holder = new Holder(); // safely published
		✔ Fully constructed before visible, ✔ Very common pattern

	✅ 2. Using synchronized
			synchronized void init() {
				holder = new Holder();
			}

			synchronized Holder get() {
				return holder;
			}
		✔ Lock release → lock acquire establishes happens-before

	✅ 3. Using final Fields (Immutable Objects)
			class Config {
				final int port;

				Config(int port) {
					this.port = port;
				}
			}
		✔ Java 5+ guarantees visibility of final fields, ✔ Best practice

	✅ 4. Static Initialization (Class Loading)
			class Singleton {
				static final Singleton INSTANCE = new Singleton();
			}

		✔ Class initialization is thread-safe, ✔ Happens-before guaranteed by JVM

	✅ 5. Thread Start Rule
			Holder h = new Holder();
			new Thread(() -> use(h)).start();
		✔ Everything before start() is visible to the new thread

🔹 Q32. What is deadlock? Conditions required for deadlock. Real-world deadlock example. How to prevent deadlock?
=============================================================================================================
1️⃣ Deadlock is a situation where two or more threads are permanently blocked, each waiting for a resource held by another, so none of them can make progress.
	👉 The system is alive but stuck forever.
	
2️⃣ Conditions Required for Deadlock (Coffman Conditions)
	Deadlock occurs only if ALL four conditions are true.
		🔴 1. Mutual Exclusion: At least one resource is non-shareable, Only one thread can hold it at a time
				synchronized (lock) { }
		
		🔴 2. Hold and Wait: Thread holds one resource, While waiting for another
				lockA acquired → waiting for lockB
		
		🔴 3. No Preemption: Resource cannot be forcibly taken, Thread releases it voluntarily only, Java locks cannot be preempted.

		🔴 4. Circular Wait: Circular chain of threads waiting for resources
				T1 → lockA → lockB
				T2 → lockB → lockA
			🎯 Interview Line: “Deadlock occurs only when all four Coffman conditions are satisfied simultaneously.”

3️⃣ Real-World Deadlock Example
	🚦 Traffic Intersection (Classic)
			Car A blocks road 1, waiting for road 2
			Car B blocks road 2, waiting for road 1
		Neither can move ➡ Deadlock

	💳 Banking System Example
			Thread A locks Account X, waits for Account Y
			Thread B locks Account Y, waits for Account X
		Money transfer system freezes

	🧠 Database Example
			Transaction 1 locks row A, waits for row B
			Transaction 2 locks row B, waits for row A
		DB detects deadlock and kills one transaction
	(Java does not auto-resolve deadlocks.)

4️⃣ How to Prevent Deadlock (Most Important): Deadlock prevention = break at least ONE Coffman condition.
	
	✅ 1. Lock Ordering (Best & Most Common): Always acquire locks in a fixed global order.
			synchronized (lockA) {
				synchronized (lockB) {
					// safe
				}
			}
		✔ If everyone follows same order → no circular wait
		🎯 Interview Line: “Consistent lock ordering is the most effective deadlock prevention technique.”

	✅ 2. Use tryLock() with Timeout (Very Important): 
			if (lockA.tryLock(1, TimeUnit.SECONDS)) {
				try {
					if (lockB.tryLock(1, TimeUnit.SECONDS)) {
						try {
							// work
						} finally {
							lockB.unlock();
						}
					}
				} finally {
					lockA.unlock();
				}
			}
		✔ Breaks hold and wait, ✔ Avoids infinite blocking

	✅ 3. Avoid Nested Locks
		❌ Bad:
			synchronized (A) {
				synchronized (B) { }
			}
		✔ Better: Reduce lock scope, Refactor design

	✅ 4. Use High-Level Concurrency Utilities
		Instead of manual locking: BlockingQueue, ConcurrentHashMap, Semaphore, ExecutorService
		They are designed to avoid deadlocks.

	✅ 5. Use Single Lock Where Possible
		synchronized (globalLock) {
			// protect related resources together
		}
		✔ Fewer locks → fewer deadlocks

	✅ 6. Deadlock Detection (Not Prevention)
		In production: Use jstack, JVM thread dump, Monitoring tools
		But: Detection is recovery, not prevention

🔹 Q33. Difference between deadlock and livelock.
=============================================================================================================
1️⃣ What Is Deadlock? 
	Deadlock occurs when two or more threads are permanently blocked, each waiting for a resource held by another.
	🧠 Key Behavior: Threads are blocked, No CPU usage, System appears frozen
	🧪 Example (Deadlock)
					synchronized (lockA) {
						synchronized (lockB) { }
					}

					synchronized (lockB) {
						synchronized (lockA) { }
					}
		Thread-1 holds lockA → waits for lockB
		Thread-2 holds lockB → waits for lockA

2️⃣ What Is Livelock? 
	Livelock occurs when threads are not blocked, but they keep responding to each other in a way that prevents progress.
	🧠 Key Behavior: Threads are running, CPU usage is high, System is active but useless
	🧪 Example (Livelock with tryLock)
				while (true) {
					if (lockA.tryLock()) {
						if (lockB.tryLock()) {
							// work
							break;
						}
						lockA.unlock();
					}
					// retry politely
				}
		Both threads: Acquire one lock, Release it to avoid deadlock, Retry at the same time, Repeat forever

3️⃣ Real-World Analogy (Interview Gold)
	🚦 Deadlock: Two cars meet head-on on a narrow road, neither can move
	🚶 Livelock:
			Two people try to pass each other in a hallway
			Both step left, then right, repeatedly
			Both are moving, but no one passes
4️⃣ Deadlock vs Livelock — Comparison Table
	| Aspect             | Deadlock         | Livelock                    |
	| ------------------ | ---------------- | --------------------------- |
	| Thread state       | BLOCKED          | RUNNABLE                    |
	| CPU usage          | Low / none       | High                        |
	| Progress           | ❌ None   	    | ❌ None           	      |
	| Threads doing work | ❌ No   	        | ✅ Yes           	          |
	| Cause              | Circular waiting | Over-reaction / retry logic |
	| Visibility         | Easy to detect   | Harder to detect            |

5️⃣ How to Prevent Them
	🔹 Prevent Deadlock: Consistent lock ordering, tryLock() with timeout, Reduce nested locks
	🔹 Prevent Livelock: Add random backoff / delay, Limit retries, Introduce priority or fairness
			Thread.sleep(randomDelay());

🔹 Q34. What is starvation? Can synchronized cause starvation? How ReentrantLock helps avoid starvation?
=============================================================================================================
1️⃣ Starvation happens when a thread is perpetually denied access to a shared resource, so it never gets a chance to execute, even though the system as a whole keeps running.
	👉 The thread is alive, but never makes progress.
	🧠 Key Characteristics: 
		Thread waits indefinitely, Other threads keep acquiring the resource, 
		No circular dependency (unlike deadlock), Often caused by unfair scheduling or locking

	🧪 Simple Example (Conceptual)
			High-priority threads keep acquiring a lock
			Low-priority thread keeps waiting
			Low-priority thread starves

2️⃣ Can synchronized Cause Starvation?
	✅ Yes — synchronized can cause starvation
	Because synchronized: Uses intrinsic (monitor) locks, Does NOT guarantee fairness, Does NOT enforce FIFO ordering
	Lock acquisition depends on: JVM implementation, OS thread scheduler, Timing and CPU availability
	👉 A thread that has been waiting longer has no priority over a newly arrived thread.

		🧪 Example Scenario
				synchronized (lock) {
					// critical section
				}
		Thread A repeatedly acquires/releases the lock quickly
		Thread B is slower and keeps getting bypassed
		Thread B may never acquire the lock

	🎯 Interview Line: “synchronized ensures mutual exclusion, but it does not guarantee fairness, so starvation is possible.”

3️⃣ Other Common Causes of Starvation
	Thread priorities (high-priority threads dominate), Infinite loops holding locks, Excessive contention, Unfair locks, Busy retry loops

4️⃣ How Does ReentrantLock Help Avoid Starvation?
	🔥 Key Advantage: Fair Locking
			ReentrantLock allows you to create a fair lock.
				Lock lock = new ReentrantLock(true); // fair lock

	🔹 What Is a Fair Lock? 
		A fair lock grants access in FIFO order, The longest-waiting thread gets the lock next, Prevents newer threads from “cutting in line”
		👉 This directly reduces starvation.

	🧠 How It Works (High Level)
		Threads are queued internally
		When lock is released: JVM gives it to the next waiting thread, No barging (or minimal barging)
		
	🧪 Fair vs Unfair ReentrantLock
	| Aspect          | Fair Lock      | Unfair Lock  |
	| --------------- | -------------- | ------------ |
	| Starvation risk | ❌ Very low    | ✅ Possible |
	| Throughput      | Lower          | Higher       |
	| Ordering        | FIFO           | No guarantee |
	| Use case        | Predictability | Performance  |

	⚠ Important Trade-off (Must Mention): Fair locks reduce starvation but also reduce throughput.
		Because: More context switching, Less opportunity for CPU-local optimizations
		That’s why: synchronized and default ReentrantLock are unfair by default

5️⃣ Why synchronized Has No Fair Option
	Because it is designed for: Simplicity, Performance, JVM-level optimizations
	Fairness: Adds overhead, Reduces scalability, Is not always required
	So Java leaves fairness control to explicit locks.

8️⃣ Best Practices in Production

🔹 Q36. Why was java.util.concurrent introduced? Difference between synchronized vs Lock interface.
=============================================================================================================
1️⃣ Why Was java.util.concurrent Introduced?
	Because synchronized alone was too low-level, inflexible, and hard to use correctly for complex concurrent programs.
	🔍 Problems with synchronized (Pre–Java 5 Era)
		Before Java 5, concurrency relied mainly on: synchronized, wait() / notify()
	
	This caused real production issues:
		❌ 1. No Fairness Control: Threads could starve, No FIFO ordering
		❌ 2. No Timeout or Try Lock: Threads could block forever, Deadlocks were hard to avoid
		❌ 3. Single Condition Queue: Only one wait set per monitor, Hard to model multiple conditions (producer/consumer)
		❌ 4. Hard to Compose Correctly: wait/notify easy to misuse, Lost notifications, Spurious wakeups, subtle bugs
		❌ 5. Poor Scalability: Coarse-grained locking, High contention, Bad performance on multi-core CPUs

	🔥 What java.util.concurrent Solved: Introduced in Java 5
		it provided: Higher-level abstractions, Better performance, More expressive APIs, Safer concurrency patterns

	✅ What It Added: 
			Lock, ReentrantLock, ReadWriteLock, Semaphore, CountDownLatch, CyclicBarrier
			ExecutorService, thread pools
			Concurrent collections (ConcurrentHashMap, BlockingQueue)
			Atomic variables (AtomicInteger, etc.)
		👉 These reduce error-prone low-level locking.

🎯 Interview Line : “java.util.concurrent was introduced to provide higher-level, safer, and more scalable concurrency abstractions than synchronized and wait/notify.”

2️⃣ Difference Between synchronized vs Lock Interface
	| Aspect              | `synchronized`   | `Lock` Interface             |
	| ------------------- | ---------------- | ---------------------------- |
	| Type                | Language keyword | API (library)                |
	| Lock acquisition    | Implicit         | Explicit                     |
	| Lock release        | Automatic        | Manual                       |
	| Fairness            | ❌ No    	     | ✅ Optional        	        |
	| Try lock            | ❌ No    	     | ✅ Yes             	        |
	| Timeout             | ❌ No    	     | ✅ Yes              	        |
	| Multiple conditions | ❌ No    	     | ✅ Yes (`Condition`) 	    |
	| Interruptible lock  | ❌ No    	     | ✅ Yes                	    |
	| Risk of misuse      | Low              | Higher (if unlock forgotten) |

3️⃣ Key Differences Explained (With Insight)
	🔥 1. Lock Acquisition & Release
		synchronized
				synchronized (lock) {
					// lock acquired & released automatically
				}
		✔ Simple ✔ Exception-safe ❌ No flexibility
		
		Lock
			lock.lock();
			try {
				// critical section
			} finally {
				lock.unlock();
			}

		✔ Flexible ❌ Developer must unlock manually

	🔥 2. Fairness (Big Advantage of Lock)
		Lock lock = new ReentrantLock(true); // fair lock
		Threads acquire lock in FIFO order, Helps avoid starvation, synchronized has no fairness guarantee.
	
	🔥 3. Try Lock & Timeout (Deadlock Avoidance)
			if (lock.tryLock(1, TimeUnit.SECONDS)) {
				try {
					// work
				} finally {
					lock.unlock();
				}
			}

		✔ Prevents infinite blocking ✔ Helps avoid deadlocks, Not possible with synchronized.
	
	🔥 4. Multiple Conditions (Very Important)
		With synchronized: Only one wait set, All threads use same wait()/notify()
		With Lock:
			Condition notEmpty = lock.newCondition();
			Condition notFull  = lock.newCondition();
		✔ Separate condition queues, ✔ Cleaner producer–consumer logic

	🔥 5. Interruptible Lock Acquisition
			lock.lockInterruptibly();
		✔ Thread can respond to interruption while waiting, synchronized threads cannot be interrupted while blocked.

4️⃣ Performance & Scalability: 
	synchronized is highly optimized in modern JVMs
		For simple use cases → perfectly fine
		For complex coordination → Lock scales better
		👉 This is not about speed alone — it’s about control.

5️⃣ When to Use What (Practical Rule)
	✅ Use synchronized when: 
			Simple critical section, No timeout / fairness required
			Low contention, You want safety & simplicity
	✅ Use Lock when:
			Need fairness, Need timeout / tryLock
			Need multiple conditions, Complex coordination, Avoiding deadlock

🔹 Q37. Explain ReentrantLock. What is fair vs non-fair lock?
=============================================================================================================
1️⃣ What Is ReentrantLock?
	ReentrantLock is an explicit, reentrant mutual-exclusion lock provided by java.util.concurrent.locks.
	Reentrant means the same thread can acquire the lock multiple times without blocking itself.
	It is a more flexible and powerful alternative to synchronized.

2️⃣ Why Is It Called Reentrant?
	Because: If a thread already holds the lock, It can lock it again, JVM maintains a hold count
			 Lock is released only when unlock() is called the same number of times
	🧪 Example
			ReentrantLock lock = new ReentrantLock();
			void methodA() {
				lock.lock();
				try {
					methodB(); // re-enters same lock
				} finally {
					lock.unlock(); // MUST be in finally
				}
			}

			void methodB() {
				lock.lock();
				try {
					// safe
				} finally {
					lock.unlock(); // MUST be in finally
				}
			}
		✔ No deadlock, ✔ Same thread re-acquires lock safely ⚠ Forgetting unlock() → deadlock risk

3️⃣ Key Features of ReentrantLock (Why It Exists)
	| Feature             | ReentrantLock | synchronized  |
	| ------------------- | ------------- | --------------|
	| Reentrant           | ✅ Yes  	  | ✅ Yes        |
	| Fairness            | ✅ Optional	  | ❌ No         |
	| tryLock()           | ✅ Yes 	      | ❌ No         |
	| Timeout             | ✅ Yes  	  | ❌ No         |
	| Interruptible lock  | ✅ Yes	      | ❌ No         |
	| Multiple conditions | ✅ Yes 	      | ❌ No         |
	| Auto release        | ❌ Manual 	  | ✅ Automatic  |

4️⃣ Fair vs Non-Fair Lock (Very Important)
	A fair lock: ReentrantLock lock = new ReentrantLock(true);
		Grants access in FIFO order, Longest-waiting thread gets the lock next, Prevents starvation
	✔ Characteristics: Predictable ordering, Reduced starvation, Lower throughput

	A non-fair lock: ReentrantLock lock = new ReentrantLock(); // default
		Allows barging, A thread can acquire the lock even if others are waiting, No ordering guarantee
	✔ Characteristics; Higher throughput, Better CPU locality, starvation is possible

6️⃣ Fair vs Non-Fair — Comparison Table
	| Aspect            | Fair Lock     | Non-Fair Lock |
	| ----------------- | ------------- | ------------- |
	| Ordering          | FIFO          | No guarantee  |
	| Starvation        | Very unlikely | Possible      |
	| Performance       | Slower        | Faster        |
	| Context switching | More          | Less          |
	| Default           | ❌    	    | ✅   	        |

7️⃣ Why Non-Fair Is the Default?
	Because: Fairness adds queue management overhead, More context switching, Lower throughput
	👉 Most applications prefer performance over strict fairness.
	🎯 Interview Line: “Fair locks reduce starvation but at the cost of throughput.”

8️⃣ How ReentrantLock Avoids Starvation (When Needed)
	Using a fair lock: ReentrantLock lock = new ReentrantLock(true);
		Threads acquire lock in arrival order, Long-waiting threads eventually run
		Useful in: Banking systems, Billing systems, SLA-sensitive workflows

🔹 Q39. What is ReadWriteLock? When to use ReadWriteLock?
=============================================================================================================
1️⃣  ReadWriteLock is a lock that maintains two separate locks:
		a read lock for read-only access
		a write lock for exclusive write access
		👉 Multiple threads can read concurrently, but writes are exclusive.

	In Java, the main implementation is: ReentrantReadWriteLock

2️⃣ Why Do We Need ReadWriteLock?
	With a normal lock (synchronized or ReentrantLock):
		Readers block each other, Even if they are only reading shared data, This limits scalability.
	
	🔥 ReadWriteLock improves throughput when: Reads are frequent, Writes are rare

3️⃣ How ReadWriteLock Works (Core Idea)
	Rules:
		Multiple threads can hold the read lock simultaneously
		Only one thread can hold the write lock
		While a writer holds the lock: No readers allowed
		While readers hold the lock: Writers must wait

	✔ Behavior: Many threads can call read() at the same time, Only one thread can call write(), Writes block both readers and writers

5️⃣ ReadWriteLock vs Normal Lock
| Aspect             | ReentrantLock   | ReadWriteLock           |
| ------------------ | --------------- | ----------------------- |
| Concurrent readers | ❌ No    	   | ✅ Yes                  |
| Writers exclusive  | ✅ Yes   	   | ✅ Yes                  |
| Best for           | Mixed workloads | Read-heavy workloads    |
| Complexity         | Low             | Higher                  |
| Risk               | Low             | Starvation (if misused) |

6️⃣ When Should You Use ReadWriteLock?
	✅ Use ReadWriteLock When: 
		Read operations dominate, Writes are infrequent, Data structure is shared and mutable, Read operations do not modify state
	
		🧪 Real-World Examples: 
				In-memory cache, Configuration store, Reference data (country codes, feature flags)
				Lookup tables, Metadata repositories

	🎯 Interview line: “ReadWriteLock is ideal when reads vastly outnumber writes.”

7️⃣ When You Should NOT Use ReadWriteLock
	❌ Avoid ReadWriteLock When: 
		Write frequency is high, Reads are short-lived and cheap, Lock upgrade (read → write) is required
		Simplicity is more important than throughput
		⚠ Lock upgrading can cause deadlock if not handled carefully.

8️⃣ Starvation Risk (Important Senior Insight)
	With non-fair ReadWriteLock: Continuous readers may starve writers
	You can enable fairness: ReadWriteLock rwLock = new ReentrantReadWriteLock(true);
	✔ Writers eventually get chance ❌ Slight performance hit

9️⃣ ReadWriteLock vs Concurrent Collections
	Often, you don’t need ReadWriteLock at all.
		Example: ConcurrentHashMap already implements internal read/write optimizations
	🎯 Interview line: “Prefer concurrent collections over manual ReadWriteLock when possible.”

🔹 Q40. Explain StampedLock. Difference between optimistic and pessimistic locking.
=============================================================================================================
1️⃣  StampedLock (Java 8+) is a high-performance lock designed for read-mostly workloads.
	It supports three modes of locking: Write lock (exclusive), Read lock (shared), Optimistic read (non-blocking)
	👉 It improves throughput by allowing lock-free reads when there’s no contention.

2️⃣ Why Was StampedLock Introduced?
	ReadWriteLock improved concurrency but: Readers still block writers, Writers still block readers, Lock acquisition still has overhead
	StampedLock solves this by introducing optimistic reads, which: Don’t block writers, Don’t block readers, Avoid locking unless necessary

	🎯 Interview line: “StampedLock is optimized for scenarios where reads are frequent and writes are rare.”

3️⃣ How StampedLock Works (Key Idea)
	Instead of returning a Lock object, StampedLock returns a stamp (long value):
		long stamp = lock.readLock();
		The stamp: Represents a version of the lock state, Must be used to unlock
		Helps detect concurrent modifications

4️⃣ Locking Modes in StampedLock
	🔴 1. Write Lock (Exclusive)
			long stamp = lock.writeLock();
			try {
				// modify shared data
			} finally {
				lock.unlockWrite(stamp);
			}

		✔ Only one writer, ✔ Blocks readers and writers

	🟡 2. Read Lock (Pessimistic Read)
			long stamp = lock.readLock();
			try {
				// read shared data
			} finally {
				lock.unlockRead(stamp);
			}

		✔ Multiple readers, ✔ Blocks writers. This is similar to ReadWriteLock.

	🟢 3. Optimistic Read (Most Important)
			long stamp = lock.tryOptimisticRead();
			// read shared data (NO LOCK!)
			if (!lock.validate(stamp)) {
				// a write occurred — fall back to read lock
			}
		
		✔ No blocking, ✔ No locking, ✔ Fastest read path
		But: Data may change, Must be validated

5️⃣ Optimistic vs Pessimistic Locking (Core Difference)

	🔹 Optimistic Locking: Assume no conflict, Proceed without locking, Validate later, Retry if conflict occurs
		✔ Characteristics: Non-blocking, High throughput, Best for read-heavy systems
		❌ Risks: Retry cost if contention is high
		🧪 Example (StampedLock)
					long stamp = lock.tryOptimisticRead();
					int value = data;
					if (!lock.validate(stamp)) {
						stamp = lock.readLock();
						try {
							value = data;
						} finally {
							lock.unlockRead(stamp);
						}
					}
	
	🔹 Pessimistic Locking: Assume conflict will happen, Lock first, Then operate
		✔ Characteristics: Safe, Simple, Predictable
		❌ Drawbacks: Blocking, Lower concurrency
		🧪 Example
				synchronized (lock) {
					readOrWriteData();
				}

6️⃣ Optimistic vs Pessimistic — Comparison Table
	| Aspect            | Optimistic Locking          | Pessimistic Locking             |
	| ----------------- | --------------------------- | ------------------------------- |
	| Locking           | None initially              | Immediate                       |
	| Blocking          | ❌ No           	          | ✅ Yes              	        |
	| Conflict handling | Validation & retry          | Wait                            |
	| Performance       | High (low contention)       | Lower                           |
	| Best for          | Read-heavy workloads        | Write-heavy / critical sections |
	| Example           | StampedLock optimistic read | synchronized / ReentrantLock    |

7️⃣ Important Limitations of StampedLock (Must Mention)
	⚠ StampedLock is NOT reentrant // Same thread cannot re-acquire lock → deadlock
	⚠ No Condition support, No automatic unlocking, More complex to use, Not suitable for simple cases
	🎯 Interview line: “StampedLock trades simplicity for performance.”

8️⃣ When Should You Use StampedLock?
	✅ Use StampedLock When: Reads vastly outnumber writes, Read operations are short, You need maximum throughput, Contention is low
	❌ Avoid StampedLock When: Code must be simple, Reentrancy is required, Write contention is high, You need condition variables
	
9️⃣ StampedLock vs ReadWriteLock (Quick Contrast)
	| Aspect           | ReadWriteLock | StampedLock         |
	| ---------------- | ------------- | ------------------- |
	| Optimistic reads | ❌ No  	   | ✅ Yes   	         |
	| Reentrant        | ✅ Yes 	   | ❌ No      	     |
	| Complexity       | Low           | High                |
	| Performance      | Good          | Better (read-heavy) |
	| Java version     | 5+            | 8+                  |

🔹 Q41. Explain Executor framework. Why should we avoid creating threads manually?
=============================================================================================================
1️⃣  The Executor Framework (java.util.concurrent) is a high-level abstraction for asynchronous task execution that decouples task submission from task execution. Instead of Creating threads, Starting threads, Managing thread lifecycle, You submit tasks, and the framework manages threads for you.

🔑 Core Idea : You tell what to execute; the Executor decides how and when to execute it.

2️⃣ Core Components of Executor Framework
	🔹 1. Executor: Basic interface with one method: void execute(Runnable command);
	🔹 2. ExecutorService extends Executor and adds Lifecycle management, Task submission with results, Graceful shutdown
			ExecutorService executor = Executors.newFixedThreadPool(5);

	🔹 3. ThreadPoolExecutor: The actual implementation behind most executors.
			Controls: Pool size, Queue, Thread creation, Rejection policy
	
	🔹 4. Task Types
			Runnable → no result
			Callable<V> → returns result, can throw exception
			Future<Integer> f = executor.submit(() -> 42);

3️⃣ Common Executor Types (Must Know)
	Executors.newFixedThreadPool(n);      // fixed number of threads
	Executors.newCachedThreadPool();     // dynamic threads
	Executors.newSingleThreadExecutor(); // one thread
	Executors.newScheduledThreadPool(n); // delayed & periodic tasks
	👉 In real systems, we usually use ThreadPoolExecutor directly for control.

4️⃣ Why Should We Avoid Creating Threads Manually?
	❌ Manual Thread Creation (Bad Practice): new Thread(() -> doWork()).start();
		Looks simple — but causes serious problems at scale.
	
5️⃣ Problems with Creating Threads Manually
	🔴 1. High Resource Cost
			Each thread: Allocates stack memory (often 1MB+), Uses OS resources, Causes context switching overhead
			Creating too many threads → High memory usage, CPU thrashing, Poor performance
		
		🎯 Interview line: “Threads are expensive; tasks are cheap.”
	
	🔴 2. No Thread Reuse
				for (...) {
					new Thread(task).start();
				}
		❌ Threads are created and destroyed repeatedly ❌ Wasteful and slow

		Executor: ✔ Reuses threads, ✔ Improves throughput								

	🔴 3. No Control Over Concurrency
		Manual threads: No limit, Can create thousands of threads, Can crash JVM with OutOfMemoryError
		Executor: newFixedThreadPool(10); // concurrency is bounded
		✔ Predictable behavior

	🔴 4. No Lifecycle Management
		Manual threads=> No way to: Shutdown gracefully, Await completion, Handle failures centrally
		Executor:
			executor.shutdown();
			executor.awaitTermination(...);
		✔ Clean shutdown, ✔ Controlled termination

	🔴 5. Poor Error Handling
		In manual threads: Exceptions may be swallowed, Hard to track failures
		Executor: Exceptions captured via Future, Centralized handling, Better observability

	🔴 6. No Scheduling or Queuing
		Manual threads: Immediate execution only
		Executor: Task queues, Scheduling, Delayed and periodic execution

6️⃣ Benefits of Executor Framework (Why It Exists)
	| Benefit              | Explanation                       |
	| -------------------- | --------------------------------- |
	| Thread reuse         | Fewer threads, better performance |
	| Resource control     | Bounded pools                     |
	| Scalability          | Works well under load             |
	| Task abstraction     | Runnable / Callable               |
	| Error handling       | Futures & policies                |
	| Lifecycle management | start / stop / shutdown           |
	| Monitoring           | Pool size, queue size             |

7️⃣ Real-World Example (Interview-Friendly)
	❌ Bad (Manual Threads in Web App): new Thread(() -> processRequest()).start();
		Under load: Thousands of threads, App crashes
	
	
	✅ Good (Executor in Web App): executor.submit(() -> processRequest());
		Limited threads, Requests queued, System remains stable

8️⃣ When Is Manual Thread Creation Acceptable?
	Very rarely: One-off background thread, JVM startup tasks, Low-level framework code
	🎯 Interview line: “Application code should almost never create threads directly.”

🔹 Q43. Expalin all types of thread pools: Fixed, Cached, Single, Scheduled
=============================================================================================================
🧵 Types of Thread Pools in Java Executor Framework
1️⃣ Fixed Thread Pool: A thread pool with a fixed number of threads.
	ExecutorService executor = Executors.newFixedThreadPool(5);
	🔧 How It Works: Creates N threads, Reuses those threads, Tasks beyond N are queued, Threads are never destroyed (until shutdown)
		Internally: Core threads = Max threads = N, Uses unbounded queue (LinkedBlockingQueue)

	✅ When to Use: CPU-bound tasks, Stable, predictable workload, You want to limit concurrency
		Examples: Processing requests, Batch jobs, Background task processing
	
	❌ When to Avoid: Very high task submission rate, Risk of unbounded queue → OutOfMemoryError
	🎯 Interview Line: “Fixed thread pool limits concurrency but can grow the queue indefinitely.”

2️⃣ Cached Thread Pool: A dynamically growing thread pool.
	ExecutorService executor = Executors.newCachedThreadPool();
	🔧 How It Works: Creates new threads as needed, Reuses idle threads, Threads idle for 60 seconds are terminated
		Internally: Core threads = 0, Max threads = Integer.MAX_VALUE, Uses SynchronousQueue (no storage)
	
	✅ When to Use: Short-lived, asynchronous tasks, Bursty workloads, I/O-bound operations
		Examples: Lightweight background tasks, Event handling
	
	❌ When to Avoid (Very Important): High load systems, Web applications, Unbounded thread creation → CPU & memory exhaustion

	🎯 Interview Line: “Cached thread pool can create unlimited threads — dangerous under heavy load.”

3️⃣ Single Thread Executor: A pool with exactly one thread.
	ExecutorService executor = Executors.newSingleThreadExecutor();
	🔧 How It Works: One worker thread, Tasks execute sequentially, If thread dies → a new one is created, Uses unbounded queue
	
	✅ When to Use: Tasks must run in order, Only one task should execute at a time, You need thread safety via serialization
		Examples: Logging, Event processing, File I/O sequencing
	
	❌ When to Avoid: Parallel processing needed, Long-running tasks blocking others

	🎯 Interview Line: “SingleThreadExecutor guarantees task ordering with automatic thread replacement.”

4️⃣ Scheduled Thread Pool: A pool that can schedule tasks to run: After a delay, Periodically
	ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(2);
	🔧 How It Works: Supports: schedule(), scheduleAtFixedRate(), scheduleWithFixedDelay()
	🧪 Examples: 
		Run once after delay: scheduler.schedule(() -> task(), 5, TimeUnit.SECONDS);
		Run periodically (fixed rate): scheduler.scheduleAtFixedRate(task, 0, 10, TimeUnit.SECONDS);
		Run periodically (fixed delay): scheduler.scheduleWithFixedDelay(task, 0, 10, TimeUnit.SECONDS);

	✅ When to Use: Periodic jobs, Heartbeats, Cleanup tasks, Retry mechanisms
	❌ When to Avoid: Heavy CPU tasks, Very long-running scheduled tasks
	
	🎯 Interview Line: “ScheduledExecutorService replaces Timer with better error handling and concurrency.”

🔥 Comparison Table (Very Important)
| Thread Pool | Threads | Queue      | Best For           | Risk             |
| ----------- | ------- | ---------- | ------------------ | ---------------- |
| Fixed       | Fixed N | Unbounded  | Stable workloads   | OOM via queue    |
| Cached      | Dynamic | None       | Short async tasks  | Thread explosion |
| Single      | 1       | Unbounded  | Ordered execution  | Bottleneck       |
| Scheduled   | Fixed N | DelayQueue | Timers & cron jobs | Task overlap     |

⚠️ Senior-Level Warning (Interview Gold)
	Avoid using Executors factory methods blindly in production.
	Why? Hidden defaults, Unbounded queues, Unbounded threads
	
	✅ Preferred: Use ThreadPoolExecutor directly:
				new ThreadPoolExecutor(core, max, keepAlive, TimeUnit.SECONDS,
					new ArrayBlockingQueue<>(100),
					new ThreadPoolExecutor.AbortPolicy()
				);

🔹 Q45. CorePoolSize vs MaximumPoolSize.
=============================================================================================================
1️⃣  CorePoolSize is the minimum number of threads that the thread pool tries to keep alive.
	These threads are: Created eagerly (or on demand), Not destroyed even when idle (by default), They form the baseline capacity of the pool

		new ThreadPoolExecutor(
			corePoolSize,
			maxPoolSize,
			keepAliveTime,
			timeUnit,
			workQueue
		);

	🧠 Key Points: Core threads handle normal workload, Tasks are first assigned to core threads, Core threads stay alive indefinitely (unless allowCoreThreadTimeOut(true))

2️⃣  MaximumPoolSize is the maximum number of threads the pool can grow to under heavy load.
	Extra threads beyond corePoolSize are called non-core threads, Created only when needed, Destroyed after being idle for keepAliveTime
	
	🧠 Key Points: Used as a burst capacity, Protects the system from creating unlimited threads, Once max is reached, tasks may be rejected

3️⃣ How ThreadPoolExecutor Decides What to Do (Very Important)
	When a new task is submitted, the pool follows this exact order:
	🔁 Task Submission Algorithm
		1️⃣ If current threads < corePoolSize → Create a new thread to run the task
		2️⃣ Else if queue is not full → Put the task into the queue
		3️⃣ Else if current threads < maximumPoolSize → Create a new non-core thread
		4️⃣ Else → Reject the task (RejectedExecutionHandler)

	🎯 Interview Line (Use This!): “Threads grow first, then the queue fills, then threads grow again up to max.”

4️⃣ Example (This Is Gold for Interviews)
			ThreadPoolExecutor executor =
				new ThreadPoolExecutor(
					2,  // corePoolSize
					5,  // maximumPoolSize
					60, TimeUnit.SECONDS,
					new ArrayBlockingQueue<>(3)
				);

				Now submit tasks 👇
				| Task # | Active Threads | Queue Size | Action     |
				| ------ | -------------- | ---------- | ---------- |
				| 1      | 1              | 0          | New thread |
				| 2      | 2              | 0          | New thread |
				| 3      | 2              | 1          | Queued     |
				| 4      | 2              | 2          | Queued     |
				| 5      | 2              | 3          | Queued     |
				| 6      | 3              | 3          | New thread |
				| 7      | 4              | 3          | New thread |
				| 8      | 5              | 3          | New thread |
				| 9      | 5              | 3          | ❌ Rejected |

				This explanation immediately impresses interviewers.

5️⃣ Core vs Maximum — Side-by-Side
	| Aspect                  | corePoolSize     | maximumPoolSize |
	| ----------------------- | ---------------- | --------------- |
	| Purpose                 | Baseline threads | Burst limit     |
	| Threads always kept     | ✅ Yes            | ❌ No            |
	| Used during normal load | ✅                | ❌               |
	| Used during peak load   | ❌                | ✅               |
	| Idle timeout            | ❌ (by default)   | ✅               |
	| Protects from overload  | ❌                | ✅               |
	
6️⃣ Relationship with Queue (Very Important Insight)
👉 Queue choice determines whether maximumPoolSize even matters
	⚠ Unbounded Queue (e.g. LinkedBlockingQueue), Tasks are always queued, Pool never grows beyond corePoolSize, maximumPoolSize is effectively ignored
	
	🎯 Interview line: “With an unbounded queue, maximumPoolSize becomes meaningless.”

	✅ Bounded Queue (Recommended): Allows pool to grow beyond core size, Enables back-pressure, Makes maxPoolSize effective

🔹 Q46. What is work queue? What happens when queue is full?
=============================================================================================================
1️⃣  A work queue is an internal task holding queue used by ThreadPoolExecutor to store submitted tasks when threads are busy.
	👉 It sits between task submission and thread execution.
		ThreadPoolExecutor executor = new ThreadPoolExecutor(corePoolSize, maximumPoolSize, 
															keepAliveTime, TimeUnit.SECONDS, workQueue);

	🧠 Why Work Queue Exists
			Threads are limited, Tasks may arrive faster than they can be processed, Queue provides buffering and back-pressure

	🎯 Interview line: “The work queue buffers tasks when all core threads are busy.”

2️⃣ How the Work Queue Is Used (Execution Flow)
		When a task is submitted:
			1️⃣ If active threads < corePoolSize → Create a new thread
			2️⃣ Else if queue is not full → Put task into work queue
			3️⃣ Else if active threads < maximumPoolSize → Create a new non-core thread
			4️⃣ Else → Reject the task
			This order is extremely important.

3️⃣ Common Types of Work Queues
		🔹 1. LinkedBlockingQueue (Unbounded): new LinkedBlockingQueue<>();
				Default for newFixedThreadPool, Can grow indefinitely
				✔ Simple ❌ Risk of OutOfMemoryError
				⚠ With unbounded queue: maximumPoolSize is effectively ignored

				🎯 Interview line: “Unbounded queues protect threads but risk memory.”

		🔹 2. ArrayBlockingQueue (Bounded): new ArrayBlockingQueue<>(100);
				Fixed capacity, Provides back-pressure
				✔ Safer for production ✔ Enables maxPoolSize usage

		🔹 3. SynchronousQueue (No Storage): new SynchronousQueue<>();
				No capacity, Direct handoff
				✔ Used by newCachedThreadPool ❌ Can create many threads

		🔹 4. PriorityBlockingQueue
				Tasks ordered by priority, Unbounded by default
				✔ Useful for priority-based processing, ❌ Still OOM risk

4️⃣ What Happens When the Queue Is Full? This is the core of the question 👇

	🔴 Case 1: Queue is Full BUT Threads < maxPoolSize: ➡ ThreadPoolExecutor creates a new non-core thread
				Queue full → grow thread pool (up to max)
				✔ Helps absorb load spikes

	🔴 Case 2: Queue is Full AND Threads == maxPoolSize; 	➡ Task is rejected
				This triggers the RejectedExecutionHandler.

✔ RejectedExecutionHandler is a strategy interface used by ThreadPoolExecutor to decide what to do when a task cannot be accepted for execution.
	A task is rejected when: The work queue is full, The thread pool has reached maximumPoolSize, The executor is shut down
	
5️⃣ Task Rejection Policies (Must Know)
	🔹 1. AbortPolicy (Default): throw RejectedExecutionException
			✔ Fail fast, ❌ Request lost if not handled

	🔹 2. CallerRunsPolicy: task.run(); // runs in calling thread
			✔ Back-pressure, ✔ Slows down producer, ✔ Very useful in production

	🔹 3. DiscardPolicy 	// silently drop task
			❌ Dangerous unless acceptable

	🔹 4. DiscardOldestPolicy: remove oldest task and retry
			✔ Useful for real-time systems

6️⃣ Real-World Example (Interview Gold)
		ThreadPoolExecutor executor = new ThreadPoolExecutor(2,  // core
																4,  // max
																60, TimeUnit.SECONDS,
																new ArrayBlockingQueue<>(2),
																new ThreadPoolExecutor.CallerRunsPolicy());
			

		Submitting many tasks: 
			First 2 → threads
			Next 2 → queue
			Next 2 → new threads
			Further → caller thread executes task
		➡ System slows gracefully instead of crashing.
		✔ Correct: Threads → Queue → More Threads → Reject

🔹 Q48. What is keepAliveTime? How to gracefully shutdown executor?
=============================================================================================================
1️⃣  keepAliveTime is the maximum time that excess (non-core) threads are allowed to remain idle before being terminated.
	🔍 Which Threads Does keepAliveTime Apply To?
		✅ By default: Applies only to non-core threads, Threads above corePoolSize, When idle longer than keepAliveTime → terminated
		❌ Does NOT apply to: Core threads (unless explicitly enabled)
	
	🧠 Why keepAliveTime Exists: Handle traffic spikes, Shrink pool back to normal size, Free CPU & memory resources
	🎯 Interview line: “keepAliveTime controls how long extra threads survive after a load spike.”

2️⃣ Example: 
	Scenario:
		Normal load → 2 threads
		Traffic spike → grows to 5 threads
		Load drops → extra 3 threads stay idle
		After 30 seconds → extra threads die
		✔ Pool returns to stable size

3️⃣ Applying keepAliveTime to Core Threads (Advanced): executor.allowCoreThreadTimeOut(true);
	Now: Even core threads can be terminated
	Useful for: Low-traffic systems, Memory-sensitive environments
	🎯 Interview line: “allowCoreThreadTimeOut makes the pool fully elastic.”

4️⃣ How to Gracefully Shut Down an Executor
	🔹 What Does Graceful Shutdown Mean? Stop accepting new tasks, allow existing tasks to finish, then release resources.

		✅ Step-by-Step Pattern
			executor.shutdown(); // 1️⃣ Stop accepting new tasks

				try {
					if (!executor.awaitTermination(60, TimeUnit.SECONDS)) {
						executor.shutdownNow(); // 2️⃣ Force shutdown
					}
				} catch (InterruptedException e) {
					executor.shutdownNow();     // 3️⃣ Cancel on interruption
					Thread.currentThread().interrupt();
				}

	🔍 What Each Method Does
	🔹 shutdown(): Stops accepting new tasks, Allows queued & running tasks to complete, Does NOT block
		✔ Preferred for graceful shutdown
	
	🔹 awaitTermination(timeout): Blocks until: All tasks finish, Timeout occurs, Returns true if terminated
	
	🔹 shutdownNow(): Attempts to: Interrupt running tasks, Return queued but unstarted tasks
		⚠ Not guaranteed to stop tasks immediately

6️⃣ What Happens If You Don’t Shut Down Executor?
	❌ JVM may never exit, ❌ Thread leaks, ❌ Resource exhaustion, ❌ Hanging deployments
	🎯 Interview line: “An executor is a resource — it must be shut down.”
	

7️⃣ shutdown vs shutdownNow (Quick Comparison)
| Aspect             | shutdown() | shutdownNow() |
| ------------------ | ---------- | ------------- |
| New tasks accepted | ❌ No       | ❌ No          |
| Running tasks      | ✅ Finish   | ❌ Interrupted |
| Queued tasks       | ✅ Executed | ❌ Returned    |
| Graceful           | ✅ Yes      | ❌ No          |
| Safe default       | ✅          | ❌             |

🔹 Q50. Difference between Runnable and Callable.
=============================================================================================================
1️⃣ High-Level Difference (Start Like This)
Runnable represents a task that does not return a result and cannot throw checked exceptions, while Callable represents a task that returns a result and can throw checked exceptions.

2️⃣ Runnable vs Callable — Core Differences
	| Aspect             | Runnable         | Callable               |
	| ------------------ | ---------------- | ---------------------- |
	| Package            | `java.lang`      | `java.util.concurrent` |
	| Method             | `void run()`     | `V call()`             |
	| Return value       | ❌ No    	    | ✅ Yes      	         |
	| Checked exceptions | ❌ No   	        | ✅ Yes       	         |
	| Introduced in      | Java 1.0         | Java 5                 |
	| Used with          | Thread, Executor | ExecutorService        |
	| Result handling    | Not supported    | Via `Future`           |

3️⃣ Runnable Explained: Runnable represents a unit of work that can be executed by a thread.
				Runnable task = () -> {
					System.out.println("Task running");
				};
	🔹 Execution: new Thread(task).start(); or executor.execute(task);
		❌ Limitations of Runnable: Cannot return a value, Cannot throw checked exceptions, Error handling is awkward
			No direct feedback mechanism

		🧠 When Runnable Is Enough: Fire-and-forget tasks, Logging, Background monitoring, Simple asynchronous work

4️⃣ Callable Explained: Callable<V> represents a task that: Returns a result, Can throw checked exceptions

				Callable<Integer> task = () -> {
					return 42;
				};

	🔹 Execution:
				Future<Integer> future = executor.submit(task);
				Integer result = future.get();

	🔥 Key Advantage: Clean error handling, Strongly typed return value, Integration with Future
	🧠 When Callable Is Preferred: Business computations, Remote calls, Database queries, Any task where result matters

5️⃣ Exception Handling Difference (Important)
	Runnable
			public void run() {
				throw new RuntimeException(); // allowed
				// checked exception ❌ not allowed
			}

		Checked exceptions must be: Caught inside run(), Wrapped in RuntimeException

	Callable
			public Integer call() throws Exception {
				return fetchFromDB();
			}
		✔ Clean exception propagation, ✔ Exceptions returned via Future.get() as ExecutionException

6️⃣ How Executor Handles Them
		executor.execute(Runnable);  // no result
		executor.submit(Runnable);   // returns Future<?> (result = null)
		executor.submit(Callable);   // returns Future<V>


🎯 Interview line: “submit works for both, execute works only for Runnable.”

7️⃣ Real-World Example (Interview Gold)
	❌ Using Runnable for computation (Bad)
		int[] result = new int[1];
		executor.execute(() -> result[0] = compute());

	✅ Using Callable (Correct)
		Future<Integer> future = executor.submit(() -> compute());
		int result = future.get();

🔹 Q51. What is Future? Limitations of Future. How CompletableFuture solves Future problems?
=============================================================================================================
1️⃣  A Future<V> represents the result of an asynchronous computation that may complete in the future.
	You submit a task, get a Future, and later ask: Is it done? Give me the result, Cancel it
			ExecutorService executor = Executors.newFixedThreadPool(2);
			Future<Integer> future = executor.submit(() -> { Thread.sleep(1000); return 42; });
		// later
			Integer result = future.get(); // blocks

2️⃣ What Can You Do with Future?
	Key methods:
				future.get();          // blocking
				future.get(timeout);  // blocking with timeout
				future.isDone();      // polling
				future.cancel(true);  // cancellation

	🎯 Interview line: “Future represents a handle to an async computation.”

3️⃣ Limitations of Future (This Is the Main Part)
	🔴 1. Blocking Nature: future.get(); // blocks current thread
			No non-blocking callback, Wastes threads, Bad for scalable systems
			🎯 Interview line: “Future forces you to block to get the result.”

	🔴 2. No Callbacks: You cannot say: “When the result is ready, do something.”
			Instead, you must: Poll using isDone(), or Block using get()
			❌ Inefficient, ❌ Ugly code

	🔴 3. No Chaining / Composition: 
			You cannot easily combine futures: Run task B after task A, Combine results of A and B, Handle pipelines
			❌ Manual orchestration, ❌ Nested blocking

	🔴 4. Poor Exception Handling
				try {
					future.get();
				} catch (ExecutionException e) {
					Throwable cause = e.getCause();
				}
			Wrapped exceptions, Hard to compose error handling, No centralized error flow

	🔴 5. Cannot Complete Manually
			With Future: Only executor controls completion, You cannot push results yourself
			❌ Not flexible for event-driven systems

4️⃣ Summary of Future Limitations
	| Limitation           | Impact                  |
	| -------------------- | ----------------------- |
	| Blocking get()       | Poor scalability        |
	| No callbacks         | Inefficient async       |
	| No chaining          | Hard to build pipelines |
	| Weak error handling  | Messy code              |
	| No manual completion | Limited flexibility     |

5️⃣ How CompletableFuture Solves These Problems
	CompletableFuture (Java 8+) is: A Future PLUS a CompletionStage, Supports non-blocking, functional, async programming
		CompletableFuture<Integer> cf = CompletableFuture.supplyAsync(() -> 42);

6️⃣ How CompletableFuture Fixes Each Problem
		✅ 1. Non-Blocking Callbacks
				cf.thenAccept(result -> {
					System.out.println(result);
				});
			✔ No blocking, ✔ Event-driven, ✔ Scales well
			🎯 Interview line: “CompletableFuture enables non-blocking callbacks.”

		✅ 2. Easy Chaining
				CompletableFuture
					.supplyAsync(() -> 10)
					.thenApply(x -> x * 2)
					.thenApply(x -> x + 5)
					.thenAccept(System.out::println);
			✔ Clean pipeline, ✔ No nested futures

		✅ 3. Combine Multiple Async Tasks
				CompletableFuture<Integer> f1 = CompletableFuture.supplyAsync(() -> 10);
				CompletableFuture<Integer> f2 = CompletableFuture.supplyAsync(() -> 20);
				f1.thenCombine(f2, Integer::sum).thenAccept(System.out::println);
			✔ Impossible with plain Future, ✔ Very powerful

		✅ 4. Better Exception Handling
					cf.exceptionally(ex -> {
						log.error("Error", ex);
						return fallbackValue;
					});
				or
					cf.handle((result, ex) -> {
						if (ex != null) return fallback;
						return result;
					});
				✔ Centralized, ✔ Declarative

		✅ 5. Manual Completion
					CompletableFuture<Integer> cf = new CompletableFuture<>();
				// later
					cf.complete(42);
					cf.completeExceptionally(new RuntimeException());
				✔ Perfect for: Event-driven systems, Callbacks, Integrations
				
7️⃣ Future vs CompletableFuture (Quick Comparison)
| Aspect             | Future     | CompletableFuture |
| ------------------ | ---------- | ----------------- |
| Blocking           | ✅ Yes      | ❌ Optional        |
| Callbacks          | ❌ No       | ✅ Yes             |
| Chaining           | ❌ No       | ✅ Yes             |
| Combine tasks      | ❌ No       | ✅ Yes             |
| Exception handling | Poor       | Strong            |
| Manual completion  | ❌ No       | ✅ Yes             |
| Programming style  | Imperative | Functional        |

8️⃣ When Should You Use What?
	✅ Use Future when: Simple async task, Blocking is acceptable, Legacy code
	✅ Use CompletableFuture when: Non-blocking systems, Async pipelines, Microservices, High-throughput apps, Reactive-style workflows
	
	🎯 Interview line: “Future is about waiting; CompletableFuture is about reacting.”

🔹 Q52. Difference between thenApply vs thenCompose and thenAccept vs thenRun.
=============================================================================================================
1️⃣ thenApply vs thenCompose: 
	🔹 thenApply — transform the result
		📌 Takes the result of a stage, Transforms it, Returns a new value

			CompletableFuture<Integer> cf =
				CompletableFuture.supplyAsync(() -> 10)
					.thenApply(x -> x * 2); // transforms value
			System.out.println(cf.join()); // 20

	🔹 thenCompose — flatten async operations
		📌 Takes the result of a stage, Calls another async operation, Flattens the result

				CompletableFuture<Integer> cf =
					CompletableFuture.supplyAsync(() -> 10)
						.thenCompose(x -> CompletableFuture.supplyAsync(() -> x * 2));
				System.out.println(cf.join()); // 20

		
🎯 Interview Line: “Use thenApply when you return a value, and thenCompose when you return a CompletableFuture.”

🧪 Real-World Example (Very Interview-Friendly)
			CompletableFuture<User> user =
				getUserId()
					.thenCompose(id -> fetchUserFromDB(id))
					.thenApply(u -> enrichUser(u));
		✔ Async DB call → thenCompose
		✔ Simple transformation → thenApply

🔥 thenApply vs thenCompose Summary
| Aspect           | thenApply                | thenCompose              |
| ---------------- | ------------------------ | ------------------------ |
| Input            | Value                    | Value                    |
| Function returns | Value                    | CompletableFuture        |
| Output           | CompletableFuture<Value> | CompletableFuture<Value> |
| Use case         | Data transformation      | Async chaining           |


2️⃣ thenAccept vs thenRun:
	🔹 thenAccept — consume the result
		📌 Takes the result, Performs a side-effect, Returns CompletableFuture<Void>
			CompletableFuture<Void> cf =
				CompletableFuture.supplyAsync(() -> "Hello")
					.thenAccept(msg -> System.out.println(msg));

	🔹 thenRun — run without result
		📌 Does not receive the result, Just runs some logic, Returns CompletableFuture<Void>
			CompletableFuture<Void> cf =
				CompletableFuture.supplyAsync(() -> "Hello")
					.thenRun(() -> System.out.println("Done"));

	🎯 Interview Line: “thenAccept uses the result; thenRun ignores it.”

🔥 thenAccept vs thenRun Summary
| Aspect           | thenAccept               | thenRun                 |
| ---------------- | ------------------------ | ----------------------- |
| Access to result |  Yes                     |   No                    |
| Return type      | CompletableFuture<Void>  | CompletableFuture<Void> |
| Use case         | Logging, saving, sending | Notifications, cleanup  |

🔹 Q54. What thread executes CompletableFuture by default?
=============================================================================================================
1️⃣  By default, CompletableFuture async tasks run on the ForkJoinPool.commonPool().
	This applies to: supplyAsync(), runAsync(), thenApplyAsync(), thenComposeAsync(), thenAcceptAsync(), etc.

		CompletableFuture.supplyAsync(() -> {
			System.out.println(Thread.currentThread().getName());
			return 42;
		});
	👉 Output will look like: ForkJoinPool.commonPool-worker-1

2️⃣ What Is ForkJoinPool.commonPool()?
	A shared, global thread pool, Introduced in Java 8
	Used by: CompletableFuture, Parallel Streams, Other async APIs

	🔹 Size of commonPool: Runtime.getRuntime().availableProcessors() - 1 : So on an 8-core machine → 7 threads.
	
	🎯 Interview line: “CompletableFuture uses the common ForkJoinPool unless an executor is explicitly provided.”

3️⃣ Important Exception: Non-Async Methods :❗ This is where many people get confused
	Methods WITHOUT Async run on the same thread that completes the previous stage.

		CompletableFuture
			.supplyAsync(() -> 10)
			.thenApply(x -> {
				System.out.println(Thread.currentThread().getName());
				return x * 2;
			});

		supplyAsync → commonPool thread, thenApply → same thread, not a new one

	🎯 Interview line: “Non-async stages run in the thread that completes the previous stage.”

4️⃣ Async vs Non-Async (Very Important)
| Method           | Thread Used                   |
| ---------------- | ----------------------------- |
| supplyAsync      | ForkJoinPool.commonPool       |
| runAsync         | ForkJoinPool.commonPool       |
| thenApply        | Same thread as previous stage |
| thenCompose      | Same thread as previous stage |
| thenAccept       | Same thread as previous stage |
| thenApplyAsync   | ForkJoinPool.commonPool       |
| thenComposeAsync | ForkJoinPool.commonPool       |

5️⃣ Providing Your Own Executor (Best Practice)
	You can override the default behavior: 	
		ExecutorService executor = Executors.newFixedThreadPool(5);
		CompletableFuture.supplyAsync(() -> task(), executor)
                 .thenApplyAsync(result -> process(result), executor);
		
		✔ Better control, ✔ Avoids starving commonPool, ✔ Recommended for production systems

	🎯 Interview line: “In production, always prefer a custom executor over the common pool.”

6️⃣ Why Using commonPool Blindly Is Dangerous
	❌ Blocking I/O tasks can: Starve the common pool
		Affect: Parallel streams, Other CompletableFutures, Cause system-wide slowdown
		Especially dangerous in: Web applications, Microservices, Blocking DB / REST calls

🔹 Q56. Why HashMap not thread safe? Difference between HashMap and ConcurrentHashMap. 
=============================================================================================================
1️⃣ HashMap is not thread-safe because it does not provide any synchronization or memory-visibility guarantees when accessed by multiple threads. But that alone is not enough for interviews. Here’s the real reason 👇

2️⃣ What Goes Wrong Internally in HashMap?
	❌ 1. Race Conditions During Put/Get
			HashMap operations like put() and get() are not atomic.
				map.put(key, value);
			Internally involves: Calculating hash, Finding bucket, Traversing list/tree, Updating pointers
			If two threads modify the same bucket: Data corruption can occur, Entries can be lost
	
	❌ 2. Infinite Loop During Rehashing (Classic Bug)
			Before Java 8 especially:
				When HashMap resizes (rehashing): Buckets are re-linked, Multiple threads rehashing simultaneously can create a cycle in the linked list
				Result: map.get(key); // CPU goes to 100%, infinite loop
		
		🎯 Interview line: “Concurrent resize in HashMap can cause infinite loops.”

	❌ 3. Memory Visibility Issues
			Without synchronization: One thread’s updates may not be visible to another, Due to CPU cache + instruction reordering
				map.put("a", 1);   // Thread A
				map.get("a");      // Thread B → may return null

	❌ 4. Structural Corruption: 
			Multiple threads can: Overwrite internal pointers, Corrupt bucket chains, Break tree structure (Java 8+)
			🧠 Key Insight: HashMap is designed for single-threaded performance, not concurrency safety.

3️⃣ Why ConcurrentHashMap Is Thread-Safe?
	🔹 Core Idea:
		ConcurrentHashMap allows safe concurrent access with high performance by minimizing locking.
		It does NOT lock the entire map.
	
4️⃣ How ConcurrentHashMap Achieves Thread Safety
	🔹 Java 7 and Earlier: Segment-based locking, Map divided into segments, Each segment had its own lock
	🔹 Java 8+ (Important for Interviews): Java 8 redesigned ConcurrentHashMap: ✔ No segments
			✔ Uses: CAS (Compare-And-Swap), Fine-grained locking, Volatile reads, Synchronized only on bucket level
	
		🧪 Put Operation (Simplified): 
			First try CAS to insert
			If contention: Lock only the bucket, Other buckets remain accessible
			✔ High concurrency, ✔ Low contention

		🧪 Read Operation: map.get(key);
			✔ Lock-free, ✔ Uses volatile reads, ✔ Extremely fast

5️⃣ Key Differences: HashMap vs ConcurrentHashMap
| Aspect                      | HashMap   | ConcurrentHashMap   |
| --------------------------- | --------- | ------------------- |
| Thread-safe                 | ❌ No      | ✅ Yes               |
| Synchronization             | ❌ None    | ✅ Fine-grained      |
| Null keys                   | ✅ One     | ❌ Not allowed       |
| Null values                 | ✅ Allowed | ❌ Not allowed       |
| Performance (single thread) | Faster    | Slightly slower     |
| Performance (multi-thread)  | ❌ Unsafe  | ✅ Scales well       |
| Fail-fast iterator          | ✅ Yes     | ❌ Weakly consistent |
| Infinite loop risk          | ✅ Yes     | ❌ No                |

6️⃣ Why ConcurrentHashMap Does NOT Allow Nulls?
	🎯 Interview favorite question.
			Reason: 
				null is used to indicate absence of mapping
				In concurrent access: get() returning null is ambiguous: key absent? value is null?
				Disallowing null removes ambiguity

7️⃣ Iterators: Fail-Fast vs Weakly Consistent
	HashMap: Fail-fast, Throws ConcurrentModificationException
	ConcurrentHashMap: Weakly consistent, Does not throw exception
	Reflects some (not necessarily all) updates
	🎯 Interview line: “ConcurrentHashMap iterators trade consistency for availability.”

8️⃣ When Should You Use What?
	✅ Use HashMap When: Single-threaded, Local method usage, No concurrent access
	✅ Use ConcurrentHashMap When: Shared mutable state, High read/write concurrency, Performance matters
	❌ Avoid Collections.synchronizedMap() for high concurrency — it uses global locking.

🔹 Q58. Difference between CopyOnWriteArrayList and synchronizedList. When should CopyOnWrite be used?
=============================================================================================================
1️⃣ What Is synchronizedList? List<Integer> list = Collections.synchronizedList(new ArrayList<>());
	🔹 How It Works: 
			Wraps a normal ArrayList, Uses one intrinsic lock for every operation
			All methods are synchronized on the same monitor
			👉 Only one thread at a time can read or write.
		🔧 Characteristics: 
				Thread-safe via coarse-grained locking
				Reads and writes block each other
				Iteration must be manually synchronized
					synchronized (list) {
						for (int i : list) {
							// safe iteration
						}
					}

2️⃣ What Is CopyOnWriteArrayList? List<Integer> list = new CopyOnWriteArrayList<>();
	🔹 How It Works
			On every write (add/remove/update): Creates a new copy of the underlying array
			Readers: Read from a stable snapshot, No locking at all
			👉 Reads are lock-free, writes are expensive.
		🔧 Characteristics:
				Thread-safe via copy-on-write
				No synchronization needed for iteration
				Iterators are snapshot-based
					for (int i : list) {
						// always safe
					}

3️⃣ Core Difference (High-Level)
	synchronizedList uses locking; CopyOnWriteArrayList uses immutability + copying.

4️⃣ CopyOnWriteArrayList vs synchronizedList (Side-by-Side)
| Aspect            | synchronizedList    | CopyOnWriteArrayList |
| ----------------- | ------------------- | -------------------- |
| Thread safety     | Via locking         | Via copying          |
| Read concurrency  | ❌ One at a time     | ✅ Fully concurrent   |
| Write cost        | Low                 | ❌ High (array copy)  |
| Read cost         | Medium (lock)       | ✅ Very fast          |
| Iteration         | Must synchronize    | Safe without sync    |
| Iterator behavior | Fail-fast           | Snapshot (no CME)    |
| Memory usage      | Low                 | ❌ Higher             |
| Best for          | Balanced read/write | Read-heavy workloads |

5️⃣ Iteration Difference (Interview Favorite)
	🔴 synchronizedList (Fail-Fast)
			for (int i : list) { } // ❌ CME if not synchronized

			Requires:
				synchronized (list) {
					for (int i : list) { }
				}

	🟢 CopyOnWriteArrayList (Snapshot Iterator)
			for (int i : list) { } // ✅ always safe
			Iterator sees a snapshot, Does not reflect later writes, Never throws ConcurrentModificationException
		🎯 Interview line: “CopyOnWriteArrayList iterates over a snapshot, not live data.”

6️⃣ When Should You Use CopyOnWriteArrayList?
	✅ Use CopyOnWriteArrayList When: 
		Reads vastly outnumber writes, Iteration is frequent
		You want lock-free reads, Slightly stale data is acceptable

	🧪 Real-World Examples: Event listeners list, Observer pattern, Configuration readers, Caches of reference data

7️⃣ When NOT to Use CopyOnWriteArrayList ❌
		Write-heavy workloads, Large lists with frequent updates, Real-time systems where updates must be immediately visible, Memory-sensitive systems
		🎯 Interview line: “CopyOnWriteArrayList is terrible for write-heavy workloads.”

8️⃣ Why CopyOnWrite Exists (Design Insight)
	The idea: “Make reads extremely cheap by making writes expensive.”
	This flips the traditional locking model.

🔹 Q59. What is fail-fast vs fail-safe iterator? Which collections are thread safe by default?
=============================================================================================================
1️⃣ What Is a Fail-Fast Iterator?
	A fail-fast iterator throws a ConcurrentModificationException if the collection is structurally modified while it is being iterated (except through the iterator’s own remove()).
	👉 It fails immediately to prevent unpredictable behavior.

	🔍 How It Works Internally: Collection maintains a modCount, Iterator captures expectedModCount
		If they differ → exception thrown
		🧪 Example (Fail-Fast)
					List<Integer> list = new ArrayList<>();
					list.add(1);
					list.add(2);

					for (Integer i : list) {
						list.add(3); // ❌ ConcurrentModificationException
					}
			🔧 Characteristics: Detects concurrent modification, Not thread-safe, Used mainly in non-concurrent collections
				Best-effort (not guaranteed in all cases)

2️⃣ What Is a Fail-Safe Iterator?
	A fail-safe iterator does not throw ConcurrentModificationException if the collection is modified during iteration.
	👉 It works on a snapshot or separate structure.
	🧪 Example (Fail-Safe)
				List<Integer> list = new CopyOnWriteArrayList<>();
				list.add(1);
				list.add(2);

				for (Integer i : list) {
					list.add(3); // ✅ No exception
				}
		🔧 Characteristics: Iterator works on a copy/snapshot, No exception on modification, Thread-safe, May not reflect latest updates

	🎯 Interview line: “Fail-safe iterators trade real-time consistency for safety.”

3️⃣ Fail-Fast vs Fail-Safe (Side-by-Side)
| Aspect                   | Fail-Fast             | Fail-Safe                               |
| ------------------------ | --------------------- | --------------------------------------- |
| Behavior on modification | Throws exception      | Continues safely                        |
| Uses snapshot            | ❌ No                  | ✅ Yes                                   |
| Thread-safe              | ❌ No                  | ✅ Yes                                   |
| Consistency              | Strong (detects bugs) | Weak (eventual)                         |
| Performance              | Fast                  | Slower (copy overhead)                  |
| Example collections      | ArrayList, HashMap    | CopyOnWriteArrayList, ConcurrentHashMap |

4️⃣ Which Collections Are Thread-Safe by Default?
	✅ Thread-Safe Collections (By Default)
		🔹 Legacy (Synchronized by design): Vector, Hashtable, Stack
			⚠ Thread-safe but poor performance due to global locking.

		🔹 Concurrent Collections (Preferred): From java.util.concurrent:
			ConcurrentHashMap, CopyOnWriteArrayList, CopyOnWriteArraySet, ConcurrentLinkedQueue, ConcurrentLinkedDeque
			BlockingQueue implementations: ArrayBlockingQueue, LinkedBlockingQueue, PriorityBlockingQueue, DelayQueue

		✔ Thread-safe, ✔ High concurrency, ✔ Designed for multi-core systems

	
	❌ NOT Thread-Safe by Default: Require external synchronization:
		ArrayList, LinkedList, HashMap, HashSet, TreeMap, TreeSet

	🟡 Conditionally Thread-Safe (Wrappers)
			List list = Collections.synchronizedList(new ArrayList<>());
			Map map = Collections.synchronizedMap(new HashMap<>());
		✔ Thread-safe via global locking, ❌ Iteration still needs external synchronization

5️⃣ Iterator Behavior in Concurrent Collections (Interview Favorite)
	🔹 ConcurrentHashMap: Iterator is weakly consistent, Does not throw CME, May or may not reflect recent updates
	🔹 CopyOnWriteArrayList: Iterator is snapshot-based, Never reflects modifications after iterator creation

🔹 Q61. What is CAS (Compare And Swap)? How AtomicInteger works internally?
=============================================================================================================
1️⃣  CAS (Compare-And-Swap) is a hardware-level atomic instruction that updates a value only if it matches an expected value.
	Conceptually: 
		CAS(memoryLocation, expectedValue, newValue)
		if (memoryLocation == expectedValue)
			memoryLocation = newValue
			return true
		else
			return false

		👉 This entire operation is atomic — no other thread can interleave.

	🧠 Why CAS Exists: 
		Avoid blocking (synchronized), Avoid context switches
		Enable lock-free concurrency, Scale better on multi-core CPUs

	🎯 Interview line: “CAS allows threads to coordinate without blocking.”

2️⃣ CAS at the CPU Level
	Modern CPUs provide CAS instructions: x86 → CMPXCHG, ARM → LDREX / STREX
	Guarantees: Atomicity, Memory visibility, Happens-before semantics
	👉 JVM directly maps CAS to these instructions.

3️⃣ How AtomicInteger Uses CAS Internally
	🔹 Key Internal Field
			public class AtomicInteger extends Number {
				private volatile int value;
			}
		
		Important points: value is volatile → visibility guaranteed, Updates are done using CAS

4️⃣ Internal Working of incrementAndGet()
	🔥 Code You Write
			AtomicInteger counter = new AtomicInteger(0);
			counter.incrementAndGet();

		🔍 What Actually Happens Internally (Simplified)
				public final int incrementAndGet() {
					for (;;) {
						int current = value;          // volatile read
						int next = current + 1;
						if (compareAndSet(current, next)) {
							return next;
						}
						// else retry
					}
				}
			This is a spin loop.

		🧠 Step-by-Step Execution
			1️⃣ Thread reads current value
			2️⃣ Calculates new value
			3️⃣ Calls CAS
			4️⃣ If CAS succeeds → done
			5️⃣ If CAS fails → retry

			CAS fails if: Another thread updated the value in between

	🎯 Interview Line: “AtomicInteger uses a CAS-based retry loop to ensure atomic updates.”

5️⃣ Why CAS Is Lock-Free but Not Wait-Free
	Lock-free: At least one thread always makes progress
	Not wait-free: A thread may retry indefinitely under heavy contention
		Thread A → CAS fails → retry
		Thread B → CAS fails → retry
	👉 But no thread is blocked.

6️⃣ Why AtomicInteger Is Better Than synchronized Counter
	❌ synchronized Counter
			synchronized void increment() {
				count++;
			}
		Thread blocking, Context switching, Lower scalability

	✅ AtomicInteger
			counter.incrementAndGet();
		No blocking, Uses CPU instructions, Scales better under contention
	
	🎯 Interview line: “CAS avoids blocking but may spin under contention.”

7️⃣ CAS vs volatile (Important Clarification)
	| Feature    | volatile | CAS      |
	| ---------- | -------- | -------- |
	| Visibility | ✅        | ✅        |
	| Atomicity  | ❌        | ✅        |
	| Blocking   | ❌        | ❌        |
	| Use case   | Flags    | Counters |

	👉 AtomicInteger = volatile + CAS

8️⃣ The ABA Problem (Advanced Topic)
	🔹 What Is ABA?
			Value = A
			Thread 1 reads A
			Thread 2 changes A → B → A
			Thread 1 CAS succeeds (but state changed!)
			CAS cannot detect this.

	🔹 How Java Solves It: AtomicStampedReference, AtomicMarkableReference, AtomicStampedReference<Integer> ref;
	🎯 Interview bonus point: “CAS alone can’t detect ABA; stamped references fix it.”

9️⃣ Why AtomicInteger Cannot Protect Invariants
		x.incrementAndGet();
		y.incrementAndGet();
	❌ Not atomic together, ❌ Invariant can break
👉 CAS works per variable, not across variables.

🔹 Q63. Difference between AtomicLong and LongAdder. When to use LongAdder?
=============================================================================================================
1️⃣ AtomicLong vs LongAdder — High-Level Difference
	AtomicLong uses a single CAS-based value, while LongAdder spreads updates across multiple counters (cells) to reduce contention under high concurrency.

2️⃣ What Is AtomicLong?
	🔹 How It Works:
		Maintains: one volatile long value, Updates via CAS (Compare-And-Swap), All threads compete on one memory location
				AtomicLong counter = new AtomicLong(0);
				counter.incrementAndGet();
		
		🔧 Characteristics: Lock-free, Strong consistency, Exact value always available, CAS retries under contention
		⚠ Problem Under High Contention:
			Many threads update the same value, CAS failures increase, CPU spins → performance degrades

🎯 Interview line: “AtomicLong doesn’t scale well under high contention.”

3️⃣ What Is LongAdder?
	🔹 How It Works
		Maintains: One base value, Multiple cells (striped counters), Threads update different cells instead of one value
				LongAdder adder = new LongAdder();
				adder.increment();
				long sum = adder.sum();

		🔧 Internal Idea (Simplified)								
				Thread 1 → Cell[0]
				Thread 2 → Cell[3]
				Thread 3 → Cell[1]
		👉 Updates are distributed → less contention
		🧠 Key Insight: “LongAdder trades exact immediacy for scalability.”

4️⃣ How LongAdder Improves Performance
	Reduces CAS contention, Minimizes false sharing, Scales linearly with threads, Uses CAS internally but on different cells

5️⃣ Key Difference: Consistency vs Throughput
	| Aspect              | AtomicLong     | LongAdder             |
	| ------------------- | -------------- | --------------------- |
	| Update mechanism    | Single CAS     | Striped CAS           |
	| Contention handling | Poor           | Excellent             |
	| Read cost           | O(1)           | O(n) (sum cells)      |
	| Value accuracy      | Always exact   | Eventually consistent |
	| Best under          | Low contention | High contention       |
	| Memory usage        | Low            | Higher                |

6️⃣ Why LongAdder.sum() Is Not Constant-Time
			long value = adder.sum();
		Reads base, Iterates all cells, Aggregates values
	
	✔ Accurate at moment of call, ❌ More expensive than AtomicLong.get()

7️⃣ When Should You Use LongAdder?
	✅ Use LongAdder When: High write contention, Frequent increments, Occasional reads, Metrics, counters, statistics
	🧪 Real-World Examples: Request counters, QPS metrics, Hit/miss counters, Monitoring systems
	🎯 Interview line: “LongAdder is ideal for hot counters.”

8️⃣ When NOT to Use LongAdder ❌
	When you need: Precise value after every update, Strong consistency guarantees, When updates are rare
	When value must be read very frequently
	👉 Use AtomicLong instead.

9️⃣ Why LongAdder Is NOT a Drop-In Replacement
		if (adder.sum() == 100) {
			// ❌ unsafe logic
		}
		Because: Value may change immediately after sum(), No CAS-style conditional update
	
	🎯 Interview line: “LongAdder is not suitable for CAS-style logic.”

🔹 Q64. What is ForkJoinPool? Difference between ForkJoinPool and ExecutorService.
=============================================================================================================
1️⃣  ForkJoinPool is a specialized thread pool designed for parallel execution of recursive, divide-and-conquer tasks.
	It works by:
		Forking a large task into smaller subtasks, Joining their results
		Using work-stealing to balance load across CPU cores, Introduced in Java 7.

2️⃣ Core Idea: Divide and Conquer
	Instead of one big task:
					Task
					 ├── Subtask A
					 ├── Subtask B
					 └── Subtask C
		Each subtask: Is small, Can run independently, Can be stolen by idle threads

3️⃣ What Is Work-Stealing?
	🔹 How It Works: 
		Each worker thread has its own deque, Threads push new subtasks to the bottom
		Idle threads steal from the top of other workers’ deques

	✔ Minimizes contention, Maximizes CPU utilization

	🎯 Interview line: “ForkJoinPool balances load using work-stealing.”

4️⃣ ForkJoinPool Components
	🔹 Task Types: 
			RecursiveTask<V> → returns result
			RecursiveAction → no result
	
		🧪 Example:
				class SumTask extends RecursiveTask<Integer> {
					private final int[] arr;
					private final int start, end;

					protected Integer compute() {
						if (end - start <= 10) {
							int sum = 0;
							for (int i = start; i < end; i++) sum += arr[i];
							return sum;
						}
						int mid = (start + end) / 2;
						SumTask left = new SumTask(arr, start, mid);
						SumTask right = new SumTask(arr, mid, end);

						left.fork();
						return right.compute() + left.join();
					}
				}

5️⃣ ForkJoinPool vs ExecutorService (High-Level)
| Aspect           | ForkJoinPool                | ExecutorService              |
| ---------------- | --------------------------- | ---------------------------- |
| Purpose          | Parallel divide-and-conquer | General async task execution |
| Task type        | Recursive                   | Independent                  |
| Load balancing   | Work-stealing               | Shared queue                 |
| Thread usage     | CPU-bound optimized         | General-purpose              |
| Blocking tasks   | ❌ Bad                       | ✅ Better                     |
| Parallel streams | ✅ Uses                      | ❌ No                         |

6️⃣ Key Differences Explained (Interview-Critical)
	🔹 1. Task Granularity 
			ForkJoinPool: Many small tasks, Designed for splitting
			ExecutorService: Fewer, independent tasks, No splitting awareness

	🔹 2. Queue Model
			ForkJoinPool: Per-thread deque, Work-stealing
			ExecutorService: Central blocking queue
	
	🔹 3. Thread Utilization
			ForkJoinPool: Keeps CPUs busy, Idle threads steal work
			ExecutorService: Threads wait on queue
	
	🔹 4. Blocking Behavior (Very Important)
			ForkJoinPool: Blocking is dangerous, Can starve worker threads
			ExecutorService: Handles blocking tasks better
	🎯 Interview line: “ForkJoinPool is for CPU-bound tasks, not blocking I/O.”

7️⃣ Common Uses of ForkJoinPool
	Parallel streams, Large array processing, Recursive algorithms, Sorting, Searching, Map-reduce style operations

8️⃣ ForkJoinPool.commonPool(): 	Global shared pool
	Used by: CompletableFuture, Parallel streams, Size ≈ CPU cores − 1
	⚠ Blocking here affects entire JVM.

9️⃣ When to Use What?
	✅ Use ForkJoinPool When: CPU-bound work, Recursive tasks, Large datasets, Parallel computation
	❌ Avoid ForkJoinPool When: Blocking I/O, Long-running tasks, External calls

	✅ Use ExecutorService When: I/O-bound tasks, Independent tasks, You need bounded queues, Predictable throughput

🔹 Q65. What is work-stealing algorithm? When should ForkJoin be used?
=============================================================================================================
1️⃣ What Is the Work-Stealing Algorithm?
	Work-stealing is a scheduling algorithm where:
		Each worker thread maintains its own task deque (double-ended queue)
		Busy threads push new subtasks to their own deque
		Idle threads steal tasks from other threads
		👉 This keeps all CPU cores busy without centralized coordination.

2️⃣ How Work-Stealing Works (Step-by-Step)
	🔹 Core Rules: Each worker thread has its own deque, New tasks are pushed to the bottom
		The owning thread: Pops tasks from the bottom (LIFO)
		Idle threads: Steal tasks from the top (FIFO) of another thread’s deque
	
	🧠 Why Two Ends? Reduces contention, Improves cache locality, Minimizes synchronization
	🎯 Interview line: “Work-stealing uses per-thread deques to minimize contention.”

3️⃣ Visual Example (Mental Model)
		Worker-1 Deque: [ TaskA | TaskB | TaskC ]
							  ↑ pop (owner)
		Worker-2 (idle) → steals TaskA from top
	
	✔ Worker-2 becomes busy, ✔ Worker-1 continues work

4️⃣ Why Work-Stealing Is Better Than a Shared Queue?
	🔴 Shared Queue (ExecutorService): All threads compete for one queue, High contention, Poor scalability
	🟢 Work-Stealing (ForkJoinPool): Mostly thread-local queues, Stealing only when needed, Scales well on multi-core CPUs

5️⃣ When Should ForkJoinPool Be Used?
	✅ Use ForkJoinPool When:
		Tasks can be recursively split, Workload is CPU-bound, Subtasks are independent, Large dataset processing
		Parallel algorithms
	
	🧪 Real-World Examples: Parallel array processing, Sorting (merge sort, quicksort), Searching, Map-reduce style tasks
		Parallel streams, CompletableFuture CPU tasks

	🎯 Interview line: “ForkJoin is ideal for fine-grained CPU-bound parallelism.”

6️⃣ When Should ForkJoinPool NOT Be Used?
	❌ Avoid ForkJoinPool When: Tasks are blocking (I/O, DB, REST calls), Tasks are long-running, Work cannot be divided
		Strict ordering is required
		Why? Blocking steals worker threads, Starves the pool, Reduces parallelism

7️⃣ How ForkJoinPool Handles Blocking (Advanced)
	Java provides: ForkJoinPool.managedBlock(...)
		Allows the pool to: Temporarily add threads, Avoid starvation
		⚠ Still not ideal for heavy I/O.

8️⃣ ForkJoin vs ExecutorService (Quick Summary)
| Aspect      | ForkJoinPool  | ExecutorService |
| ----------- | ------------- | --------------- |
| Scheduling  | Work-stealing | Shared queue    |
| Task nature | Recursive     | Independent     |
| Best for    | CPU-bound     | I/O-bound       |
| Blocking    | ❌ Bad         | ✅ OK            |
| Scalability | Very high     | Moderate        |

🔹 Q66. How parallel streams work internally? Why parallel streams can be dangerous? How to control parallel stream thread pool?
=============================================================================================================
1️⃣  A parallel stream splits a stream pipeline into multiple subtasks and executes them in parallel using ForkJoinPool.
		list.parallelStream()
			.map(this::process)
			.forEach(System.out::println);
	
	🔧 Internal Mechanics (Step by Step)
			1️⃣ Source Splitting (Spliterator): Each stream source has a Spliterator
				It knows how to: Estimate size, Split itself into smaller chunks
					Data → Spliterator → split → split → split
			
			2️⃣ Task Creation: 
				Each split becomes a ForkJoinTask, Tasks are recursively subdivided until they are “small enough”

			3️⃣ Execution via ForkJoinPool
				Tasks are submitted to ForkJoinPool.commonPool(), Threads use work-stealing
				Idle threads steal subtasks from busy threads

			4️⃣ Result Combination: Partial results are merged, Final result is produced

		🎯 Interview line: “Parallel streams are implemented using ForkJoinPool and work-stealing.”

2️⃣ Which Thread Pool Do Parallel Streams Use?
	🔹 Default Behavior: Parallel streams always use ForkJoinPool.commonPool()
		Same pool used by: CompletableFuture (by default), Other parallel streams, Size ≈ CPU cores - 1
		⚠ This is global and shared across the JVM.

3️⃣ Why Parallel Streams Can Be Dangerous. This is the most important part.
	🔴 1. Blocking Operations Can Starve the JVM
			list.parallelStream()
				.map(this::callRemoteService) // blocking I/O
				.forEach(System.out::println);
			❌ Each blocking call occupies a ForkJoin worker, ❌ Other tasks (CF, streams) wait, ❌ System-wide slowdown

			🎯 Interview line: “Blocking inside parallel streams can starve the common pool.”

	🔴 2. No Control Over Thread Pool (By Default)
			You cannot: Set queue size, Set rejection policy,Isolate workloads, All parallel streams share one pool.

	🔴 3. Side Effects & Race Conditions
			int sum = 0;
			list.parallelStream().forEach(i -> sum += i); // ❌ broken
			
			Parallel streams require stateless, non-mutating operations, Shared mutable state → race conditions

	🔴 4. Performance Can Be Worse Than Sequential
			Parallel streams add overhead: Task splitting, Context switching, Result merging
			For: Small collections, Cheap operations ➡ Sequential streams are faster

		🎯 Interview line: “Parallel streams are not automatically faster.”

	🔴 5. Debugging & Observability Issues
			Non-deterministic execution order, Harder stack traces, Harder to reason about failures

4️⃣ When Are Parallel Streams a Good Fit?
	✅ Use Parallel Streams When: Data size is large, Operations are CPU-bound, Tasks are independent
		No shared mutable state, Order doesn’t matter
		
		🧪 Good Examples: Large array transformations, Numeric computations, CPU-heavy map/filter/reduce

5️⃣ How to Control Parallel Stream Thread Pool
	⚠ Important Truth (Interview Gold): You cannot directly configure the thread pool used by parallel streams.
	But you have workarounds.

		✅ Option 1: Use Custom ForkJoinPool (Recommended)
					ForkJoinPool pool = new ForkJoinPool(4);

					pool.submit(() ->
						list.parallelStream()
							.map(this::process)
							.forEach(System.out::println)
					).join();

			✔ Isolates workload, ✔ Controls parallelism, ✔ Prevents commonPool starvation

		🎯 Interview line: “To control parallel streams, run them inside a custom ForkJoinPool.”

		❌ Option 2: JVM Property (Global, Not Recommended)
			-Djava.util.concurrent.ForkJoinPool.common.parallelism=4
			Problems: Affects entire JVM, Risky in shared environments, Hard to tune per workload

6️⃣ Parallel Streams vs ExecutorService (Important Comparison)
| Aspect         | Parallel Stream | ExecutorService |
| -------------- | --------------- | --------------- |
| Thread pool    | commonPool      | Custom          |
| Control        | ❌ Low           | ✅ High          |
| Best for       | CPU-bound       | I/O or mixed    |
| Error handling | Poor            | Better          |
| Debugging      | Hard            | Easier          |

🔹 Q67. How does JVM handle thread stack? What is stack overflow in multithreading?
=============================================================================================================
1️⃣ How Does JVM Handle Thread Stack?
	🔹 Per-Thread Stack (Very Important): 👉 Each Java thread has its own private stack.
		Stack is not shared, Created when thread is created, Destroyed when thread terminates
			Thread-1 → Stack-1
			Thread-2 → Stack-2
			Thread-3 → Stack-3
		🎯 Interview line: “Stack memory is thread-confined.”

2️⃣ What Is Stored in the Thread Stack? 	Each thread stack consists of stack frames, one per method call.
	🔹 Each Stack Frame Contains: 
		Local variables, Method parameters, Operand stack, Return address, Reference to constant pool
			void a() {
				int x = 10;
				b();
			}

			void b() {
				int y = 20;
			}
		Call sequence:
					Stack:
					| b() frame |
					| a() frame |

3️⃣ Stack Size in JVM: Stack size is fixed per thread
	Configurable using JVM option: -Xss512k, -Xss1m
	Default: Depends on OS + JVM, Typically 512 KB – 1 MB per thread

	🎯 Interview line: “Stack size limits recursion depth and local variable usage.”

4️⃣ Why Stack Is Important in Multithreading
	Because: Every thread consumes its own stack memory, Too many threads → huge memory consumption
	Example: 1000 threads × 1MB stack = 1GB memory
	❌ JVM crash, ❌ OutOfMemoryError

5️⃣ What Is StackOverflowError? StackOverflowError occurs when a thread exhausts its stack space.
	Common causes: Infinite recursion, Deep recursion, Large local variables, Excessive method calls
	🧪 Simple Example
			void recursive() {
				recursive();
			}
		➡ Each call adds a new stack frame, ➡ Stack fills up, ➡ StackOverflowError

6️⃣ StackOverflow in Multithreading (Key Question)
	🔥 Important Clarification: 
		StackOverflowError is per-thread, not global.
		One thread overflowing stack does NOT affect others
		Each thread crashes independently
		
		🧠 Example Scenario
					new Thread(() -> recursive()).start();
					new Thread(() -> doWork()).start();

			Thread-1 → StackOverflowError
			Thread-2 → Continues normally

		🎯 Interview line: “Stack overflow affects only the failing thread.”

7️⃣ Multithreading + Stack Overflow = Hidden Danger
	🔴 Problem 1: Too Many Threads
				Even without recursion: Creating thousands of threads, Each thread needs stack memory
				➡ JVM may throw: java.lang.OutOfMemoryError: unable to create new native thread
	
	🔴 Problem 2: Recursion + Thread Pools
				Recursive logic inside: Executor threads, ForkJoinPool tasks
				➡ Can kill worker threads, ➡ Reduce pool capacity, ➡ Throughput degradation

8️⃣ Stack vs Heap in Multithreading (Interview Favorite)
| Aspect        | Stack              | Heap                  |
| ------------- | ------------------ | --------------------- |
| Shared?       | ❌ No               | ✅ Yes                 |
| Thread safety | Inherent           | Needs synchronization |
| Stores        | Local variables    | Objects               |
| Lifetime      | Method scope       | Until GC              |
| Error         | StackOverflowError | OutOfMemoryError      |

9️⃣ How JVM Protects Stack Memory
	Stack is: Fixed size, Bounds-checked
	JVM throws: StackOverflowError, No silent corruption

1️⃣ How Many Threads Can a JVM Create?
There is no fixed number. The maximum number of threads depends on OS limits, available memory, and JVM configuration (especially stack size).

🔹 Q69. Difference between concurrency and parallelism. What is safepoint?
=============================================================================================================
1️⃣ Difference Between Concurrency and Parallelism
	🔹 Concurrency is about structuring a program to handle multiple tasks that overlap in time, even if they are not 	executing at the same instant.
		👉 It’s about correctness, coordination, and responsiveness.
		
		🧠 Key Idea: Tasks make progress together, Execution may be interleaved, Can exist on single-core CPU
			Task A → pause → Task B → pause → Task A
		
		🧪 Example: Web server handling thousands of requests, Each request waits for I/O, Threads/tasks are interleaved
		🎯 Interview line: “Concurrency is about managing multiple tasks, not executing them simultaneously.”

	🔹 Parallelism is about executing multiple tasks simultaneously using multiple CPU cores.
		👉 It’s about performance and throughput.

		🧠 Key Idea: Tasks run at the same time, Requires multi-core hardware
					Core 1 → Task A
					Core 2 → Task B
					Core 3 → Task C
		
		🧪 Example: Parallel streams, ForkJoinPool,Matrix multiplication
		
		🎯 Interview line: “Parallelism is about doing more work faster.”

🔥 Concurrency vs Parallelism — Side by Side
| Aspect              | Concurrency              | Parallelism                |
| ------------------- | ------------------------ | -------------------------- |
| Focus               | Structure & coordination | Execution speed            |
| Execution           | Interleaved              | Simultaneous               |
| Requires multi-core | ❌ No                     | ✅ Yes                      |
| Goal                | Responsiveness           | Throughput                 |
| Example             | Web server               | Data processing            |
| Java tools          | Locks, executors         | ForkJoin, parallel streams |

🧠 Important Insight (Interview Gold): Concurrency does not imply parallelism, but parallelism implies concurrency.

2️⃣ What Is a Safepoint? Now the JVM internals part.
	A safepoint is a state where all Java threads are paused (or at a known safe state) so that the JVM can perform global operations safely.
	👉 At a safepoint, the JVM knows: Where every thread is, What objects they reference
		That no thread is modifying critical structures

3️⃣ Why Does JVM Need Safepoints?
	Some JVM operations cannot run concurrently with application threads:
	🔥 Common Safepoint Operations:
			Garbage Collection (especially Stop-The-World GC)
			Deoptimization, Class redefinition (HotSwap), Biased lock revocation, Thread stack inspection
	
	🎯 Interview line: “Safepoints allow the JVM to reach a globally consistent state.”

4️⃣ How Safepoints Work Internally
	🔹 Key Mechanism: 
		JVM inserts safepoint checks at specific locations:
			Method calls, Loop back edges, Return points
				while (true) {
					work(); // safepoint check here
				}
			When a safepoint is requested: Threads run until they reach a safepoint, Then they pause

	🧠 Important Detail
		Threads are not stopped at arbitrary instructions.
		They stop only at well-defined safe locations.

5️⃣ What Happens at a Safepoint?
	All application threads: Are paused, OR Are in a known safe state, JVM performs the operation, Threads are resumed
	This is often called: Stop-The-World (STW)

6️⃣ Safepoint vs Blocking (Important Distinction)
	| Safepoint           | Blocking                 |
	| ------------------- | ------------------------ |
	| JVM-initiated       | Application-initiated    |
	| Affects all threads | Affects specific threads |
	| For JVM internals   | For synchronization      |
	| Example: GC         | Example: synchronized    |

	🎯 Interview line: “Safepoints are JVM pauses, not application locks.”

7️⃣ Performance Impact of Safepoints
	Frequent safepoints → latency spikes
	Long GC safepoints → application pauses
	Tight loops without safepoint checks → safepoint bias problems
	Modern JVMs optimize this heavily.

🔹 Q70. How does GC interact with running threads? Can GC stop application threads?
=============================================================================================================
1️⃣ How Does GC Interact with Running Threads?
	Yes, the JVM can stop application threads for garbage collection — but how and when depends on the GC algorithm.
	GC and application threads coexist, but sometimes GC needs exclusive access to the heap.

2️⃣ Key Concept: Safepoint
	GC never stops threads randomly.
	👉 JVM brings all application threads to a safepoint.
		At a safepoint: Thread state is well-defined, Object references are known, No heap mutation happens

	🎯 Interview line: “GC pauses threads only at safepoints, not at arbitrary instructions.”

3️⃣ Can GC Stop Application Threads?
	✅ Yes — This Is Called Stop-The-World (STW)
	During STW: All application (mutator) threads are paused, GC threads run, Application resumes afterward
	This is required for: Heap consistency, Correct object reachability, Compacting memory

4️⃣ When Does GC Stop Threads?
	🔴 Always STW Phases (All GCs)
		Some GC phases must stop all threads: 
			Initial root marking, Final remark, Heap compaction, Class unloading, Deoptimization
			Even “concurrent” collectors have short STW pauses.
	
		🎯 Interview line: “No GC is 100% pause-free.”

5️⃣ Types of GC vs Thread Interaction
	🔹 1. Serial GC: Single GC thread, Long STW pauses, All app threads stopped
		❌ Not for production servers
	
	🔹 2. Parallel GC: Multiple GC threads, Still fully STW, Faster than Serial, but pauses scale with heap size
	
	🔹 3. CMS (Concurrent Mark Sweep) (legacy): Most work done concurrently
			Still has: Initial Mark → STW, Remark → STW, Shorter pauses, but more complex.
	
	🔹 4. G1 GC: Region-based, Mostly concurrent, Predictable, short STW pauses, Designed for low latency
	
	🔹 5. ZGC / Shenandoah: 
		Ultra-low latency GCs, Do almost everything concurrently
		STW pauses are very short (milliseconds), Heap size independent pauses

	🎯 Interview line: “Modern GCs minimize pause time, not eliminate it.”

6️⃣ What Happens to Threads During STW?
	When GC is triggered:
		1️⃣ JVM requests safepoint
		2️⃣ All application threads: Finish current instruction, Reach a safepoint, Pause execution
		3️⃣ GC threads run
		4️⃣ Threads are resumed
		
	Important: Threads do not lose stack data, Execution continues normally afterward

7️⃣ Do All Threads Stop?
	🔹 What Stops: Application (mutator) threads
	🔹 What Keeps Running: GC threads, JVM internal threads (limited)

8️⃣ Why GC Must Stop Threads (Sometimes)
	Because without stopping: Objects could move while being accessed, References could become invalid, Heap could become inconsistent
	
	🎯 Interview line: “Stopping threads ensures a consistent view of the heap.”

9️⃣ Performance Impact of GC Pauses
	Effects of long STW: Latency spikes, Request timeouts, “Application freeze”, SLA violations
	That’s why: Low-latency systems prefer G1 / ZGC, Heap tuning matters

🔹 Q71. ThreadLocal – how it works internally? Memory leak with ThreadLocal?
=============================================================================================================
1️⃣ What is ThreadLocal?
	ThreadLocal<T> provides thread-confined variables — each thread has its own independent copy of the variable.
		ThreadLocal<Integer> tl = new ThreadLocal<>();
		tl.set(10);
		Integer value = tl.get();
		
	👉 Same ThreadLocal object, different value per thread.
	🎯 Interview line: “ThreadLocal gives per-thread state without synchronization.”

2️⃣ How ThreadLocal Works Internally (Very Important)
	This is where most people fail interviews.
	🔑 Key Insight: ThreadLocal values are NOT stored inside ThreadLocal. They are stored inside the Thread object.

3️⃣ Internal Structure
	Each Thread has a field:
		ThreadLocal.ThreadLocalMap threadLocals;
		Structure:
					Thread
					 └── ThreadLocalMap
						  ├── Entry (key = ThreadLocal, value = Object)
						  ├── Entry (key = ThreadLocal, value = Object)
						  └── ...

4️⃣ ThreadLocalMap Entry (Critical Detail)
		static class Entry extends WeakReference<ThreadLocal<?>> {
			Object value;
		}

	⚠ Important: Key → ThreadLocal (WeakReference), Value → Actual object (Strong reference)
	🎯 Interview line (very important): “ThreadLocalMap uses weak keys but strong values.”

5️⃣ Step-by-Step: set() / get()
	🔹 set(value): Get current thread, Get thread.threadLocals, Put (ThreadLocal → value) into map
	
	🔹 get(): Get current thread, Look up current ThreadLocal key, Return associated value

6️⃣ Why ThreadLocal Is Fast & Thread-Safe
	No locking, No contention, Data is thread-confined, Map is accessed by only one thread
	
	🎯 Interview line: “ThreadLocal achieves thread safety through isolation, not synchronization.”

7️⃣ Memory Leak with ThreadLocal (Most Important Part)
	❗ Yes — ThreadLocal can cause memory leaks, especially with thread pools.
	🔥 How the Leak Happens
		Scenario:
				ThreadLocal<User> tl = new ThreadLocal<>();
				tl.set(new User()); // large object
		Later:
				tl = null; // ThreadLocal reference lost

		What happens internally?
			Key (ThreadLocal) → GCed (weak reference)
			Value (User) → ❌ NOT GCed
			Entry remains in ThreadLocalMap with:
				key = null
				value = User (strong reference)
			➡ Memory leak

8️⃣ Why This Is Worse with Thread Pools
	Thread pools: Threads live for a long time, ThreadLocalMap lives as long as thread lives
	So: Request finishes, Thread reused, Old ThreadLocal values still there
	
	🎯 Interview line: “ThreadLocal leaks are dangerous in thread pools because threads don’t die.”

9️⃣ When Does Cleanup Happen?
	ThreadLocalMap cleans entries: During get(), set(), remove() NOT automatically on GC
	If you never touch ThreadLocal again → leak persists.

	🔴 Real Production Example: Web application
		ThreadLocal stores: User context, Security context, DB connection, Thread pool reused, Missing remove()
		Heap grows slowly → OutOfMemoryError

🔟 How to Prevent ThreadLocal Memory Leaks
	✅ Always call remove()
						try {
							threadLocal.set(value);
							// use it
						} finally {
							threadLocal.remove();
						}

	✅ Use frameworks carefully: Spring Security cleans context, MDC logging cleans ThreadLocal
	
	❌ Do NOT store: Large objects, DB connections, Request-scoped data without cleanup

11️⃣ ThreadLocal vs Shared Variable (Quick Contrast)
| Aspect            | ThreadLocal  | Shared Variable |
| ----------------- | ------------ | --------------- |
| Synchronization   | ❌ Not needed | ✅ Required      |
| Visibility issues | ❌ No         | ✅ Yes           |
| Memory risk       | ⚠ Leak risk  | ❌ No            |
| Best for          | Context data | Shared state    |

🔹 Q72. Design a producer-consumer system.
=============================================================================================================
3️⃣ Correct Java Design (Recommended Solution)
✅ Using BlockingQueue (Best Practice)
🔹 Data Model
			class Task {
				private final int id;

				Task(int id) {
					this.id = id;
				}

				int getId() {
					return id;
				}
			}
🔹 Producer
			class Producer implements Runnable {

				private final BlockingQueue<Task> queue;
				private static int counter = 0;

				Producer(BlockingQueue<Task> queue) {
					this.queue = queue;
				}

				@Override
				public void run() {
					try {
						while (!Thread.currentThread().isInterrupted()) {
							Task task = new Task(counter++);
							queue.put(task); // blocks if queue is full
							System.out.println("Produced: " + task.getId());
							Thread.sleep(500);
						}
					} catch (InterruptedException e) {
						Thread.currentThread().interrupt();
					}
				}
			}
🔹 Consumer
			class Consumer implements Runnable {
				private final BlockingQueue<Task> queue;

				Consumer(BlockingQueue<Task> queue) {
					this.queue = queue;
				}

				@Override
				public void run() {
					try {
						while (!Thread.currentThread().isInterrupted()) {
							Task task = queue.take(); // blocks if queue empty
							System.out.println("Consumed: " + task.getId());
							Thread.sleep(1000);
						}
					} catch (InterruptedException e) {
						Thread.currentThread().interrupt();
					}
				}
			}
🔹 Main Driver
		public class ProducerConsumerApp {
			public static void main(String[] args) throws InterruptedException {

				BlockingQueue<Task> queue = new ArrayBlockingQueue<>(5);

				ExecutorService executor = Executors.newFixedThreadPool(4);

				executor.submit(new Producer(queue));
				executor.submit(new Producer(queue));

				executor.submit(new Consumer(queue));
				executor.submit(new Consumer(queue));

				Thread.sleep(10000); // run system

				executor.shutdownNow(); // graceful shutdown
			}
		}

	✅ Thread Safety: BlockingQueue handles synchronization internally, No explicit locks needed
	✅ Back-Pressure: Queue is bounded, Producers block when queue is full
	✅ No Busy Waiting: put() and take() block efficiently
	✅ Scalable: Add producers/consumers easily, Tune queue size independently

Producer ---> put() ---> [ BlockingQueue ] ---> take() ---> Consumer
              (blocks if full)               (blocks if empty)

🔹 Q73. Implement rate limiter using Java concurrency.
=============================================================================================================
1️⃣  A rate limiter controls how many requests/actions are allowed per unit of time, protecting systems from overload.
	Examples: 
		100 requests / second per user
		1000 API calls / minute
		10 DB writes / second

2️⃣ Which Algorithm to Use?
	Token Bucket is the best choice because:Simple, Smooth traffic, Allows bursts, Easy to implement using Java concurrency

🪣 Token Bucket Algorithm (Quick)
	Bucket has max capacity, Tokens are added at a fixed rate, Each request consumes 1 token
	If no token → request rejected

3️⃣ Token Bucket – Java Implementation (Recommended)
	✅ Thread-safe, ✅ Non-blocking, ✅ Production-ready
	🔹 Core Design Choices
			AtomicLong → thread-safe token count
			ScheduledExecutorService → refill tokens
			No synchronized blocks
			Fast under high concurrency

🔹 Implementation
					import java.util.concurrent.*;
					import java.util.concurrent.atomic.AtomicLong;

					public class RateLimiter {

						private final long capacity;
						private final long refillRate; // tokens per second
						private final AtomicLong tokens;

						private final ScheduledExecutorService scheduler =
								Executors.newSingleThreadScheduledExecutor();

						public RateLimiter(long capacity, long refillRate) {
							this.capacity = capacity;
							this.refillRate = refillRate;
							this.tokens = new AtomicLong(capacity);

							startRefill();
						}

						private void startRefill() {
							scheduler.scheduleAtFixedRate(() -> {
								long current;
								long updated;
								do {
									current = tokens.get();
									updated = Math.min(capacity, current + refillRate);
								} while (!tokens.compareAndSet(current, updated));
							}, 1, 1, TimeUnit.SECONDS);
						}

						public boolean allowRequest() {
							long current;
							do {
								current = tokens.get();
								if (current == 0) {
									return false;
								}
							} while (!tokens.compareAndSet(current, current - 1));
							return true;
						}

						public void shutdown() {
							scheduler.shutdown();
						}
					}

🔹 Usage Example
				public class RateLimiterTest {

					public static void main(String[] args) throws InterruptedException {

						RateLimiter limiter = new RateLimiter(5, 2); // 5 max, 2/sec

						ExecutorService executor = Executors.newFixedThreadPool(10);

						for (int i = 0; i < 20; i++) {
							executor.submit(() -> {
								if (limiter.allowRequest()) {
									System.out.println("Request allowed");
								} else {
									System.out.println("Request rejected");
								}
							});
							Thread.sleep(200);
						}

						executor.shutdown();
						limiter.shutdown();
					}
				}

5️⃣ Alternative Implementations (Interview Bonus)
	🔹 Using Semaphore (Simpler but Less Precise)
				Semaphore semaphore = new Semaphore(10);

				boolean allow() {
					return semaphore.tryAcquire();
				}
		❌ No smooth refill, Less control
	
	🔹 Fixed Window Counter (Not Recommended)
			Counter resets every second, Causes burst at window boundary, Easy but inaccurate
	
	🔹 Sliding Window (More Complex): More accurate, Requires timestamps, Higher memory & CPU

🔹 Q74. Design a thread-safe cache. How to ensure ordering of tasks?
=============================================================================================================
PART 1️⃣ Design a Thread-Safe Cache
	1️⃣ Core Requirements (Say This First)
		A good cache should provide: Thread safety, Fast reads, Controlled writes, Optional eviction (LRU / TTL)
		No corruption under concurrency
	
	2️⃣ Best Default Choice in Java: ConcurrentHashMap + proper coordination
		Why? Lock-free reads, Fine-grained locking for writes, Scales under high concurrency

	3️⃣ Simple Thread-Safe Cache (No Eviction)
		✅ Basic Design
						class Cache<K, V> {

							private final ConcurrentHashMap<K, V> map = new ConcurrentHashMap<>();

							public V get(K key) {
								return map.get(key);
							}

							public void put(K key, V value) {
								map.put(key, value);
							}

							public V computeIfAbsent(K key, Function<K, V> loader) {
								return map.computeIfAbsent(key, loader);
							}
						}

		🔥 Interview line: “ConcurrentHashMap provides thread safety without global locking.”

	4️⃣ Prevent Cache Stampede (Very Important)
		❌ Problem: Multiple threads miss the cache and load the same value repeatedly.
		✅ Solution: computeIfAbsent
					V value = cache.computeIfAbsent(key, k -> loadFromDB(k));
		
		✔ Loader called only once, ✔ Other threads wait for result
		🎯 Interview line: “computeIfAbsent prevents cache stampede.”

	5️⃣ Thread-Safe Cache with LRU Eviction
		❌ Why plain ConcurrentHashMap is not enough: No ordering, No eviction
		✅ Option 1: Synchronized LRU (Simple, but limited)
					class LRUCache<K, V> {

						private final Map<K, V> cache =
							Collections.synchronizedMap(
								new LinkedHashMap<>(16, 0.75f, true) {
									protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
										return size() > 100;
									}
								}
							);
					}

				✔ Maintains access order, ❌ Global lock, ❌ Not great for high concurrency

		✅ Option 2: Production-Grade (Recommended)
			Use Caffeine / Guava Cache
			Why? Lock-free reads, Window-TinyLFU eviction, Time-based expiration, Async loading
			🎯 Interview line: “In production, use Caffeine instead of reinventing cache.”

		6️⃣ Summary: Cache Design Choices
			| Use Case         | Recommended             |
			| ---------------- | ----------------------- |
			| Simple cache     | ConcurrentHashMap       |
			| Avoid stampede   | computeIfAbsent         |
			| LRU / TTL        | Caffeine                |
			| High concurrency | ConcurrentHashMap-based |
			| Strong ordering  | Separate mechanism      |

PART 2️⃣ How to Ensure Ordering of Tasks
 This is often the hidden second part of the question.
	7️⃣ What Does “Ordering” Mean? 
		Tasks must execute: In submission order, Or per key (same key → ordered), Or globally ordered

	8️⃣ Global Ordering of Tasks
		✅ Best Solution: Single-Thread Executor
				ExecutorService executor =
						Executors.newSingleThreadExecutor();

				executor.submit(task1);
				executor.submit(task2);


		✔ Tasks execute one by one, ✔ Strict FIFO ordering
		🎯 Interview line: “SingleThreadExecutor guarantees ordering.”
	
	9️⃣ Ordering Per Key (Very Important Real-World Case)
		🔹 Example: Cache updates per user, Same user → ordered, Different users → parallel

			✅ Design: Key-based executors
						class OrderedExecutor {

							private final int stripes = 16;
							private final ExecutorService[] executors;

							OrderedExecutor() {
								executors = new ExecutorService[stripes];
								for (int i = 0; i < stripes; i++) {
									executors[i] = Executors.newSingleThreadExecutor();
								}
							}

							public void submit(Object key, Runnable task) {
								int index = Math.abs(key.hashCode() % stripes);
								executors[index].submit(task);
							}
						}

			✔ Same key → same executor → ordered, ✔ Different keys → parallel

			🎯 Interview line: “Striped executors preserve per-key ordering with concurrency.”

	🔟 Ordering Inside Cache Updates
			Example: Ordered cache writes
						executor.submit(() -> cache.put(key, value));
			Guarantees: No race condition, Deterministic ordering

	11️⃣ When NOT to Use Locks for Ordering ❌
		❌ Using synchronized for ordering, ❌ Busy waiting, ❌ Sleeping threads
		Ordering is an execution concern, not a locking concern.

🔹 Q75. How to limit concurrent API calls?
=============================================================================================================
1️⃣ What Does “Limit Concurrent API Calls” Mean?
	It means: Only N API calls should be executing at the same time, (extra calls must wait, queue, or fail fast)
	⚠ This is NOT the same as rate limiting (X requests per second).
	🔥 Key distinction (say this in interview)
		Concurrency limit → how many calls run at once
		Rate limit → how many calls per time window

2️⃣ Best & Simplest Solution: Semaphore
	✅ Semaphore is MADE for this problem
	🔹 How Semaphore Works: Maintains a fixed number of permits
		Each API call: acquires a permit, executes, releases permit
	
	✅ Implementation (Interview-Ready)
				import java.util.concurrent.Semaphore;

				public class ApiLimiter {

					private final Semaphore semaphore;

					public ApiLimiter(int maxConcurrentCalls) {
						this.semaphore = new Semaphore(maxConcurrentCalls);
					}

					public void callApi(Runnable apiCall) {
						try {
							semaphore.acquire();   // blocks if limit reached
							apiCall.run();
						} catch (InterruptedException e) {
							Thread.currentThread().interrupt();
						} finally {
							semaphore.release();
						}
					}
				}
	
	🔹 Usage
				ApiLimiter limiter = new ApiLimiter(5); // max 5 concurrent calls

				limiter.callApi(() -> {
					System.out.println("Calling API");
				});

	🎯 Interview line: “Semaphore is the cleanest way to cap concurrent API calls.”

3️⃣ Fail-Fast Instead of Blocking (Very Important Variant)
	Sometimes you don’t want to block threads.
	✅ Use tryAcquire()
				public boolean callApiIfAllowed(Runnable apiCall) {
					if (!semaphore.tryAcquire()) {
						return false; // reject immediately
					}
					try {
						apiCall.run();
						return true;
					} finally {
						semaphore.release();
					}
				}

		✔ Fast failure, ✔ Protects system under overload
	
	🎯 Interview line: “Fail-fast is better than blocking in high-load systems.”

4️⃣ Using Thread Pool to Limit Concurrency
	Another very common and valid approach.
	✅ Fixed Thread Pool
				ExecutorService executor =
						Executors.newFixedThreadPool(5);

				executor.submit(() -> callApi());

		✔ At most 5 API calls at once, ✔ Others are queued
		⚠ Drawback: Queue can grow unbounded, No easy rejection unless customized

	✅ Better: Bounded Queue + Rejection
				ThreadPoolExecutor executor =
					new ThreadPoolExecutor(
						5, 5,
						0, TimeUnit.MILLISECONDS,
						new ArrayBlockingQueue<>(50),
						new ThreadPoolExecutor.AbortPolicy()
					);

		✔ Limits concurrency, ✔ Limits queue, ✔ Rejects overload
		🎯 Interview line: “Thread pool + bounded queue gives concurrency + back-pressure.”

5️⃣ Semaphore vs Thread Pool (Important Comparison)
	| Aspect    | Semaphore          | Thread Pool    |
	| --------- | ------------------ | -------------- |
	| Controls  | Concurrency        | Execution      |
	| Blocking  | Optional           | Usually        |
	| Fail fast | Easy               | Harder         |
	| Queue     | ❌ No               | ✅ Yes          |
	| Use case  | External API calls | Task execution |

	🎯 Interview insight: “Use Semaphore when threads already exist; use thread pool to manage threads.”

6️⃣ Combining Rate Limiting + Concurrency Limiting (Senior Answer)
	Real systems usually need both.
	Example: Max 5 concurrent calls, Max 100 calls per minute
	Solution: Semaphore → concurrency, Token Bucket → rate
	🎯 Interview line: “Concurrency protects resources; rate limiting protects fairness.”

7️⃣ Reactive / Async Systems (Bonus)
	If API calls are async / non-blocking: Semaphore still works
	Or use: Reactive back-pressure, Bounded schedulers
But for Java interviews → Semaphore wins.

🔹 Q76. How to handle back-pressure?
=============================================================================================================
1️⃣  Back-pressure is a mechanism by which a system slows down producers when consumers cannot keep up.
	In simple words:
		Producer is too fast, Consumer is too slow, System must say: “Slow down or stop”
	
	🎯 Interview line: “Back-pressure prevents overload by propagating pressure upstream.”

2️⃣ Why Back-Pressure Is Critical
	Without back-pressure: Memory grows unbounded, Queues explode, Latency spikes, System crashes (OOM)
	Real-world analogy: A restaurant stops accepting orders when the kitchen is full.

3️⃣ Common Back-Pressure Strategies (Very Important)
	1️⃣ Blocking (Slow Down Producer)
		🔹 Using BlockingQueue (Classic)
				BlockingQueue<Task> queue = new ArrayBlockingQueue<>(100);
				queue.put(task); // blocks if queue is full
		
			✔ Simple, ✔ Natural throttling ❌ Producer threads get blocked

		🎯 Interview line: “BlockingQueue provides built-in back-pressure.”

	2️⃣ Fail Fast (Reject Requests)
		Instead of blocking, reject immediately.
		🔹 Semaphore example
				if (!semaphore.tryAcquire()) {
					throw new RuntimeException("System overloaded");
				}

			✔ Protects system, ✔ Predictable latency, ❌ Caller must retry

		🎯 Interview line: “Fail-fast is preferred in high-load systems.”

	3️⃣ Bounded Queues (Most Important)
		Never allow unbounded buffering.
				ThreadPoolExecutor executor =
						new ThreadPoolExecutor(
							10, 10,
							0, TimeUnit.MILLISECONDS,
							new ArrayBlockingQueue<>(100),
							new ThreadPoolExecutor.AbortPolicy()
						);

			✔ Controls memory, ✔ Applies back-pressure via rejection
		🎯 Interview line: “Unbounded queues hide problems; bounded queues expose them.”

	4️⃣ Rate Limiting (Slow Down Input)
		Limit how fast producers can generate work. Token Bucket, Leaky Bucket

					if (!rateLimiter.allow()) {
						return HTTP_429;
					}
			✔ Prevents bursts, ✔ Smooth traffic
		🎯 Interview line: “Rate limiting is a form of back-pressure.”
	
	5️⃣ Load Shedding (Drop Work Intentionally)
		When overloaded: Drop low-priority requests, Sample traffic
		Examples: Drop logs, Skip metrics, Reject background tasks
		🎯 Interview line: “Dropping work is better than crashing.”
	
	6️⃣ Caller Runs Policy (Executor Trick)
				new ThreadPoolExecutor(
					5, 5,
					0, TimeUnit.MILLISECONDS,
					new ArrayBlockingQueue<>(50),
					new ThreadPoolExecutor.CallerRunsPolicy()
				);
		What happens: When queue is full, Producer thread executes task itself, Producer slows down naturally
		✔ Elegant back-pressure ✔ No data loss
	
		🎯 Interview line: “CallerRunsPolicy pushes back pressure to the caller.”
	
	4️⃣ Back-Pressure in Different Architectures
		🔹 Synchronous APIs: Block, Reject (HTTP 429)
		🔹 Asynchronous / Event Systems: Bounded queues, Message broker credits, Consumer acknowledgments
		🔹 Reactive Systems: Demand-based flow, Subscriber controls rate
		🎯 Interview line: “Reactive systems make back-pressure explicit.”

6️⃣ Choosing the Right Strategy (Interview Gold)
| Scenario        | Best Strategy    |
| --------------- | ---------------- |
| CPU overload    | Fail fast        |
| I/O overload    | Bounded queue    |
| External API    | Rate limit       |
| Background jobs | Drop / delay     |
| Thread pools    | CallerRunsPolicy |

	🔥 Common CPU-Killer Patterns

				while (true) {
					// no blocking
				}
				Thread dump: java.lang.Thread.State: RUNNABLE

		❌ 2. Spin Waiting / CAS Storm
				while (!flag.compareAndSet(false, true)) {}
				Symptoms: RUNNABLE, Atomic operations, High contention

			🎯 Interview line: “CAS can cause CPU spin under contention.”

		❌ 3. Excessive Synchronization / Lock Contention
			Threads stuck trying to acquire lock: JVM spends CPU managing contention
			Look for: - waiting to lock <0x...>
		
		❌ 4. ForkJoinPool / Parallel Streams Abuse
			Parallel streams, Blocking inside them, Common pool starvation
			
			🎯 Interview line: “Blocking ForkJoinPool threads leads to CPU spikes.”
		
		❌ 5. GC Thrashing
			Symptoms: CPU high, Throughput low
			Check GC logs: -XX:+PrintGCDetails
			Look for: Too frequent GC, High GC CPU time

5️⃣ Correlate with JVM Metrics
	🔹 Use JConsole / VisualVM / JMC
		Check: Thread count, Runnable threads, GC activity, Heap pressure
		🎯 Interview line: “CPU spikes often correlate with GC or runnable thread explosion.”

6️⃣ Production-Safe Commands (Very Important)
	| Tool                | Safe in Prod?         |
	| ------------------- | --------------------- |
	| `top -H`            | ✅ Yes                 |
	| `jstack`            | ✅ Yes                 |
	| `jcmd GC.heap_info` | ✅ Yes                 |
	| `jmap -histo`       | ⚠ Carefully           |
	| `jmap -dump`        | ❌ Avoid unless needed |

7️⃣ Typical Root Causes Summary
	| Cause                   | Symptom                |
	| ----------------------- | ---------------------- |
	| Infinite loop           | Single thread 100% CPU |
	| Busy wait               | Many RUNNABLE threads  |
	| Lock contention         | CPU + low throughput   |

	| GC pressure             | CPU + latency          |
	| Parallel streams misuse | ForkJoinPool hot       |

8️⃣ How to Fix (After Finding Root Cause)
	Replace busy loops with blocking calls, Add back-pressure, Reduce thread count, Fix synchronization, Avoid blocking ForkJoinPool, Tune GC / heap, Use async APIs
	🎯 Interview line: “Fix is always simpler than diagnosis — diagnosis is the hard part.”

🔹 Q79. How do you test multithreaded code?

2️⃣ Core Principles for Testing Multithreaded Code

3️⃣ Unit Testing with Controlled Concurrency
	🔹 Use ExecutorService Instead of Raw Threads
			ExecutorService executor = Executors.newFixedThreadPool(10);
		This gives: Controlled concurrency, Deterministic shutdown
	
	🔹 Use CountDownLatch to Coordinate Threads
		Example: Testing race condition
						@Test
						void testConcurrentIncrement() throws InterruptedException {

							AtomicInteger counter = new AtomicInteger();
							int threads = 10;

							CountDownLatch start = new CountDownLatch(1);
							CountDownLatch done = new CountDownLatch(threads);

							for (int i = 0; i < threads; i++) {
								new Thread(() -> {
									try {
										start.await();
										counter.incrementAndGet();
									} catch (InterruptedException ignored) {
									} finally {
										done.countDown();
									}
								}).start();
							}

							start.countDown(); // start all threads
							done.await();

							assertEquals(threads, counter.get());
						}


			🎯 Interview line: “CountDownLatch helps start threads at the same time.”

🔹 Q80. How to make singleton thread safe?
=============================================================================================================
1️⃣ Option 3: Double-Checked Locking (Correct Only in Java 5+)
	✅ This is VERY IMPORTANT
			class Singleton {

				private static volatile Singleton instance;

				public static Singleton getInstance() {
					if (instance == null) {               // first check (no lock)
						synchronized (Singleton.class) {
							if (instance == null) {       // second check (with lock)
								instance = new Singleton();
							}
						}
					}
					return instance;
				}
			}

			🔥 Why volatile is required
				Without volatile, JVM may reorder: Allocate memory, Assign reference, Initialize object
				Another thread may see a half-constructed object.
			
			🎯 Interview line: “Double-checked locking is broken without volatile.”
			✅ Pros: Lazy initialization, High performance, Thread-safe
			❌ Cons: Complex, Easy to get wrong

2️⃣ Option 4: Eager Initialization (Simple & Safe)
		class Singleton {
			private static final Singleton INSTANCE = new Singleton();

			public static Singleton getInstance() {
				return INSTANCE;
			}
		}

		✅ Pros: Thread-safe (class loading guarantees),Simple, No synchronization
		❌ Cons: Instance created even if never used
		🎯 Interview line: “Class loading guarantees thread safety.”

3️⃣ ⭐ BEST OPTION: Initialization-on-Demand Holder Idiom: 👉 Most recommended for interviews
		
		class Singleton {

			private Singleton() {}

			private static class Holder {
				private static final Singleton INSTANCE = new Singleton();
			}

			public static Singleton getInstance() {
				return Holder.INSTANCE;
			}
		}
		
		🔥 Why this is best: Lazy initialization, No synchronization, Thread-safe, Clean & elegant
		How? JVM loads Holder only when accessed, Class loading is thread-safe
		🎯 Interview gold line: “This uses JVM class-loading guarantees.”

4⃣ ⭐ BEST & SIMPLEST: Enum Singleton (Effective Java)
		enum Singleton {
			INSTANCE;
		}

		✅ Pros: Thread-safe, Prevents reflection attack, Prevents serialization issues, Simplest
		❌ Cons: Not flexible (no lazy params), Some teams avoid enums
		🎯 Interview line: “Enum singleton is the safest singleton.”

				protected Object readResolve() {
