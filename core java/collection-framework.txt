🔹 Q1. What is the Java Collection Framework? Why was it introduced?
=============================================================================================================
✅ The Java Collection Framework is a unified architecture in Java that provides interfaces, implementations, and algorithms to store and manipulate groups of objects.
It is part of the java.util package and includes:
	Interfaces → List, Set, Queue, Map
	Implementations → ArrayList, HashSet, HashMap, TreeMap, etc.
	Utility Algorithms → sorting, searching, synchronization (via Collections class)

✅ Why Was Java Collection Framework Introduced?
	Before Java Collection Framework (pre-Java 1.2), Java had: Vector, Stack, Hashtable, Enumeration
	These were: Inconsistent, Not part of a unified hierarchy, Mostly synchronized (performance overhead), Lacked common interfaces
	
	🎯 Key Reasons for Introducing Java Collection Framework
		1️⃣ Standardization: 
			Provide a common set of interfaces so developers can code against interfaces instead of implementations.
			Example:
					List<String> list = new ArrayList<>();
			This allows flexibility to switch to:
					List<String> list = new LinkedList<>();
		
		2️⃣ Reusability & Interoperability
			All collections follow common interfaces → algorithms work across implementations.
			Example:
					Collections.sort(list);
			Works for any List implementation.

		3️⃣ Performance Improvements
			Non-synchronized alternatives (e.g., ArrayList instead of Vector)
			Better data structures (e.g., HashMap, TreeMap)
	
		4️⃣ Algorithm Abstraction
			The Collections utility class provides: Sorting, Searching, Reversing, Synchronization wrappers
			Example:
					Collections.reverse(list);
		
		5️⃣ Cleaner Design (OOP Principles): Interface-based design, Polymorphism, Extensibility, Separation of concerns

🔹 Q3. Explain the Collection hierarchy in detail.
=============================================================================================================
1️⃣ Root: Iterable
			Iterable
			   └── Collection

		🔹 Why Iterable? Introduced to support for-each loop
			Defines: Iterator<T> iterator(); 
					Without Iterable, this would not work: for(String s : list) { }

2️⃣ Collection Interface: This is the root interface for all collections except Map.
	Core methods: add(), remove(), size(), contains(), iterator()
	It represents a group of objects.

3️⃣ Subinterfaces of Collection
	Collection
	 ├── List
	 ├── Set
	 └── Queue

	🔹 A) List Interface: 
			Characteristics: Ordered, Allows duplicates, Allows null, Index-based access
			Implementations:
				| Class      | Internal Structure | Use Case               |
				| ---------- | ------------------ | ---------------------- |
				| ArrayList  | Dynamic array      | Fast read              |
				| LinkedList | Doubly linked list | Frequent insert/delete |
				| Vector     | Synchronized array | Legacy                 |
				| Stack      | Extends Vector     | LIFO (legacy)          |

	🔹 B) Set Interface
			Characteristics: No duplicates, No index-based access
			Implementations:
				| Class         | Internal Structure | Order           |
				| ------------- | ------------------ | --------------- |
				| HashSet       | Hash table         | No order        |
				| LinkedHashSet | Hash + Linked List | Insertion order |
				| TreeSet       | Red-Black Tree     | Sorted          |

	🔹 C) Queue Interface
			Characteristics: FIFO structure, Used for processing order
			Implementations:
				| Class         | Internal Structure | Use Case          |
				| ------------- | ------------------ | ----------------- |
				| PriorityQueue | Heap               | Priority-based    |
				| ArrayDeque    | Resizable array    | Faster than Stack |
				| LinkedList    | Doubly linked list | Queue/Deque       |

4️⃣ Map Hierarchy (Separate from Collection)
	⚠ Map does NOT extend Collection. Because Map stores key-value pairs, not individual elements.
			Map
			 ├── HashMap
			 ├── LinkedHashMap
			 ├── TreeMap
			 ├── Hashtable (Legacy)
			 ├── ConcurrentHashMap
			 ├── WeakHashMap
			 └── EnumMap

	Key Differences:
		| Map Type          | Ordering         | Thread-safe | Internal Structure |
		| ----------------- | ---------------- | ----------- | ------------------ |
		| HashMap           | No               | No          | Hash table         |
		| LinkedHashMap     | Insertion/Access | No          | Hash + Linked      |
		| TreeMap           | Sorted           | No          | Red-Black Tree     |
		| ConcurrentHashMap | No               | Yes         | CAS + Buckets      |
		| Hashtable         | No               | Yes         | Legacy             |

5️⃣ Abstract Classes (Important for Senior Devs)
	Between interfaces and implementations, Java provides skeletal implementations:
		AbstractCollection
		   ├── AbstractList
		   ├── AbstractSet
		   ├── AbstractQueue
		   └── AbstractMap
		   
		Why? To reduce boilerplate when creating custom collections.
		Example: If you implement List, better extend AbstractList.

6️⃣ Complete Structure in Tree Form
		Iterable
		 └── Collection
			  ├── List
			  │     ├── ArrayList
			  │     ├── LinkedList
			  │     ├── Vector
			  │     └── Stack
			  │
			  ├── Set
			  │     ├── HashSet
			  │     ├── LinkedHashSet
			  │     └── TreeSet
			  │
			  └── Queue

					├── ArrayDeque
					└── LinkedList

		Map (Separate)
		 ├── HashMap

🔹 Q5. Difference between Iterable and Iterator?
=============================================================================================================
🔥 Difference Between Iterable and Iterator
	| Feature          | `Iterable`                                   | `Iterator`                        |
	| ---------------- | -------------------------------------------- | --------------------------------- |
	| Type             | Interface                                    | Interface                         |
	| Package          | `java.lang`                                  | `java.util`                       |
	| Purpose          | Represents a collection that can be iterated | Used to iterate over elements     |
	| Key Method       | `iterator()`                                 | `hasNext()`, `next()`, `remove()` |
	| For-each support | Yes                                          | No                                |
	| Role             | Provides iterator                            | Traverses elements                |

1️⃣ Iterable – The Source of Iteration
	Iterable represents something that can be iterated.
	It has only one abstract method:
			Iterator<T> iterator();
		Why was it introduced? To support enhanced for-each loop (Java 5).
		Example:
			List<String> list = new ArrayList<>();
			for (String s : list) {
				System.out.println(s);
			}
		Internally, this works because List implements Iterable.
		How For-Each Works Internally
				Iterator<String> it = list.iterator();
				while(it.hasNext()) {
					String s = it.next();
				}

2️⃣ Iterator – The Traversal Mechanism
	Iterator is used to actually traverse the collection.
	Key methods:
			boolean hasNext();
			E next();
			void remove();

		Example:
			Iterator<String> it = list.iterator();
			while (it.hasNext()) {
				System.out.println(it.next());
			}

🎯 Core Concept Difference
	Iterable = Collection side
	Iterator = Traversal side
	Think of it like:
		Iterable → "I can provide something to iterate"
		Iterator → "I am doing the iteration"

🔹 Q6. Why does Collection extend Iterable?
=============================================================================================================
🔥 Collection extends Iterable so that all collection types can be used in enhanced for-each loops and provide a standard way to traverse elements.
	📌 Design Reason
		Iterable defines only one method:Iterator<T> iterator();
		By extending Iterable, every collection automatically guarantees: 
			It can provide an Iterator, It supports for-each loop, It follows a consistent traversal mechanism

	🔍 Without Iterable, This Would Not Work
			List<String> list = new ArrayList<>();
				for (String s : list) {
					System.out.println(s);
				}
			For-each works only because List → Collection → Iterable.

🔄 Internal Working of For-Each
	This: for(String s : list)
	Is converted by compiler into:
		Iterator<String> it = list.iterator();
		while(it.hasNext()) {
			String s = it.next();
		}
		So without Iterable, the compiler would not know how to get an iterator.

📌 Hierarchy Relationship
Iterable
   └── Collection
          ├── List
          ├── Set
          └── Queue

🎯 Why Not Put iterator() Directly in Collection?
	Because: Iterable is a more general contract, Any class (even non-collection) can be iterable
	Example:
			class Range implements Iterable<Integer> {
			   ...
			}
	So Iterable separates: Traversal capability, From collection behavior, This follows Single Responsibility Principle.

🔹 Q7. What are the benefits of using generics with collections?
=============================================================================================================
🔥 Generics were introduced in Java 5 to provide type safety at compile time.
	Before generics, collections stored Object type only.
		1️⃣ Type Safety (Compile-Time Checking)
			❌ Before Generics
				List list = new ArrayList();
				list.add("Java");
				list.add(10);   // Allowed
				
				String s = (String) list.get(1);  // Runtime ClassCastException
				Error happens at runtime.

			✅ With Generics
				List<String> list = new ArrayList<>();
				list.add("Java");
				list.add(10);  // Compile-time error
				Error caught at compile time. 👉 This is the biggest benefit.

		2️⃣ No Explicit Type Casting
			Without generics: String s = (String) list.get(0);
			With generics: String s = list.get(0);
			Cleaner, safer code.
		
		3️⃣ Improved Code Readability
				Map<String, Integer> map = new HashMap<>();
			
				Immediately clear: Key → String, Value → Integer
				without generics, intent is unclear.

		4️⃣ Eliminates ClassCastException (Mostly)
			Generics shift errors from:	❌ Runtime
			✅ To Compile-time This makes systems more reliable.

		5️⃣ Enables Generic Algorithms
			Example:
					public <T> void printList(List<T> list) {
						for (T item : list) {
							System.out.println(item);
						}
					}
			Reusable for any type.

		6️⃣ Better API Design
			Collections API heavily uses generics: List<E>, Set<E>, Map<K, V>, Comparator<T>
			This makes framework flexible and type-safe.

		7️⃣ Supports Bounded Types
			Example:
				public <T extends Number> void process(List<T> list)
			Ensures only numeric types are allowed.

📌 Internal Note (Very Important for 8+ Years)
	Generics in Java are implemented using Type Erasure.
		Meaning: Type information is removed at runtime, Generics exist only at compile-time
		Example:
			List<String>
			List<Integer>
		At runtime both become: List

🔹 Q8. Can you explain fail-fast vs fail-safe iterators?
=============================================================================================================
1️⃣ Fail-Fast Iterator
A fail-fast iterator throws ConcurrentModificationException if the collection is structurally modified after the iterator is created (except through iterator’s own remove()).
	Example:
		List<String> list = new ArrayList<>();
		list.add("A");
		list.add("B");
		
		Iterator<String> it = list.iterator();
		while (it.hasNext()) {
			list.add("C");  // Structural modification
			System.out.println(it.next()); // Throws exception
		}

	🔍 How Fail-Fast Works Internally
		Collections like: ArrayList, HashMap, HashSet maintain an internal variable called: modCount
		When: Element is added, Element is removed, Structure changes, modCount increments.
		When iterator is created: It stores current modCount as expectedModCount.
		During iteration:
			if (modCount != expectedModCount)
				throw ConcurrentModificationException;

		📌 Key Points About Fail-Fast: 
		Works on best-effort basis, Not guaranteed (not a synchronization mechanism), Detects bugs early, Not thread-safe

2️⃣ Fail-Safe Iterator
Fail-safe iterators do NOT throw exception when collection is modified.
They operate on a clone (snapshot) of the collection.
	Example:
		CopyOnWriteArrayList<String> list = new CopyOnWriteArrayList<>();
		list.add("A");
		list.add("B");
		
		Iterator<String> it = list.iterator();
		list.add("C");  // No exception
		No ConcurrentModificationException.

	🔍 How Fail-Safe Works
		Collections like: CopyOnWriteArrayList, ConcurrentHashMap create a separate copy for iteration.
		Changes go to: Original collection, Not the snapshot

⚠ Important Clarification
	Fail-safe is not an official term in Java documentation. It’s commonly used in interviews.
	Official docs say: “The iterators are fail-fast.”

🔹 Q9. What is UnsupportedOperationException in collections?
=============================================================================================================
🔥 UnsupportedOperationException is a runtime exception thrown when a collection does not support a particular operation.
	It belongs to: java.lang.UnsupportedOperationException
	It usually occurs when you try to: Add, Remove, Modify
	Elements in a collection that is fixed-size or unmodifiable.

	📌 Common Scenarios
		1️⃣ Using Arrays.asList()
			List<String> list = Arrays.asList("A", "B", "C");
			list.add("D");   // ❌ UnsupportedOperationException

			Why? Arrays.asList() returns a fixed-size list backed by the array.
			You can: list.set(0, "X");   // ✅ Allowed
			But cannot:
				list.add("D");      // ❌ Not allowed
				list.remove(0);     // ❌ Not allowed

		2️⃣ Using List.of() (Java 9+)
			List<String> list = List.of("A", "B");
			list.add("C");   // ❌ UnsupportedOperationException

			This is a completely immutable list.

		3️⃣ Using Collections.unmodifiableList()
			List<String> original = new ArrayList<>();
			original.add("A");
			List<String> unmodifiable = Collections.unmodifiableList(original);
			unmodifiable.add("B");  // ❌ UnsupportedOperationException
			
			But: original.add("B");  // ✅ Modifies underlying list
			This is unmodifiable, not immutable.

📌 Why Does This Exception Exist?
	To enforce: Immutability, Fixed-size behavior, Read-only views, API safety
	It prevents structural modification when the collection design does not allow it.
	
📊 Types of Collections That Throw It
| Collection Type      | Supports add/remove? |
| -------------------- | -------------------- |
| ArrayList            | Yes                  |
| Arrays.asList()      | No (fixed size)      |
| List.of()            | No (immutable)       |
| Unmodifiable wrapper | No                   |
| CopyOnWriteArrayList | Yes                  |

📌 Internal Design Reason
	Java follows interface-based design.
	Example: 
		List interface defines:
			boolean add(E e);
		But implementation is free to decide: Support it Or throw UnsupportedOperationException
		This allows: Read-only collections, Partial implementations, Optimized immutable collections

📌 Important Interview Clarification
	UnsupportedOperationException is:
		❌ NOT checked exception, ❌ NOT compile-time error
		✅ Runtime exception

🔹 Q10. How does equals() and hashCode() affect collections?
=============================================================================================================
🔥 They directly impact hash-based collections, mainly:
	HashMap, HashSet, LinkedHashMap, ConcurrentHashMap
	
	📌 Why Are They Important?
		Hash-based collections use:
			1️⃣ hashCode() → To decide bucket
			2️⃣ equals() → To compare objects within bucket

	🔍 Internal Working (HashMap Example)
		When you do: map.put(key, value);
			Step 1 → hashCode() is called, It determines bucket index.
			Step 2 → If collision occurs equals() is used to check if key already exists.

	📌 The Contract (Very Important)
		Java defines strict contract:
			1️⃣ If two objects are equal using equals(): 👉 They MUST have same hashCode()
			2️⃣ If two objects have same hashCode(): 👉 They MAY or MAY NOT be equal

	🔥 What Happens If You Violate Contract?
		❌ Case 1: Override equals() but NOT hashCode()
				class Person {
				   String name;

				   public boolean equals(Object o) {
					   return true;
				   }
				}
			Result: HashSet may store duplicates, HashMap lookup fails, Behavior becomes inconsistent

		❌ Case 2: Mutable Keys in HashMap
				class Person {
				   String name;
				}

			If used as key:
				map.put(person, "data");
				person.name = "changed";
				map.get(person);  // ❌ May return null
				
				Because: hashCode changed, Bucket changed, Object becomes unreachable
		
		📌 HashSet Behavior
			HashSet internally uses: HashMap<E, Object>
				So: Duplicate prevention depends on equals() + hashCode()
				If incorrectly implemented → duplicates allowed.
		
		📌 Performance Impact
			Poor hashCode() implementation like:
				public int hashCode() {
				   return 1;
				}
			All elements go into same bucket →
			HashMap becomes linked list →
			Performance drops from: O(1) → O(n)
		
		📌 Where equals() Alone Is Used?
			In: List.contains(), List.remove(), TreeSet (if comparator not used)
			These do not use hashCode().

🔹 Q11. what is ArrayList,it default capacity,How does resizing happen, how does ArrayList internally work and
What happens internally when ArrayList grows?
=============================================================================================================
🔥 ArrayList is a resizable array implementation of the List interface.
	It provides: Ordered collection, Allows duplicates, Allows null, Fast random access (O(1))
	It belongs to java.util package and extends AbstractList.

🔥 2️⃣ Default Capacity of ArrayList
🔹 Before Java 8: Default capacity = 10
🔹 Java 8+ Important change:
	When you do: new ArrayList<>();
	It does NOT immediately create array of size 10.
	Instead: It creates an empty array (DEFAULTCAPACITY_EMPTY_ELEMENTDATA)
	Capacity becomes 10 only when first element is added

🔥 3️⃣ Internal Structure of ArrayList
	Internally it maintains: 
			transient Object[] elementData;
			private int size;
		elementData → Actual array storing elements
		size → Number of elements currently stored

🔥 4️⃣ How Does ArrayList Internally Work?
		When you do: list.add("A");
			Step 1 → Check Capacity
					If size < elementData.length → Insert directly.
			Step 2 → If Full → Resize
			
🔥 5️⃣ What Happens When ArrayList Grows?
		When capacity is full:
			1️⃣ New larger array is created
			2️⃣ Old elements copied to new array
			3️⃣ Reference updated
			🔹 Growth Formula (Java 8+)
				New capacity = oldCapacity + (oldCapacity >> 1)
				Which means: oldCapacity * 1.5
				
			🔹 Internal Method Used
				Arrays.copyOf(elementData, newCapacity);
				This internally uses: System.arraycopy()
				Which is very fast (native method).

🔥 6️⃣ Time Complexity
| Operation         | Complexity     |
| ----------------- | -------------- |
| get(index)        | O(1)           |
| add()             | O(1) amortized |
| add() when resize | O(n)           |
| remove(index)     | O(n)           |
Why amortized O(1)? Because resize doesn’t happen every time.
	
🔥 7️⃣ What Happens Internally During Resize (Step-by-Step)
		Suppose capacity = 10 and size = 10
		When adding 11th element:
			1️⃣ ensureCapacityInternal() called
			2️⃣ grow() method triggered
			3️⃣ New array created (capacity 15)
			4️⃣ System.arraycopy() copies 10 elements
			5️⃣ elementData updated
			6️⃣ New element inserted

🔥 8️⃣ Memory Consideration
		ArrayList may waste unused memory: If capacity = 100 But size = 10, Remaining 90 slots unused.
		You can reduce using: list.trimToSize();

🔥 9️⃣ Important Interview Edge Cases
	❓ What if we know size in advance?
		Use: new ArrayList<>(1000); Improves performance by avoiding resizing.
	❓ Why ArrayList is not thread-safe?
		Because multiple threads modifying elementData without synchronization can corrupt state.
	❓ What if initial capacity is negative? Throws IllegalArgumentException.

🔹 Q12. Why ArrayList allows duplicate and null values? Difference between ensureCapacity() and trimToSize()?
=============================================================================================================
📌 A) Why Duplicates Are Allowed?
	Because ArrayList implements the List interface.
		List contract says: Ordered collection, Allows duplicate elements, Unlike Set, which enforces uniqueness
		List is designed to preserve: Insertion order, Positional access (index-based), Repeated values
		Internally, ArrayList is just a dynamic array: transient Object[] elementData;
		It does NOT check: hashCode(), equals(), uniqueness
		So:
			list.add("A");
			list.add("A");  // Allowed
			No validation is performed.

📌 B) Why Null Is Allowed?
	Because: ArrayList stores elements in Object[], null is a valid reference in Java
	There is no restriction in List contract against null.
		list.add(null);  // Allowed
	Internally it just stores: elementData[index] = null;

📌 Why Some Collections Do NOT Allow Null?
	| Collection        | Null Allowed? | Reason                         |
	| ----------------- | ------------- | ------------------------------ |
	| HashMap           | One null key  | Design decision                |
	| TreeMap           | No null key   | Uses compareTo()               |
	| ConcurrentHashMap | No null       | Avoid ambiguity in concurrency |
	| ArrayList         | Yes           | No restriction                 |

2️⃣ Difference Between ensureCapacity() and trimToSize()
	These are capacity management methods in ArrayList.
		📌 A) ensureCapacity(int minCapacity) 
			Purpose: Increases internal array capacity if needed.
			Use Case: When you know in advance how many elements will be added.
			Example:
				ArrayList<String> list = new ArrayList<>();
				list.ensureCapacity(1000);

			This: Avoids multiple resizes, Improves performance
			Internally:
				If: minCapacity > currentCapacity, Then grow() is triggered. Otherwise, nothing happens.

		📌 B) trimToSize()
			Purpose: Reduces internal capacity to match current size.
			Example: list.trimToSize();
				If: Capacity = 100 Size = 10
				After trim: Capacity becomes 10

❓ When would you use ensureCapacity()?
	When: Loading large dataset from DB, Parsing big file, Bulk insert known size
	Example:
			List<String> data = new ArrayList<>(100000);
		Or:
			list.ensureCapacity(100000);

❓ When would you use trimToSize()?
	When: After removing many elements, Before caching list, Before serializing list, To reduce memory footprint

🔹 Q14. Why ArrayList is not thread-safe? When should you NOT use ArrayList?
=============================================================================================================
📌 Because ArrayList does not use synchronization, so multiple threads modifying it simultaneously can cause:
	Data corruption, Lost updates, Inconsistent state, ArrayIndexOutOfBoundsException

	🔍 Internal Reason
		Internally, ArrayList has:
			transient Object[] elementData;
			private int size;
		
		When you call: list.add("A");
		Internally:
			1️⃣ Check capacity
			2️⃣ Insert element
			3️⃣ Increment size
			None of these operations are synchronized.

	📌 Race Condition Example
		Suppose two threads run:
			list.add("A");
			list.add("B");
		Possible issue: Both read same size, Both write to same index, One value gets overwritten, size becomes inconsistent

🔥 What Happens During Resize in Multithreading?
	If two threads trigger resize simultaneously:
		Both create new arrays, Copy happens inconsistently, elementData reference may be corrupted
		You may get: ArrayIndexOutOfBoundsException or NullPointerException

🔥 Why Vector Is Thread-Safe?
	Because Vector methods are: public synchronized boolean add(E e)
	But synchronization causes: Performance overhead, Lock contention
	That’s why Vector is considered legacy.

🔥 How to Make ArrayList Thread-Safe?
	Option 1: Collections.synchronizedList(new ArrayList<>());
	Option 2 (Better for read-heavy): CopyOnWriteArrayList<>();

🔥 When Should You NOT Use ArrayList?
	🚫 1️⃣ In Multi-Threaded Environment (Without Synchronization)
		Use: CopyOnWriteArrayList, Concurrent collections
	
	🚫 2️⃣ Frequent Insert/Delete at Beginning or Middle
		ArrayList shifts elements: O(n)
		Use: LinkedList

	🚫 3️⃣ When You Need Sorted Structure
		Use: TreeSet, TreeMap
	
	🚫 4️⃣ When You Need Constant-Time Insert/Delete
		Use: LinkedList (for frequent structural modification)

	🚫 5️⃣ When Memory Is Highly Constrained
		ArrayList: Keeps extra unused capacity, Can waste memory

	🚫 6️⃣ Real-Time Systems (Where Resize Is Risky)
		Resize operation: O(n), Can cause latency spikes

🔹 Q15. What is LinkedList and Internal structure of LinkedList?
=============================================================================================================
🔥 LinkedList is a doubly linked list implementation of: List, Deque, Queue, It belongs to java.util package.
	It allows: Duplicate elements, Null values, Insertion at beginning/middle/end
	Unlike ArrayList, it does NOT use a dynamic array.

	🔥 Internal Structure of LinkedList
		Internally, LinkedList is implemented as a doubly linked list. Each element is stored in a Node object.

	📌 Internal Node Class (Simplified)
				private static class Node<E> {
					E item;
					Node<E> next;
					Node<E> prev;
				}
				Each node contains:
					item → actual data
					next → pointer to next node
					prev → pointer to previous node

	📌 LinkedList Internal Fields
			transient Node<E> first;
			transient Node<E> last;
			private int size;
			
			first → head pointer
			last → tail pointer
			size → number of elements
			
🔥 How LinkedList Works Internally
	🔹 Adding Element at End : list.add("A");
		Internally:
			1️⃣ Create new node
			2️⃣ Set newNode.prev = last
			3️⃣ last.next = newNode
			4️⃣ Update last pointer
			5️⃣ Increment size
			Time Complexity → O(1)

	🔹 Adding at Beginning: list.addFirst("A");
			1️⃣ Create new node
			2️⃣ newNode.next = first
			3️⃣ first.prev = newNode
			4️⃣ Update first pointer
			Time Complexity → O(1)

	🔹 Adding in Middle
			1️⃣ Traverse to index → O(n)
			2️⃣ Update previous & next links
			Time Complexity → O(n)

	🔹 Removing Element
		Removal does: 
			prev.next = next;
			next.prev = prev;
			No shifting required.

🔥 Why Random Access Is Slow?
	If you do: list.get(500);
	LinkedList: Traverses node by node, From head or tail (whichever is closer)	Time Complexity → O(n)
	While ArrayList → O(1)

🔥 Time Complexity Comparison
	| Operation       | LinkedList | ArrayList      |
	| --------------- | ---------- | -------------- |
	| get(index)      | O(n)       | O(1)           |
	| add() end       | O(1)       | O(1) amortized |
	| add() middle    | O(n)       | O(n)           |
	| remove() middle | O(n)       | O(n)           |
	| insert at start | O(1)       | O(n)           |

🔥 Memory Overhead: 
	LinkedList consumes more memory because:
		Each node stores: Data, Next pointer, Prev pointer, So memory overhead is higher than ArrayList.

🔥 When to Use LinkedList?
	✅ Frequent insert/delete at beginning
	✅ Implementing stack/queue
	✅ Deque operations
	🚫 Not good for: Random access, Heavy read scenarios

🔹 Q17. Why LinkedList is preferred for frequent insert/delete?
=============================================================================================================
🔥 Why LinkedList Is Preferred for Frequent Insert/Delete?
	📌 Core Reason: Because LinkedList does not require shifting elements.
		Unlike ArrayList, which shifts elements during insert/delete, LinkedList just updates node references.

	🔹 In ArrayList (Insert in Middle)
		Example: list.add(2, "X");
		Internally:
			1️⃣ All elements from index 2 onward are shifted right
			2️⃣ System.arraycopy() used
			3️⃣ Time complexity → O(n)

	🔹 In LinkedList (Insert in Middle)
		Internally:
			1️⃣ Traverse to index (O(n))
			2️⃣ Update pointers:
				prev.next = newNode;
				newNode.prev = prev;
				newNode.next = next;
				next.prev = newNode;
				No shifting.

🔥 But Here Is the Senior-Level Truth
	Most people stop here. But interviewer expects more 👇
	⚠ Important Clarification; LinkedList insertion is O(1) only if you already have node reference.
	If you insert by index: list.add(5000, "X");
	LinkedList must: Traverse to index → O(n), Then update links → O(1), Total → O(n)
	Same complexity as ArrayList (but without shifting cost).

🔥 When LinkedList Truly Shines
	✅ Frequent insert/remove at head: addFirst(), removeFirst(), Time → O(1)
	✅ Implementing Queue / Deque
		Because: Head/tail operations are constant time.
	
🔥 Real-World Insight
	In practice: ArrayList often outperforms LinkedList, Because CPU cache favors contiguous memory
	LinkedList has pointer chasing overhead So LinkedList is preferred only when:
		Heavy structural modification at ends, Not random access, Not heavy iteration

📊 Performance Reality Table

| ---------------- | --------- | ---------- |

🔹 Q22. What is Hashset and How does HashSet internally work? How HashSet ensures uniqueness? Why HashSet allows only one null?
=============================================================================================================
🔥 HashSet is an implementation of the Set interface that Does NOT allow duplicates, Does NOT maintain insertion order
Allows one null, Provides average O(1) time complexity and It belongs to java.util package.

🔥 Internal Implementation (Very Important)
	👉 HashSet is internally backed by a HashMap.
		Simplified source code idea: 
				private transient HashMap<E, Object> map;
				private static final Object PRESENT = new Object();
			
			When you do: set.add("A");
			Internally:
				map.put("A", PRESENT);
			So HashSet stores:
				Element → as key
				Dummy object → as value

🔥 How HashSet Internally Works
	When you call: set.add(element);
		Step 1️⃣ → Call hashCode(), Determines bucket index.
		Step 2️⃣ → Go to bucket, Index calculated as: hash & (n - 1), Where n = table size.
		Step 3️⃣ → Check Collision, If bucket empty → insert, If not → compare using equals()
		Step 4️⃣ → If equal found Do NOT insert (duplicate prevented)
		Step 5️⃣ → If not equal, Add to bucket (linked list or tree)

🔥 Java 8 Improvement
	If bucket size > 8 and table size ≥ 64:
		Linked list converts into: 👉 Red-Black Tree
		Improves worst-case performance from: O(n) → O(log n)

🔥 How HashSet Ensures Uniqueness?
	Uniqueness depends on: 	1️⃣ hashCode(), 2️⃣ equals()
		Two elements are considered duplicates if:
			existing.equals(newElement) == true, they land in same bucket (same hashCode).
			If equals() returns true: 👉 Insertion rejected.

⚠ Important Interview Trap
	If you override equals() but NOT hashCode(): 👉 HashSet may allow duplicates.
	Because: Different hashCode → different bucket, equals() never compared

🔥 Why HashSet Allows Only One Null?
	Because: HashSet is backed by HashMap, HashMap allows only one null key
	Internally:
		hash(null) = 0
		All null values go to bucket 0.
		When second null is added: equals(null, null) → true
		So duplicate → rejected.

📌 What Happens Internally When Adding null?
	set.add(null);
		Internally: map.put(null, PRESENT);
		HashMap allows one null key → so HashSet allows one null element.

📊 Time Complexity
| Operation  | Average | Worst Case |
| ---------- | ------- | ---------- |
| add()      | O(1)    | O(log n)   |
| remove()   | O(1)    | O(log n)   |
| contains() | O(1)    | O(log n)   |

🔹 Q23. How equals() and hashCode() work together? What happens if hashCode changes after insertion?
=============================================================================================================
🔥 They work together in hash-based collections like: HashMap, HashSet, LinkedHashMap, ConcurrentHashMap
	📌 Step-by-Step Flow (HashMap Example)
		When you do: map.put(key, value);
		1️⃣ hashCode() is called: Used to determine the bucket index.
			int hash = key.hashCode();
			int index = hash & (table.length - 1);
		
		2️⃣ Go to that bucket
			If bucket empty → insert directly.
			If bucket contains elements → collision handling begins.

		3️⃣ equals() is used
			Inside the bucket: if (existingKey.equals(newKey))
				If true → value replaced.
				If false → add to bucket (linked list or tree).

📌 The Contract (Very Important)
	Java defines:
		Rule 1️⃣ 	If two objects are equal (equals() returns true), 👉 they MUST have same hashCode().
		Rule 2️⃣  If two objects have same hashCode(), 👉 they MAY or MAY NOT be equal.

🔥 Why Both Are Needed?
	hashCode() → Fast bucket lookup
	equals() → Exact object comparison
	Without hashCode: All elements go to same bucket → O(n)
	Without equals: Cannot verify actual equality inside bucket.

🔥 What Happens If hashCode() Changes After Insertion?
		Consider:
				class Person {
				   String name;

				   public int hashCode() {
					   return name.hashCode();
				   }

				   public boolean equals(Object o) {
					   return this.name.equals(((Person)o).name);
				   }
				}

			Now:
				Person p = new Person("John");
				map.put(p, "Data");
			
			Then: p.name = "David";   // hashCode changes
			Now: map.get(p);   // ❌ Returns null

		📌 Why Does This Happen?
			Because:
				1️⃣ When inserted, bucket calculated using old hashCode.
				2️⃣ After mutation, new hashCode points to different bucket.
				3️⃣ HashMap searches wrong bucket.
				4️⃣ Object becomes unreachable.

🔥 Consequences of Changing hashCode After Insert
	Key becomes unreachable: Memory leak, containsKey() returns false, remove() fails, Data inconsistency
	Very dangerous in production systems.
	🔥 Golden Rule
		👉 Keys in HashMap should be immutable.
		That’s why: String is immutable, Integer is immutable
		Most recommended keys are immutable

🔹 Q24. What is  LinkedHashSet.How LinkedHashSet maintains insertion order? 
=============================================================================================================
🔥 LinkedHashSet is a Set implementation that:
		Does NOT allow duplicates, Maintains insertion order, Allows one null, Provides near O(1) performance
		It belongs to java.util package.

📌 Key Difference from HashSet
| Feature            | HashSet         | LinkedHashSet             |
| ------------------ | --------------- | ------------------------- |
| Order              | No order        | Maintains insertion order |
| Internal Structure | HashMap         | LinkedHashMap             |
| Performance        | Slightly faster | Slightly slower           |

🔥 Internal Working of LinkedHashSet
	👉 LinkedHashSet is backed internally by a LinkedHashMap, not just HashMap.
		Simplified internal code idea:
				public class LinkedHashSet<E> extends HashSet<E> {

					public LinkedHashSet() {
						super(16, .75f, true);
					}
				}

		It uses: LinkedHashMap<E, Object>
		Instead of: HashMap<E, Object>

🔥 How LinkedHashSet Maintains Insertion Order?
	The magic happens in LinkedHashMap.
	LinkedHashMap maintains:
		1️⃣ A hash table (like HashMap)
		2️⃣ A doubly linked list running through all entries
	Each entry stores: before, after references.

🔥 Internal Entry Structure (Conceptually)
		In LinkedHashMap, each entry has:
			static class Entry<K,V> extends HashMap.Node<K,V> {
					Entry<K,V> before, after;
			}

		So besides bucket links, it maintains a linked list connecting all elements in insertion order.
	
🔥 Insertion Flow: When you do:
		set.add("A");
		set.add("B");
		set.add("C");
		
		Internally:
			1️⃣ Insert into hash table (like HashSet)
			2️⃣ Append entry at end of doubly linked list

	Iteration follows: head → A → B → C → tail

🔥 Performance Characteristics
| Operation  | Complexity |
| ---------- | ---------- |
| add()      | O(1)       |
| remove()   | O(1)       |
| contains() | O(1)       |
| iteration  | O(n)       |
Slightly slower than HashSet due to maintaining linked list.

🔥 Memory Overhead
	LinkedHashSet uses more memory because:
		Each entry stores: hash, key, next (bucket link), before, after
		So extra 2 references compared to HashSet.

🔥 Important Interview Insight
	LinkedHashSet maintains: 👉 Insertion order, NOT sorting order.
	If you need sorted order → use TreeSet.

🔥 Does LinkedHashSet Allow Access Order? No.
		But LinkedHashMap supports: new LinkedHashMap<>(16, 0.75f, true);
		Where: true = access-order (used for LRU cache)
		LinkedHashSet does NOT expose this directly.

🔹 Q27. What is TreeSet, How TreeSet maintains sorting?
=============================================================================================================
🔥 TreeSet is a Set implementation that: Does NOT allow duplicates, Maintains elements in sorted order, Does NOT allow null (in natural ordering case), Provides O(log n) time complexity, It belongs to java.util package.

📌 Internal Implementation
	👉 TreeSet is internally backed by a TreeMap.
		Simplified idea: private transient NavigableMap<E,Object> m;
		When you do: set.add("A");
		Internally: map.put("A", PRESENT);
		Exactly like HashSet uses HashMap, TreeSet uses TreeMap.

🔥 What Data Structure Does TreeMap Use? 👉 Red-Black Tree (self-balancing binary search tree)
🔥 How TreeSet Maintains Sorting?
	Sorting is maintained using:
			1️⃣ Natural Ordering (Comparable)
				If elements implement Comparable:
					class Person implements Comparable<Person> {
					   public int compareTo(Person p) {
						   return this.age - p.age;
					   }
					}
				TreeSet automatically uses compareTo().
			
			2️⃣ Custom Comparator: 
					TreeSet<Person> set = new TreeSet<>(new Comparator<Person>() {
					   public int compare(Person p1, Person p2) {
						   return p1.age - p2.age;
					   }
					});
				TreeSet uses compare() method.

🔥 Internal Working During Insertion
		When you add an element:
			set.add(50);
			set.add(20);
			set.add(70);
		TreeSet:
			1️⃣ Compares with root
			2️⃣ Goes left or right
			3️⃣ Inserts at correct position
			4️⃣ Rebalances tree to maintain red-black properties
			So order is automatically maintained.

🔥 Important Interview Insight
		In TreeSet:
			👉 Duplicate detection is based on: compareTo() == 0, NOT equals()
				If compareTo() returns 0: Element is considered duplicate, Not inserted, Even if equals() returns false.

🔥 Why TreeSet Does NOT Allow Null?
	Because: During insertion, it calls: compareTo()
		If element is null → NullPointerException.
	There is no meaningful way to compare null values in sorting.

🔥 Time Complexity
| Operation  | Complexity |
| ---------- | ---------- |
| add()      | O(log n)   |
| remove()   | O(log n)   |
| contains() | O(log n)   |
Slower than HashSet (O(1)), but maintains sorted order.

🔥 When to Use TreeSet?
		✅ When you need sorted data
		✅ When you need range queries
		✅ When you need first(), last(), higher(), lower()
	Not ideal for: ❌ Fast random lookup, ❌ High-performance hashing scenarios

🔹 Q28. Difference between Comparable and Comparator? What happens if compareTo returns 0?
=============================================================================================================
📌 1️⃣ Comparable: Interface in java.lang, Defines natural ordering, Implemented inside the class itself
		Method: int compareTo(T o);
		Example:
			class Person implements Comparable<Person> {
				int age;

				public int compareTo(Person p) {
					return this.age - p.age;
				}
			}
		Now objects are sorted by default using age.

📌 2️⃣ Comparator: Interface in java.util, Defines custom ordering, Implemented separately from the class
		Method: int compare(T o1, T o2);
		Example:
			Comparator<Person> nameComparator = (p1, p2) -> p1.name.compareTo(p2.name);
		Used like: new TreeSet<>(nameComparator);

🔥 Comparison Table
| Feature                 | Comparable  | Comparator     |
| ----------------------- | ----------- | -------------- |
| Package                 | java.lang   | java.util      |
| Method                  | compareTo() | compare()      |
| Defined in              | Same class  | Separate class |
| Ordering                | Natural     | Custom         |
| Multiple sorting logic  | No          | Yes            |
| Modifies original class | Yes         | No             |

🔥 What Happens If compareTo() Returns 0?
	In sorted collections like: TreeSet, TreeMap
	If: compareTo() == 0, 👉 The two elements are considered equal
	Even if: equals() == false
	🔹 Example
			TreeSet<Integer> set = new TreeSet<>();
			set.add(10);
			set.add(10);
		Second insertion rejected because: compareTo() returns 0

	🔹 Dangerous Case
				class Person implements Comparable<Person> {
					int age;
					String name;

					public int compareTo(Person p) {
						return this.age - p.age;
					}
				}
		Now:
			Person p1 = new Person(25, "John");
			Person p2 = new Person(25, "David");
			set.add(p1);
			set.add(p2);
		👉Second one will NOT be inserted.
		Why? Because: compareTo() returns 0 (same age), Even though names are different.

🔥 Important Interview Insight
	In TreeSet and TreeMap: 👉 Equality is based on comparison logic NOT equals()
		So if compareTo() returns 0: Element treated as duplicate, Not inserted

🔥 Why This Is Dangerous? 
	If compareTo() is inconsistent with equals():
		Violates general contract, Leads to strange behavior
	Java recommends: compareTo should be consistent with equals.

🔹 Q31. How does HashMap internally work (Java 8+)? Difference between HashMap in Java 7 vs Java 8?
=============================================================================================================
1️⃣ Core Data Structure
	HashMap uses: Array of Node<K,V>
		transient Node<K,V>[] table;
		Each bucket contains: Single node OR Linked list OR Red-Black Tree (Java 8+)

2️⃣ What Happens During put(key, value)?
	Step 1️⃣ → Calculate Hash
			int hash = key.hashCode();
			hash = hash ^ (hash >>> 16); This spreads higher bits to reduce collision.
	
	Step 2️⃣ → Find Bucket Index
			index = (n - 1) & hash;
			Where: n = table length (always power of 2)
	
	Step 3️⃣ → Insert Logic
			Case 1: Bucket empty: Insert directly.
			Case 2: Bucket not empty: Check first node:
					if (key.equals(existingKey))
						replace value
					Else: Traverse linked list OR traverse tree (if treeified)
	
	Step 4️⃣ → Treeification (Java 8 Feature)
		If: Bucket size > 8 AND Table size ≥ 64 Linked list converts to: 👉 Red-Black Tree
		Improves worst-case performance.

3️⃣ Resize Mechanism
	Triggered when: size > capacity * loadFactor
	Default: Initial capacity = 16
	Load factor = 0.75
	Resize: Capacity doubles
	Rehashing occurs. Nodes redistributed

🔥 Why Load Factor 0.75?
		Balance between: Memory usage, Performance
		Higher load factor → fewer resizes but more collisions
		Lower load factor → more memory but fewer collisions

🔥 Difference Between HashMap Java 7 vs Java 8
| Feature               | Java 7                         | Java 8                 |
| --------------------- | ------------------------------ | ---------------------- |
| Collision handling    | Linked List only               | Linked List → Tree     |
| Worst-case complexity | O(n)                           | O(log n)               |
| Tree structure        | ❌ No                           | ✅ Yes                  |
| Hash function         | Basic spread                   | Improved bit spreading |
| Resize transfer       | Head insertion (reverse order) | Maintains order        |

🔥 Java 7 Problem
	In Java 7: All collisions stored in linked list, Worst case → O(n)
	Under heavy collision → performance degradation
	In concurrent resize → could cause infinite loop (rare bug)

🔥 Java 8 Improvement
	Java 8 introduced: 👉 Tree bins (Red-Black Tree)
	When collision count > 8: Linked list → converted to tree, Search becomes O(log n)
	When count drops below 6: Tree → converted back to linked list

🔥 Important Internal Classes (Java 8)
		static class Node<K,V>
		static final class TreeNode<K,V> extends LinkedHashMap.Entry<K,V>
		TreeNode represents Red-Black tree nodes.

🔹 Q36. Can HashMap contain mutable keys? Why HashMap is not thread-safe?
=============================================================================================================
1️⃣  Technically yes, but it is strongly discouraged.
	👉 Mutable keys can break HashMap behavior.
	🔍 Why Is It Dangerous?
		HashMap uses:
			1️⃣ hashCode() → to determine bucket
			2️⃣ equals() → to compare keys inside bucket
			If key state changes after insertion and that state affects hashCode() or equals():👉 The key becomes unreachable.

			📌 Example
				class Person {
					String name;

					public int hashCode() {
						return name.hashCode();
					}

					public boolean equals(Object o) {
						return this.name.equals(((Person)o).name);
					}
				}
			Now:
				Person p = new Person();
				p.name = "John";
				map.put(p, "Data");
				p.name = "David";  // Mutation
				map.get(p);  // ❌ returns null

			Step-by-step:
				1️⃣ Insert → bucket determined using hashCode("John")
				2️⃣ After mutation → hashCode("David")
				3️⃣ get() searches wrong bucket
				4️⃣ Key not found
			
			🚨 Consequences: containsKey() returns false, remove() fails, Memory leak (entry still exists but unreachable)
					Data corruption
			
			🔥 Best Practice: 👉 Always use immutable keys in HashMap.
			Examples of good keys: String, Integer, UUID, Custom immutable objects

🔥 2️⃣ Why HashMap Is NOT Thread-Safe?
	Because its internal structure is not synchronized. 
	Multiple threads modifying it concurrently can cause: Race conditions, Data corruption, Infinite loops (Java 7 resize bug), Lost updates
	
	🔍 Internal Reason
		HashMap has: Node<K,V>[] table; int size;
		Operations like put() involve:
				1️⃣ Compute hash
				2️⃣ Find bucket
				3️⃣ Insert node
				4️⃣ Update size
				5️⃣ Possibly resize
		None of these steps are atomic.

	📌 Race Condition Example
		Thread 1: map.put("A", 1);
		Thread 2: map.put("B", 2);
		Possible issues: Both read same size, Both write to same bucket, Overwrite each other, Size becomes inconsistent
	
	📌 Resize Problem (Java 7 Classic Issue)
		During resize: Table doubles, Entries rehashed
		If two threads resize simultaneously: 👉 Bucket links can form a cycle, 👉 Infinite loop during traversal
		This was a serious concurrency bug in Java 7.

🔥 Why Read Operations Also Not Safe?
	Even: map.get(key); 
	is unsafe during concurrent write because: table reference may change, structure may be inconsistent

🔹 Q37. Difference between HashMap and LinkedHashMap? How access-order works in LinkedHashMap?
=============================================================================================================
🔥 Difference Between HashMap and LinkedHashMap
	1️⃣ Internal Data Structure
		🔹 HashMap: 
				Uses hash table
				Each bucket contains: Node, Linked list Or Red-Black Tree (Java 8+)
				No ordering maintained
		
		🔹 LinkedHashMap:
				Extends HashMap
				Uses: Hash table (like HashMap), PLUS a doubly linked list connecting all entries
				This linked list maintains order.

	2️⃣ Feature Comparison
	| Feature      | HashMap            | LinkedHashMap       |
	| ------------ | ------------------ | ------------------- |
	| Order        | No order           | Maintains order     |
	| Order Type   | —                  | Insertion or Access |
	| Performance  | Slightly faster    | Slightly slower     |
	| Memory usage | Lower              | Higher              |
	| Null allowed | 1 key, many values | Same                |

	3️⃣ How LinkedHashMap Maintains Order?
		Each entry is linked in a doubly linked list: head → entry1 → entry2 → entry3 → tail
		When new entry added:
			Insert into bucket (like HashMap)
			Append to tail of linked list
			Iteration follows linked list order.
	
🔥 What is Access-Order in LinkedHashMap?
	By default: new LinkedHashMap<>();
	Maintains: 👉 Insertion Order
	But if you use: new LinkedHashMap<>(16, 0.75f, true);
		The third parameter = accessOrder
	If true: 👉 Maintains Access Order

🔥 How Access-Order Works
	When: map.get(key); That entry moves to the end of the linked list.
	Example:
		Initial insertion: A → B → C
		If you access: map.get("A");
		Order becomes: B → C → A
		Recently accessed moves to tail.

🔥 Why Access-Order Is Important?
	Used for: 👉 LRU Cache (Least Recently Used), Oldest accessed entry is at head.

🔥 LRU Implementation Example
		LinkedHashMap<String, String> lru =  new LinkedHashMap<>(16, 0.75f, true) {
				protected boolean removeEldestEntry(Map.Entry eldest) {
					return size() > 100;
				}
			};
		This automatically removes least recently used entry.

| --------- | ------- | ------------------------ |
| put()     | O(1)    | O(1)                     |
| get()     | O(1)    | O(1)                     |
| remove()  | O(1)    | O(1)                     |
| Iteration | O(n)    | O(n) (predictable order) |

LinkedHashMap slightly slower due to linked list maintenance.

🔹 Q38. How to implement LRU cache using LinkedHashMap? Real-time use cases of LinkedHashMap?
=============================================================================================================
🔥 How to Implement LRU Cache Using LinkedHashMap
	📌 Why LinkedHashMap? 
		Because it supports: new LinkedHashMap<>(capacity, loadFactor, true); The third parameter = accessOrder
		When true: Recently accessed entries move to the end, Least recently used stays at the head
		Perfect for LRU logic.

🔥 Basic LRU Implementation
				import java.util.LinkedHashMap;
				import java.util.Map;

				public class LRUCache<K, V> extends LinkedHashMap<K, V> {

					private final int capacity;

					public LRUCache(int capacity) {
						super(capacity, 0.75f, true);  // accessOrder = true
						this.capacity = capacity;
					}

					@Override
					protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
						return size() > capacity;
					}
					
					public V getValue(K key) {
						return super.get(key);
					}

					public void putValue(K key, V value) {
						super.put(key, value);
					}
				}

	🔥 How It Works Internally
		1️⃣ Insertion: Entry added to hash table, Appended to tail of doubly linked list
		2️⃣ Access (get): Entry moved to tail (most recently used)
		3️⃣ When size exceeds capacity: removeEldestEntry() removes head (least recently used)

	🔥 Example Usage
		LRUCache<String, String> cache = new LRUCache<>(3);
		cache.put("A", "Apple");
		cache.put("B", "Banana");
		cache.put("C", "Cherry");
		cache.get("A");   // A becomes most recently used
		cache.put("D", "Date");  // Removes B (least recently used)

	🔥 Time Complexity
| Operation | Complexity |
| --------- | ---------- |
| get()     | O(1)       |
| put()     | O(1)       |
| remove()  | O(1)       |

⚠ Important: Thread-Safety: LinkedHashMap is NOT thread-safe.
	For multi-threaded usage: Use Collections.synchronizedMap() Or use ConcurrentHashMap + custom LRU logic
	Or use libraries like Caffeine (production-grade)

🔥 Real-Time Use Cases of LinkedHashMap
	1️⃣ LRU Cache: API response caching, DB query caching, Session storage, Recently viewed products
	2️⃣ Preserving Insertion Order
		Example: Configuration loading, Maintaining predictable iteration, JSON parsing order preservation
	
	3️⃣ Access-Order Based Tracking: Tracking recently accessed users, Rate limiting systems, Analytics (top recent events)
	4️⃣ Lightweight In-Memory Cache
		Small microservices where: No distributed caching required, Simple eviction logic sufficient
	
	5️⃣ Framework Usage
		Used internally in: Spring, Hibernate, Various caching mechanisms

🔥 Why Not Always Use LinkedHashMap for Cache?
	Because: ❌ Not thread-safe, ❌ No expiry time support, ❌ No size-based eviction tuning, ❌ No statistics

For production caching: Use Caffeine, Use Redis, Use EHCache

🔹 Q39. How TreeMap internally works? TreeMap vs HashMap vs LinkedHashMap?
=============================================================================================================
🔥 How TreeMap Internally Works
📌 1️⃣ Core Data Structure
		TreeMap is implemented using a: 👉 Red-Black Tree (self-balancing binary search tree)
		Internally:
					static final class Entry<K,V> {
						K key;
						V value;
						Entry<K,V> left;
						Entry<K,V> right;
						Entry<K,V> parent;
						boolean color;  // RED or BLACK
					}

			It maintains: transient Entry<K,V> root;

🔥 2️⃣ How Insertion Works
		When you do: map.put(key, value);
			Step 1️⃣ → Comparison 
				TreeMap uses: compareTo() (if key implements Comparable) OR Provided Comparator
				To determine: Go left (if smaller), Go right (if larger)
			
			Step 2️⃣ → Insert as leaf: Insert at correct BST position.
			Step 3️⃣ → Rebalance Tree: 
				Red-Black Tree rules enforced: 
					Root must be black, No two red nodes in a row
					Equal black height on all paths, Rotations and recoloring are done.

🔥 3️⃣ How Search Works
	Search follows BST traversal: Compare → go left or right
	Time complexity: O(log n)
	
🔥 Why TreeMap Does NOT Use hashCode?
	Because: No buckets, No hashing, Position decided by comparison

🔥 TreeMap vs HashMap vs LinkedHashMap
| Feature         | HashMap    | LinkedHashMap                   | TreeMap        |
| --------------- | ---------- | ------------------------------- | -------------- |
| Data structure  | Hash table | Hash table + Doubly linked list | Red-Black Tree |
| Ordering        | No order   | Insertion / Access order        | Sorted order   |
| Time complexity | O(1) avg   | O(1) avg                        | O(log n)       |
| Null key        | 1 allowed  | 1 allowed                       | ❌ Not allowed  |
| Memory usage    | Lower      | Medium                          | Higher         |

🔥 When to Use Which?
	✅ Use HashMap: Fast lookup required, Order doesn’t matter, High throughput systems
		Example: Caching, Indexing, Frequency counting
	
	✅ Use LinkedHashMap: Need predictable iteration order, Implementing LRU cache, Config loading where order matters
	
	✅ Use TreeMap: Need sorted data, Range queries, firstKey(), lastKey(), ceilingKey(), Leaderboards / ranking systems

🔥 Performance Insight
	HashMap fastest (O(1))
	TreeMap slower (O(log n)) because tree traversal
	LinkedHashMap slightly slower than HashMap due to list maintenance

🔥 What Happens If compareTo() Returns 0 in TreeMap?
	If compareTo() returns 0, the two keys are considered equal in TreeMap.
		👉 The new value replaces the existing value. The key is NOT inserted as a new entry.
	
	📌 Why? : TreeMap is backed by a Red-Black Tree (BST).
		Insertion logic:
			1️⃣ Compare new key with existing node
			2️⃣ If result < 0 → go left
			3️⃣ If result > 0 → go right
			4️⃣ If result == 0 → treat as duplicate key
		So equality in TreeMap is determined by: compareTo() == 0, NOT by equals().

🔹 Q41. How Comparator affects TreeMap behavior?
=============================================================================================================
1️⃣ TreeMap Uses Comparator for Ordering
	When you create a TreeMap: TreeMap<K,V> map = new TreeMap<>(comparator);
	TreeMap will:
		👉 Use comparator.compare(k1, k2), NOT use compareTo(), NOT use hashCode(), NOT use equals()

	Example: TreeMap<Integer, String> map = new TreeMap<>((a, b) -> b - a);  // descending order
	Now: Keys sorted in reverse order
	Comparator completely defines how keys are arranged in tree.

3️⃣ Comparator Defines Key Equality
	In TreeMap: compare(k1, k2) == 0
	👉 Means keys are considered equal, Second insertion replaces first
		Even if: k1.equals(k2) == false
	Example:
		Comparator<String> comp = (a, b) -> a.length() - b.length();

		map.put("AA", "First");
		map.put("BB", "Second");
		
		Both length = 2, Comparator returns 0

	Meaning: compare(a,b) == 0 → a.equals(b) should be true

	Example:
		A > B
		B > C
		C > A   ❌
	
	Tree becomes invalid. Possible outcomes: Infinite loops, IllegalArgumentException, Corrupted tree structure

6️⃣ Performance Impact
	Comparator is called during: Every insertion, Every lookup, Every removal
	So: 👉 Slow comparator → slow TreeMap
	Example: Heavy string processing, Complex DB calls inside comparator (bad practice)

🔹 Q42. Why ConcurrentHashMap is faster than Hashtable? Internal working of ConcurrentHashMap (Java 8)?

🔥 Why ConcurrentHashMap Is Faster Than Hashtable?
	📌 1️⃣ Hashtable Uses Coarse-Grained Locking
			In Hashtable, every method is: public synchronized V put(K key, V value)
			That means: Entire map locked for every operation, Only one thread can access at a time
			Result: High contention, Poor scalability, Low throughput
	
	📌 2️⃣ ConcurrentHashMap Uses Fine-Grained Locking
			Instead of locking entire map: Locks only bucket (bin) level, Uses CAS for many operations, Allows concurrent reads and writes
			Result: Better parallelism, Higher throughput, Reduced contention

🔥 Internal Working of ConcurrentHashMap (Java 8)
	Java 8 significantly changed CHM.
	📌 1️⃣ Data Structure: transient volatile Node<K,V>[] table;
		Same idea as HashMap: Array of buckets
		Each bucket contains: Node OR TreeNode (Red-Black Tree if collisions high)
	
	📌 2️⃣ No Segments (Java 8 Change)
		Java 7:
			Used: Segment[] (each segment was separate lock)
		Java 8: Removed segments.
			Now: Single table, Lock per bucket, Uses CAS

🔥 How put() Works in Java 8 CHM
	Step 1️⃣ → Calculate hash: int hash = spread(key.hashCode());
	Step 2️⃣ → Find bucket: index = (n - 1) & hash;
	Step 3️⃣ → Insert Logic
		Case 1: Bucket empty
			Uses CAS: casTabAt(table, index, null, new Node), No locking needed.
		
		Case 2: Bucket not empty
			Lock only that bucket: synchronized (firstNode), Then: Traverse linked list OR tree

📌 4️⃣ Treeification
	Like HashMap: If bucket size > 8, Convert to Red-Black Tree, Improves worst-case from O(n) to O(log n)

🔥 5️⃣ Why Reads Are Fast?
	get() method: Does NOT lock, Uses volatile reads, Safe due to memory visibility guarantees
				  Very high performance for read-heavy systems.

🔥 6️⃣ Why ConcurrentHashMap Does NOT Allow Null?
	Because:
		If: map.get(key) == null, We cannot distinguish: Key not present OR Key mapped to null
		In concurrent environment, this ambiguity is dangerous.
		So null keys and values are not allowed.

🔥 7️⃣ Resize in ConcurrentHashMap
	Resize is: Cooperative, Multiple threads can help transfer, Uses ForwardingNode to mark moved bins, Very sophisticated implementation.

| Operation  | Complexity |
| ---------- | ---------- |
| get()      | O(1)       |
| put()      | O(1)       |
| remove()   | O(1)       |
| Worst case | O(log n)   |

🔹 Q44. How thread-safety is achieved in ConcurrentHashMap? What is CAS in ConcurrentHashMap?
=============================================================================================================
🔥 How Thread-Safety Is Achieved in ConcurrentHashMap (Java 8)
	Java 8 redesigned ConcurrentHashMap. 
	It achieves thread-safety using:
		1️⃣ Volatile variables
		2️⃣ CAS (Compare-And-Swap)
		3️⃣ Fine-grained locking (bin-level locking)
		4️⃣ Tree bins for high collision
		5️⃣ Cooperative resizing

📌 1️⃣ Internal Data Structure
	transient volatile Node<K,V>[] table;
	Important: table is volatile, Ensures visibility across threads
	Each bucket (bin) contains: Node (linked list) OR TreeNode (Red-Black tree)

📌 2️⃣ Read Operations (get)
	map.get(key); Completely non-blocking, No synchronization, Uses volatile reads
	Why safe? table is volatile, Node fields are volatile, Memory visibility guaranteed, This makes reads extremely fast.

📌 3️⃣ Write Operations (put)
		Case A: Bucket is empty
				Uses CAS: casTabAt(table, index, null, new Node(...)), No locking required.
		Case B: Bucket is not empty
				It locks only that specific bin: synchronized (firstNode)
				So: Other buckets remain accessible, Only that bucket is locked, This is called fine-grained locking.

📌 4️⃣ Treeification
		If bucket size > 8: Linked list → Red-Black Tree, Improves worst-case performance
		Still thread-safe: Bin-level synchronization applied

📌 5️⃣ Resize Mechanism
	Resize is: Multi-threaded, Cooperative, Uses ForwardingNode, Multiple threads help transfer entries, Very advanced implementation.

🔥 What is CAS in ConcurrentHashMap?
	CAS = Compare-And-Swap, It is a CPU-level atomic instruction.
	📌 How CAS Works
		CAS does: If (currentValue == expectedValue) update to newValue, Else do nothing
		All done atomically.

		📌 Example (Conceptual)
			if (table[index] == null) {
				table[index] = newNode;
			}
		In concurrent world this is unsafe.
		Instead CHM does: casTabAt(table, index, null, newNode)
		Meaning: Insert only if bucket still null, If another thread inserted → CAS fails, Retry, No locking required.

	📌 Why CAS Is Powerful: Lock-free, Non-blocking, Avoids thread contention, Improves scalability
		Used in: AtomicInteger, ConcurrentHashMap, Lock-free algorithms

	Both: Are fail-fast, Throw ConcurrentModificationException If collection structurally modified outside iterator.

🔥 When to Use What?
	✅ Use Iterator: Generic collection traversal, Removing elements safely
	✅ Use ListIterator: Need reverse traversal, Need add() during iteration, Need index-based traversal

🔹 Q46. Why remove() from collection during iteration causes exception? How to safely remove elements while iterating?
=============================================================================================================
🔥 Why Removing from Collection During Iteration Causes Exception?
		When you iterate like this:
			for (String s : list) {
				list.remove(s);   // ❌
			}
		You get: ConcurrentModificationException
		📌 Reason: Fail-Fast Mechanism
		Most Java collections (ArrayList, HashMap, HashSet, etc.) are: Fail-Fast
		They detect structural modification during iteration.

	🔍 How Fail-Fast Works Internally
		Collections maintain a variable: modCount
		Every structural modification: add(), remove(), clear(), resize() 👉 Increments modCount
		When Iterator Is Created
			Iterator stores: expectedModCount = modCount;
			During iteration:
				if (modCount != expectedModCount)
					throw ConcurrentModificationException;
					
🔥 What is Structural Modification?
	Structural modification means: Changing size, Adding or removing elements, Rehashing
	Not structural: Updating value via set(), Replacing existing value in Map

🔥 Why Direct remove() Causes Exception?
	Example: 
			Iterator<String> it = list.iterator();
			while (it.hasNext()) {
				String s = it.next();
				list.remove(s);   // ❌
			}
		Because: list.remove() increments modCount, Iterator's expectedModCount not updated, Mismatch detected, Exception thrown

🔥 How to Safely Remove Elements While Iterating?
	✅ 1️⃣ Use Iterator’s remove()
			Iterator<String> it = list.iterator();
			while (it.hasNext()) {
				String s = it.next();
				if (s.equals("A")) {
					it.remove();  // ✅ Safe
				}
			}
		Why safe? Because: Iterator.remove() updates expectedModCount,No mismatch occurs

	✅ 2️⃣ Use removeIf() (Java 8+)
			list.removeIf(s -> s.equals("A"));
		Internally safe.

	✅ 3️⃣ Collect to Separate List
			List<String> toRemove = new ArrayList<>();
			for (String s : list) {
				if (condition) {
					toRemove.add(s);
				}
			}
			list.removeAll(toRemove);
	
	✅ 4️⃣ Use Concurrent Collections
			If multi-threaded: Use ConcurrentHashMap, Use CopyOnWriteArrayList
			These use: Weakly consistent iterators, No ConcurrentModificationException

🔥 Important Interview Insight
	Fail-fast is: Best-effort mechanism, Not guaranteed, Used to detect programming bugs, It is NOT thread-safety mechanism.

Because in for-each: You don’t have reference to iterator, You cannot call iterator.remove() directly

🔹 Q50. Internal working of CopyOnWriteArrayList? 
=============================================================================================================
🔥 What is CopyOnWriteArrayList?
	CopyOnWriteArrayList is a thread-safe list implementation where Reads are lock-free, Writes create a new copy of the array, It belongs to java.util.concurrent.

🔥 Internal Data Structure
	Internally it maintains:
		private transient volatile Object[] array;
	
	Important: Backed by array (like ArrayList), Array reference is volatile, Ensures visibility across threads

🔥 Core Idea: Copy-On-Write
	When you perform: list.add("A");
	It does:
		1️⃣ Acquire lock
		2️⃣ Create new array (oldSize + 1)
		3️⃣ Copy old elements
		4️⃣ Add new element
		5️⃣ Replace old array reference
		6️⃣ Release lock
	Old array remains untouched.
	
🔥 How Read Operations Work
		list.get(index);
	It simply reads from: array[index]; No locking.
	Because: Array reference is volatile, Visibility guaranteed, No structural modification on read

🔥 Locking Strategy
	Writes use: ReentrantLock
	Simplified internal flow:
				lock.lock();
				try {
				   Object[] newArray = Arrays.copyOf(oldArray, oldArray.length + 1);
				   newArray[last] = element;
				   array = newArray;
				} finally {
				   lock.unlock();
				}

🔥 Iterator Behavior (Very Important)
	Iterators are: Fail-safe, Snapshot-based
	When iterator is created: Object[] snapshot = array;
	Iterator iterates over snapshot.
	Even if list changes: Iterator continues safely, No ConcurrentModificationException

🔥 Important Insight
	Iterator sees old state.
	Example:
		Iterator it = list.iterator();
		list.add("X");
	Iterator will NOT see "X".

🔥 Performance Characteristics
| Operation | Complexity |
| --------- | ---------- |
| get()     | O(1)       |
| add()     | O(n)       |
| remove()  | O(n)       |
| iteration | O(n)       |

🔥 When Is It Suitable: Best for: Read-heavy systems, Event listeners, Observer patterns, Subscriber lists, Rare writes
🔥 When NOT Suitable: ❌ Write-heavy systems, Large lists, Memory-sensitive environments
	Because: Every write copies entire array, High memory churn, GC pressure

🔥 Why No ConcurrentModificationException?
	Because: No modCount, Iterator uses snapshot, Underlying array never modified in-place

🔥 Real-World Example
	Used in: Event dispatch systems, Configuration lists, Plugin lists, Listeners in frameworks

🔹 Q51. How to make custom collection thread-safe?
=============================================================================================================
🔥 There is no single way. You choose strategy based on:
	Read vs write ratio, Performance requirements, Memory constraints, Consistency needs

✅ Approach 1: Use Synchronized Methods (Coarse-Grained Locking) Simplest approach.

		public class MySafeList<E> {
			private final List<E> list = new ArrayList<>();

			public synchronized void add(E e) {
				list.add(e);
			}

			public synchronized E get(int index) {
				return list.get(index);
			}

			public synchronized int size() {
				return list.size();
			}
		}
	Pros: Simple, Easy to implement
	Cons: Entire object locked, Low scalability, High contention
	
✅ Approach 2: Use Explicit Lock (ReentrantLock) Better control.
		import java.util.concurrent.locks.ReentrantLock;
		public class MySafeList<E> {
			private final List<E> list = new ArrayList<>();
			private final ReentrantLock lock = new ReentrantLock();

			public void add(E e) {
				lock.lock();
				try {
					list.add(e);
				} finally {
					lock.unlock();
				}
			}
		}
	
	Pros: More flexible, Fair locking option, TryLock support
	Cons: More complex

✅ Approach 3: Use ReadWriteLock (Best for Read-Heavy Systems)
		import java.util.concurrent.locks.ReentrantReadWriteLock;
		public class MySafeList<E> {

			private final List<E> list = new ArrayList<>();
			private final ReentrantReadWriteLock lock =	new ReentrantReadWriteLock();

			public void add(E e) {
				lock.writeLock().lock();
				try {
					list.add(e);
				} finally {
					lock.writeLock().unlock();
				}
			}

			public E get(int index) {
				lock.readLock().lock();
				try {
					return list.get(index);
				} finally {
					lock.readLock().unlock();
				}
			}
		}
	Pros: Multiple readers allowed, Better scalability
	Cons: More complex logic

✅ Approach 4: Use Concurrent Data Structures Internally
	Instead of building from scratch: 
				private final ConcurrentHashMap<K,V> map;
			OR
				private final CopyOnWriteArrayList<E> list;
		
		Let Java handle concurrency. Best approach in most cases.

✅ Approach 5: Use Immutable Design
	Make collection immutable: No modification allowed, Safe by design
	Example:
		List<E> list = List.copyOf(original);
		Immutable collections are naturally thread-safe.

🔥 Advanced Strategy: Lock-Free (Using CAS)
	Very advanced.
	Use: AtomicReference, CAS operations
	Example idea:
		AtomicReference<List<E>> ref;
	Mostly used in high-performance systems.

🔥 Important Considerations
	When designing custom thread-safe collection:
		1️⃣ Avoid exposing internal mutable references
		2️⃣ Protect iteration logic
		3️⃣ Handle compound operations carefully
		
		Example:
			if (!map.containsKey(k)) {
				map.put(k, v);   // Race condition
			}
		
		Better: map.putIfAbsent(k, v);

🔥 Common Mistake
	Just synchronizing add() and get() is not enough if you have compound operations.
	Example:
		public synchronized boolean contains(E e) {
			return list.contains(e);
		}
	
	If caller does: 
		if (!list.contains(e)) {
			list.add(e);
		}
	Still not atomic.

🔹 Q52. Difference between immutable and unmodifiable collections? How Collections.unmodifiableList works internally?
=============================================================================================================
🔥 Difference Between Immutable and Unmodifiable Collections
	📌 1️⃣ Immutable Collection : Cannot be modified after creation, No structural changes allowed, Backing data cannot change
	Examples:
		List.of(...) (Java 9+), Set.of(...), Map.of(...)
		Once created: No add(), No remove(), No update(), Underlying data fixed

	📌 2️⃣ Unmodifiable Collection: 
		 Read-only view of another collection
		 Structural modification not allowed through wrapper
		 But underlying collection CAN change
		Example:
			List<String> original = new ArrayList<>();
			original.add("A");
			List<String> unmodifiable = Collections.unmodifiableList(original);
		
		If: original.add("B"); Then: unmodifiable
		Also reflects "B". Because: It’s just a wrapper.

🔥 Key Differences Table
| Feature                  | Immutable   | Unmodifiable                   |
| ------------------------ | ----------- | ------------------------------ |
| Can modify structure?    | ❌ No        | ❌ No (via wrapper)             |
| Underlying data changes? | ❌ No        | ✅ Yes                          |
| Thread-safe by design?   | Usually yes | Not necessarily                |
| Defensive copy required? | No          | Often yes                      |
| Example                  | List.of()   | Collections.unmodifiableList() |

🔥 How Collections.unmodifiableList() Works Internally
	When you call: List<String> list = Collections.unmodifiableList(original);
	Java creates: new Collections.UnmodifiableList<>(original);

📌 Internal Structure (Simplified)
		static class UnmodifiableList<E> 
				extends UnmodifiableCollection<E>
				implements List<E> {

			final List<? extends E> list;

			UnmodifiableList(List<? extends E> list) {
				this.list = list;
			}

			public E get(int index) {
				return list.get(index);
			}

			public E set(int index, E element) {
				throw new UnsupportedOperationException();
			}

			public void add(int index, E element) {
				throw new UnsupportedOperationException();
			}

			public E remove(int index) {
				throw new UnsupportedOperationException();
			}
		}

📌 Important Points: Delegates read methods to original list, Overrides modifying methods, Throws UnsupportedOperationException

🔥 Why It Is Not Truly Immutable?
	Because: original.add("C"); Changes visible in wrapper.
	So it is: A read-only view, Not a defensive copy

🔥 If You Want True Immutability
	Use: 
		List<String> immutable = List.copyOf(original);
	Or:
		List<String> immutable = List.of("A", "B");

	These create: Independent copy, No backing reference

🔥 Thread-Safety Insight
	Immutable collections: Naturally thread-safe, No synchronization required
	Unmodifiable wrapper: Not thread-safe if original list is mutable and shared
	
🔥 Why Unmodifiable Collection Can Be Modified Indirectly?
	When you create:
		List<String> original = new ArrayList<>();
		original.add("A");
		List<String> unmodifiable =     Collections.unmodifiableList(original);
		
	What happens? unmodifiable is just a wrapper, It holds a reference to original, It does NOT create a copy

📌 If Original Changes: 
	original.add("B");
	System.out.println(unmodifiable);
	Output: [A, B]
	
	Even though: unmodifiable.add("C"); // ❌ UnsupportedOperationException
	So: 
		Direct modification via wrapper → ❌ Not allowed
		Indirect modification via original → ✅ Allowed

🔥 So What Is Actually Protected?
	unmodifiableList protects: add(), remove(), clear(), set()
	But it does NOT protect: Changes in original collection, Mutation of elements inside collection

🔥 How to Prevent Indirect Modification?
	✅ Option 1: Defensive Copy
		List<String> safe = Collections.unmodifiableList(new ArrayList<>(original));
		Now: Wrapper over new copy, Original changes won’t affect safe list

	✅ Option 2: Use Immutable Collection (Java 9+)
			List<String> immutable = List.copyOf(original);
		OR
			List<String> immutable = List.of("A", "B");
		These are truly immutable.

🔥 Key Interview Insight
	Unmodifiable ≠ Immutable
	Unmodifiable: Read-only view, Backing data can change
	Immutable: Data cannot change, No reference to mutable structure

🔹 Q56. Why Big-O matters while choosing collections? Memory overhead comparison of List, Set, Map?
=============================================================================================================
🔥 Why Big-O Matters While Choosing Collections?
	Big-O tells you how performance scales as data grows.
	Example:
		If you store: 
			100 elements → everything feels fast
			10 million elements → wrong choice becomes disaster
			
📌 Example 1: ArrayList vs LinkedList
| Operation     | ArrayList | LinkedList |
| ------------- | --------- | ---------- |
| get(index)    | O(1)      | O(n)       |
| add at end    | O(1)*     | O(1)       |
| insert middle | O(n)      | O(n)       |

If your app: 
	Frequently reads by index → ArrayList wins
	Rare random access → LinkedList maybe ok
	Choosing wrong one can turn: 10ms → 5 seconds

📌 Example 2: HashMap vs TreeMap
| Operation | HashMap  | TreeMap  |
| --------- | -------- | -------- |
| get()     | O(1) avg | O(log n) |
| add()     | O(1) avg | O(log n) |
| Sorted    | No       | Yes      |
If sorting not required → TreeMap is unnecessary overhead.

Notice: 
	O(1) grows flat
	O(log n) grows slowly
	O(n) grows steeply

🔥 Memory Overhead Comparison
	Time complexity is not everything. Memory overhead also matters.
	📌 1️⃣ List Implementations
		🔹 ArrayList: Backed by Object[], Stores references only, Extra unused capacity
			Memory per element: Reference only (~8 bytes), Most memory-efficient list.
		
		🔹 LinkedList: Each element stored in Node: Object data, Node next, Node prev
			Memory per element: 3 references + object header, Much higher memory overhead.
	
	📌 2️⃣ Set Implementations
		🔹 HashSet: Internally: HashMap, Node object, hash, key reference, next reference, Higher memory than ArrayList.
		
		🔹 TreeSet: Red-Black Tree node, key, left/right/parent references, color flag, More memory than HashSet.
		
	📌 3️⃣ Map Implementations
		🔹 HashMap: Each entry stores: key, value, hash, next pointer, Memory moderate.
		
		🔹 LinkedHashMap: Everything in HashMap, PLUS before/after pointers, Higher memory than HashMap.
		
		🔹 TreeMap: Each entry stores:, key, value, left/right/parent, color, Highest memory among these three.

🔹 Q57. Why Map lookup is generally faster than List search? Why TreeMap is slower than HashMap?
=============================================================================================================
🔥 Why Map Lookup Is Generally Faster Than List Search?
📌 1️⃣ List Search (Linear Search)
		If you search in a List: list.contains("A");
		Internally: Starts from index 0, Compares element one by one, Stops when found
		Time complexity: O(n)
		Worst case: Element at end Or not present
	
📌 2️⃣ Map Lookup (HashMap)
		When you do: map.get(key);
		Internally:
			1️⃣ Calculate hashCode()
			2️⃣ Compute bucket index
			3️⃣ Go directly to that bucket
			4️⃣ Compare only within that bucket
			Time complexity (average): O(1)
			Because: Direct index calculation, No full traversal	

🔥 Why Map Is Faster?
	Because Map (HashMap): Uses hashing, Jumps directly to bucket, Avoids scanning entire structure
	While List: Must scan elements sequentially

🔥 Important Exception
	If: You use TreeMap Or HashMap has extreme collisions, Then complexity changes.

🔥 Why TreeMap Is Slower Than HashMap?
	📌 1️⃣ TreeMap Uses Red-Black Tree
		TreeMap stores elements in: Self-balancing binary search tree
		Every operation: Traverse tree, Compare keys, Rebalance tree
		Time complexity: O(log n)
		
	📌 2️⃣ HashMap Uses Hashing
		HashMap: No tree traversal (except high collision case), Direct bucket access
		Time complexity: O(1) average

🔹 Q59. How would you detect duplicate objects using collections?
=============================================================================================================
🔥 How to Detect Duplicate Objects Using Collections?
	The best approach depends on: Data size, Object type, Performance requirement, Whether duplicates must be preserved
	✅ Approach 1: Using HashSet (Most Efficient)
		Because: HashSet does NOT allow duplicates, Backed by HashMap, O(1) average lookup
		📌 Example
				Set<Person> set = new HashSet<>();
				Set<Person> duplicates = new HashSet<>();

				for (Person p : list) {
					if (!set.add(p)) {
						duplicates.add(p);
					}
				}
			If add() returns false → duplicate found.

	🔥 Important: equals() and hashCode() Must Be Correct
		If you don’t override properly: HashSet may fail to detect duplicates
		Rule: If equals() returns true → hashCode must be same

	✅ Approach 2: Using Map with Counting
		If you need frequency count:
			Map<Person, Integer> map = new HashMap<>();
			for (Person p : list) {
				map.put(p, map.getOrDefault(p, 0) + 1);
			}
		Then: if (map.get(p) > 1) You found duplicate.

	✅ Approach 3: Using Stream (Java 8+)
			Set<Person> seen = new HashSet<>();

			List<Person> duplicates =
				list.stream()
					.filter(p -> !seen.add(p))
					.collect(Collectors.toList());

	✅ Approach 4: Sorting + Comparison (Less Efficient)
		If objects are Comparable:
			Collections.sort(list);
			for (int i = 1; i < list.size(); i++) {
				if (list.get(i).equals(list.get(i - 1))) {
					System.out.println("Duplicate found");
				}
			}

		Time complexity: O(n log n), Slower than HashSet approach.

🔥 Performance Comparison
| Approach         | Time Complexity |
| ---------------- | --------------- |
| HashSet          | O(n)            |
| HashMap counting | O(n)            |
| Sorting          | O(n log n)      |
| Nested loop      | O(n²) ❌         |

🔥 Real-World Scenarios
	If millions of records: Use HashSet / HashMap
	If memory constraint: Consider streaming / external sorting
	If DB-level duplicates: Use SQL GROUP BY

🔹 Q61. How to sort a Map by value?
=============================================================================================================
🔥 How to Sort a Map by Value?
	Important: HashMap is not sorted, We must convert entries to a List, Then sort, Then collect into LinkedHashMap

✅ Approach 1: Using Java 8 Stream (Recommended)
		Map<String, Integer> map = new HashMap<>();
		map.put("A", 3); 
		map.put("B", 1);
		map.put("C", 2);

		Map<String, Integer> sorted =
			map.entrySet()
			   .stream()
			   .sorted(Map.Entry.comparingByValue()) //.sorted(Map.Entry.comparingByValue().reversed()) => Descending Order
			   .collect(Collectors.toMap(
				   Map.Entry::getKey,
				   Map.Entry::getValue,
				   (e1, e2) -> e1,
				   LinkedHashMap::new
			   ));
	
		👉 LinkedHashMap is used to preserve sorted order.

✅ Approach 2: Using List + Collections.sort (Pre-Java 8)
		List<Map.Entry<String, Integer>> list =  new ArrayList<>(map.entrySet());
		Collections.sort(list, (e1, e2) -> e1.getValue().compareTo(e2.getValue()));
		
		Map<String, Integer> sorted = new LinkedHashMap<>();
		for (Map.Entry<String, Integer> entry : list) {
			sorted.put(entry.getKey(), entry.getValue());
		}
	
	🔥 Time Complexity 
		Sorting requires: O(n log n)
		Because: We are sorting n entries.

🔥 Why Not Use TreeMap?
	TreeMap sorts by key, not by value.
	So: new TreeMap<>(map) Will sort by key only.

🔥 Alternative: Using TreeSet with Custom Comparator
	Less common approach:
		TreeSet<Map.Entry<String, Integer>> set = new TreeSet<>(Comparator.comparing(Map.Entry::getValue));
	
	⚠ Risk: If two values are equal → comparator returns 0 → One entry may be lost.
	Better to include key comparison too:
		Comparator.comparing(Map.Entry::getValue).thenComparing(Map.Entry::getKey)

🔥 Senior-Level Insight
	If: Sorting frequently required, Data large
	Better to: Use PriorityQueue Or maintain sorted structure incrementally Instead of sorting every time.

🔹 Q64. How to handle NullPointerException in collections?
=============================================================================================================
🔥 Why NullPointerException Happens in Collections?
	Common scenarios:
		1️⃣ Collection reference is null
		2️⃣ Null element inside collection
		3️⃣ Null key in Map
		4️⃣ Calling method on null value from map
		5️⃣ Using stream on null collection
	
	Example:
		List<String> list = null;
		list.add("A");   // ❌ NPE

🔥 1️⃣ Always Initialize Collections
	Instead of: List<String> list = null;
	Use: List<String> list = new ArrayList<>(); OR List<String> list = Collections.emptyList();
	Best practice: Never return null collections from methods.

🔥 2️⃣ Handle Null Elements Safely
| Collection        | Null Allowed?       |
| ----------------- | ------------------- |
| ArrayList         | ✅ Yes               |
| HashSet           | ✅ Yes (one null)    |
| HashMap           | ✅ One null key      |
| TreeMap           | ❌ No null key       |
| ConcurrentHashMap | ❌ No null key/value |

And with: map.get(key).toString();  // ❌ if value is null

🔥 4️⃣ Safe Map Access Pattern
	Instead of: map.get(key).length();   // ❌ risk
	Use:
		String value = map.get(key);
		if (value != null) {
			value.length();
		}

	Or: map.getOrDefault(key, "default");
	Or: Optional.ofNullable(map.get(key)).ifPresent(v -> System.out.println(v));

🔥 5️⃣ Using Optional for Safer Handling
	Optional.ofNullable(list)

🔹 Q65. How to avoid memory leaks caused by collections?
=============================================================================================================
🔥 Why Collections Can Cause Memory Leaks?
	Java has GC — so how can memory leak happen?
	Because: Objects are still reachable, Collection still holds reference, GC cannot clean them
	A memory leak in Java means: Objects are no longer used but still strongly referenced.

🔥 1️⃣ Not Removing Unused Objects from Collections
	Example:
		List<Object> cache = new ArrayList<>();
		public void add(Object o) {
			cache.add(o);
		}
	If never removed: List grows forever, Heap increases, Eventually OutOfMemoryError
	
	✅ Fix: Remove unused objects, Clear collection when done
		cache.clear();

🔥 2️⃣ Static Collections (Very Common Leak)
		private static List<User> users = new ArrayList<>();
	
	Static variables: Live for entire application lifetime, GC never collects them
		If you keep adding → permanent memory growth.
	
	✅ Fix: Avoid static collections, Use proper lifecycle management, Use bounded cache

🔥 3️⃣ Using Unbounded Cache
	Bad: Map<String, Object> cache = new HashMap<>();
	If used as cache: No eviction, Grows forever
	
	✅ Fix: Use LRU Cache, LinkedHashMap with removeEldestEntry()
		OR use: Caffeine, Redis

4️⃣ Listener / Callback Leaks
	Example: listeners.add(this);
	If never removed: Object remains referenced, GC cannot clean it
	Very common in: GUI apps, Event-driven systems
	
	✅ Fix: Always remove listeners, Use WeakReference, Use WeakHashMap

🔥 5️⃣ ThreadLocal Memory Leak
	ThreadLocal<User> userContext = new ThreadLocal<>();
	If using thread pool: Thread never dies, ThreadLocal value remains, Memory leak
	
	✅ Fix: Always:
		try {
			userContext.set(user);
		} finally {
			userContext.remove();
		}

🔥 6️⃣ Using WeakHashMap for Auto Cleanup
	WeakHashMap: Keys stored as weak references, If key not referenced elsewhere, Entry automatically removed
	Good for: Metadata mapping, Cache with GC support

🔥 7️⃣ Clearing Large Collections After Use
	Example:
		List<BigObject> list = new ArrayList<>();
		process(list);
		list.clear();  // Important
	
	Better: list = null; If object no longer needed.

🔥 8️⃣ Avoid Retaining References Accidentally
	Example: List<LargeObject> temp = largeList.subList(0, 10);
	subList keeps reference to entire original list. Memory of full list retained.
	
	✅ Fix: new ArrayList<>(largeList.subList(0, 10)); Creates independent copy.

🔥 9️⃣ Profiling & Detection Tools
	In real systems use: VisualVM, JProfiler, Eclipse MAT, Heap dump analysis
	Look for: Large collections, Growing maps, Static references

🔹 Q66. Why WeakHashMap exists? Use cases of WeakHashMap in real systems?
=============================================================================================================
🔥 Why WeakHashMap Exists?
	In normal HashMap: Keys are stored using strong references, As long as key exists in map → GC cannot remove it
	This can cause memory leaks if: You forget to remove entries, Keys are no longer used elsewhere
	
	📌 Problem with HashMap
		Map<Object, String> map = new HashMap<>();
		Object key = new Object();
		map.put(key, "Data");
		key = null;   // Remove external reference
	
	Question: will entry be garbage collected?
		❌ No. Because: HashMap still strongly references key. So entry stays forever.

🔥 How WeakHashMap Solves This
	WeakHashMap stores keys as: Weak references
	If key is not referenced anywhere else: GC clears weak reference, Entry automatically removed

🔥 How WeakHashMap Internally Works
	Internally:
		static class Entry<K,V> extends WeakReference<K> {
			V value;
		}
	
	Important: Keys wrapped in WeakReference, Uses ReferenceQueue, When GC collects key → entry removed

🔥 When GC Runs
	If:
		Object key = new Object();
		weakMap.put(key, "Value");
		key = null;
		System.gc();
	
	Now: Key has no strong reference, GC clears weak reference, Entry removed automatically

🔥 Real Use Cases of WeakHashMap
	✅ 1️⃣ Caching Metadata
		Example:
			Framework storing metadata about objects: WeakHashMap<Object, Metadata>
			If object destroyed: Metadata automatically removed
			Used in: Reflection caching, ORM frameworks, Class metadata storage

	✅ 2️⃣ Avoid Memory Leaks in Listeners
		Store listeners as weak keys: WeakHashMap<Listener, Object>
		If listener object no longer used: Automatically removed
	
	✅ 3️⃣ Canonical Mapping / Interning
		Store computed values associated with object identity: WeakHashMap<Object, ComputedValue>
		If object gone → computed value removed

	✅ 4️⃣ ClassLoader-Based Caching
		Very important in application servers.
		If: ClassLoader unloaded, But cached strongly in map → Memory leak
		WeakHashMap avoids this.

🔥 Important Limitations
	⚠ Only keys are weak, Values are still strong references, Cleanup happens only during GC, Not deterministic

🔥 When NOT to Use WeakHashMap
	❌ If you need stable cache, If you need deterministic eviction, For large performance-critical caches
	Better use: Caffeine, SoftReference cache, LRU

🔥 Weak vs Soft Reference (Quick Insight)
| Type   | GC Behavior                                    |
| ------ | ---------------------------------------------- |
| Strong | Never collected while referenced               |
| Weak   | Collected immediately when no strong reference |
| Soft   | Collected only under memory pressure           |

🔹 Q67. What is IdentityHashMap? Difference between HashMap and IdentityHashMap?
=============================================================================================================
🔥 IdentityHashMap is a special Map implementation where Keys are compared using == (reference equality) NOT using equals()
	It also uses: System.identityHashCode(), NOT key.hashCode()
	
🔥 Normal HashMap Behavior
	In HashMap: Two keys are considered equal if: key1.equals(key2) == true And: key1.hashCode() == key2.hashCode()

🔥 IdentityHashMap Behavior
	In IdentityHashMap: Two keys are equal only if: key1 == key2
	Meaning: They must be the exact same object in memory.

🔥 Example
	String a = new String("hello");
	String b = new String("hello");

	HashMap<String, Integer> map1 = new HashMap<>();
	map1.put(a, 1);
	map1.put(b, 2);
	System.out.println(map1.size());  // 1
	
	Because: a.equals(b) == true
	
	Now with IdentityHashMap:
		IdentityHashMap<String, Integer> map2 = new IdentityHashMap<>();
		map2.put(a, 1);
		map2.put(b, 2);
		System.out.println(map2.size());  // 2
	
	Because: a == b  // false

🔥 Internal Working of IdentityHashMap
	Unlike HashMap: Does NOT use buckets + linked list, Uses flat array structure
	Stores key-value pairs in alternating slots: [ key1, value1, key2, value2, ... ]
	Comparison uses: == 
	Hashing uses: System.identityHashCode(key)

🔥 Key Differences Table
| Feature          | HashMap         | IdentityHashMap          |
| ---------------- | --------------- | ------------------------ |
| Equality check   | equals()        | ==                       |
| Hash function    | hashCode()      | identityHashCode()       |
| Logical equality | Yes             | No                       |
| Use case         | General purpose | Object identity tracking |
| Null key         | 1 allowed       | 1 allowed                |

🔥 When to Use IdentityHashMap?
	Very specific scenarios:
		✅ 1️⃣ Object Graph Traversal
			During: Serialization, Deep cloning, Object mapping, You must track visited objects by identity.
			Example: Map<Object, Boolean> visited = new IdentityHashMap<>();
		
		✅ 2️⃣ Proxy / Framework Internals
			Used in: JVM internals, Serialization frameworks, ORM tools, Dependency injection frameworks
		
		✅ 3️⃣ Caching Based on Object Instance
			When you care about: Specific object instance, Not logical equality

🔥 When NOT to Use IdentityHashMap
	❌ General business logic, When logical equality matters, For typical key-value storage
	Because: It violates Map contract expectations.

🔹 Q68. EnumMap use cases? Why EnumMap is faster than HashMap?
=============================================================================================================
🔥 EnumMap is a specialized Map implementation designed only for enum keys.
	Example:
		enum Status {
				NEW, PROCESSING, COMPLETED, FAILED
			}
		EnumMap<Status, String> map = new EnumMap<>(Status.class);
		
		👉 Keys must be enum type, Extremely efficient

🔥 Why EnumMap Is Faster Than HashMap?
📌 1️⃣ Internal Data Structure
	Unlike HashMap: No hashing, No buckets, No linked lists, No tree bins
	Instead: Internally backed by an array, Index of enum is determined by: enum.ordinal()
	
📌 2️⃣ Direct Index Access
	Example: map.put(Status.NEW, "Task Created");
	Internally: array[Status.NEW.ordinal()] = "Task Created";
	So lookup is: O(1) with direct index, No hash calculation required.

📌 3️⃣ Memory Efficient
	HashMap Entry stores: key, value, hash, next pointer
	EnumMap stores: Only array of values, Keys implied by enum order, Much lower memory overhead.

🔥 Performance Comparison
| Feature         | HashMap            | EnumMap                |
| --------------- | ------------------ | ---------------------- |
| Key type        | Any object         | Only enum              |
| Lookup          | O(1) avg (hashing) | O(1) direct index      |
| Memory usage    | Higher             | Lower                  |
| Null key        | 1 allowed          | ❌ Not allowed          |
| Iteration order | No order           | Enum declaration order |

🔥 Why It’s Faster in Practice?
	Because:
		1️⃣ No hash computation
		2️⃣ No collision handling
		3️⃣ No linked list traversal
		4️⃣ No rehashing
		5️⃣ Better cache locality
	Array access is extremely fast.

🔥 Use Cases of EnumMap
	✅ 1️⃣ State Machine Implementation
		Example: EnumMap<State, Handler> handlers;
		Very common in: Workflow engines, Order processing systems
	
	✅ 2️⃣ Role-Based Logic
		EnumMap<Role, Permission> permissions;
		Instead of: if (role == ADMIN) { }
		Cleaner and faster.
	
	✅ 3️⃣ Configuration by Enum Type
		Example: EnumMap<DayOfWeek, Schedule> schedule;

	✅ 4️⃣ Replacing Switch Statements
		Instead of:
		switch(status) {
			case NEW:
		}
		
		Use: handlers.get(status).execute(); More extensible.

🔥 When NOT to Use EnumMap
	❌ If keys are not enum, If dynamic keys required, If enum type not fixed

🔥 Real Production Insight
	EnumMap is widely used in: Compilers, JVM internals, Spring framework, State-driven systems

			Map
			  ↓
			SortedMap

🔹 Q70. Explain subMap(), headMap(), tailMap()
=============================================================================================================
🔥 These methods provide range views of a sorted map.
	They are available in: SortedMap, NavigableMap (with inclusive/exclusive options)
	Main implementation: TreeMap, ConcurrentSkipListMap
	
	1️⃣ headMap(): Returns portion of map whose keys are less than given key.
		SortedMap version: headMap(toKey)
		👉 Keys < toKey, 👉 toKey is exclusive
		
		NavigableMap version: headMap(toKey, inclusive)
		Example: map.headMap(20, true);	👉 Includes key 20
		
		📌 Example: 
					TreeMap<Integer, String> map = new TreeMap<>();
					map.put(10, "A");
					map.put(20, "B");
					map.put(30, "C");
					map.headMap(20);

					Result:{10=A}

	2️⃣ tailMap(): Returns portion whose keys are greater than or equal to given key.

		SortedMap version: tailMap(fromKey), keys >= fromKey

		NavigableMap version: tailMap(fromKey, inclusive)
		Example: map.tailMap(20, false); 	👉 Excludes 20

		📌 Example
				map.tailMap(20);
				
				Result: {20=B, 30=C}

	3️⃣ subMap(): Returns portion between two keys.
		SortedMap version: subMap(fromKey, toKey)
			👉 fromKey inclusive
			👉 toKey exclusive

		NavigableMap version: subMap(fromKey, fromInclusive, toKey, toInclusive)
		Full control over inclusivity.
			Example: map.subMap(10, 30);
			Result: {10=A, 20=B}

🔹 Q71. Internal working of PriorityQueue? PriorityQueue vs TreeSet? Why PriorityQueue is not sorted iteration-wise?
=============================================================================================================
🔥 PriorityQueue in Java is implemented using: Binary Heap, Backed by array
	By default: Min-Heap, Smallest element at root
	
	Internal Structure
		Internally:
			transient Object[] queue;
			int size;
			Comparator<? super E> comparator;
		It uses a complete binary tree stored in an array.

🔥 Heap Index Formula
	If element at index i:
		Left child → 2*i + 1
		Right child → 2*i + 2
		Parent → (i - 1) / 2

🔥 Operations: 
	1️⃣ add() / offer()
		Steps: Insert at end of array, Perform heapify-up (sift-up)
		Time complexity: O(log n)

	2️⃣ poll()
		Steps: Remove root, Replace with last element, Perform heapify-down (sift-down)
		Time complexity: O(log n)
	
	3️⃣ peek(): Return root. Time complexity: O(1)

🔥 Why PriorityQueue Is NOT Sorted Iteration-Wise?
	Because:
		Heap guarantees: Parent ≤ Children
		It does NOT guarantee: Left subtree < Right subtree, Full sorted order
	The internal array is a heap structure, not a sorted array.
	So: for (Integer i : pq)	Will NOT give sorted order.
	To get sorted order:
		while (!pq.isEmpty()) {
			System.out.println(pq.poll());
		}

🔥 Example
	Insert: 5, 1, 3, 2
	Heap may look like:
		  1
		/   \
	   2     3
	  /
	 5

	Array: [1, 2, 3, 5]
	But if you insert different order, structure changes. Only root guaranteed smallest.

🔥 PriorityQueue vs TreeSet
| Feature          | PriorityQueue | TreeSet        |
| ---------------- | ------------- | -------------- |
| Structure        | Binary Heap   | Red-Black Tree |
| Sorted globally  | ❌ No          | ✅ Yes          |
| Root min/max     | ✅ Yes         | Yes            |
| Iteration sorted | ❌ No          | ✅ Yes          |

🔥 Time Complexity Comparison
| Operation        | PriorityQueue | TreeSet  |
| ---------------- | ------------- | -------- |
| add              | O(log n)      | O(log n) |
| remove           | O(log n)      | O(log n) |
| peek/min         | O(1)          | O(log n) |
| contains         | O(n)          | O(log n) |
| Iteration sorted | ❌ No          | ✅ Yes    |

🔥 Important Differences
	1️⃣ Duplicate Elements
			PriorityQueue → allows duplicates
				TreeSet → does NOT allow duplicates

	2️⃣ Lookup Performance
			PriorityQueue → contains() is O(n)
			TreeSet → contains() is O(log n)

	3️⃣ Use Case
		Use PriorityQueue when:	Need fast min/max extraction, Implement Dijkstra, Task scheduling, 	Top-K problems
		Use TreeSet when: Need sorted set, Need range queries, Need floor/ceiling operations

🔥 Why PriorityQueue Is Faster for Min Retrieval?
	Because: Root always smallest, No tree traversal needed, Direct array access

🔹 Q72. Difference between Collection and Stream? How streams internally process collections?
=============================================================================================================
1️⃣ Conceptual Difference
| Feature          | Collection         | Stream                      |
| ---------------- | ------------------ | --------------------------- |
| Purpose          | Store data         | Process data                |
| Mutable          | Yes                | No (does not modify source) |
| Reusable         | Yes                | No (one-time use)           |
| Traversal        | External iteration | Internal iteration          |
| Eager/Lazy       | Eager              | Lazy (intermediate ops)     |
| Parallel support | Manual             | Built-in                    |

🔹 Collection
	Example:
		List<String> list = new ArrayList<>();
		list.add("A");
		👉 Holds data, You control iteration

🔹 Stream
	Example: 
		list.stream()
			.filter(s -> s.startsWith("A"))
			.forEach(System.out::println);

		👉 Does not store data, Processes data in pipeline

🔥 External vs Internal Iteration
	🔹External Iteration (Collection)
		for (String s : list) {
			...
		}
		👉 Developer controls loop, Pull-based model

	
	🔹Internal Iteration (Stream)
		list.stream().forEach(...)
		👉 Stream controls iteration, Push-based model

🔥 How Streams Internally Process Collections?
	Now let’s go deeper. When you write:
		list.stream()
			.filter(x -> x > 5)
			.map(x -> x * 2)
			.forEach(System.out::println);

	Internally:
		Step 1️⃣ → Stream Creation: list.stream()
			Collection provides: Spliterator, Stream gets a Spliterator from collection.
		
		Step 2️⃣ → Build Pipeline
			Each intermediate operation: filter(), map(), sorted()
				Creates a new stream stage. No processing happens yet. 👉 Lazy evaluation.
		
		Step 3️⃣ → Terminal Operation Triggers Execution
			When: forEach()
				Stream starts processing.

🔥 Internal Architecture
	Internally Stream API uses: AbstractPipeline, ReferencePipeline, Sink, Spliterator
	Data flows through chained stages.

🔥 Important Concepts
	📌 1️⃣ Lazy Evaluation
			list.stream()
				.filter(...)
				.map(...)

		Nothing happens until: terminal operation
		
📌 2️⃣ Vertical Processing (Not Horizontal)
	Instead of: Filter entire list, Then map entire list
	Stream does:
		Element1 → filter → map → output
		Element2 → filter → map → output
		Improves performance.

📌 3️⃣ Spliterator
		Collection provides: Spliterator<E> spliterator()
		Spliterator: Traverses elements, Splits for parallel processing
		Used in: parallelStream()

🔥 How Parallel Stream Works?
	Spliterator splits data, ForkJoinPool used, Multiple threads process chunks

🔥 Performance Comparison
| Operation           | Collection        | Stream          |
| ------------------- | ----------------- | --------------- |
| Simple loop         | Faster            | Slight overhead |
| Complex chaining    | Verbose           | Cleaner         |
| Parallel processing | Manual threads    | Built-in        |
| Memory usage        | Mutable structure | No storage      |

🔥 Important Limitation
	Stream: Cannot be reused, Consumed once
		Stream s = list.stream();
		s.forEach(...);
		s.forEach(...);  // ❌ IllegalStateException

🔹 Q75. Parallel streams with collections – risks?
=============================================================================================================
🔥 What Are Parallel Streams?
			list.parallelStream()
				.filter(x -> x > 10)
				.map(x -> x * 2)
				.forEach(System.out::println);
		
		Internally: Uses ForkJoinPool.commonPool(), Splits data using Spliterator, Processes chunks in parallel threads

🔥 Risks of Parallel Streams
	⚠ 1️⃣ Shared Mutable State (Race Conditions)
		Bad example:
				List<Integer> result = new ArrayList<>();
				list.parallelStream()
					.forEach(result::add);  // ❌ Not thread-safe

		Problem: Multiple threads modify same list, Data corruption possible
		✅ Fix:
				list.parallelStream()
					.collect(Collectors.toList());
			Use proper collectors.

	⚠ 2️⃣ Performance Overhead
		Parallel streams are NOT always faster.
		Overhead includes: Thread management, Task splitting, Context switching, Synchronization
		For small datasets: 👉 Sequential is faster.
	
	⚠ 3️⃣ Blocking Operations Inside Parallel Stream
		Example: list.parallelStream().forEach(x -> callExternalAPI(x));
		If API call blocks: Threads get blocked, ForkJoinPool threads exhausted, Other parallel tasks starve
		ForkJoinPool is designed for: CPU-bound tasks, Not I/O-bound tasks.

	⚠ 4️⃣ Using Parallel Streams in Application Servers
		ForkJoinPool.commonPool() is shared globally.
		If you use parallel streams heavily: You block common pool, Other framework tasks suffer, Spring / Servlet container impact possible.

	⚠ 5️⃣ Non-Associative Operations
		Reduction operations must be associative.
		Safe: .reduce(0, Integer::sum);
		Unsafe: .reduce(0, (a, b) -> a - b);  // ❌ Not associative
		Parallel results unpredictable.
	
	⚠ 6️⃣ Order-Sensitive Operations
		Parallel stream does NOT guarantee order unless: forEachOrdered() But forEachOrdered() reduces performance.

	⚠ 7️⃣ Poor Splitting Performance
		Collections like: 
			ArrayList → Splits well
			LinkedList → Poor splitting
			I/O streams → Cannot split
		Parallelism efficiency depends on Spliterator.

🔥 When Parallel Streams Make Sense
	✅ Large dataset
	✅ CPU-intensive tasks
	✅ Stateless operations
	✅ No shared mutable state
	✅ Associative reductions

	Example: IntStream.range(1, 1_000_000).parallel().sum();

🔥 When NOT to Use
	❌ Small datasets, I/O operations, Database calls, Logging inside stream, Shared mutable state, Transactional code

🔥 Senior-Level Insight
	Parallel streams are NOT a replacement for: ExecutorService, CompletableFuture, Reactive programming
	They are best for: 👉 Data processing workloads.

🔹 Q76. Why using mutable objects as Map keys is dangerous?
=============================================================================================================
🔥 Because HashMap relies on: 1️⃣ hashCode(), 2️⃣ equals()
	If key state changes after insertion → map structure breaks.

🔥 The Problem with Mutable Keys
	Consider:
		class User {
			String email;

			@Override
			public int hashCode() {
				return email.hashCode();
			}

			@Override
			public boolean equals(Object o) {
				return this.email.equals(((User)o).email);
			}
		}

	Step 1️⃣ Insert
			User user = new User("a@gmail.com");
			map.put(user, "Data");
			Hash computed based on "a@gmail.com".

	Step 2️⃣ Modify Key
			user.email = "b@gmail.com";
			Now: hashCode changed, equals behavior changed
	
	Step 3️⃣ Try to Retrieve
			map.get(user);
		HashMap: Computes new hash (based on new email), Looks in different bucket, Doesn’t find entry
		Result: null, Even though entry exists!

🔥 Worse: Entry Becomes “Lost”
		The old entry: Still in old bucket, But no way to retrieve it
		This causes: Logical corruption, Memory leak, Hard-to-debug production issues

🔥 What Exactly Breaks?
	HashMap assumes: hashCode and equals are stable
	Contract: If a key is in a Map, its hashCode must not change while it is in the Map.
	If violated: Bucket index mismatch, equals comparison fails, Retrieval fails

🔥 Even TreeMap Is Affected
	If using TreeMap: If compareTo() depends on mutable field:
	After mutation: Tree structure becomes invalid, Search fails

🔥 Best Practices
	✅ 1️⃣ Use Immutable Keys
		Examples: String, Integer, UUID
		Custom immutable class
	
	✅ 2️⃣ Make Fields Final
		class User {
			private final String email;
		}
	
	✅ 3️⃣ Do NOT Include Mutable Fields in hashCode/equals
		Bad: List<String> roles;  // mutable
		If roles used in hashCode → dangerous.

🔥 Real Production Example
	Common bug: Using List or Date as key, Date modified later, Map lookup fails, Very hard to debug.

🔹 Q78. Why ConcurrentModificationException is unchecked?
=============================================================================================================
🔥 First: What is ConcurrentModificationException?
	It is thrown when: A collection is structurally modified, While being iterated, Outside the iterator
	Example:
			List<String> list = new ArrayList<>();
			list.add("A");
			list.add("B");

			for (String s : list) {
				list.remove(s);   // ❌ ConcurrentModificationException
			}

🔥 Why Is It Unchecked (RuntimeException)?
	Because: It indicates a programming error, not a recoverable condition.

🔥 Checked vs Unchecked Philosophy
	✅ Checked Exception
		Used for: Recoverable conditions, External failures, I/O issues, Network failures
		Example: IOException, SQLException
		You are expected to handle them.
	
	❌ Unchecked Exception
		Used for: Programming mistakes, Logical errors, API misuse, Violated contract
		Examples: NullPointerException, IllegalArgumentException, ConcurrentModificationException
		You are NOT expected to recover.

🔥 Why CME Is a Programming Error?
	Because: 
		You violated iterator contract:Do not structurally modify collection during iteration except via Iterator.remove().
		It’s not: Thread-safety exception, Concurrency control mechanism
		It is: 👉 A fail-fast detection mechanism.

🔥 Fail-Fast Is Best-Effort
	Internally:
		Collections maintain: modCount
		Iterator stores: expectedModCount
		If mismatch detected: throw new ConcurrentModificationException();

🔥 Why Not Checked?
	If CME were checked: Every loop would need try-catch, Code becomes noisy, It encourages wrong recovery patterns
	You cannot meaningfully recover:
		try {
		   for (...) { }
		} catch (ConcurrentModificationException e) {
		   // What would you do here?
		}
	It usually indicates: Your logic is wrong.
	Important: CME Is NOT About Multi-Threading Only
	Common misconception: It can happen even in single thread:
		for (String s : list) {
			list.add("X");
		}
	No concurrency involved.
	So: It is about contract violation, not thread collision.

🔥 Design Principle
	Java designers made it unchecked because: It represents API misuse, It’s a developer bug ,It should fail fast
	It should not force boilerplate handling

🔹 Q79. Why HashSet size may appear incorrect?
=============================================================================================================
🔥When someone says: “HashSet size is incorrect”
It usually means: Duplicates are not removed, Or expected duplicates are still present, Or unexpected removals happened

🔥 1️⃣ equals() and hashCode() Not Implemented Properly
	This is the most common reason.
		Remember: HashSet internally uses HashMap.
		Duplicate detection depends on: hashCode(), equals()

		📌 Example (Wrong Behavior)
				class User {
					String email;
				}

			User u1 = new User("a@gmail.com");
			User u2 = new User("a@gmail.com");
			Set<User> set = new HashSet<>();
			set.add(u1);
			set.add(u2);
			System.out.println(set.size());  // 2 ❌

			Because: Default equals() compares reference
			Different objects → treated as different

		✅ Fix
			Override properly:
					@Override
					public boolean equals(Object o) {
						return this.email.equals(((User)o).email);
					}

					@Override
					public int hashCode() {
						return email.hashCode();
					}
🔥 2️⃣ Mutable Objects Used in HashSet
	Another very dangerous scenario.
	If key fields change after insertion: 
		user.email = "new@gmail.com";
	Hash code changes.
	Now: Object is stored in wrong bucket, Contains() may return false, Removal may fail
	Set size looks correct but behavior broken.

🔥 3️⃣ Inconsistent equals() and hashCode()
	Rule: If: equals() == true Then: hashCode() must be same
	If violated: Duplicate detection fails, Set size increases unexpectedly

🔥 4️⃣ Using Custom Comparator in TreeSet (Not HashSet)
	Sometimes confusion happens:
		In TreeSet: Duplicate detection depends on: compareTo() == 0 Not equals().
		If compareTo inconsistent: Size may seem wrong

🔥 5️⃣ Concurrency Issues
	If using HashSet in multi-threaded environment: set.add(x);
	Without synchronization: Race conditions, Corrupted structure, Incorrect size, HashSet is NOT thread-safe.

🔥 6️⃣ Hash Collision Edge Cases
	If many objects return same hashCode(): return 1 => Performance degrades. But size remains correct.
	Still may cause confusion in debugging.

🔥 Real Production Example
	Common bug:
		Set<Order> processedOrders = new HashSet<>();
		Order object includes: mutable status, mutable timestamp, If those fields used in hashCode:
		After update → set broken

🔥 Debugging Checklist
	When HashSet size looks wrong:
		1️⃣ Check equals() implementation
		2️⃣ Check hashCode() implementation
		3️⃣ Check immutability of fields
		4️⃣ Check multi-threaded access
		5️⃣ Check null handling

🔹 Q80. Why TreeSet may silently drop elements?
=============================================================================================================
🔥 Why TreeSet May Silently Drop Elements?
	Because: TreeSet does NOT use equals() to determine duplicates, It uses compareTo() (or Comparator)
	If: compare(a, b) == 0
		TreeSet treats them as duplicates
		Even if: a.equals(b) == false
		So second element is silently ignored.

🔥 How TreeSet Internally Works
	TreeSet is backed by: TreeMap
	Internally:
		private transient NavigableMap<E,Object> m;
		Insertion logic: if (compare(key, existingKey) == 0) do not insert
		
	🔥 Example 1: Wrong compareTo()
			class Person implements Comparable<Person> {
				int age;
				String name;

				public int compareTo(Person p) {
					return this.age - p.age;
				}
			}

		Now:
			Person p1 = new Person(25, "John");
			Person p2 = new Person(25, "David");
			Set<Person> set = new TreeSet<>();
			set.add(p1);
			set.add(p2);
			System.out.println(set.size());  // 1 ❌
		
		Why?
			Because: compareTo() returns 0
			TreeSet treats them as same element.
			Even though: p1.equals(p2) == false

	🔥 Example 2: Bad Comparator
		Comparator<String> comp = (a, b) -> a.length() - b.length();
		TreeSet<String> set = new TreeSet<>(comp);
		set.add("AA");
		set.add("BB");
		System.out.println(set.size());  // 1 ❌
		Because both length = 2 → comparator returns 0.
	
	🔥 Why It Happens Silently?
		TreeSet: Calls compare(), If result == 0, Simply ignores insertion, No exception thrown, This is by design.
		
🔥 Important Rule
	Java docs say: Comparator should be consistent with equals.
	Meaning: compare(a,b) == 0  →  a.equals(b) should be true
	If violated: Silent drops, Logical corruption, Unexpected behavior

🔥 Difference from HashSet
| Feature          | HashSet           | TreeSet                      |
| ---------------- | ----------------- | ---------------------------- |
| Duplicate check  | equals()          | compareTo()/Comparator       |
| Silent drop risk | Rare (bad equals) | Very common (bad comparator) |

🔥 Real Production Example
	Very common mistake:
		Sorting by: Comparator.comparing(Person::getAge)
		But two persons same age but different ID.
		TreeSet will drop one.
		Better comparator:
			Comparator.comparing(Person::getAge).thenComparing(Person::getId)

🔥 How to Fix
	Always ensure:
		1️⃣ Comparator includes all fields defining uniqueness
		2️⃣ compareTo consistent with equals
		3️⃣ No mutable fields used in comparison
