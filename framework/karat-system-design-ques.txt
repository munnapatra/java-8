Q1. This is “e-sign” legal documents system. A document may need hundreds of signatures, and after it’s fully signed, the owner gets a notification.
	 We had a bug that made about half of the notifications fail.
	 We have two pieces of information:
		1) A database that tracks documents, including status if they are fully complete.
		2) Flat-file logs from 500 production servers. The logs track the document IDs for successfully sent notifications,
			but failed attempts are missing.
	Now that the bug is fixed, how can we find all the missed notifications?
	⚡The service handles billions of documents daily, so we need a scalable solution.
=================================================================================================================================
🔥What we know
	Database Contains all documents:
			docId | status
			----------------
			1     | COMPLETE
			2     | PENDING
			3     | COMPLETE

	✅Logs Contain only  Successful notifications: docId=1, docId=5, docId=10
	✅Failed notifications: Missing

🎯Goal is Find: (COMPLETE documents - Successfully notified documents) 	Result: Missed notifications

📌⚡Scalable Solution (MapReduce / Spark)
		Step 1: Extract COMPLETE documents From DB: (doc123, COMPLETE), (doc456, COMPLETE)
				Emit: (doc123, DB), (doc456, DB)
				
		Step 2: Parse 500 server logs For every successful notification: doc123, doc999
				Emit: (doc123, LOG), (doc999, LOG)
		
		Step 3: Shuffle by document ID
				Group: 
					doc123 -> [DB, LOG]
					doc456 -> [DB]
					doc999 -> [LOG]
		
		Step 4: Reduce Logic:
				if(values contains DB && values does NOT contain LOG) {
					missed notification
				}
				
				Example:
					doc123 -> [DB, LOG]  => OK
					doc456 -> [DB]       => MISSED
					doc999 -> [LOG]      => ignore
					Output: doc456
				
	Why this scales Suppose: 10 billion completed documents and 500 log files
			MapReduce/Spark: 
				Reads files in parallel
				Reads DB partitions in parallel
				Shuffles by document ID
				Reducers independently process subsets
				Memory usage per machine stays manageable.

		Complexity: O(N) where N = total records.

📌Alternative Interview Answer (Best Production Design)
	If designing from scratch, don't rely on logs. Maintain a notification table:
	Document( docId, status)
	Notification ( docId, notificationSent, sentTimestamp)
	
	When document becomes COMPLETE and notificationSent = false
		Notification worker: SELECT * WHERE notificationSent = false
		After success: notificationSent = true
		Then recovery is trivial: SELECT docId FROM Notification WHERE notificationSent = false;
		No log reconciliation needed.

Q2. Kids play zone Card Recharge Design
		Having a kids play zone card application and deploy that application to 1,00,000 places.
		What will be the approach and how will you handle this.
=================================================================================================================================
💡Answer: 
	We have a Kids Play Zone Card system where users can recharge cards, check balance, and use the card to access rides and games.
	The challenge is that the system is deployed across 100,000 locations, so we need high scalability, low latency, and support for intermittent internet connectivity.

✅Step 1: Identify Core Operations
		The main operations are: Card recharge, Balance check, Ride/game access, Transaction history
		Balance checks and ride access will be the most frequent operations.

✅Step 2: High-Level Architecture
		I would use a centralized cloud backend with multiple microservices such as Recharge Service, Balance Service, and Transaction Service behind an API Gateway and Load Balancer.

		All requests from mobile apps, kiosks, and POS devices will go through the API Gateway.
		Client -> Load Balancer -> API Gateway -> Services
		
✅Step 3: Handle 100,000 Locations
		Since we have 100,000 locations, directly calling the cloud for every card swipe may cause latency and create dependency on internet connectivity. Therefore, I would introduce an Edge Server at each location.
		Card Reader -> Local Edge Server -> Cloud
		The Edge Server maintains local card balances and recent transactions. This allows ride access decisions to happen locally within milliseconds.

✅Step 4: Offline Support
		One major challenge is internet failure. 
		If connectivity is lost, the Edge Server continues processing card swipes and stores transactions locally in a queue.
		Once connectivity is restored, the queued transactions are synchronized with the cloud.
		This ensures the play zone never stops operating because of network issues.

✅Step 5: Database and Caching
		In the cloud, I would use a primary database as the source of truth and Redis cache for frequently accessed balances.
		Read replicas can be added to scale balance queries.

✅Step 6: Reliability
		For recharge operations, I would use idempotency keys to prevent duplicate recharges caused by retries.
		Services should be stateless so they can scale horizontally behind load balancers.

✅Step 7: Monitoring
		I would monitor transaction throughput, recharge failures, synchronization delays, and edge server health across all locations.
		Alerts should be generated if an edge server goes offline or transaction queues start growing.

🔥Closing (Strong Interview Finish)
	So my design would use cloud-based microservices for centralized management and edge servers at each location for low latency and offline capability. This approach provides scalability for 100,000 locations, high availability, fast card processing, and uninterrupted operations even during network outages.

Q3. Design a search engine and How will you decide how much database space is required for search engine.
=================================================================================================================================
A search engine has 3 major parts: Crawling, Indexing, Query/Search Serving

✅1. Functional Requirements
		A search engine should: Crawl web pages, Index content, Accept user search queries, Return relevant ranked results quickly
		The key requirement is low-latency search even when indexing billions of pages.

✅2. High-Level Architecture: 
			Crawler → Parser → Indexer → Search Index
											   ↑
			User → Query Service → Ranker → Results
			
		1)Crawler: Visits websites and downloads pages.
		2)Parser: Extracts text, links, metadata.
		3)Indexer: Creates an inverted index.
		4)Search Service: Receives search queries.
		5)Ranking Service: Ranks results based on relevance, popularity, freshness, etc.
		
✅3. Why Inverted Index?
		Instead of scanning all documents for every search:
			"java" → Doc1, Doc7, Doc10
			"spring" → Doc2, Doc7
		
		Searching becomes: O(1) lookup rather than scanning billions of documents.

	✅4. Scaling
			Since there may be billions of pages:
				Partition indexes across multiple servers (Sharding)
				Use replicas for high availability
				Cache popular searches
				
								
				Load Balancer
					  |
				Search Nodes
					  |
				Sharded Indexes

🔥Storage Estimation
	Suppose:
		Assumptions: 1 Billion web pages and Average page size = 100 KB
		Total raw data: 1 Billion × 100 KB = 100 TB
		Inverted Index Size: Typically index size is 20–40% of raw content. Let's assume 30%. So 100 TB × 30% = 30 TB
		Replication: For reliability we generally keep 3 copies. So it will be 30 TB × 3 = 90 TB

		Additional Metadata: Store=> URL, Page title, Ranking scores, Crawl timestamps, So Assume: 10 TB

	✅Total Storage
	Raw Data        = 100 TB
	Search Index    = 30 TB
	Metadata        = 10 TB
	Replication     = Included
	==========================
	Approx Storage = 140 TB

	If storing replicated copies separately: 140 TB × 3 ≈ 420 TB


✅Challenges
	Freshness: New pages appear constantly.
	Solution: Incremental crawling, Near real-time indexing
	
	Hot Queries: Many users search same terms.
	Solution: Redis cache, Query result cache
	
	Availability
		Solution: Replication, Multi-region deployment

🔥Common Bottlenecks
	| Bottleneck      | Solution             |
	| --------------- | -------------------- |
	| Large index     | Compression          |
	| Slow queries    | Caching              |
	| Hot queries     | CDN/cache            |
	| Crawl overload  | Distributed crawlers |
	| Ranking latency | Precompute scores    |

Q4. Having a single server application what are the pros and cons of it to deploy on single server.
	How will you fix this and what will be the pros and cons for solution provided
=================================================================================================================================
💡What is a Single Server Application?
	All components run on one machine: Application + Database + Cache + Files + Background Jobs
	Example: One VM/server hosting: frontend, backend, DB, storage
	
🧠Pros of Deploying on a Single Server
	1) Simple Architecture: Easy to => deploy, debug, monitor, maintain, Good for => startups, MVPs, internal tools
	2) Low Cost: Only one machine needed => Low infrastructure cost, Low operational overhead, Low DevOps effort
	3) Faster Communication: Processes communicate locally. => Example: App ↔ DB via localhost, Very low latency.
	4) Easier Development: No distributed system complexity: no network partitions, no distributed tracing, no service discovery
	5) Easier Transactions: Single database transaction management.

🧠Cons of Single Server Deployment
	1) Single Point of Failure (Biggest Problem): If server crashes: Entire application down => Causes: hardware failure, OS crash, power/n/w issue
	2) Poor Scalability: Limited by => CPU, RAM, disk, network bandwidth, Eventually vertical scaling stops helping.
	3) Resource Contention: Example: backup job consumes CPU, DB slows, app becomes slow, All services fight for same resources.
	4) Difficult Maintenance: Server restart: Downtime, OS patching affects whole system.
	5) Security Risk: Compromise of one machine may expose: database, application, user files
	6) No Fault Isolation: Bug in one component affects everything.	Example: Memory leak in app → DB starves → system crashes
		
🏗️How to Fix 
	Single Server to Load Balancer
			Load Balancer
				|
			-------------------
			| App Server 1    |
			| App Server 2    |
			| App Server 3    |
			-------------------
					|
			   Database Cluster
					|
			   Cache/Storage
		   
🧠Improvements / Solution
	1) Separate Application and Database: 
			Benefits: better isolation, independent scaling, improved security
			Cons: network latency, more infrastructure
	
	2) Add Load Balancer: Distribute requests among multiple app servers.
		Benefits: high availability, horizontal scaling, traffic distribution
		Cons: added complexity, LB can become bottleneck if not redundant
	
	3) Replication: Database Replication
		Benefits: failover, read scaling, backups easier
		Cons: replication lag, consistency challenges
		
	4) Use Cache Layer: App → Redis → DB
		Benefits: reduced DB load, faster response
		Cons: cache invalidation complexity, stale data risk
	
	5) Containerization + Orchestration
		Use: Docker, Kubernetes
		Benefits: auto-healing, autoscaling, portability
		Cons: operational complexity, learning curve
		
🧠Pros of the Improved Distributed Solution
	| Benefit            | Explanation                       |
	| ------------------ | --------------------------------- |
	| High Availability  | One server failure won’t kill app |
	| Horizontal Scaling | Add more servers easily           |
	| Better Performance | Load distributed                  |
	| Fault Isolation    | One component failure isolated    |
	| Easier Maintenance | Rolling deployments possible      |
	| Disaster Recovery  | Replicas/backups available        |

🧠Cons of Distributed Solution
	| Problem               | Explanation                          |
	| --------------------- | ------------------------------------ |
	| Complexity            | Distributed systems are hard         |
	| Cost                  | More servers/tools                   |
	| Network Failures      | Inter-service communication can fail |
	| Consistency Issues    | Replication lag                      |
	| Monitoring Complexity | Need centralized observability       |
	| Deployment Complexity | CI/CD harder                         |

Q5. We have a video streaming system that fetches aggregate statistics (both historical and live) from third-party APIs.
	These APIs are unreliable and frequently experience outages or failures, which impacts the performance of your system.
	How would you improve the design to make the system more resilient and performant despite these external API failures?
=================================================================================================================================
💡Problem: External APIs fail and affect your streaming system.
⭐Resilient Design
	✅a) Circuit Breaker: Prevents thread/resource exhaustion.
		If API continuously fails: stop calling temporarily, return fallback data

	✅b) Retry with Exponential Backoff: Avoid retry storms. 1s → 2s → 4s → 8s
	
	✅c) Timeouts: Never wait forever. Example: API timeout = 2 seconds
	
	✅d) Cache Responses: Historical data should be cached heavily.
	
	✅e) Bulkhead Isolation: One failing dependency shouldn’t crash everything.
			Separate thread pools:
				Sports API threads
				Finance API threads
				Chat API threads
			
	✅f) Graceful Degradation: instead of full outage.
			Show: stale data, partial data, “temporarily unavailable”
	
	✅g) Queue-based Async Fetching: Do not fetch synchronously during user request.
			Better: Worker fetches APIs periodically, Store results locally, Users read local DB/cache

Q6. There is a batch program which takes backup on centralized server at 1am every night. What is the flaw in the above system design?
=================================================================================================================================
💡At first glance it works, but there are several design flaws.
	a) Single Point of Failure: 
		Since all backups are stored on one centralized server, if that server crashes, gets corrupted, or the data center becomes unavailable, we could lose all backups.
		
		✅Solution: Replicate backups to multiple servers., Store copies in different regions or cloud storage.
		
	b) Backup Traffic Spike: 
		If thousands of systems start backup exactly at 1 AM, the network and backup server may become overloaded.
			10000 Machines -> Backup Server: This creates a "thundering herd" problem.
		
		✅Solution: Stagger backup times, Use distributed backup agents, Use queues and rate limiting.
		
	c) Long Recovery Point Objective (RPO):
		Backups occur only once per day. If a system fails at 11 PM, we could lose almost 24 hours of data.
		
		✅Solution: Incremental backups, Continuous replication, More frequent snapshots.
		
	d) Backup Window Failure:
		If the backup job fails at 1 AM because of network issues or server problems, the next backup may not happen until the following night.
		
		✅Solution: Retry mechanisms, Monitoring and alerts, Automatic re-execution.
	
	e) Scalability Issues:
		As the number of systems grows, a single backup server may not handle the storage and throughput requirements.
		
		✅Solution: Distributed backup architecture, Multiple backup servers, Cloud object storage.
	
	f) Security Risk:
		Centralized backups contain sensitive data. If compromised, all systems are affected.
		
		✅Solution: Encryption at rest and in transit, Access controls, Backup immutability.

🔥The major flaws are the centralized server becoming a single point of failure, traffic spikes at 1 AM, potential data loss due to daily backups, scalability limitations, and security concerns. I would address these by using distributed backup storage, replication, staggered backup schedules, incremental backups, monitoring, and encryption. This would make the backup system more scalable, reliable, and fault tolerant.

🔥Better Design
	Use: Incremental backups, Distributed backup nodes, Snapshot-based backups, Staggered schedules

	Example:
		Region A → 1 AM
		Region B → 2 AM
		Region C → 3 AM

Q8. We have users who stores photos on server farm. We have 26 servers.
	User with username starting with "a" photos go to server 1, username starting with "b" goes to server 2 and so on.
	Do you see any issue with this design?
=================================================================================================================================
💡Problems in Design
	 While this looks simple, it has several scalability and reliability issues.

	✅Uneven Data Distribution (Hotspots): Usernames are not uniformly distributed.
				For example:
						A -> 5 Million Users
						Q -> 100 Thousand Users
						X -> 50 Thousand Users
					Some servers become overloaded while others remain underutilized.
					This leads to: Storage imbalance, CPU imbalance, Network imbalance

	✅Server Capacity Problem: 
				Suppose Server A becomes full. Users starting with A -> Server 1 (Full)
				We cannot easily move half of the A users elsewhere because the partitioning logic is tied to the first character.

	✅Poor Scalability:
				Today we have 26 servers. Tomorrow may be 26 → 100 Servers, How do we use the new servers?
				The current scheme only supports 26 partitions. Adding servers requires moving huge amounts of data.

	✅Server Failure: 
				Suppose Server 1 fails. Now every user whose name starts with 'A' loses access to photos. 
				A single server failure affects an entire group of users.

	✅Data Migration Cost:
				Suppose we add a 27th server. We may need to rebalance data.
				With this design: Move Millions of Photos, This is expensive and time-consuming.

🔥Better Solution: Hash-Based Partitioning
	Instead of: First Character Use: hash(username)
	Example: hash(user1) % 26, hash(user2) % 26
	Benefits: Uniform distribution, Better load balancing, Avoid hotspots

🔥Even Better: Consistent Hashing
	In large-scale systems, I would use consistent hashing.
	Benefits: Easy server addition/removal, Minimal data movement, Better scalability
	Examples of systems using similar ideas: Apache Cassandra, Amazon DynamoDB, 
	
🔥Reliability Improvements
	I would also: Replicate photos to multiple servers. Keep at least 3 copies. Use health checks and automatic failover.
			Primary Server
				  |
				  +---- Replica 1
				  |
				  +---- Replica 2

		So a server failure doesn't affect users.

✅Better Approaches
	a) Consistent Hashing: Distribute uniformly.
	b) Sharding by User ID Hash: Example=> hash(userId) % N
	c) Add Replication: Primary + replica.

Q9. There is one single thread application which has some bug in production.
	The application is failing and it is loosing all the 10 requests sent to it.
	The bug fixing requires some time for code changes. What are the immediate fixes you can implement for the above problem?
=================================================================================================================================
💡 We have a single-threaded application running in production. Due to a bug, whenever the application fails, all 10 incoming requests are lost. Since code changes require time, I would focus on immediate mitigation strategies to prevent request loss and improve availability.

✅Immediate Fixes Before Code Fix
	✔️1. Introduce a Queue (Most Important)
			Currently: Client → Application If the application crashes: 10 Requests → Lost
			I would immediately place a message queue in front of the application. Client → Queue → Application
			Examples: Apache Kafka, RabbitMQ
			Now when the application crashes: Requests stay in Queue and can be processed after recovery. This is the biggest immediate improvement.

	✔️2. Run Multiple Instances
			Currently: 1 Application Instance which is a single point of failure.
			I would deploy multiple instances: Load Balancer -> App1, App2, App3
			If one instance crashes, traffic is routed to healthy instances.

	✔️3. Automatic Restart:
			If the process crashes: Application Down
			I would configure: Process supervisor, Container restart policy, Auto-healing
			Examples: Linux systemd, Docker restart policies, Kubernetes
			This reduces downtime immediately.

	✔️4. Reduce Traffic:
			If a specific request pattern triggers the bug: I would Disable the affected feature and will add rate limiting, Block problematic requests to reduce crash frequency.

	✔️5. Graceful Failure:
			Instead of losing requests: 500 Internal Server Error
			I would temporarily return: Service Temporarily Unavailable and ask clients to retry.
			At least requests are not silently lost.

	✔️6. Monitoring and Alerts: 
			I would immediately add: Crash alerts, Error-rate monitoring, Queue-depth monitoring
			so the operations team knows when failures occur.

Q11. We have an application where children can draw characters.
	 We then automatically generate animations based on those drawings and store them on our servers.
	 As the number of users grows, the storage and infrastructure costs increase significantly. 
	 How would you design a system to reduce storage costs and optimize space usage while still ensuring animations remain accessible and performant?
=================================================================================================================================
💡	As user growth increases, storing every generated animation permanently becomes expensive.
	I would optimize storage using a combination of compression, tiered storage, deduplication, and caching

		✅1. Store Drawings Instead of Full Animations:
				The child's drawing is the source data and is usually much smaller than the generated animation.
				1) Store the original drawing and animation metadata.
				2) Regenerate animations on demand if generation cost is cheaper than long-term storage.
				3) This converts a storage problem into a compute problem.
		
		✅2. Use Efficient Formats and Compression:
				1) Store drawings in vector format when possible instead of large bitmap images.
				2) Compress images using WebP/AVIF.
				4) Compress videos using H.265 or AV1 codecs.
				5) Remove unnecessary frames and optimize resolution based on device requirements.
			
		✅3. Deduplication
				Many animations may share the same backgrounds, character templates, or effects.
				1) Store common assets once.
				2) Reference them using IDs.
				3) This significantly reduces duplicate storage.
		
		✅4. Storage Tiering: 
				Not all content is accessed equally.
				1) Frequently accessed animations → Hot Storage (SSD/Object Storage).
				2) Older content → Cold Storage (cheaper archive storage).
				3) Use lifecycle policies to automatically move files after 30/60/90 days.
		
		✅5. CDN and Caching
				1) Store animations in object storage.
				2) Serve through a CDN.
				3) Frequently viewed animations are cached at edge locations, reducing repeated reads from storage servers.
		
		✅6. Data Retention Policies
				1) Keep inactive animations for a defined period.
				2) Archive or delete content that hasn't been accessed for a long time.
				3) Notify users before deletion if required.
		
		✅7. Scalable Architecture
				User Upload -> API Gateway -> Animation Service -> Object Storage (S3/GCS) -> Metadata DB -> CDN Cache

✅Tradeoff
	If regeneration is fast, store only drawings and metadata. If generation is expensive, store compressed animations but move old ones to cold storage.

Q14. Doctor appointment are scheduled based on queue and notification is send to first person and wait for 15 mins in queue to get response back from the person wether to book or reject and go to next person in queue, What is the flaw and drawback here
==============================================================================================================================================
📌Major Drawbacks
		1. Very High Waiting Time:
				If the first few users don't respond, everyone behind them waits unnecessarily.
				Example: P1 doesn't respond → 15 min, P2 doesn't respond → 15 min, P3 doesn't respond → 15 min
				P4 waits 45 minutes even if they are ready immediately.
		
		2. Poor Resource Utilization: 
				The doctor's slot remains unbooked while the system is waiting for responses.
				Result: Empty appointment slots. Reduced doctor utilization.
		
		3. Single Point of Delay
				One unresponsive user blocks the entire queue.
				This is similar to head-of-line blocking in distributed systems.

		4. Notification Failures:
				If SMS, email, or push notification fails: User never sees the message. System still waits full 15 minutes.
		
		5. Scalability Issues
				For thousands of users: Large number of timers. Long queues. Poor user experience.
		
		6. User Frustration:
				Users far back in the queue have no visibility: "Am I next?", "Should I wait?", "Will I get a slot today?"

⭐Better Design Options
	Option 1: Notify Top N Users Simultaneously
			Instead of one user notify first 10 users
			First valid acceptance gets slot. Others are informed that the slot is filled.
			Pros: Faster booking, Higher utilization, Less waiting
	
	Option 2: Soft Hold + Expiry
			Notify User A, Reserve slot for 5 mins
			If not accepted: Release slot, Notify next batch Shorter timeout.
	
	Option 3: Priority Score
			Use score based on: Queue position, Previous misses, Urgency, Preferred doctor, Instead of pure FIFO.

	Option 4: Lottery Among Eligible Users
			Common in high-demand booking systems.
			Eligible Users -> Random Selection -> Offer Slot
			Prevents long waiting chains.

Q15. An app has live scoring, statistics ,video streaming and entertainment, statistics of 50 user scores which queries data to get all those information and details and give to user During peak hours the load is high, What is the issue and how to handle it
==============================================================================================================================================
🔥This is a classic high-read, low-write system design scenario like sports apps (e.g., Cricbuzz, ESPN).

⚡Problems During Peak Hours
	1. Database Overload
		Suppose 1 million users open the match page.
		Each page requires: Live score, Team statistics, Player statistics, Top 50 player scores, Match details, Commentary
		If every request hits the database: 1M users × 6 queries = 6M DB queries
		Database becomes the bottleneck.

	2. Hotspot Data
		Everyone requests the same match. India vs Australia
		Current Score = 250/3
		Millions of users ask for identical data.
		Repeated DB reads are wasteful.

	3. Video Streaming Bandwidth
		Video consumes much more resources than scores.
		Example:
			Score API = KBs
			Video Stream = MBs
		Network saturation becomes a problem.

	4. Cache Stampede
		When cache expires: Cache Miss -> 100,000 requests -> All hit DB, Database crashes.
	
	5. Statistics Aggregation Cost
		Computing: Top 50 scorers, Strike rates, Leaderboards, Rankings on every request is expensive.
	
	6. High Latency
		Multiple backend calls: Score Service, Player Service, Statistics Service, Video Service Response time increases.

🧠How to Handle It
	1. Use Caching Aggressively: Users -> CDN/Cache -> Backend
		Store: Match score, Statistics, Top players in cache (e.g., Redis).
		95%+ requests should come from cache.
	
	2. Push Updates Instead of Polling
		Bad: 1M users poll every 2 sec
		Good: WebSocket Server Push When score changes: Score Updated -> Broadcast to all users
		Reduces millions of requests.
	
	3. Precompute Statistics
		Instead of: SELECT top 50 players for every user, Maintain leaderboard continuously: Match Event -> Leaderboard Service -> Redis Cache
		Users read already-computed results.
	
	4. CDN for Video Streaming
		Origin Server -> CDN -> Users
		Avoid streaming directly from application servers.
	
	5. Microservice Separation
		Separate: Score Service, Statistics Service, Streaming Service, User Service
		Video traffic should not affect score traffic.
	
	6. Event-Driven Architecture
		Match Event -> Kafka -> Score Service -> Statistics Service -> Notification Service
		Each service updates independently.
	
	7. Read Replicas
		Primary DB -> Read Replica 1, Read Replica 2, Read Replica 3
		Heavy read traffic goes to replicas.
	
	8. Rate Limiting
		Prevent: User refreshing 100 times/sec
		Apply throttling.
		
🎯What Interviewers Expect When they say: "Live scoring, statistics, video streaming, millions of users during peak match hours"

	📌Key answers:
		✅Cache frequently accessed data (Redis).
		✅Use WebSockets/SSE instead of polling.
		✅Precompute leaderboards and statistics.
		✅Use CDN for video delivery.
		✅Use message queues/event streaming (e.g., Apache Kafka).
		✅Scale services independently.
		✅Use read replicas for read-heavy workloads.
		✅Handle cache stampede and hot keys.

		Upload files, Download files, Share files with users, Public/private links, Folder structure, File versioning, Search files

⭐Horizontal scalability
2. High-Level Architecture
				Client
				   |

				   |

		FILE_SHARE ( share_id, file_id, user_id, permission)

	Example: bucket/file123, bucket/file124

	When a file is uploaded: Read the file content. Generate a content hash. hash = SHA256(file_bytes)

		Movie.mp4  -> ABC123
		Avengers.mp4   -> ABC123

	Since the hash is the same, the content is identical.

				Client -> Get File Metadata -> API Server -> Returns Storage URL -> Download directly from Storage

	Frequently downloaded files: Images, Videos, Documents Cached in CDN.
	Flow: Client -> CDN -> Storage
	Benefits: Lower latency, Reduced storage traffic

	Generate: share_token

	Store: token, file_id, expiry, permission

10. Reliability

		Durability target: 99.999999999%

12. Handling Millions of Users

	Scale horizontally: 
						LB
						 |
						 +--- API1
						 +--- API2
						 +--- API3
						 +--- API100

	Encryption: 
		At Rest: AES-256
		In Transit: HTTPS/TLS
		
14. Bottlenecks and Solutions
	| Problem         | Solution           |
	| --------------- | ------------------ |
	| Large uploads   | Chunking           |
	| Duplicate files | Deduplication      |
	| Slow downloads  | CDN                |
	| DB overload     | Sharding           |
	| Server overload | Horizontal scaling |
	| Data loss       | Replication        |
	| Hot files       | CDN caching        |
	| Upload failures | Resumable uploads  |

Interview Answer (2-Minute Version)
	I would separate metadata from file storage.
	Metadata such as file name, owner, permissions, and storage location would be stored in a sharded database while file contents would be stored in a distributed object storage system.
	Large files would be uploaded in chunks to support resumable and parallel uploads.
	Downloads would bypass application servers and use pre-signed storage URLs, optionally accelerated by a CDN.
	Files would be replicated across multiple storage nodes for durability, and deduplication using content hashes would reduce storage usage.
	The application tier would remain stateless behind a load balancer, allowing horizontal scaling to millions of users and petabytes of data.

Q20. We are working on a clone of Google Docs. The software has the following features and limitations:
	1)Multiple users may work on a single document at the same time.
	2)A document must be handled by a single server, no matter how many users are using it.
	3)We have a fixed number of servers which will be sufficient to handle our expected load if used properly.
	4)Our load balancer uses a round-robin system to permanently assign documents to each server, so each server will have an equal number of documents.
	Do you have any concerns about this load balancing system?
==============================================================================================================================================
🔥Yes. The major concern is that equal number of documents ≠ equal load.

	Example Assume 3 servers:
		Server A: Doc1 (10,000 active users), Doc2 (1 user)
		Server B: Doc3 (2 users), Doc4 (1 user)
		Server C: Doc5 (3 users), Doc6 (2 users)

	Round-robin assignment gives:
		Server A -> 2 docs
		Server B -> 2 docs
		Server C -> 2 docs
	
	Looks balanced. But Actual load:
		Server A -> 10,001 users
		Server B -> 3 users
		Server C -> 5 users

	Server A becomes overloaded while the others are almost idle.

💡Hot Document Problem
	Google Docs workloads are highly skewed.
	Most documents: 1-2 active users
	A few documents: hundreds or thousands of users
	Examples: Company-wide meeting notes, Shared project plans, Public collaborative documents
	These become hot documents.
	
	Since requirement #2 says:
		A document must be handled by a single server.
		A hot document can overwhelm its assigned server regardless of how many documents are on that server.

💡Another Concern: No Rebalancing
	Suppose: Doc123 was created six months ago and assigned to Server A.
	Today it suddenly becomes viral and gets: 50,000 concurrent users
	With permanent assignment: Doc123 -> Server A forever You cannot move it.

	Result: 
		Server A overloaded
		Server B idle
		Server C idle

✅Better Strategy
	Instead of balancing by: Number of documents
	balance by: Active users, CPU usage, Memory usage, WebSocket connections, Edit operations/sec
	Maintain metadata: Document -> Assigned Server, Current Load and allow document migration when necessary.

	Example:
		Doc123: Server A → Server C during low activity or using a controlled handoff.

🧠Interview Answer
	Yes. The load balancer assumes that all documents generate equal load, which is unlikely in a collaborative editing system.
	A document with thousands of concurrent editors can consume far more CPU, memory, network bandwidth, and WebSocket connections than hundreds of inactive documents.
	As a result, round-robin assignment by document count can create severe load imbalance.
	The system is also vulnerable to "hot documents" because each document must be handled by a single server and assignments are permanent.
	A better approach is to balance based on actual resource usage or active users and support document reassignment when servers become overloaded

Q22. We are working on a clone of Facebook.
	We want to add a numeric count to every post showing how many friends the post’s author has at the time of viewing the post, like this:
	Our database has two tables:
 		USER( user_id pk, name, created_date)
		USER_RELATIONSHIP( friendship_id pk, user1_id (indexed), user2_id (indexed), start_date )
 
	Focusing on the database, how would you implement the friend-count feature?
	Note we will soon be more popular than Facebook, so the solution needs to scale.
==============================================================================================================================================
Showing friend count for every post should not require counting friendships from the relationship table every time a post is viewed.

1. Naive Approach (Bad)
	When rendering a post: SELECT COUNT(*) FROM USER_RELATIONSHIP WHERE user1_id = :userId OR user2_id = :userId;
	
	Problems: 
		Huge table scans. 
		Every post on a feed triggers a count query. 
		Feed may contain hundreds of posts.
		Millions of users viewing feeds simultaneously.

	Complexity: Read Cost = O(number of friendships) Not scalable.

2. Store Friend Count in USER Table
	Add a column:
		USER ( user_id, name, created_date, friend_count)
		
	When friendship is created: UPDATE USER SET friend_count = friend_count + 1 WHERE user_id IN (:user1, :user2);
	When friendship is removed: UPDATE USER SET friend_count = friend_count - 1 WHERE user_id IN (:user1, :user2);
	Reading becomes: SELECT friend_count FROM USER WHERE user_id = :userId;
	
	Benefits:
		Feed query: SELECT p.post_id, p.content, u.name, u.friend_count FROM POSTS p JOIN USER u ON p.author_id = u.user_id;
		
	Read complexity: O(1) Perfect for read-heavy workloads.

3. Use Event-Driven Updates (Better)
	At Facebook scale: Friendship Created
		User A sends request -> Friendship Service -> Create relationship row -> Publish Event -> FriendshipCreated(A,B)

	Consumers update counters:
		Counter Service
			friend_count(A)+1
			friend_count(B)+1

	Advantages: Asynchronous, High throughput, Decoupled services
	
4. Caching Layer
	Popular users' counts can be stored in: Redis
	Key: friend_count:12345
	Read flow: Feed Service -> Redis -> Miss? -> Database -> Update Redis
	
5. Handling Massive Scale
	Suppose: 1 Billion users and 500 Billion friendships
	Store friendships as:
		USER_RELATIONSHIP ( friendship_id, user1_id, user2_id)
	
	Sharded by: hash(user_id)
	Friend count remains: USER.friend_count because counting across shards on every request would be extremely expensive.

6. Consistency Consideration
	Friend count is not a financial transaction. A small delay is acceptable.
	Example: Friendship created -> Count updates after 2 seconds
	User sees: 999 friends instead of: 1000 friends for a short period.
	This is eventual consistency, which is acceptable for social networks.

Interview Answer (Concise)
	I would denormalize the friend count by storing it in the USER table (or a dedicated counter store).
	When a friendship is created or removed, an event updates the count for both users.
	Feed rendering reads the precomputed friend_count in O(1) time rather than counting rows in USER_RELATIONSHIP.
	For large-scale systems, friendship updates are handled asynchronously through an event stream 
	and frequently accessed counts are cached in Redis.
	This optimizes for the read-heavy nature of social networks while accepting eventual consistency.

Q23. Which consistency model is more appropriate for each of these applications: strong consistency, or eventual consistency? Why?
	1)An API call that needs to respond within 20 milliseconds, used to retrieve metadata about a video stream. The metadata includes things like the author name, rating, and view count.
	3)A web analytics platform recording every single click on a popular web page.
	4)A banking system that makes deposits and payments to checking accounts.
==============================================================================================================================================
The interviewer is testing whether you can choose between:
	🔥 Strong Consistency: Every read gets the latest committed write. No stale data. Higher latency.
	🔥 Eventual Consistency: Reads may see stale data temporarily. Faster and highly scalable. Data converges eventually.
	
1️⃣ Video Metadata API (20ms Response Time)
	Metadata: Author Name, Rating, View Count
	✅ Choice: Eventual Consistency
	
	Why?
	🎯 Requirement: Respond within 20ms
		⚡ Low latency is more important than absolute freshness.
		Examples: 
		View Count = 1,000,000 and Actual Count = 1,000,005 Showing a slightly stale count is acceptable.
		Users won't notice: 1,000,000 vs 1,000,005

	Typical Architecture: User -> CDN -> Redis Cache -> Database


3️⃣ Web Analytics Platform Recording Every Click
	Example: Google Analytics, Adobe Analytics, Mixpanel
	✅ Choice: Eventual Consistency
	
	Why? Suppose: 10 Million Clicks / Minute
		Requiring strong consistency for every click would be extremely expensive.
	
	Analytics systems prioritize: High Throughput, Scalability, Write Availability over immediate consistency.
	
	Typical Flow: Browser -> Kafka -> Analytics Pipeline -> Data Warehouse
	Dashboard may show: Current: 99,950 clicks but Actual: 100,000 clicks This is acceptable.


4️⃣ Banking System (Deposits and Payments)
	Operations: Deposit ₹10,000 Withdraw ₹5,000 
	✅ Choice: Strong Consistency
	Why? Banking cannot tolerate stale reads.
	Example: Balance = ₹1000
		Two ATMs read simultaneously: ATM1 → Withdraw ₹1000, ATM2 → Withdraw ₹1000
		Without strong consistency: Final Balance = -₹1000 Money is lost.

	Required Properties:  ACID Transactions, Serializability, Immediate Consistency
	Typical Architecture:  Application -> Primary Database ->  Transaction Log
	Reads often come from the primary database for critical operations.


📌 Final Summary
| Application                     | Consistency Model      | Reason                                                        |
| ------------------------------- | ---------------------- | ------------------------------------------------------------- |
| 🎥 Video Metadata API           | ✅ Eventual Consistency | Low latency is more important than perfectly fresh metadata   |
| 📈 Web Analytics Click Tracking | ✅ Eventual Consistency | Massive write volume; slight delays are acceptable            |
| 🏦 Banking Deposits & Payments  | 🔥 Strong Consistency  | Financial correctness is critical; stale data is unacceptable |

🎯 Interview Rule of Thumb
	🔥Use Strong Consistency when: Money, Inventory, Account Balances, Seat Reservations, Payment Processing
	🔥Use Eventual Consistency when: Analytics, Social Media Likes, View Counts, Recommendations, Caching

Q25. Lets say we have a puzzle mobile application, for solving puzzle we have hints, now we have 2 approach to fetch the hints from back end -
	* Preload the hint beforehand 
	* load hints after each step
	can you give pros and coons for each approach?
==============================================================================================================================================
✅This is a classic latency vs resource utilization tradeoff.

	🔥Approach 1: Preload All Hints Beforehand: When the puzzle is loaded, fetch all hints at once.
		Pros: 
			•Instant hint display: No waiting when the user requests a hint
			•Works offline: Once downloaded, hints are available even if network drops.
			•Fewer API calls: One request instead of many requests.
			•Better user experience: No loading spinner when asking for a hint.
			•Reduced backend traffic: Less request overhead.
		
		Cons: 
			•Higher initial load time: Puzzle screen takes longer to open.
			•Wasted bandwidth: User may never use most of the hints.
			•Higher memory usage: Storing all hints in the app.
			•Security concern: All hints are already on the client, Advanced users may inspect network traffic or app storage.
			•Scalability issue: If some puzzles have hundreds of hints, payload size grows significantly.
		
	🔥Approach 2: Load Hint On Demand (After Each Step): Fetch the next hint only when needed.
		Pros:
			• Faster initial puzzle load: Only puzzle data is downloaded.
			• Lower bandwidth consumption: Download only the hints actually used.
			• Lower memory footprint: Store only current hint(s).
			• Better content protection: Future hints remain on the server.
			• Easy to update hints: Backend can modify hints without app updates.
		
		Cons:
			• Network latency: User waits for API response.
			• More API calls: Every hint request hits the backend.
			• Backend load increases: More requests per user session.
			• Poor experience on weak networks: Hint may take several seconds to appear.
			• No offline support: User cannot get hints without connectivity.
		
🔥Interview Discussion
	A strong system design answer would mention a hybrid approach.
	Hybrid Approach
		• Load Puzzle + First 2-3 hints initially.
		• Prefetch the next hint in the background when the user is solving.
		• Cache already fetched hints locally.
	
		Example: 
		Puzzle Opened
			|
			+--> Fetch Puzzle
			+--> Fetch Hint1
			+--> Fetch Hint2

		User Views Hint1
			|
			+--> Background Fetch Hint3

		User Views Hint2
			|
			+--> Hint3 already available
			+--> Background Fetch Hint4
	
	Benefits
		• Fast hint display.
		• Lower initial payload than downloading all hints.
		• Reduced backend calls compared to pure on-demand loading.
		• Better user experience on slow networks.
	
🔥What Interviewers Usually Expect
	Think through:
		• Latency, Bandwidth usage, Memory consumption, Backend load, Offline support, Data security, Scalability, User experience

Q26. Lets say we have a system which process files, a lot of files, and we have files which is larger than our RAM - how the system will process those files?
==============================================================================================================================================

A good answer would be:
	✅Key Idea:
		• Never load the entire file into memory.
		• Process the file in chunks (streaming).

✅Approaches
	1. Chunk-Based Processing
		• Read 1 MB, 10 MB, or some fixed-size block.
		• Process the chunk.
		• Write results if needed.
		• Discard the chunk and read the next one.
		
		File
		 ├── Chunk 1 -> Process
		 ├── Chunk 2 -> Process
		 ├── Chunk 3 -> Process
		 └── ...

		Memory Usage: O(chunk size)

	2. Streaming
		Instead of: byte[] data = Files.readAllBytes(path);
		Use: 
			BufferedReader br = new BufferedReader(new FileReader(file));
			String line;
			while ((line = br.readLine()) != null) {
				process(line);
			}

			• Only one line (or buffer) is in memory at a time.

	3. External Sorting (For Huge Files)
		Suppose a 500 GB file must be sorted and RAM is only 8 GB.
		Steps:
			• Divide file into smaller chunks.
			• Sort each chunk in memory.
			• Write sorted chunks to disk.
			• Merge sorted chunks using K-way merge.
			This is called External Merge Sort.

		Time Complexity: • O(N log N)
		Used by: Databases, Hadoop, Spark

	4. Memory-Mapped Files
		Using OS support: MappedByteBuffer buffer = fileChannel.map(...);
		• OS loads pages on demand.
		• Useful for random access to huge files.
	
✅System Design Considerations
	• File size
	• Available RAM
	• Sequential vs Random Access
	• Throughput requirements
	• Disk I/O bottleneck
	• Parallel processing
	• Fault tolerance (resume from last processed chunk)

✅Interview Answer (Short Version)
	• Do not load the entire file into memory.
	• Process the file using streaming/chunking.
	• For sorting or aggregation of huge files, use external algorithms such as External Merge Sort.
	• For high throughput, process chunks in parallel and maintain checkpoints for recovery.

Q29. You are designing an e-commerce website like Amazon.
	Product price is updated on one server, but due to network issues, other servers still show the old price.
	a) Is it okay to show slightly outdated prices?
	b) Which CAP property is more important here?
	c)Should the system block users until all servers are updated?

🔥 For an e-commerce website, serving users is usually more important than showing the absolutely latest price on every server instantly.
	Scenario
		Price update: Old Price = ₹999, New Price = ₹949
		Price updated on Server A. Server A = ₹949
		But due to network partition: Server B = ₹999, Server C = ₹999

	Yes, usually acceptable. For a short period: Server A → ₹949, Server B → ₹999
	Users may see different prices temporarily.
	This is called: Eventual Consistency
	After network recovery: All servers → ₹949
	Most e-commerce systems tolerate this.

	Why? Because: Website remains operational, Users can continue browsing, Revenue is not impacted significantly

	Choose between: CP or AP
	For product catalog and pricing pages: AP = Availability + Partition Tolerance
	Reason: Serving slightly stale data is usually better than: 503 Service Unavailable
	Amazon-like systems prioritize: ✅ Availability over ❌ Perfect consistency for product browsing.

Q31. Scenario:You are designing an online banking system.
	A user transfers ₹10,000 from Account A to Account B.
	During this transaction, the network between servers goes down.
	a) According to CAP Theorem, which two properties will you choose?
	b) Will you prefer Consistency or Availability?
	c) What should the system do if one server is down
==============================================================================================================================================
🔥For an online banking system, correctness of money is more important than serving every request.
	Scenario:
		User transfers: Account A → Account B of ₹10,000
		During the transaction: Server A ----X---- Server B Network partition occurs.
		
a) According to CAP Theorem, which two properties will you choose?
	Since network partition has happened: P = Mandatory You must choose between: CP  or  AP
	For banking: CP = Consistency + Partition Tolerance
	✅ Choose CP

b) Will you prefer Consistency or Availability?
	Prefer Consistency
		Reason: 
			Suppose: Account A = ₹50,000, Account B = ₹20,000, Transfer ₹10,000. 
			After transfer: Account A = ₹40,000, Account B = ₹30,000
			
			Now imagine: Debit happened on Server A, Credit did NOT happen on Server B because of Network failed
			Result: Account A = ₹40,000, Account B = ₹20,000 and ₹10,000 disappeared.
			Or another case: Account A = ₹50,000, Account B = ₹30,000. Money got duplicated.
			Both are unacceptable.
		
		Therefore: ✅ Consistency > Availability

c) What should the system do if one server is down?
	Correct Banking Behavior: Do NOT continue the transaction with uncertain state.
	Instead: Transaction Started -> Network Failure -> Abort / Rollback Transaction -> Show Error
	User sees: "Transaction could not be completed. Please try again." or "Transaction is being processed. Please check status later."

	❌What should NOT happen?
		Debit Account A and not credit Account B
		Credit Account B twice
		Show inconsistent balances

🔥 How Real Systems Handle This
	Using:
		1. ACID Transactions: BEGIN Debit A Credit B COMMIT If failure occurs: ROLLBACK
		2. Two-Phase Commit (2PC)
			Coordinator
					|
					+--> Server A
					|
					+--> Server B

			Phase 1: Prepare, Can you commit? Both must say: YES
			Phase 2: COMMIT If any server fails: ROLLBACK

🔥Interview Answer
	For a banking transaction, I would choose Consistency and Partition Tolerance (CP).
	During a network partition, consistency is more important than availability because financial data must remain correct.
	If one server is down or communication fails, the system should reject, delay, or roll back the transaction rather than risk inconsistent account balances or loss/duplication of money.
	This is typically implemented using ACID transactions, distributed transaction protocols like 2PC, or consensus mechanisms.

Q33. one app is hosted on multiple servers. what are the pros and cons of this?
==============================================================================================================================================
🎯Hosting an application on multiple servers is a common approach for scalability and high availability, but it comes with trade-offs.

Architecture
           Users
             |
      Load Balancer
       /    |    \
      /     |     \
Server1  Server2  Server3

✅Pros
	1. High Availability: 
		If one server crashes Users can still access the application.
		Benefit: Reduced downtime.

	2. Horizontal Scalability
			As traffic increases: 1 Server  -> 1000 users, 5 Servers -> 5000 users
			You can add more servers instead of upgrading a single machine.
			Benefit: Better handling of traffic spikes.

	3. Load Distribution: Requests are spread across servers.
			Instead of: 10000 requests -> Single Server
			you get: 10000 requests -> Load Balancer -> 2500 each
			Benefit: Better response times.

	4. Fault Tolerance: Hardware failures are inevitable.
		Examples: Disk failure, Network issue, OS crash
		With multiple servers, the application can continue operating.

	5. Rolling Deployments
		You can deploy gradually:
			Server1 -> New Version, Server2 -> Old Version, Server3 -> Old Version
		
		Monitor for issues before upgrading all servers.
		Benefit: Reduced deployment risk.

	6. Geographic Distribution: Servers can be deployed in multiple regions.
		Example: Mumbai, Singapore, London Users connect to the nearest region.
		Benefit: Lower latency.

🛑Cons
	1. Session Management Issues:
		Suppose a user logs in: Request 1 -> Server1 Session stored in Server1 memory.
		Next request: Request 2 -> Server2 Server2 doesn't know the session.
		
		Solutions: Sticky sessions, Distributed cache like Redis, JWT tokens
		
	2. Data Consistency Challenges
		Example: Server1 updates data, Server2 reads old data This can cause inconsistent behavior.
		Need: Shared database, Cache synchronization, Replication strategies
	
	3. Increased Operational Complexity: 
		Single server: Deploy -> Done
		Multi-server: Deploy, Health Checks, Load Balancer Updates, Monitoring, Rollback Strategy
		Much more operational overhead.

	4. Distributed System Problems
		New failure modes appear: Network Partition
			Server1 ↔ Server2, Connection Lost, Clock Synchronization, Different timestamps across servers, Partial Failures
			One component fails while others are healthy.
			These problems don't exist in a single-server setup.

	5. More Expensive
		Need: Multiple VMs, Load balancers, Monitoring tools, Storage replication, Infrastructure costs increase.

	6. Cache Synchronization
		Example: Server1 Cache: Price = 100 and Server2 Cache: Price = 120 Users may see different results.
		Need distributed caching strategies.

	7. Harder Debugging
		Single server: Check one log file
		Multi-server: Check logs from: Server1, Server2, Server3, Load Balancer, Database, Cache
		Requires centralized logging and tracing.

	8. Database Becomes Bottleneck: 
		Even if application servers scale: 10 App Servers -> 1 Database
		Database may become the limiting factor.
		Need: Read replicas, Sharding, Connection pooling
		
⭐Common Interview Follow-up
	What additional components are needed?
		Users
		  |
		Load Balancer
		  |
		App Servers
		  |
		Redis Cache
		  |
		Database

	And often: Centralized Logging, Monitoring, Distributed Tracing, Service Discovery


🔥When Should You Use Multiple Servers?
	✅Use multiple servers when: 
		High traffic, High availability required, Zero/minimal downtime deployments, Fault tolerance needed
	
	❌Avoid unnecessary complexity when:
		Small internal application, Few users, Low availability requirements

A common interview answer is:
The main advantages are scalability, availability, and fault tolerance.
The main challenges are session management, data consistency, cache synchronization, deployment complexity, and distributed-system failures.

Q34. There is a system where user can upload, browse and download the file.
	Rule - system is checking files byte-for-byte to check whether same file is available in the system.
	If its already available then user can access that file.
	Do you find any concern in this ? how to enhance this ? what more rules can be applied ?
==============================================================================================================================================
💡Yes. There are several security, privacy, scalability, and correctness concerns with a pure "byte-for-byte duplicate detection" approach.

🛑Current Design
	User uploads file 
		  ↓
	System compares file bytes with all existing files
		  ↓
	If identical file exists
		  ↓
	User gets access to existing file

	This looks like content deduplication, but it introduces some risks.

1. Major Security Concern: Unauthorized Data Disclosure
	Suppose: User A uploads: SalaryReport2026.pdf System stores it.
	Later: User B uploads same PDF System detects identical content and grants access to the existing file.

	Now User B has effectively learned: Someone already uploaded this file. Potentially gains access to User A's document.
	This is a privacy breach.
	
	Better Approach
		Separate: File Content from Access Permissions
		Example: File F1, Hash = ABC123
		Owner: User A
		User B uploads same file: Hash = ABC123
		Reuse storage, but create a new ownership/reference record:
		User A -> F1
		User B -> F1
		
		Do not automatically share access.

2. Performance Issue: Byte-by-byte comparison against all files is expensive.
	Imagine: 10 million files of 1 GB each, Comparing bytes for every upload is not scalable.
	Better Approach
		Use cryptographic hashes: SHA-256
		Upload flow: Compute SHA-256 -> Check hash exists -> If yes verify if required Else store file
	
	Complexity becomes: O(1) lookup instead of scanning storage.

3. Hash Collision Risk
	If using weak hashes: MD5, SHA-1 => collisions can occur.
	Better Use: SHA-256, SHA-512 => For highly sensitive systems:
	Hash + File Size or Hash + Secondary Verification

4. Information Disclosure Attack
	An attacker can upload known files.
	Example: Confidential_Merger_Document.pdf
	If system responds: File already exists attacker learns the document is present.
	This is called a confirmation attack.
	
	Better Never reveal: "This file already exists."
	Instead: Upload successful, Internally deduplicate.

5. Malware Problem
	Suppose malware file exists. virus.exe
	User B uploads same malware.
	Deduplication causes reuse of infected content.

	Additional Rule
		Before storing: Virus Scan, Malware Scan, Content Validation
		
6. Ownership and Auditing
	Need to know: Who uploaded? When uploaded? Who downloaded?
	Deduplication should not lose audit trails.
	
	Example schema: 
		File( fileId, hash, size, storageLocation)
		UserFileMapping(userId, fileId, uploadDate, permissions)
		
7. Near-Duplicate Files
	Current system detects only: 100% identical files
	Misses: report_v1.pdf, report_v2.pdf where only one page changed.
	
	Depending on business requirements, consider: Versioning, Similarity detection, Document fingerprinting
	
8. Large File Uploads
	Computing hash after upload wastes bandwidth.
	Enhancement
		Client computes hash: SHA-256 and sends: filename, size, hash
		Server checks: Hash exists? If yes: Instant upload (No file transfer needed)
		Used by cloud storage systems.

9. Access Control Rules
	Even when file content is shared: User A, User B must have separate permissions.
	Rules: Owner, Read, Write, Share, Download should be independent of storage deduplication.

10. Compliance and Legal Concerns
	Some regulations require: Data isolation, Right to delete, Retention policies
	Example: User A deletes file You cannot physically delete content if: User B still references it
	Need reference counting.

📌Improved Design
	User Upload
		  ↓
	Virus Scan
		  ↓
	SHA-256 Calculation
		  ↓
	Hash Lookup
		  ↓
	File Exists?
	   ├─ Yes → Create User-File Mapping
	   └─ No  → Store File + Create Mapping
		  ↓
	Apply ACLs
		  ↓
	Audit Logging

📌Additional Rules
	Use SHA-256 for deduplication.
	Never auto-share files between users.
	Scan files for malware.
	Maintain ownership and ACLs separately.
	Keep upload/download audit logs.
	Use reference counting before deletion.
	Do not reveal whether another user already uploaded the file.
	Support versioning if business needs it.
	Encrypt files at rest.
	Enforce file size/type restrictions.

Q36. one system accepts password in English characters only. But we have enforced below rules for password validation :
	1. it should be 8-16 chars
	2. it should accept one uppercase, one lowercase, one symbol, one digit
	Do you find any concern in above rules. If yes, what are those and how to resolve it.
==============================================================================================================================================
🎯Yes. Those rules are common, but they have several security and usability concerns.

Current Rules: Length: 8–16 characters, At least 1 uppercase letter, At least 1 lowercase letter, At least 1 digit, At least 1 symbol

💡Concerns
	1. Maximum length of 16 is too restrictive
		A user may want a strong passphrase such as: MyFavoriteCoffee@Morning2026
		This could be more secure than a short complex password but would be rejected.
		
		Recommendation:
			Increase maximum length to at least 64 characters.
			Prefer 128+ if technically feasible.
	
	2. Complexity requirements encourage predictable patterns
		Users typically create passwords like: Password@1, Welcome@123, Company@2026
		These satisfy all rules but are easily guessed. Attackers know users follow such patterns.
		
		Recommendation:
			Remove mandatory composition rules.
			Focus on password length and strength checking.
	
	3. Minimum length of 8 is outdated
		Modern guidance recommends longer passwords.
		Examples: P@ssw0rd, Admin@123 are only 8–9 characters and vulnerable to dictionary attacks.
		
		Recommendation: Minimum length: 12 characters. Better: 14+ characters.
		
	4. No protection against common passwords
		The rule accepts: Password@1, Qwerty@1, Welcome@1 even though they are extremely weak.
		
		Recommendation:
			Check against breached/common password lists.
			Reject commonly used passwords.
	
	5. English-only characters reduce password entropy
		Restricting users to English characters excludes: Unicode characters, Non-English alphabets
		This reduces possible password combinations.
		However, some systems intentionally restrict to ASCII to avoid encoding issues.
		
		Recommendation:
			If possible, support Unicode safely.
			If not, clearly document ASCII-only support.
	
	6. No password blacklist checks
		Users may choose: CompanyName@1, Username@1 which are easy to guess.
		
		Recommendation:
			Reject passwords containing: Username, Email, Company name, First/last name
	
	7. No protection against breached passwords
		A password can satisfy all rules but already exist in public breach databases.
		Recommendation:
			Check against breach databases such as: Have I Been Pwned Passwords API
	
	8. Password rotation policies can backfire
		If users are forced to change passwords frequently, they often do: Password@1, Password@2, Password@3
		
		Recommendation:
			Change passwords only when compromise is suspected.
			Follow modern guidance from standards bodies.

🔥Modern Recommended Policy
	A stronger policy would be:
		1. Minimum length: 12 characters
		2. Maximum length: 64+ characters
		3. Allow spaces
		4. Do not require uppercase/lowercase/symbol/digit combinations
		5. Reject common and breached passwords
		6. Reject passwords containing username or personal information
		7. Store passwords using bcrypt, Argon2, or PBKDF2
		8. Support MFA (2FA)

	Examples of acceptable passwords: 
		correct horse battery staple
		coffee train sunset river 2026
		MyDogLikesPlayingCricket
		These are typically stronger and easier to remember than: P@ssw0rd1, Welcome@123

 💳 Payment Flow
	Step 1: Client Sends Payment:
				POST /payments
				Idempotency-Key: txn-12345
				Amount: 1000
				Currency: INR

	Step 2: Validate Request:
				Checks: User Exists, Merchant Active, Card Valid, Limits Not Exceeded
	
	Step 3: Idempotency Check
				Before processing: Idempotency-Key -> Redis/DB Lookup
				Existing Key: Found -> Return Previous Response
				New Key: Not Found -> Continue Processing
	
🔄 Idempotency Design
	SQL Table: IDEMPOTENCY ( key, request_hash, payment_id, status, response, created_at)
	
	Flow: Request -> Check Key -> Already Exists? 
		if Yes → Return Stored Response
		if No  → Execute Transaction

	✅Benefits: Prevent duplicate charges, Safe retries, Handles network failures

📒 Ledger Design (Most Critical)
	Double Entry Accounting
		Never store: Balance = 1000 Only.
		Store immutable entries.

	SQL Table: LEDGER_ENTRY( entry_id, account_id, debit, credit, transaction_id, timestamp)
	
	Example: Payment ₹1000
		Customer Account: Debit ₹1000
		Merchant Account: Credit ₹1000
		Invariant: Total Debit = Total Credit Always.

⚖️ Ledger Consistency
	Rule: Money can never be created or lost.
	Validation: Σ Debit = Σ Credit for every transaction.
	Transaction Boundary: Payment Record + Ledger Entries + Outbox Event Committed together.

📦 Transactional Outbox Pattern
	Avoid: DB Commit Success, Kafka Publish Failed
	Solution: DB Transaction: Payment + Ledger + Outbox Event COMMIT
	Outbox Worker: Read Outbox -> Publish Kafka -> Mark Sent

	✅Benefits: No event loss, Exactly-once business effect

📜 Audit Trail: Every action must be traceable.
	Audit Table: AUDIT_LOG ( audit_id, actor, action, old_value, new_value, timestamp, ip )
	
	Examples: Payment Created, Refund Processed, Status Updated, User Modified
	Requirements Immutable: No Update, Delete, only Append.

🔒 PCI-DSS Compliance
	Card information must never be stored directly.
	Bad: 4111111111111111 Stored in DB.
	Good: Token Stored in DB.

	Card Vault: Card Number -> Tokenization Service -> Token Returned
	Store: tok_abc123 Only.

	PCI Zone: Card Data -> Vault Isolated environment.

👤 PII Protection
	Sensitive Data: Name, Email, Phone, Address
	Encryption At Rest: AES-256
	Encryption In Transit: TLS 1.3
	Field-Level Encryption: Email, Phone Encrypted before storage.
	Masking Example: john@gmail.com Stored/displayed as: j***@gmail.com
	
🌍 Multi-Region High Availability
	Active-Active Architecture
         Region A
      (Primary Active)
             │
             ▼
      Global Load Balancer
             ▲
             │
         Region B
      (Active Active)
	  
	Database Strategy
		Ledger Database
			Use: Synchronous Replication, Strong Consistency
			Examples: CockroachDB, YugabyteDB, Spanner-like systems
			Why? Money movement requires: Strong Consistency, Not eventual consistency.
			Failover: Region A Failure -> Region A Down -> Traffic Routed -> Region B No manual intervention.

🔄 Reconciliation Service: Daily verification.
	Compare: Ledger vs Bank Settlement File
	Detect: Missing Payment, Duplicate Settlement, Failed Settlement
	
📊 Monitoring
	✅Business Metrics: Payments/sec, Success Rate, Refund Rate
	✅System Metrics: API Latency, DB Latency, Kafka Lag, Error Rate
	✅Financial Metrics: Ledger Mismatch, Settlement Failure, Reconciliation Errors

🚨 Failure Scenarios
	Payment Retry: Client Timeout -> Retry -> Same Idempotency Key -> No Double Charge
	Kafka Failure: DB Commit -> Outbox -> Retry Publish No data loss.
	Region Failure: Region A Down -> Failover -> Region B Service continues.

To protect cardholder data and Personally Identifiable Information (PII) both in transit and at rest, implement the following security controls:

1. Data Protection in Transit
	Use TLS 1.2/1.3 for all network communications.
	Enforce HTTPS for web applications and APIs.
	Use mutual TLS (mTLS) where applicable for service-to-service communication.
	Disable insecure protocols (SSL, TLS 1.0/1.1).
	
2. Data Protection at Rest
	Encrypt sensitive data using strong algorithms such as AES-256.
	Encrypt databases, file systems, backups, and storage volumes.
	Apply tokenization or data masking for card numbers where possible.
	Store only the minimum required sensitive data.
	
3. Key Management
	Store encryption keys separately from encrypted data.
	Use a centralized Key Management System (KMS) or Hardware Security Module (HSM).
	Implement key rotation policies and periodic key renewal.
	Restrict access to encryption keys based on least privilege.
	Maintain key backup and recovery procedures.
	
4. Access Controls
	Follow the Principle of Least Privilege (PoLP).
	Implement Role-Based Access Control (RBAC).
	Enforce Multi-Factor Authentication (MFA) for privileged users.
	Regularly review and revoke unnecessary access.
	Use strong password policies and account lockout mechanisms.
	
5. Monitoring and Auditing
	Log all access to cardholder data and PII.
	Enable real-time security monitoring and alerting.
	Conduct periodic access reviews and audits.
	Retain audit logs according to compliance requirements.
	
6. Compliance Considerations
	Follow PCI DSS requirements for cardholder data.
	Adhere to privacy regulations such as GDPR, CCPA, or applicable local laws.
	Perform regular vulnerability assessments and penetration testing.

Q53. Design SSO with step-up authentication for high-risk operations.
==============================================================================================================================================
🎯 Objectives
	Provide seamless Single Sign-On (SSO) across applications.
	Enforce additional authentication for high-risk operations.
	Minimize user friction while maintaining strong security.
	Support centralized identity and access management.
	
🏗️ Architecture Components
	🆔 Identity Provider (IdP): Authenticates users. Issues SAML/OAuth2/OIDC tokens.
	💻 Service Providers (Applications): Trust the IdP. Consume identity tokens.
	🎟️ Access Token / ID Token: Carries user identity and authentication context.
	📊 Risk Engine: Evaluates transaction risk. Uses factors such as amount, location, device, IP reputation, and user behavior.
	🔐 MFA Service: OTP, Push Notification, Biometrics, Security Key (FIDO2/WebAuthn).
	
🔄 Authentication Flow
	✅ Normal Login (SSO)
		1. User accesses Application A.
		2. Redirect to IdP.
		3. User authenticates (Password + MFA if required).
		4. IdP issues token.
		5. User accesses Application A.
		6. User accesses Application B without re-login (SSO).

	⚠️ Step-Up Authentication Flow
		When a user performs a sensitive action:
		Examples: Add/Change payment card, Transfer funds, Change user roles, Modify security settings, Download sensitive reports
		
		Flow:
			1. User initiates high-risk operation.
			2. Application sends request to Risk Engine.
			3. Risk Engine evaluates risk score.
			4. If risk exceeds threshold: Trigger Step-Up Authentication.
			5. User completes additional verification: OTP, Push approval, Biometric, Security key
			6. IdP issues a new token containing elevated assurance level.
			7. Application allows operation.
			
🛡️ Risk-Based Decision Factors
	New geographic location, Unknown device, Suspicious IP address, Unusual login time, Large transaction amount
	Privileged/admin operation, Velocity anomalies (multiple actions in short period)
	
🎟️ Token Claims Example
{ "sub": "12345", "role": "Admin", "acr": "mfa", "auth_time": 1717500000, "step_up": true }

	✔️acr → Authentication Context Class Reference
	✔️auth_time → Time of latest authentication
	✔️step_up → Indicates elevated authentication
	
🔒 Security Controls
	1. OAuth 2.0 / OpenID Connect
	2. MFA for step-up events
	3. Short-lived elevated tokens
	4. Session timeout for privileged actions
	5. Audit logging of all step-up requests
	6. Token revocation support
	7. Secure cookie settings (HttpOnly, Secure, SameSite)
	
⭐ Interview Answer (2-Minute Version)
	1. Use a centralized IdP implementing SAML/OIDC-based SSO.
	2. Users authenticate once and receive tokens accepted by all applications.
	3. A risk engine continuously evaluates sensitive operations.
	4. For high-risk actions, trigger step-up authentication using MFA.
	5. After successful verification, issue a token with elevated assurance (acr=mfa).
	6. Elevated access is temporary and scoped only to the sensitive operation.
	7. Log all authentication and authorization events for auditing and compliance.
	8. This approach balances user convenience through SSO while providing strong security for critical transactions.
