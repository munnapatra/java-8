✅ Difference Between ArrayList and LinkedList in Java
+----------------------------+-----------------------------+------------------------------+
| Aspect                    | ArrayList                   | LinkedList                  |
+----------------------------+-----------------------------+------------------------------+
| Underlying Data Structure | Dynamic array (resizable)   | Doubly linked list          |
| Memory Usage              | Less (contiguous array)     | More (data + 2 pointers)    |
| Access Time (get/index)   | Fast (O(1) random access)   | Slow (O(n) traversal)       |
| Insertion at End          | Fast (O(1) amortized)       | Fast (O(1) if tail kept)    |
| Insertion in Middle       | Slow (O(n) due to shifting) | Fast (O(n) to locate node)  |
| Deletion from Middle      | Slow (O(n) due to shifting) | Faster (O(n) + unlink)      |
| Search Performance        | Fast with index, else O(n)  | Always O(n)                 |
| Thread Safety             | Not thread-safe             | Not thread-safe             |
| When to Use               | Frequent reads/index access | Frequent inserts/deletes    |
+----------------------------+-----------------------------+------------------------------+
🧠 Short Verbal Explanation for Interviews
“ArrayList uses a dynamic array internally, which makes it very fast for random access using indices—like get(i). But adding/removing elements in the middle of the list can be costly due to element shifting.
On the other hand, LinkedList uses a doubly linked list, which makes insertion and deletion in the middle efficient, as there's no shifting—but accessing elements by index is slow because it has to traverse the list node by node."

Example:
List<String> arrayList = new ArrayList<>();
List<String> linkedList = new LinkedList<>();
arrayList.add("A");  // fast
linkedList.add("A"); // also fast
arrayList.get(0);    // O(1) - fast
linkedList.get(0);   // O(1) - fast for head, but O(n) for index in general

@ExtendWith(MockitoExtension.class)

    private TeacherService teacherService; // real object, we can partially mock
    @InjectMocks
    private StudentService studentService; // injects spy into this

✅ Runnable vs Callable in Java
During the interview, I was asked to explain the difference between Runnable and Callable. Here's a concise breakdown I prepared for the question:

+------------------------+-------------------------------+-----------------------------------------------+
| Feature                | Runnable                      | Callable<V>                                   |
+------------------------+-------------------------------+-----------------------------------------------+
| Introduced in          | Java 1.0                      | Java 5 (java.util.concurrent)                 |
| Return Type            | void – does not return result | Returns a result of type V                    |
| Exception Handling     | Cannot throw checked exceptions | Can throw checked exceptions                 |
| Usage                  | Used with Thread or Executor  | Used with ExecutorService.submit()            |
| Result Retrieval       | No return value               | Returns a Future<V> to get the result         |
| Typical Use Case       | Fire-and-forget tasks         | Tasks needing a result or exception handling  |
+------------------------+-------------------------------+-----------------------------------------------+
🔹 Runnable Example
Runnable task = () -> System.out.println("Running a task");
Thread thread = new Thread(task);
thread.start();
🔹 Callable Example

Callable<String> task = () -> {
    Thread.sleep(1000); // simulate some delay
    return "Task result";
};

ExecutorService executor = Executors.newSingleThreadExecutor();
Future<String> future = executor.submit(task);
String result = future.get(); // blocks until result is available
executor.shutdown();
🔹 Summary
Use Runnable when you don't need a result or exception handling. Use Callable when the task needs to return a value or might throw a checked exception.

🌐 Global Exception Handling in Spring Boot
During the interview, I was asked:

“How do you handle exceptions globally in a Spring Boot application?”

This is a fundamental question that often reveals how cleanly you separate concerns in your backend architecture.

Spring Boot provides a very elegant way to handle exceptions globally using two annotations:

@ControllerAdvice — to mark a global exception handler class.
@ExceptionHandler — to define methods that handle specific exceptions.
By using these together, you can keep your controller logic clean while delivering structured, meaningful error responses to the client.

🔹 Step-by-Step: Custom Exception Handling
1. Define a Custom Exception
public class ResourceNotFoundException extends RuntimeException {
    public ResourceNotFoundException(String message) {
        super(message);
    }
}
2. Create a Global Exception Handler
@ControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<String> handleNotFound(ResourceNotFoundException ex) {
        return new ResponseEntity<>(ex.getMessage(), HttpStatus.NOT_FOUND);
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<String> handleGeneralError(Exception ex) {
        return new ResponseEntity<>("Internal Server Error: " + ex.getMessage(), HttpStatus.INTERNAL_SERVER_ERROR);
    }
}
3. Use the Custom Exception in Your Controller
@GetMapping("/user/{id}")
public User getUser(@PathVariable Long id) {
    return userRepository.findById(id)
        .orElseThrow(() -> new ResourceNotFoundException("User not found with ID: " + id));
}
4. Optional: Custom Error Response Structure
To return structured error responses (especially useful in REST APIs), I often use a custom ErrorResponse class:

Become a member
And modify the handler method:

@ExceptionHandler(ResourceNotFoundException.class)
public ResponseEntity<ErrorResponse> handleNotFound(ResourceNotFoundException ex) {
    ErrorResponse error = new ErrorResponse(ex.getMessage(), HttpStatus.NOT_FOUND.value());
    return new ResponseEntity<>(error, HttpStatus.NOT_FOUND);
}
✅ Why It Matters
This pattern is especially important in production-grade REST APIs and microservices. It:

Ensures consistency in error responses.
Reduces code duplication across controllers.
Enhances client-side error handling (e.g., front-end apps can easily parse JSON error structures).

Liskov Substitution Principle and Interface Segregation Principle — Interview Deep Dive
During my interview at EPAM for the Senior Java Engineer position, I was asked to explain two core SOLID principles: Liskov Substitution Principle (LSP) and Interface Segregation Principle (ISP). These questions were designed to assess my understanding of clean object-oriented design and my ability to avoid fragile architecture in real-world systems.

✅ Liskov Substitution Principle (LSP)
Definition:

“Subtypes must be substitutable for their base types without altering the correctness of the program.”

This principle ensures that if a class S is a subclass of class T, then objects of type T in a program can be replaced with objects of type S without breaking the behavior of the program.

🔧 Common Violation Example:
class Bird {
    void fly() { System.out.println("Bird is flying"); }
}
class Ostrich extends Bird {
    @Override
    void fly() {
        throw new UnsupportedOperationException("Ostriches can't fly!");
    }
}
In this example, Ostrich violates LSP by overriding the fly() method in an invalid way. Substituting an Ostrich where a Bird is expected could result in a runtime failure.

✅ Better Design:
interface Bird { void eat(); }

interface FlyingBird extends Bird { void fly(); }

class Sparrow implements FlyingBird {
    public void eat() {}
    public void fly() {}
}

class Ostrich implements Bird {
    public void eat() {}
}
Here, we avoid forcing non-flying birds to implement flying behavior, respecting the LSP and enabling safer polymorphism.

✅ Interface Segregation Principle (ISP)
Definition:

“Clients should not be forced to depend on methods they do not use.”

ISP emphasizes creating smaller, role-specific interfaces instead of large, monolithic ones.

🔧 Common Violation Example:
interface Worker {
    void work();
    void eat();
}
class Robot implements Worker {
    public void work() {}
    public void eat() {
        throw new UnsupportedOperationException("Robots don't eat.");
    }
}
The Robot is forced to implement eat() even though it doesn't need to — a clear violation of ISP.

✅ Better Design:
interface Workable {
    void work();
}

interface Eatable {
    void eat();
}
class Human implements Workable, Eatable {
    public void work() {}
    public void eat() {}
}
class Robot implements Workable {
    public void work() {}
}
Now, each class implements only what it needs. This leads to more maintainable and decoupled code.

📚 Bonus: Why @Transactional Doesn't Work on Private Methods in Spring
Spring’s @Transactional is powerful—but it won’t work on private methods. Here’s why:

⚙️ How It Works
Spring uses proxy-based AOP
Only public methods are proxied
Proxies can’t intercept calls made within the same class
🔍 Non-working Example
@Service
public class OrderService {

@Transactional
    private void processOrder() {
        // This won't trigger a transaction!
    }
    public void placeOrder() {
        processOrder(); // Self-invocation - no proxy involved
    }
}
✅ Correct Usage
@Service
public class OrderService {

@Transactional
    public void processOrder() {
        // Transaction is applied here
    }
}
💡 Workaround
@Autowired
private ApplicationContext context;

public void placeOrder() {
    context.getBean(OrderService.class).processOrder(); // Proxy invocation
}
🧠 Summary Table
Case Transaction Applied Public Method ✅ Yes
Private/Protected Method ❌ No
Self-invocation ❌ No
Cross-bean invocation ✅ Yes

📌 Final Advice
Understand Spring’s AOP proxying mechanism to avoid silent bugs. Use @Transactional correctly and design your service layers with this in mind.

🚀 Git Features Every Senior Developer Should Master
As a senior developer, using Git effectively is more than just knowing how to clone, commit, and push. You’re expected to understand and leverage Git’s powerful features to keep your team’s development workflow clean, efficient, and resilient.

Below is a curated list of advanced Git features and concepts that have helped me debug faster, collaborate more smoothly, and maintain cleaner histories in real-world projects.

🔹 Branching & Merging
Creating feature branches and merging them cleanly is fundamental. I often use git branch and git merge to isolate work and combine it in a controlled manner.

🔹 Rebase
git rebase helps maintain a clean, linear commit history — especially useful before merging a feature branch. It rewrites the base of your branch to the latest commit in main (or dev), making code review much easier.

🔹 Cherry-pick
Need to pull a specific bug fix from another branch without merging everything else? git cherry-pick saves the day.

🔹 Stash
When I need to switch context quickly but don’t want to commit my changes, git stash lets me store them temporarily and pick up where I left off with git stash pop.

🔹 Reflog
Ever accidentally deleted a branch or lost a commit? git reflog is your Git time machine. It lets you recover lost history even after rebasing or resetting.

🔹 Interactive Rebase
With git rebase -i, I can squash, reorder, or reword commits before merging — perfect for cleaning up messy commit histories before opening a PR.

🔹 Amend
Need to update your last commit (e.g., forgot to add a file or fix a message)? git commit --amend allows you to make changes without creating a new commit.

🔹 Reset vs Revert
Understanding the difference between git reset (rewrites history) and git revert (creates a new inverse commit) is crucial for safely undoing changes, especially on shared branches.

🔹 .gitignore
A simple but essential tool to avoid tracking unnecessary files like logs, build outputs, or local configs.

🔹 Git Hooks
With Git hooks like pre-commit and pre-push, I can automate tasks such as running linters or tests before changes are accepted — ensuring code quality and consistency.

🔹 Bisect
Tracking down bugs introduced across many commits? git bisect uses binary search to efficiently pinpoint the exact commit that broke things.

🔹 Submodules
For projects that depend on other repositories, Git submodules let me include and manage external codebases while keeping them separate.

🔹 Tagging
Tagging specific commits helps with version control and releases. I use git tag to mark stable builds and production deployments.

🔹 Git Log, Blame, and Diff
These tools are my go-to for investigating who made changes, what changed, and when — perfect for debugging or code reviews.

🧠 Bonus Concepts Every Senior Should Know:
Rebase vs Merge — When to squash history or preserve commit timelines.
Conflict Resolution — Efficiently resolving merge and rebase conflicts.
GitFlow & Trunk-Based Development — Choosing and implementing the right branching strategy.
GPG Commit Signing — Securing contributions in regulated or open-source environments.
CI/CD Integration — Leveraging Git with GitHub Actions, GitLab CI, or Jenkins for automated pipelines.
